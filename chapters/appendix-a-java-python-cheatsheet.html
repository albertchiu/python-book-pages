GIL：執行緒不加速 CPU 密集工作（佐證）。下面這段可自行重跑，比較同一段純計算「序列跑兩次」vs.「兩個執行緒並行」：

python 
import time, threading

def count(n):
    while n > 0:
        n -= 1

N = 20_000_000

t0 = time.perf_counter()
count(N); count(N)                       # 序列
seq = time.perf_counter() - t0

t0 = time.perf_counter()
ts = [threading.Thread(target=count, args=(N,)) for _ in range(2)]
for t in ts: t.start()
for t in ts: t.join()                    # 兩執行緒並行
par = time.perf_counter() - t0

print(f"sequential: {seq:.2f}s")
print(f"2 threads:  {par:.2f}s")
print(f"speedup:    {seq/par:.2f}x")

某次單機輸出（數字依硬體而異，重點是 speedup ≈ 1.0，不是精確值）：

text 
sequential: 0.35s
2 threads:  0.35s
speedup:    1.02x   # 接近 1.0，GIL 把 CPU 密集工作序列化了

思維差異：這是 Java 工程師最大的認知衝擊——在 Java 裡多執行緒就是 CPU 平行的同義詞，在 CPython 裡（GIL 仍預設啟用，free-threading 為 3.13+ 的選用建置）執行緒只對 I/O 密集有用。CPU 密集要靠 multiprocessing 或 C 擴充。決策樹：I/O 密集且量大 → asyncio；I/O 密集且想沿用既有阻塞程式碼 → 執行緒池；CPU 密集 → 行程池（詳見第 14 章）。

最關鍵的 10 個思維轉換

  1. 沒有編譯器把關——private／final／checked exception 都不存在；嚴謹度靠 mypy + ruff + 測試 + 命名慣例補回（詳見第 9、12、13 章）。
  2. Optional<T> 就是 T | None——沒有包裝物件，直接用 is None 分支；別找 .map()／.orElse()（詳見第 9 章）。
  3. Stream 就是 comprehension／generator——[...] 即時、(...) 惰性；超過兩層請改用 yield 函式（詳見第 6 章）。
  4. interface 一分為二——零侵入用 Protocol（結構子型），強制契約用 abc.ABC（名義子型）（詳見第 8 章）。
  5. record／Lombok 就是 @dataclass——一行生出 __init__／__repr__／__eq__，frozen=True 即不可變且可雜湊（詳見第 7 章）。
  6. 多執行緒不等於 CPU 平行——GIL 讓執行緒只對 I/O 密集有用；CPU 密集走 multiprocessing（詳見第 14 章）。
  7. int 不溢位——任意精度整數讓你不再操心 overflow；精確小數仍用 Decimal（詳見第 2 章）。
  8. is vs ==——is 是參考相等（只給 None／singleton 用），== 是值相等（走 __eq__）（詳見第 2 章）。
  9. dict 保證插入順序——不再需要 LinkedHashMap；要不可變改用 tuple／frozenset（詳見第 4 章）。
  10. pyproject.toml 是新的 pom.xml——uv 管依賴與環境，型別提示是接上整條現代工具鏈的接口（詳見第 1、11 章）。

心法：不要把 Java 的寫法直譯成 Python——先用這張表找到對應的 Python 慣用語，再翻回正文理解「Python 為什麼這樣想」。當對照表與你的 Java 直覺衝突時，通常是 Python 在提醒你有更簡潔的寫法。
