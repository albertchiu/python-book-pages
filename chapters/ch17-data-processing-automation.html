第 17 章：資料處理與自動化腳本

身為 Java 工程師，「寫個小工具」對你來說常常是個沉重的決定：要開 Maven 專案、拉 OkHttp、拉 Jackson、拉 Apache POI、寫 main、打包成 fat jar，最後得到一個三百行才能抓資料寫 Excel 的東西。於是大多數人退而求其次，回去寫 bash——然後在 awk、sed、引號跳脫的地獄裡度過下午。Python 在這個區間是壓倒性的：它是業界公認的「膠水語言」（glue language），抓 HTTP、清資料、寫報表、串子行程，每一段都短到像虛擬碼，卻又有型別、有例外、可測、跨平台。這一章把第 14、15 章建立的工具（async、pathlib、subprocess、argparse、logging）和四個重量級套件（requests/httpx、pandas、BeautifulSoup、openpyxl）串起來，目標是讓你能在一個下午做出「抓資料 → 整理 → 出報表」的自動化腳本。過程中要記住兩個工程心法：向量化（vectorization，別用 for-loop 逐列算）與可重現（reproducibility，測試用本地資料、不依賴外部世界）。

HTTP client：requests vs httpx

Java 11 之前抓 HTTP 是場災難（HttpURLConnection 的 API 像懲罰），所以大家用 OkHttp 或 Apache HttpClient。Java 11 之後有了官方 java.net.http.HttpClient，總算像樣。Python 這邊長年的事實標準是 requests——「HTTP for Humans」，幾乎每個 Python 專案都依賴它。近年崛起的 httpx 則是 requests 的精神續作：API 幾乎一模一樣，但多了 async（承接第 14 章的 async/await）與 HTTP/2 支援。經驗法則：純同步、不在乎 async，就用 requests；需要 async 或 HTTP/2，就用 httpx。兩者學會一個，另一個只要改 import。

requests：同步的業界標準

java 
// Java 11 HttpClient
HttpClient client = HttpClient.newHttpClient();
HttpRequest req = HttpRequest.newBuilder()
        .uri(URI.create("https://httpbin.org/get?q=python"))
        .header("User-Agent", "demo/1.0")
        .timeout(Duration.ofSeconds(10))
        .GET()
        .build();
HttpResponse<String> resp = client.send(req, HttpResponse.BodyHandlers.ofString());
if (resp.statusCode() / 100 != 2) throw new IOException("HTTP " + resp.statusCode());
String body = resp.body();   // 還要自己丟給 Jackson 解析 JSON
python 
# requests（需連網）
import requests

resp = requests.get(
    "https://httpbin.org/get",
    params={"q": "python"},                       # query string，自動編碼
    headers={"User-Agent": "demo/1.0"},
    timeout=10,                                     # 務必設！見下方陷阱
)
resp.raise_for_status()                            # 非 2xx 直接丟 HTTPError
data = resp.json()                                 # 內建 JSON 解析，不用 Jackson

短了一半，而且 params 自動處理 URL 編碼、resp.json() 內建解析——這就是「HTTP for Humans」的意思。raise_for_status() 是關鍵習慣：requests 預設不會因為 404/500 丟例外（它認為「收到回應」本身就是成功），你必須主動呼叫它把非 2xx 轉成 HTTPError，對應 Java 那段手寫的 statusCode() / 100 != 2 檢查。

⚠️ 陷阱：requests.get(...) 預設沒有 timeout，會無限等待。在生產環境這是頭號事故來源——一個慢掉的下游能把你的 thread pool 全部卡死。每一個 request 都要傳 timeout。Java 的 HttpClient 至少還有連線 timeout 概念，requests 連那都沒有預設。可傳 timeout=10（連線＋讀取都 10 秒）或 timeout=(3.05, 27)（連線 3.05 秒、讀取 27 秒）。

POST JSON body 一樣俐落，用 json= 參數會自動序列化並設好 Content-Type：

python 
resp = requests.post(
    "https://httpbin.org/post",
    json={"name": "Alice", "age": 30},   # 自動 json.dumps + Content-Type: application/json
    timeout=10,
)
resp.raise_for_status()
print(resp.json()["json"])               # httpbin 回顯我們送的 body

對照之下，若要送傳統表單（application/x-www-form-urlencoded）則用 data=，送檔案用 files=。json= vs data= 的差別，正是 Java 裡你要手動設 Content-Type 並選對 body publisher 的那一步。

Session：連線重用（連線池）

這是新手最常忽略、最影響效能的一點。每次 requests.get(...) 都會新建一條 TCP 連線、重做一次 TLS 握手。抓一個 URL 沒差，但在迴圈裡抓一百個就是一百次握手。解法是用 Session——它內部維護一個連線池（connection pool），對同一 host 的請求會重用 keep-alive 連線，並且讓你設一次 header/cookie 就套用到所有請求。

java 
// Java：HttpClient 本身就是執行緒安全、可重用的，建一個用到底
HttpClient client = HttpClient.newHttpClient();   // 重用這一個
// OkHttp 同理：一個 OkHttpClient 走全程，背後有連線池
python 
# requests（需連網）：Session 對應 Java 重用同一個 HttpClient
import requests

with requests.Session() as session:
    session.headers.update({"User-Agent": "python-book-demo/1.0"})
    for page in range(1, 4):
        resp = session.get(
            "https://httpbin.org/get",
            params={"q": "python", "page": page},
            timeout=10,
        )
        resp.raise_for_status()
        print(resp.json()["args"])      # {'page': '1', 'q': 'python'} ...

執行上面這段會印出三次 args，三個請求共用同一條底層連線。Session 用 with 確保收尾時釋放連線池。

🔁 Java 對照：requests.get()（無 Session）≈ 每次 HttpClient.newHttpClient() 再丟掉——浪費。requests.Session() ≈ 建一個 HttpClient/OkHttpClient 用到底——這才是正確姿勢。Session 之於 requests，就像 connection pool 之於 JDBC。

httpx：同樣的 API，外加 async 與 HTTP/2

httpx 的同步 API 幾乎是 requests 的拷貝（httpx.get(...)、Client 對應 Session），但它真正的賣點是 async。回想第 14 章：當你要併發抓很多 URL，同步逐一抓會被 I/O 等待拖死，而 async 能在等待時去做別的請求。

python 
# httpx async（需連網）：併發抓多個 URL
import asyncio
import httpx

async def fetch(client: httpx.AsyncClient, page: int) -> dict:
    resp = await client.get(
        "https://httpbin.org/get",
        params={"page": page},
        timeout=10.0,
    )
    resp.raise_for_status()
    return resp.json()["args"]

async def main() -> None:
    async with httpx.AsyncClient() as client:        # 對應 requests.Session
        # gather 同時發出三個請求，總耗時 ≈ 最慢那一個，而非三者相加
        results = await asyncio.gather(
            *(fetch(client, p) for p in range(1, 4))
        )
        for r in results:
            print(r)

asyncio.run(main())

關鍵字 async with / await 就是第 14 章那套；asyncio.gather 把多個 coroutine 同時跑，相當於 Java 的 CompletableFuture.allOf(...)。同步版本抓三頁要等三輪 round-trip，async 版本三個請求重疊，總時間趨近於單一最慢請求——這正是 async I/O 的甜蜜點。

Java Python 說明
java.net.http.HttpClient（同步 send） requests / httpx（同步） 阻塞式，最直觀
HttpClient.sendAsync + CompletableFuture httpx.AsyncClient + await 非阻塞併發
OkHttp OkHttpClient（一個用到底） Session / Client / AsyncClient 連線池重用
resp.statusCode() 手動檢查 resp.raise_for_status() 非 2xx 轉例外
Jackson readValue(body, ...) resp.json() 內建 JSON 解析

💡 Pythonic 重點：先用 requests，需要 async/HTTP/2 時整段換 httpx——import 改掉、Session 改 Client、加 async/await，邏輯幾乎不動。不要一開始就為「以後可能要 async」過度設計；YAGNI 在 Python 比在 Java 更管用，因為換的成本極低。

pandas 入門：DataFrame 思維

pandas 是 Python 資料處理的核心，但對 Java 工程師而言最大的障礙不是 API，而是思維轉換。你習慣「資料 = 一堆物件，處理 = 對 collection 跑迴圈或 Stream」。pandas 要你換成另一套世界觀：把資料想成一張表，整欄整欄地操作。請同時用三個熟悉的心智模型來理解 DataFrame：

  • 程式化的試算表（Excel）——有列有欄、有欄名。
  • 記憶體裡的 SQL 表——可以 select、where、group by、join。
  • 欄導向資料集（columnar）——每一欄是同型別的連續陣列，這是它快的根源。

Series 是單一欄（帶索引的一維陣列），DataFrame 是多個 Series 並排（共享列索引）。

建立、選取、過濾

python 
import pandas as pd

orders = pd.DataFrame({
    "order_id": [1, 2, 3, 4, 5],
    "customer": ["Alice", "Bob", "Alice", "Carol", "Bob"],
    "category": ["book", "book", "toy", "book", "toy"],
    "amount": [120, 80, 200, 60, 150],
})
print(orders)

0         1    Alice     book     120
1         2      Bob     book      80
2         3    Alice      toy     200
3         4    Carol     book      60
4         5      Bob      toy     150

選取與過濾，請直接對照 SQL：

python 
>>> orders["amount"]              # 選一欄 → Series，等同 SELECT amount
0    120
1     80
2    200
3     60
4    150
Name: amount, dtype: int64

>>> orders[orders["amount"] > 100]    # boolean mask → WHERE amount > 100
   order_id customer category  amount
0         1    Alice     book     120
2         3    Alice      toy     200
4         5      Bob      toy     150

>>> orders.loc[orders["category"] == "book", "amount"].sum()   # WHERE + 聚合
260

orders["amount"] > 100 不是回傳 bool，而是回傳一整欄 True/False 的 Series（boolean mask），再用它去索引 DataFrame 篩列。這就是向量化過濾——你描述「條件」，pandas 一次套用到整欄。.loc[列條件, 欄] 是最常用的二維存取（label-based），.iloc[數字, 數字] 則是純位置存取（integer-based），對應的差別就像 SQL 的欄名 vs JDBC ResultSet.getObject(1)。

java

Map<String, Integer> totals = orders.stream()

向量化思維：最重要的一課

這是整個 pandas 章節的核心。Java 工程師的本能是「遍歷每一列、逐筆處理」，在 pandas 裡這是反模式（anti-pattern）。

python 
# 反模式：逐列 for-loop（慢 10～100 倍，別這樣）
totals = []
for i in range(len(orders)):
    row = orders.iloc[i]
    totals.append(row["amount"] * 1.05)
orders["bad_total"] = totals
python 
# Pythonic / pandas way：向量化，一行
orders["good_total"] = orders["amount"] * 1.05

兩者結果一樣，但效能天差地遠。原因在底層：pandas 的每一欄是 NumPy 陣列，是連續記憶體裡的同型別資料；orders["amount"] * 1.05 這個乘法是在 C 層級對整塊記憶體跑迴圈，沒有 Python 直譯器的逐次 dispatch、沒有逐列建立 Python 物件的開銷。而 for i in range(...) + iloc[i] 每一圈都要：跨越 Python/C 邊界、把一列打包成 Series 物件、再拆出值——每一步都是純 Python 的慢速路徑。

這跟 Java Stream 表面上像（都是「宣告式描述轉換」），但本質不同：Stream 仍在 JVM 逐元素跑你的 lambda；pandas 向量化是把運算整批下放到編譯好的 C/NumPy。所以「用 Stream 還是 for 迴圈」在 Java 裡多半是風格選擇，「用向量化還是 for 迴圈」在 pandas 裡是數量級的效能差異。

條件邏輯也要向量化。要做「if-else 產生新欄」，別跑迴圈，用 numpy.where：

python 
import numpy as np
orders["tier"] = np.where(orders["amount"] >= 100, "high", "low")

💡 Pythonic 重點：在 pandas 裡看到自己寫 for 跑列、iterrows()、apply(axis=1)，先停下來問「這能不能用整欄運算、boolean mask、groupby、np.where 表達？」九成可以。把「逐列思考」改成「整欄思考」，是吃透 pandas 的關鍵 mindset shift。

網頁爬蟲：BeautifulSoup

抓網頁有兩步：取得 HTML（用上面的 requests/httpx）與解析 HTML（用 BeautifulSoup，套件名 beautifulsoup4）。對照 Java，BeautifulSoup ≈ Jsoup——都是把 HTML 解析成可查詢的樹，支援 CSS selector。差別是 Python 這邊解析器可插拔（內建 html.parser，或裝 lxml 求快）。

💡 Pythonic 重點 / 可重現心法：教學與測試絕對不要真的去爬線上網站——網站會改版、會擋你、會讓你的測試時好時壞。正確做法是把 HTML 當成本地字串（或測試 fixture 檔）來解析，這樣範例 100% 可重現、可進 CI、不依賴連網。真實程式碼裡才把這段字串換成 resp.text。下面整段就用本地 HTML 字串示範。

python 
from bs4 import BeautifulSoup

# 真實情境：HTML = requests.get(url, timeout=10).text
# 教學/測試：用本地字串，完全可重現
HTML = """
<html>
  <body>
    <h1 id="title">2026 暢銷書榜</h1>
    <ul class="book-list">
      <li class="book" data-isbn="978-1"><span class="name">流暢的 Python</span><span class="price">880</span></li>
      <li class="book" data-isbn="978-2"><span class="name">設計資料密集型應用</span><span class="price">980</span></li>
      <li class="book" data-isbn="978-3"><span class="name">Effective Java</span><span class="price">650</span></li>
    </ul>
    <a href="https://example.com/next" class="next">下一頁</a>
  </body>
</html>
"""

soup = BeautifulSoup(HTML, "html.parser")

find 取第一個符合的元素、find_all 取全部、get_text 取文字、["attr"] 取屬性：

python 
>>> soup.find("h1").get_text(strip=True)        # 第一個 <h1> 的文字
'2026 暢銷書榜'

>>> for li in soup.find_all("li", class_="book"):   # 注意是 class_（避開關鍵字）
...     name = li.find("span", class_="name").get_text(strip=True)
...     price = li.find("span", class_="price").get_text(strip=True)
...     isbn = li["data-isbn"]                    # 取屬性
...     print(f"{isbn}: {name} -> {price}")
978-1: 流暢的 Python -> 880
978-2: 設計資料密集型應用 -> 980
978-3: Effective Java -> 650

⚠️ 陷阱：用 class_= 不是 class=。class 是 Python 保留字，BeautifulSoup 用尾底線 class_ 規避——這跟你在 Java 用 clazz 當變數名是同一個尷尬。

更強大的是 select/select_one，吃標準 CSS selector，跟 Jsoup 的 select 一模一樣，也跟你前端調 document.querySelectorAll 的肌肉記憶相同：

python 
>>> [span.get_text(strip=True) for span in soup.select("li.book span.name")]
['流暢的 Python', '設計資料密集型應用', 'Effective Java']

>>> soup.select_one("a.next")["href"]            # 取連結 → 可拿去抓下一頁
'https://example.com/next'
java 
// Java Jsoup 對照
Document doc = Jsoup.parse(html);
Elements names = doc.select("li.book span.name");

幾乎一行對一行。find_all 對應 find，select 對應 select，get_text() 對應 .text()，["href"] 對應 .attr("href")。

禮貌爬蟲守則

抓資料是技術問題，怎麼抓是工程倫理與法遵問題。資深工程師務必守住：

  1. 先看 robots.txt（https://site/robots.txt），它聲明哪些路徑不歡迎機器人。標準庫的 urllib.robotparser 可以程式化判讀。尊重它，這是業界基本禮貌也可能涉及法律。
  2. 設誠實的 User-Agent，標明你是誰、附聯絡方式，別偽裝成瀏覽器去躲偵測。
  3. 限速（rate limit）：請求之間 time.sleep 一下，別在迴圈裡全速狂打。把對方伺服器打爆，輕則被封 IP，重則構成攻擊。
  4. 優先用官方 API：很多站有 JSON API，抓 API 比解析 HTML 穩定（不會因改版而碎），也通常是對方允許的方式。
  5. 遵守法遵與授權：服務條款（ToS）、著作權、個資。爬公開資料不等於可以任意散布或商用。
python 
import time
import requests
from urllib.robotparser import RobotFileParser

rp = RobotFileParser()
rp.set_url("https://example.com/robots.txt")
rp.read()                                          # 需連網
ua = "python-book-bot/1.0 (+mailto:me@example.com)"
if rp.can_fetch(ua, "https://example.com/list"):
    resp = requests.get("https://example.com/list",
                        headers={"User-Agent": ua}, timeout=10)
    # ... 解析 ...
    time.sleep(1.0)                                # 限速：每秒最多一次

Excel／報表自動化：openpyxl

「把結果產成 Excel 給老闆/客戶」是企業環境最務實的需求。Java 這邊的答案是 Apache POI——功能強大但 API 極其囉嗦（XSSFWorkbook、createSheet、createRow、createCell、CellStyle、DataFormat……寫個三欄表要四十行）。Python 的對應是 openpyxl，讀寫 .xlsx，API 短得多。

java 
// Apache POI（精簡示意）
Workbook wb = new XSSFWorkbook();
Sheet sheet = wb.createSheet("報表");
Row header = sheet.createRow(0);
header.createCell(0).setCellValue("產品");
header.createCell(1).setCellValue("Q1");
// ... 每個 cell 都要 createCell + setCellValue
try (FileOutputStream out = new FileOutputStream("report.xlsx")) {
    wb.write(out);
}

# 寫一整列直接用 append；單格用 ws["A1"] 或 ws.cell(row, column)

data = [("筆電", 120, 150), ("滑鼠", 300, 280), ("鍵盤", 90, 110)]

    ws.cell(row=i, column=1, value=name)
    ws.cell(row=i, column=2, value=q1)
    ws.cell(row=i, column=3, value=q2)
    ws.cell(row=i, column=4, value=f"=B{i}+C{i}")   # 寫入 Excel 公式

out = Path("/tmp/report_demo.xlsx")
wb.save(out)

注意兩個 Python 化的細節：ws["A1"] 用 __getitem__ 多載成「A1 儲存格」，賦值就是寫入；ws.cell(row, column, value=...) 則用數字座標（方便迴圈）。座標是 1-based（Excel 慣例），別用 Java 的 0-based 直覺去算。公式直接寫成字串 "=B2+C2"，openpyxl 原樣存進去，由 Excel 開檔時計算。

讀回驗證——這是自動化腳本該有的紀律，寫完馬上讀回確認：

print("title:", ws2.title)
print("A2:", ws2["A2"].value)

A2: 筆電
D2 formula: =B2+C2

⚠️ 陷阱：load_workbook 預設讀回的 D2 是公式字串 "=B2+C2"，不是計算結果——因為 openpyxl 不是 Excel，它不算公式。若你需要「上次 Excel 算好的值」，要用 load_workbook(out, data_only=True)，但前提是該檔曾被 Excel（或其他試算表軟體）開啟並存檔過快取值；對一個 openpyxl 剛寫出、從未被 Excel 開過的檔，data_only=True 那格會是 None。這是 POI 沒有的坑（POI 有公式求值器），務必心裡有數。

實務上，如果你的資料已經在 pandas，根本不必手刻 openpyxl——df.to_excel("out.xlsx", index=False) 一行就好，它底層就是呼叫 openpyxl。手寫 openpyxl 的時機是你需要精細控制樣式、公式、多工作表、凍結窗格這類版面細節時。下一節的整合範例會兩種都用上。

排程任務與系統腳本：Python 取代 shell script

到這裡你已經有了所有零件。最後一塊拼圖是：把這些零件組成一個可排程、可維護的自動化腳本，取代你以前用 bash 寫的那些東西。

為什麼用 Python 取代 bash

bash 適合「三行以內的膠水」。一旦腳本要處理錯誤、解析資料、有條件分支，bash 就會變成負債：

面向 bash Python
可讀性 引號、跳脫、$() 地獄 像虛擬碼
錯誤處理 set -euo pipefail 也只能到這 try/except、明確例外型別
資料處理 awk/sed/cut 拼接 pandas、json、csv
型別 全是字串 有型別提示、可 mypy
可測試 幾乎不可能 pytest（第 13 章）
跨平台 Windows 直接死 macOS/Linux/Windows 一致

Python 腳本能 import、能寫測試、能加型別、有真正的例外——這些對「會長期維護、會出事要追責」的自動化任務是決定性的。承接第 15 章，你已經會用 pathlib（路徑）、subprocess（呼叫外部命令，取代 backtick）、argparse（CLI 參數，取代 $1 $2）；承接第 12 章，你會用 logging（取代 echo 到 stderr）與結構化例外處理。

排程選項

寫好腳本後，「怎麼定時跑」有幾條路，對照 Java 生態：

Java Python 適用情境
ScheduledExecutorService sched / time.sleep 迴圈 程式內最陽春的延遲/週期
Spring @Scheduled schedule 套件 程式內、語法甜的週期任務
Quartz APScheduler 套件 程式內、需 cron 表達式/持久化/錯過補跑
作業系統 crontab 作業系統 crontab 呼叫 Python 最穩、與應用解耦、生產首選

經驗法則：生產環境的「每日跑報表」這類任務，首選作業系統的 cron（Linux）或工作排程器（Windows）去呼叫你的 Python 腳本，而不是讓 Python 程序常駐。理由是解耦與可靠性：cron 由系統保證觸發，腳本本身保持「跑一次就結束」的單純無狀態形式，崩了重跑也乾淨。schedule/APScheduler 適合「已經有一個常駐服務、想在裡面塞週期任務」的情境（對應你在 Spring Boot 裡用 @Scheduled）。一行 crontab 範例：

cron 
# 每天 06:30 跑報表腳本，stdout/stderr 導到 log
# uv 的絕對路徑請以 `which uv` 為準（官方安裝腳本通常放 ~/.local/bin，
# Homebrew 在 macOS 為 /opt/homebrew/bin）；cron 的 PATH 很精簡，務必寫絕對路徑
30 6 * * *  ~/.local/bin/uv run /opt/jobs/daily_report.py >> /var/log/daily_report.log 2>&1

整合範例：抓資料 → pandas 整理 → 出報表

把全章串起來。下面是一個可實跑的自動化腳本骨架：抓資料（這裡用內嵌 CSV 字串保證可重現，真實版把 load_raw 換成 requests.get(...).text）、用 pandas 聚合、輸出 Excel 報表、讀回驗證，全程有 logging 與例外處理。

python 
from __future__ import annotations

import io
import logging
import sys
from pathlib import Path

import pandas as pd

logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s %(levelname)s %(name)s: %(message)s",
    datefmt="%Y-%m-%d %H:%M:%S",  # 不加這行，asctime 預設會帶毫秒（如 ...53,946）
)
log = logging.getLogger("daily_report")

# 真實版：RAW = requests.get(URL, timeout=10).text （或 httpx）
# 教學/測試版：內嵌字串 → 可重現、可進 CI
RAW = """date,product,region,units,price
2026-01-05,laptop,APAC,3,1200
2026-01-05,mouse,APAC,10,25
2026-01-06,laptop,EU,2,1250
2026-01-06,mouse,EU,5,28
"""


def load_raw() -> pd.DataFrame:
    log.info("loading raw data")
    return pd.read_csv(io.StringIO(RAW), parse_dates=["date"])


def transform(df: pd.DataFrame) -> pd.DataFrame:
    df = df.copy()
    df["revenue"] = df["units"] * df["price"]            # 向量化
    summary = (
        df.groupby("region", as_index=False)["revenue"]  # GROUP BY region
        .sum()
        .sort_values("revenue", ascending=False)         # ORDER BY revenue DESC
    )
    return summary


def export(df: pd.DataFrame, out: Path) -> None:
    df.to_excel(out, index=False, sheet_name="summary")  # 底層用 openpyxl
    log.info("wrote %d rows to %s", len(df), out)


def main() -> int:
    out = Path("/tmp/region_report.xlsx")
    try:
        summary = transform(load_raw())
        export(summary, out)
        back = pd.read_excel(out, sheet_name="summary")  # 讀回驗證
        log.info("verification:\n%s", back.to_string(index=False))
    except Exception:                                    # 真實版要分類例外
        log.exception("report job failed")               # 印出 traceback
        return 1
    return 0


if __name__ == "__main__":
    sys.exit(main())                                     # 回傳 exit code 給 cron

實跑這支腳本的輸出：

text 
2026-06-15 00:56:53 INFO daily_report: loading raw data
2026-06-15 00:56:53 INFO daily_report: wrote 2 rows to /tmp/region_report.xlsx
2026-06-15 00:56:53 INFO daily_report: verification:
region  revenue
  APAC     3850
    EU     2640

要跑它，連依賴都不用先裝——uv run --with pandas --with openpyxl daily_report.py（第 1、11 章的 uv）會在隔離環境臨時帶上套件執行。注意幾個工程細節：main() 回傳 exit code 給 sys.exit，讓 cron 能判斷成敗；log.exception 在 except 區塊裡自動附上 traceback（對照 Java 的 logger.error("...", e)）；每個階段拆成可單獨測試的函式（load_raw/transform/export），transform 是純函式、不碰 I/O，最好測——這正是第 13 章「把副作用推到邊界」的實踐。

🔁 Java 對照：這支 40 行的腳本，在 Java 等價於一個 Maven 專案 + OkHttp + Jackson + Apache POI + 一個 main，外加打包成 jar 才能讓 cron 呼叫。Python 把它壓縮成一個檔案、一行 uv run。這就是「膠水語言」在自動化場景的壓倒性優勢——不是 Python 哪個功能比 Java 強，而是從想法到能跑的腳本之間的摩擦力低一個數量級。

本章小結

  1. HTTP client：requests 是同步業界標準，httpx 是 API 近似但多了 async／HTTP/2 的續作。永遠設 timeout、用 raise_for_status() 把非 2xx 轉例外、用 resp.json() 解析、用 Session/Client 重用連線（等同重用 Java HttpClient/OkHttpClient）。
  2. DataFrame 思維：把 DataFrame 想成「程式化試算表／記憶體 SQL 表／欄導向資料集」。選取／過濾／groupby 直接對照 SQL 與 Stream，但寫法短得多。
  3. 向量化是最重要的一課：別在 pandas 裡跑 for-loop 逐列算，用整欄運算、boolean mask、np.where、groupby。底層下放到 C/NumPy，效能差一到兩個數量級——這與 Java Stream「仍在 JVM 跑 lambda」本質不同。
  4. BeautifulSoup ≈ Jsoup：find/find_all/select（CSS selector）解析 HTML，get_text() 取文字、["attr"] 取屬性。class_ 要加底線。
  5. 可重現心法：教學與測試用本地 HTML 字串／內嵌資料，不要真的爬線上網站；真實程式碼才換成 resp.text。爬蟲要守禮貌守則：看 robots.txt、誠實 User-Agent、限速、優先用官方 API、遵守法遵。
  6. openpyxl ≈ Apache POI 但短得多：座標 1-based，公式以字串寫入但 openpyxl 不求值（讀回是公式字串，除非 data_only=True 且檔案被 Excel 算過）。資料在 pandas 時直接 df.to_excel（底層即 openpyxl）。
  7. Python 取代 bash：腳本一旦要錯誤處理／資料解析／分支，就用 Python——可讀、可測、有型別、跨平台。組合 pathlib/subprocess/argparse/logging。
  8. 排程：生產首選作業系統 cron 呼叫「跑一次就結束」的 Python 腳本；常駐服務內的週期任務才用 schedule/APScheduler（對應 Spring @Scheduled/Quartz）。

練習

  1. 連線重用驗證：寫一支腳本，用 requests（無 Session）連抓 https://httpbin.org/get 十次，再用 requests.Session() 抓十次，各自計時並印出總耗時，量化連線重用的差異。（提示：time.perf_counter()；需連網。再加碼把 Session 版改寫成 httpx.AsyncClient + asyncio.gather 併發版對照。）

  2. 向量化 vs 迴圈：建一個十萬列的 DataFrame（pd.DataFrame({"x": range(100_000)})），分別用 (a) for + iloc 逐列、(b) df.apply、(c) 純向量化 df["x"] * 2 + 1 計算同一個新欄，用 time.perf_counter() 比較三者耗時，解釋為何向量化最快。

  3. 本地爬蟲：把本章的書榜 HTML 字串擴充（多加幾本書、再加一個 <table>），用 BeautifulSoup 解析成 list[dict]，再用 pd.DataFrame(...) 轉成 DataFrame，過濾出價格大於 700 的書並依價格排序。（提示：select + list comprehension 組 dict。）

  4. 報表腳本：把整合範例改成真正的 CLI 工具：用 argparse（第 15 章）接收 --input（CSV 路徑）與 --output（xlsx 路徑），加上 --group-by 選擇聚合欄位。為 transform 函式寫一個 pytest 測試（第 13 章），餵入小 DataFrame 驗證聚合結果——示範純函式好測的價值。

  5. 加樣式的報表：把練習 4 的輸出改用手寫 openpyxl（而非 to_excel），讓表頭加粗上色、數字欄加千分位格式（cell.number_format = "#,##0"）、凍結首列（ws.freeze_panes = "A2"）。讀回驗證樣式是否寫入。（提示：先 df.to_excel 再用 load_workbook 後製，或全程手寫。）
