第 12 章：例外處理與防禦性程式設計

身為 Java 工程師，你的編譯器幫你管了一輩子的例外：哪些方法會丟什麼、throws 要宣告什麼、catch 漏了哪個都過不了 javac。Python 把這套全拆了——沒有 checked exception，沒有 throws，編譯器不會逼你接任何東西。第一次轉過來，你會覺得像沒繫安全帶開車。本章要做的，是把你那套「工程紀律」從「編譯器強制」搬到「自律 + 文件 + 工具」上：精準捕捉、fail fast、用 chaining 保留根因、用 logging 取代 print、用 warnings 管棄用。學完你會發現，沒有 checked exception 反而讓你寫出更乾淨的程式碼——前提是你得自己扛起責任。

例外階層與「沒有 checked exception」

沒有 checked exception，這是哲學選擇

先把這件事講清楚，因為它顛覆你最深的直覺。Java 把例外分兩類：

Java 範例 編譯器要求
checked exception IOException、SQLException 必須 catch 或在 throws 宣告，否則編譯失敗
unchecked exception NullPointerException、IllegalArgumentException（繼承 RuntimeException） 不強制處理

Python 只有一種：全部都是 unchecked。沒有 throws 宣告，沒有任何例外會被編譯器要求你處理。下面這段，呼叫端完全不知道、也不被強制知道 connect() 會丟 ConnectionError：

python 
def connect(host: str) -> Socket:
    if not reachable(host):
        raise ConnectionError(f"無法連線到 {host}")   # 沒人逼呼叫端 catch 它
    ...
java 
// Java：簽章被例外汙染，呼叫鏈一路往上宣告 throws
Socket connect(String host) throws ConnectionException { ... }

這不是 Python「偷懶」，而是刻意的設計取向。Java 社群自己對 checked exception 的爭議已經吵了二十年——Bruce Eckel、Anders Hejlsberg（C# 設計者）都公開反對它，理由是：checked exception 在大型系統不易擴展（scalability）、會逼出 catch (Exception e) {} 這種吞例外的反模式、也讓介面隨實作細節改變而被迫改簽章。C#、Kotlin、Scala 後來都不採用 checked exception，Python 從一開始就站在這一邊。

Python 的替代方案是它一以貫之的哲學：

  • EAFP（第 5 章）：先做再說，錯了再接，而不是事前 throws 宣告一堆可能性。
  • 靠文件而非編譯器：函式的 docstring 用 :raises ValueError: 之類標注會丟什麼，型別提示（第 9 章）標好輸入輸出，但這些都不在執行期強制——跟型別提示一樣，是給人和工具看的契約，不是給直譯器的命令。

🔁 Java 對照：Java 的 throws 是編譯期契約；Python 的「會丟什麼」是 docstring 上的君子協定。沒有編譯器當靠山，是你的紀律要補上這一塊。

💡 Pythonic 重點：不要試圖在 Python 重建 checked exception（例如回傳 Result 物件包住錯誤）。除非你在寫函數式風格的程式庫，否則這在 Python 不慣用、也得不到語言支援。慣用做法是直接 raise，讓例外往上拋到能處理它的層級。

例外階層：`BaseException` 在頂端

Python 所有例外的根是 BaseException，但你日常 raise 和 except 的東西繼承自它底下的 Exception：

text 
BaseException
 ├── SystemExit          ← sys.exit() 丟的，不要接
 ├── KeyboardInterrupt   ← Ctrl-C，不要接
 ├── GeneratorExit
 └── Exception           ← 你所有的「業務錯誤」都在這層以下
      ├── ValueError        型別對、但內容不合法（int("abc")）
      ├── TypeError         型別不對（"a" + 1）
      ├── LookupError
      │    ├── KeyError     dict 找不到 key
      │    └── IndexError   list 索引越界
      ├── ArithmeticError
      │    └── ZeroDivisionError
      ├── OSError           IO/檔案/網路（FileNotFoundError 等都繼承它）
      ├── AttributeError    物件沒有那個屬性
      ├── RuntimeError
      └── ...

關鍵在 SystemExit 和 KeyboardInterrupt 不是 Exception 的子類，是故意放在 Exception 旁邊的：

python 
>>> issubclass(KeyboardInterrupt, Exception)
False
>>> issubclass(KeyboardInterrupt, BaseException)
True
>>> issubclass(KeyError, LookupError)
True

這個設計有實際用途：你寫 except Exception: 時，不會誤接 Ctrl-C 和 sys.exit()。所以使用者按 Ctrl-C 永遠能中斷你的程式，這正是你要的。

⚠️ 陷阱：千萬不要寫 except BaseException: 或裸 except:（等同接 BaseException）。那會把 KeyboardInterrupt 和 SystemExit 一起吞掉，使用者按 Ctrl-C 你的迴圈卻不停，這是惡名昭彰的「殭屍程式」成因。要接「所有業務錯誤」就用 except Exception:，永遠到此為止。

常見內建例外的語意值得記熟，因為精準捕捉靠的就是選對類別：

python 
>>> int("abc")          # ValueError：是字串但內容不是合法整數
>>> "a" + 1             # TypeError：型別不相容
>>> {"x": 1}["y"]       # KeyError
>>> [1, 2][5]           # IndexError
>>> open("nope.txt")    # FileNotFoundError（OSError 子類）

try / except / else / finally：四個區塊各司其職

Java 只有 try / catch / finally。Python 多了一個你沒看過的 else，而且四個區塊的分工比 Java 更講究。

python 
def load_user(user_id: int) -> dict:
    try:
        raw = db.fetch(user_id)        # 1. 可能出錯的「嘗試」
    except KeyError:
        raise UserNotFound(user_id)    # 2. 出錯時的處理
    else:
        return parse(raw)              # 3. 沒出錯才執行
    finally:
        db.release()                   # 4. 不管怎樣都執行

`else`：try 沒拋例外才執行

else 是 Java 工程師最該停下來想一下的區塊。它的語意是「try 區塊正常結束（沒拋例外）時才執行」。為什麼不直接把 parse(raw) 寫進 try？因為這樣能精準界定「哪一行的例外是你要接的」：

python 
# ❌ 把太多東西塞進 try，捕捉範圍失準
try:
    raw = db.fetch(user_id)
    return parse(raw)          # 如果 parse 內部也丟 KeyError，會被誤接成 UserNotFound！
except KeyError:
    raise UserNotFound(user_id)
python 
# ✅ 用 else 把「不該被這個 except 接的程式碼」移出 try
try:
    raw = db.fetch(user_id)    # 只有這行的 KeyError 該被視為「找不到使用者」
except KeyError:
    raise UserNotFound(user_id)
else:
    return parse(raw)          # parse 的錯誤會原樣往上拋，不會被誤判

這正是防禦性設計的核心：try 區塊應該只包真正可能拋出你要接的那個例外的程式碼，其餘搬進 else。Java 沒有 else，所以 Java 工程師習慣把後續邏輯也塞進 try，連帶擴大了捕捉範圍——這個壞習慣要在 Python 戒掉。

`finally`：清理語意與一個陷阱

finally 跟 Java 一樣：無論正常、例外、還是 return，都會執行，通常用於釋放資源。但有個 Java 也有的坑要提醒：finally 裡的 return 會吞掉例外。

python 
def bad():
    try:
        raise ValueError("原始錯誤")
    finally:
        return 42        # ❌ 例外被默默吃掉，呼叫端拿到 42，永遠不知道出過錯

>>> bad()
42

註：Python 3.14 起，finally 內的 return 會額外發出 SyntaxWarning: 'return' in a 'finally' block，正是因為這個吞例外的行為太常被誤用——更不該這樣寫。（3.12/3.13 上不會有此警告，輸出與上面相同。）

實務上資源清理多半不用手寫 finally，而是用 with 上下文管理器（context manager），更乾淨也更難出錯（詳見第 15 章）。finally 留給沒有 context manager 包裝的情況。

多個 except 與順序：子類在前

except 子句由上往下比對，第一個匹配的就贏。所以子類別要放在父類別前面，否則永遠輪不到：

python 
try:
    process()
except FileNotFoundError:        # 具體的放前面
    print("檔案不存在")
except OSError:                  # 一般的放後面（FileNotFoundError 是 OSError 子類）
    print("其他 IO 錯誤")
python 
# ❌ 順序錯了：OSError 先匹配，FileNotFoundError 永遠走不到
try:
    process()
except OSError:
    ...
except FileNotFoundError:        # dead code：FileNotFoundError 永遠走不到（注意 mypy/ruff 預設並不會警告此種順序錯誤，需自行留意）
    ...

要一次接多種不相關的例外，用 tuple：

python 
try:
    data = json.loads(s)
except (ValueError, TypeError) as e:
    logger.warning("解析失敗: %s", e)

exception groups 與 `except*`（3.11+）

Python 3.11 引入了 ExceptionGroup 與 except*，解決「同時發生多個例外」的場景——典型是並行任務（第 14 章的 asyncio.TaskGroup）一次跑十個，三個失敗，你要全部回報而不是只看到第一個：

python 
def task(n: int) -> None:
    if n % 2 == 0:
        raise ValueError(f"even {n}")
    raise TypeError(f"odd {n}")

errors = []
for n in range(4):
    try:
        task(n)
    except Exception as e:
        errors.append(e)

# 把一批例外打包成一個群組往上拋
try:
    raise ExceptionGroup("批次任務失敗", errors)
except* ValueError as eg:                          # 注意是 except*（星號）
    print("ValueErrors:", [str(e) for e in eg.exceptions])
except* TypeError as eg:
    print("TypeErrors:", [str(e) for e in eg.exceptions])
text 
ValueErrors: ['even 0', 'even 2']
TypeErrors: ['odd 1', 'odd 3']

except* 跟一般 except 不同：它會把群組「拆開」，每個 except* 子句都可能被執行（不是只執行第一個匹配），各自處理它認得的那一類。日常開發不一定常用，但理解它，因為 asyncio.TaskGroup 失敗時丟的就是 ExceptionGroup。

自訂例外與 exception chaining

建立專案的例外階層

跟 Java 一樣，你應該為專案定義一個例外階層。慣例是先有一個基底例外，所有自家例外都繼承它，這樣呼叫端可以「一網打盡」自家錯誤，也能精準接某一種：

python 
class OrderError(Exception):
    """訂單模組所有例外的基底。"""

class OrderNotFound(OrderError):
    def __init__(self, order_id: int) -> None:
        super().__init__(f"找不到訂單 {order_id}")
        self.order_id = order_id        # 例外帶足資訊，方便上層程式化處理

class PaymentDeclined(OrderError):
    def __init__(self, order_id: int, reason: str) -> None:
        super().__init__(f"訂單 {order_id} 付款被拒：{reason}")
        self.order_id = order_id
        self.reason = reason
java 
// Java：概念完全一致
class OrderException extends RuntimeException { ... }
class OrderNotFoundException extends OrderException { ... }

用起來，呼叫端可選擇接基底（管它哪種）或具體類別：

python 
try:
    place_order(req)
except PaymentDeclined as e:
    notify_user(e.reason)      # 程式化讀 e.reason，不是 parse 錯誤字串
except OrderError as e:
    logger.error("訂單處理失敗: %s", e)

💡 Pythonic 重點：例外要帶足資訊。別只 raise OrderError("失敗")，把 order_id、reason 存成屬性，讓上層能程式化處理，而不是去 str(e) 解析字串。這跟 Java 自訂例外加 getter 是同一個道理。

exception chaining：保留根因

這是 Python 比 Java 更優雅、也更該善用的功能。當你在 except 裡把低階例外轉換成高階例外時，根因不該丟失。

Python 有兩種鏈結：

隱式鏈結（__context__）：在 except 區塊內又拋出新例外，Python 自動把原例外掛在 __context__ 上，traceback 會印「During handling of the above exception, another exception occurred」：

python 
def load():
    try:
        int("abc")
    except ValueError:
        raise RuntimeError("設定載入失敗")    # 沒寫 from，但根因自動保留在 __context__

顯式鏈結（__cause__，用 from）：明確表達「這個新例外是由那個舊例外引起的」。traceback 印「The above exception was the direct cause of the following exception」，語意更強：

python 
class ConfigError(Exception):
    pass

def load_config(raw: str) -> int:
    try:
        return int(raw)
    except ValueError as e:
        raise ConfigError(f"無法解析設定值: {raw!r}") from e   # 顯式鏈結
python 
>>> try:
...     load_config("abc")
... except ConfigError as e:
...     print("cause:", repr(e.__cause__))
...     print("context:", repr(e.__context__))
cause: ValueError("invalid literal for int() with base 10: 'abc'")
context: ValueError("invalid literal for int() with base 10: 'abc'")

兩者差別在語意與 traceback 措辭：from e 是「我故意把它轉成這個」，隱式是「在處理 A 時碰巧又爆了 B」。對「轉換例外」的場景一律用 from，意圖清楚。

🔁 Java 對照：raise NewError(...) from e ≈ Java 的 new NewException(msg, cause)（帶 cause 的建構子）；e.__cause__ ≈ Throwable.getCause()。Python 的差別是它預設就會隱式鏈結，你什麼都不寫根因也不會消失——這比 Java 容易忘記傳 cause 的情況更安全。

`from None`：刻意隱藏根因

有時根因是實作細節，對呼叫端是雜訊（例如你內部用 dict 查表，但不想洩漏「其實是 KeyError」）。用 from None 切斷鏈結：

python 
def get_country(code: str) -> str:
    try:
        return _COUNTRY_TABLE[code]
    except KeyError:
        raise ValueError(f"未知國碼: {code!r}") from None   # 隱藏 KeyError 細節
python 
>>> try:
...     get_country("ZZ")
... except ValueError as e:
...     print("cause:", e.__cause__)
...     print("suppress_context:", e.__suppress_context__)
cause: None
suppress_context: True

from None 把 __suppress_context__ 設為 True，traceback 就不會印那串「During handling...」。節制使用——根因通常是除錯時的救命稻草，只在它真的是無關內部細節時才隱藏。

logging：千萬別用 print

print() 除錯是 Python 新手的通病，對你這種寫過 Java 的人來說，這等於在 production code 裡用 System.out.println——不可接受。Python 的 logging 模組就是它的 SLF4J/Logback，概念幾乎一對一對應。

logger 階層、level、handler、formatter

Java（SLF4J/Logback） Python（logging） 說明
LoggerFactory.getLogger(Foo.class) logging.getLogger(__name__) 取得具名 logger
Logger 名稱用 . 分層 同樣用 . 分層 a.b.c 的 parent 是 a.b
Appender Handler 決定輸出到哪（console/file/...）
PatternLayout Formatter 決定輸出格式
Level（TRACE..ERROR） Level（DEBUG..CRITICAL） 過濾門檻
logback.xml dictConfig / basicConfig 設定

logger 是樹狀階層，用名稱的點號分層，跟 Java 一模一樣。慣例是每個模組頂部寫：

python 
import logging

logger = logging.getLogger(__name__)   # 模組是 myapp.service 時，logger 名就是 myapp.service
python 
>>> import logging
>>> logging.getLogger("myapp")          # 先讓中介 logger 存在
<Logger myapp (WARNING)>
>>> child = logging.getLogger("myapp.parser")
>>> child.parent.name
'myapp'

⚠️ 陷阱：logger.parent 指向的是目前已存在的最近祖先 logger，不是字面上的父名稱。若你沒先呼叫過 logging.getLogger("myapp")，那麼 logging.getLogger("myapp.parser").parent.name 會直接是 'root'——因為中介的 myapp logger 從未被建立，myapp.parser 的 parent 就跳到 root logger。這是 logging 階層常見的誤解點：名稱用點號分層只是命名慣例，parent 鏈則取決於哪些 logger 真的被 getLogger 過。上面範例先建立 myapp 才看得到 'myapp'。

訊息會沿著階層往上「傳播」（propagate）到 root logger 的 handler 輸出。所以你通常只在程式進入點（main）設定一次 handler，各模組只管 logger.xxx()：

python 
import logging

logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s [%(levelname)s] %(name)s: %(message)s",
)
logger = logging.getLogger("myapp.service")
logger.info("服務啟動")
# 2026-06-14 20:07:16,151 [INFO] myapp.service: 服務啟動

💡 Pythonic 重點：log 訊息用 %s 占位符 + 參數，不要自己用 f-string 拼。logger.info("處理 %s 筆", n) 而非 logger.info(f"處理 {n} 筆")。理由跟 SLF4J 的 {} 一樣：當這條 log 的 level 被過濾掉時，字串格式化根本不會執行，省下成本。

函式庫不要動 root logger，不要用 print

這是寫可重用程式庫時最重要的紀律，對照 SLF4J 的設計哲學：

  • 函式庫絕不呼叫 basicConfig()、不加 handler、不設 level。那是應用程式的權責。函式庫亂設，會搶走使用你函式庫的人的設定權。
  • 函式庫只要 logger = logging.getLogger(__name__) 然後正常記 log。要避免「No handlers could be found」的雜訊，慣例是在套件 __init__.py 加一個 NullHandler：
python 
# 函式庫的 mypackage/__init__.py
import logging
logging.getLogger(__name__).addHandler(logging.NullHandler())

這正是 SLF4J「函式庫只依賴 API、不綁定實作」的 Python 版：函式庫產生 log 事件，但由應用程式決定要不要輸出、輸出到哪。print() 完全破壞這套——它無法被過濾、無法調 level、無法重導向，永遠噴到 stdout，所以函式庫裡看到 print 等於 bug。

`logger.exception()`：在 except 內記 traceback

在 except 區塊裡記錯誤，用 logger.exception()，它會自動附上完整 traceback（等同 logger.error(..., exc_info=True)）：

python 
logger = logging.getLogger("myapp.service")

def divide(a: float, b: float) -> float | None:
    try:
        return a / b
    except ZeroDivisionError:
        logger.exception("除法失敗 a=%s b=%s", a, b)   # 自動帶 traceback
        return None

divide(1, 0)
text 
2026-06-14 20:07:16,151 [ERROR] myapp.service: 除法失敗 a=1 b=0
Traceback (most recent call last):
  File "...", line 5, in divide
    return a / b
ZeroDivisionError: division by zero

logger.exception() 只能在 except 區塊內呼叫（它去抓「當前正在處理的例外」）。這對應 SLF4J 的 logger.error("msg", ex)——把 Throwable 當最後一個參數傳進去印 stack trace。

⚠️ 陷阱：別寫 logger.error(str(e)) 或 logger.error(f"出錯: {e}")，那只印錯誤訊息、丟失 traceback，等於 Java 裡只 log.error(e.getMessage()) 不傳 exception——除錯時你會恨自己。在 except 內一律 logger.exception(...)。

結構化日誌（structured logging，把 log 輸出成 JSON 給 ELK/Datadog 之類消費）在 production 很常見，標準庫可用 logging.config.dictConfig 配合自訂 Formatter，或直接上 structlog 這類套件，這裡點到為止。

warnings 與棄用策略

最後一塊是 warnings，對應 Java 的 @Deprecated。先釐清它跟 logging 的分工，這是 Java 工程師常混淆的點：

warnings logging
對象 開發者（用你 API 的人） 維運者 / 你自己
典型用途 API 棄用、語意可疑的用法 執行期事件、錯誤、追蹤
預設行為 同一處只印一次、可被過濾、可轉成錯誤 依 level 輸出
控制 -W 旗標 / warnings.filterwarnings log level / handler

用 `warnings.warn` 標記棄用

棄用一個 API 時，用 DeprecationWarning：

python 
import warnings

def old_api() -> int:
    warnings.warn(
        "old_api() 已棄用，將於 2.0 移除，請改用 new_api()",
        DeprecationWarning,
        stacklevel=2,        # 關鍵：見下方說明
    )
    return new_api()

// Java：靠註解 + javadoc，編譯期由 IDE/javac 提示
@Deprecated
public int oldApi() { ... }

差別很關鍵：Java 的 @Deprecated 是編譯期提示（IDE 畫刪除線、-Xlint:deprecation 警告）；Python 的 DeprecationWarning 是執行期才觸發的事件。它的好處是更靈活——可以被過濾、被靜音、或升級成錯誤讓 CI 抓出來：

python 
>>> warnings.simplefilter("error", DeprecationWarning)
>>> old_api()
Traceback (most recent call last):
  ...
DeprecationWarning: old_api() 已棄用，將於 2.0 移除，請改用 new_api()

在 CI 用 python -W error::DeprecationWarning 跑測試，就能強制「任何用到棄用 API 的程式碼直接讓測試紅燈」——這比 Java 的編譯警告更有強制力。pytest 預設也會把 DeprecationWarning 顯示出來。

`stacklevel` 為什麼重要

stacklevel 決定警告訊息指向哪一行程式碼。預設 stacklevel=1，警告會指向 warnings.warn() 那一行——但那是你函式庫內部，對使用者毫無意義。設 stacklevel=2，警告就指向呼叫 old_api() 的那一行，也就是使用者自己的程式碼，他才知道該去改哪裡：

python 
# stacklevel=1（預設）：訊息指向 deprecation.py 第 5 行（函式庫內部）→ 沒用
# stacklevel=2：訊息指向 user_script.py 第 42 行（呼叫處）→ 使用者一眼看到該改哪

這跟 SLF4J/Log4j 算「呼叫者位置」是同一個問題。寫棄用警告幾乎都該設 stacklevel=2（若多包一層 wrapper 就再往上加）。漏掉它是棄用警告最常見的 bug——訊息對，但指錯地方。

💡 Pythonic 重點：棄用是「給開發者的訊息」用 warnings，不是用 logging。因為 warning 能被使用者的 CI 轉成錯誤、能精準指到呼叫處、且同一處只吵一次。混用會讓使用者無法用標準工具（-W、filter）管理它。

防禦性設計：把紀律從編譯器搬到自己身上

收束本章，把散落的原則整理成你日常該守的幾條鐵律：

  1. 精準捕捉，別裸 except:。接你打算處理、也知道怎麼處理的那個例外。接不該接的等於把 bug 藏起來。

    python 
    # ❌ 把所有錯誤（連程式 bug）都當成「設定錯誤」吞掉
try:
    cfg = load_config(path)
except Exception:
    cfg = DEFAULT_CONFIG

# ✅ 只接真正預期的那個
try:
    cfg = load_config(path)
except FileNotFoundError:
    cfg = DEFAULT_CONFIG          # 檔案不存在用預設，合理
# 其他錯誤（解析失敗、權限不足）原樣往上拋——那是真 bug，該炸

  2. fail fast。輸入不合法就在邊界立刻 raise，別讓壞資料滲進系統深處才爆，那時 traceback 早就離現場十萬八千里。這跟 Java 的 Objects.requireNonNull / 參數驗證同理。

  3. 例外帶足資訊。把 order_id、path、reason 存成屬性，讓上層能程式化處理，而不是去 parse 錯誤字串。

  4. 用 chaining 保留根因。轉換例外時 raise HighLevelError(...) from low_level，除非根因真是無關細節才 from None。

  5. except 內用 logger.exception()，永遠保留 traceback。

  6. 不要吞例外。except SomeError: pass 是高風險動作，至少要 log。真的要靜默忽略，用 contextlib.suppress 明確表達意圖：

    python 
    import contextlib
with contextlib.suppress(FileNotFoundError):
    os.remove(tmp_path)        # 檔案本來就不在，無所謂——意圖清楚，不是手滑漏接

最後，回到開頭那個「沒繫安全帶」的不安：沒有 checked exception，不代表 Python 比較不安全，而是它把「哪些錯誤要處理、處理到哪一層」的決定權從編譯器交還給你。對一個資深工程師而言，這其實是好事——你本來就比編譯器更懂業務該怎麼降級、該不該重試、該對使用者說什麼。本章這套紀律，就是你接下這份權力的代價與配套。

本章小結

  • Python 沒有 checked exception，全部例外都是 unchecked，沒有 throws。這是刻意的設計選擇，呼應 EAFP 哲學與「靠文件/型別而非編譯器強制」的一貫立場。
  • 例外根是 BaseException；SystemExit、KeyboardInterrupt 故意不在 Exception 之下，所以 except Exception: 安全地不誤接 Ctrl-C。永遠別用裸 except: 或 except BaseException:。
  • try/except/else/finally 四區塊：else 是 Java 沒有的，用它把「不該被這個 except 接的程式碼」移出 try，讓捕捉範圍精準。
  • 多個 except 子類在前父類在後；except (A, B) 接多種；3.11 的 ExceptionGroup + except* 處理「同時多個例外」（asyncio TaskGroup 的失敗形態）。
  • 為專案建立繼承 Exception 的例外階層；轉換例外時用 raise New(...) from original 保留根因（__cause__），無關細節才 from None（__context__ 是預設的隱式鏈結）。
  • 用 logging 不用 print：logging.getLogger(__name__)、%s 占位符、except 內用 logger.exception()。函式庫絕不動 root logger，加 NullHandler 即可，輸出由應用程式決定。
  • 棄用 API 用 warnings.warn(..., DeprecationWarning, stacklevel=2)（對應 @Deprecated，但執行期觸發、可被 CI 轉成錯誤）；stacklevel=2 讓警告指向呼叫處而非函式庫內部。
  • 防禦性設計鐵律：精準捕捉、fail fast、例外帶足資訊、chaining 保留根因、不吞例外（要忽略用 contextlib.suppress）。

練習

  1. 戒掉裸 except：下面這段把所有錯誤都當「使用者不存在」。請改寫成只接真正代表「找不到」的例外，其餘原樣往上拋，並用 raise ... from ... 保留根因。

    python 
    def get_user(uid):
    try:
        row = db.query("SELECT * FROM users WHERE id=?", uid)
        return User(row)
    except Exception:
        raise UserNotFound(uid)

    提示：先想清楚「找不到」對應哪個底層例外（可能是 KeyError 或自訂的 NoRowsError），其餘（連線斷、SQL 語法錯）不該被偽裝成 UserNotFound。

  2. 用 else 收斂 try：把一段「try 裡塞了 fetch + parse + return 三件事、只想接 fetch 的 TimeoutError」的程式碼，重構成 try（只放 fetch）/ except TimeoutError / else（放 parse + return）三段式，確保 parse 的錯誤不會被誤接。

  3. 建立例外階層：為一個 payment 模組設計例外階層：基底 PaymentError，子類 CardDeclined（帶 card_last4 與 reason 屬性）、InsufficientFunds（帶 shortfall 金額）。寫一個 charge() 函式在內部把第三方 SDK 丟的 SdkError 轉成你的例外（用 from），並寫呼叫端示範「接基底一網打盡」與「接具體類別讀屬性」兩種用法。

  4. logging 取代 print：把任一個你用 print 除錯的小腳本改成 logging：模組頂部 logger = logging.getLogger(__name__)，main 裡 basicConfig，把錯誤處的 print(e) 換成 logger.exception(...)。觀察 traceback 是否完整出現，並試著把 level 調成 WARNING 看 INFO 訊息消失。

  5. 棄用警告 + CI 攔截：寫一個 legacy_parse(s)，在內部 warnings.warn(..., DeprecationWarning, stacklevel=2) 並轉呼叫新的 parse(s)。用 uv run --with pytest pytest -W error::DeprecationWarning 跑一個會呼叫 legacy_parse 的測試，確認它因棄用警告被轉成錯誤而紅燈；再把 stacklevel 改成 1，觀察警告指向的行號如何從「呼叫處」變成「函式庫內部」。
