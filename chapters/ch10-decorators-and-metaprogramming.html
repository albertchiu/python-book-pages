第 10 章：裝飾器與元程式設計

在 Java 裡，要在不改動原始碼的前提下「橫切」一段邏輯——記時、記日誌、開交易、做權限檢查——你會想到 annotation 加上一個框架：寫個 @Transactional，然後讓 Spring AOP 或 AspectJ 在執行期用動態代理、或在編譯期織入（weaving）位元組碼，把行為塞進去。Python 把同一件事做得直接得多：裝飾器（decorator）就是一個吃函式、回傳函式的普通函式，沒有反射、沒有代理、沒有框架，在 def 執行的那一刻就把包裝套好了。本章從裝飾器原理一路往下挖到 Python 元程式設計（metaprogramming）的底層——__getattr__、descriptor、metaclass——你會發現 @property、@dataclass、甚至「方法為什麼能綁定 self」這些每天用的東西，背後都是同一套機制。重點先講在前頭：裝飾器是日常工具，descriptor 是理解 @property 的鑰匙，metaclass 是 99% 用不到的重武器。

裝飾器原理：`@deco` 只是語法糖

承接第 3 章——函式是一等物件，能被當參數傳入、當回傳值傳出。裝飾器把這兩件事合起來：一個接收函式、回傳函式的高階函式（higher-order function）。

先看最樸素的形式，完全不用 @ 符號：

python 
from collections.abc import Callable
from typing import Any

def my_logger(func: Callable[..., Any]) -> Callable[..., Any]:
    def wrapper(*args: Any, **kwargs: Any) -> Any:
        print(f"呼叫 {func.__name__}，參數 {args} {kwargs}")
        result = func(*args, **kwargs)
        print(f"{func.__name__} 回傳 {result!r}")
        return result
    return wrapper

def add(a: int, b: int) -> int:
    return a + b

add = my_logger(add)        # 關鍵這一行：把 add 換成包裝後的版本
python 
>>> add(3, 4)
呼叫 add，參數 (3, 4) {}
add 回傳 7
7

add = my_logger(add) 是整個裝飾器的精髓：my_logger 收到原始的 add，回傳一個新函式 wrapper，我們把 wrapper 重新綁定到 add 這個名字上。之後呼叫 add(3, 4)，跑的其實是 wrapper，它在前後印日誌、中間轉發給原本的函式。

@ 符號只是把這行「重新賦值」寫得漂亮一點：

python 
@my_logger          # 完全等價於 add = my_logger(add)
def add(a: int, b: int) -> int:
    return a + b

💡 Pythonic 重點：@deco 與 f = deco(f) 是完全相同的兩種寫法，前者只是語法糖（syntactic sugar）。每次看到 @something，心裡就翻譯成「把下面這個函式餵給 something，再用回傳值蓋掉原名」。理解這條等式，裝飾器就沒有任何神祕之處了。

手寫一個計時裝飾器

把 logging 換成記時，結構一模一樣：

python 
import time
from collections.abc import Callable
from functools import wraps
from typing import Any

def timed(func: Callable[..., Any]) -> Callable[..., Any]:
    @wraps(func)                                   # 稍後解釋為什麼需要它
    def wrapper(*args: Any, **kwargs: Any) -> Any:
        start = time.perf_counter()
        try:
            return func(*args, **kwargs)
        finally:
            elapsed = time.perf_counter() - start
            print(f"{func.__name__} 花了 {elapsed * 1000:.2f} ms")
    return wrapper

@timed
def slow_square(n: int) -> int:
    time.sleep(0.05)
    return n * n
python 
>>> slow_square(9)
slow_square 花了 50.12 ms
81

用 try/finally 確保「即使被裝飾的函式拋例外，記時依然會印出來」——這是橫切關注點（cross-cutting concern）的典型細節。

對照 Java：目的相同，機制天差地別

Java（annotation + AOP） Python（decorator）
本質 annotation 是被動的中繼資料（metadata），自己什麼都不做 decorator 是主動的函式，本身就執行包裝
何時生效 靠框架在執行期用反射 / 動態代理，或編譯期 weaving（AspectJ） 在 def 被執行、模組載入的那一刻就套好
需要框架嗎 通常要（Spring 容器、AspectJ 編譯器） 不需要，純語言特性
攔截範圍 方法呼叫（proxy 邊界） 任何可呼叫物件，連巢狀函式都行
java 
// Java：annotation 本身不做事，要 Spring 容器掃到它、建立代理才生效
@Transactional
public void transfer(Account from, Account to, BigDecimal amount) { ... }
python 
# Python：decorator 在 def 執行時就把 transfer 換成包裝版，沒有容器、沒有代理
@transactional
def transfer(src: Account, dst: Account, amount: Decimal) -> None:
    ...

🔁 Java 對照：@Transactional 是「貼張標籤，等框架來讀」；@transactional 是「立刻把函式換掉」。Java 的 annotation 沒有 Spring 就是死的中繼資料；Python 的 decorator 沒有任何框架就能運作，因為它本身就是會執行的程式碼。

⚠️ 陷阱：裝飾器在「定義時」就執行，不是「呼叫時」。@timed def f(): ... 一被直譯器讀到，timed(f) 就立刻跑了一次。如果你的裝飾器在最外層做了重量級的事（連資料庫、註冊全域狀態），那是在模組載入時發生的，不是第一次呼叫函式時。

`functools.wraps`：別讓包裝抹掉身分

上面 timed 裡那行 @wraps(func) 不是裝飾品。沒有它，包裝會把原函式的 metadata 全部蓋掉——因為呼叫者拿到的是 wrapper，不是原函式：

python 
from collections.abc import Callable
from typing import Any

def bad_deco(func: Callable[..., Any]) -> Callable[..., Any]:
    def wrapper(*args: Any, **kwargs: Any) -> Any:    # 沒有 @wraps
        return func(*args, **kwargs)
    return wrapper

@bad_deco
def greet(name: str) -> str:
    """跟某人打招呼。"""
    return f"Hi {name}"
python 
>>> greet.__name__
'wrapper'                       # 慘了，名字變成 wrapper
>>> greet.__doc__
>>> # None，docstring 也不見了

greet.__name__ 變成 'wrapper'、docstring 消失——這會搞壞日誌、除錯訊息、help()、以及任何靠 __name__ 運作的工具（包括其他裝飾器）。functools.wraps 把原函式的 __name__、__doc__、__module__、__qualname__ 等複製到 wrapper 上，並額外設一個 __wrapped__ 指回原函式：

python 
from functools import wraps

def good_deco(func: Callable[..., Any]) -> Callable[..., Any]:
    @wraps(func)
    def wrapper(*args: Any, **kwargs: Any) -> Any:
        return func(*args, **kwargs)
    return wrapper

@good_deco
def greet(name: str) -> str:
    """跟某人打招呼。"""
    return f"Hi {name}"
python 
>>> greet.__name__
'greet'
>>> greet.__doc__
'跟某人打招呼。'
>>> greet.__wrapped__            # @wraps 留的逃生門：可以拿回原始未包裝的函式
<function greet at 0x...>

💡 Pythonic 重點：寫任何回傳 wrapper 的裝飾器，一律加 @functools.wraps(func)。這是肌肉記憶等級的紀律。__wrapped__ 還讓你能在測試時繞過裝飾器、直接呼叫原函式（greet.__wrapped__("x")）。

帶參數的裝飾器：三層巢狀

@timed 不帶參數，直接 timed(func)。但 @retry(times=3) 這種帶參數的呢？關鍵觀察：retry(times=3) 是一個函式呼叫，它的回傳值才是真正的裝飾器。所以結構多了一層：

python 
import time
from collections.abc import Callable
from functools import wraps
from typing import Any

def retry(times: int = 3, delay: float = 0.1) -> Callable[..., Any]:
    # 第 1 層：retry 接收「參數」，回傳真正的裝飾器
    def decorator(func: Callable[..., Any]) -> Callable[..., Any]:
        # 第 2 層：decorator 接收「函式」，回傳包裝
        @wraps(func)
        def wrapper(*args: Any, **kwargs: Any) -> Any:
            # 第 3 層：wrapper 接收「呼叫時的引數」，做實際的事
            last_exc: Exception | None = None
            for attempt in range(1, times + 1):
                try:
                    return func(*args, **kwargs)
                except Exception as exc:
                    last_exc = exc
                    print(f"第 {attempt} 次失敗：{exc!r}")
                    time.sleep(delay)
            assert last_exc is not None
            raise last_exc
        return wrapper
    return decorator

對應的呼叫順序這樣拆解：

python 
@retry(times=3, delay=0.05)
def flaky() -> str:
    ...

# 完全等價於：
# step1 = retry(times=3, delay=0.05)   # step1 是 decorator
# flaky = step1(flaky)                  # flaky 變成 wrapper

實際跑一個一定會失敗的函式：

python 
@retry(times=3, delay=0.0)
def always_fail() -> None:
    raise ValueError("壞掉了")
python 
>>> always_fail()
第 1 次失敗：ValueError('壞掉了')
第 2 次失敗：ValueError('壞掉了')
第 3 次失敗：ValueError('壞掉了')
Traceback (most recent call last):
    ...
ValueError: 壞掉了

三層各司其職：最外層吃裝飾器參數、中間層吃函式、最內層吃呼叫時的引數。記住這個對應，帶參數的裝飾器就不會繞暈。

⚠️ 陷阱：@retry 與 @retry() 不一樣！@retry（不加括號）會把 retry 當成不帶參數的裝飾器，於是 func 被當成 times 傳進去，整個壞掉。帶參數的裝飾器一定要加括號，即使全部用預設值也得寫 @retry()。這是 Java 工程師（習慣 @Retryable 可加可不加括號）最常踩的坑。

💡 Pythonic 重點：如果你覺得三層巢狀很煩，標準函式庫的 functools.partial 或第三方的 wrapt 能簡化；但三層結構是基礎功，看懂它你才看得懂任何別人寫的裝飾器。

類別裝飾器：裝飾的是類別

裝飾器不限於函式——@ 放在 class 上面，收到的就是類別物件，回傳值（通常是修改後的同一個類別）取代原類別。最常見的用途是自動註冊到 registry：

python 
from collections.abc import Callable
from typing import Any

HANDLERS: dict[str, type] = {}

def register(name: str) -> Callable[[type], type]:
    def decorator(cls: type) -> type:
        HANDLERS[name] = cls          # 把類別登記到全域表
        return cls                    # 原封不動回傳，只是順手登記
    return decorator

@register("json")
class JSONHandler:
    def parse(self, raw: str) -> Any:
        import json
        return json.loads(raw)

@register("csv")
class CSVHandler:
    def parse(self, raw: str) -> Any:
        return [row.split(",") for row in raw.splitlines()]
python 
>>> HANDLERS
{'json': <class 'JSONHandler'>, 'csv': <class 'CSVHandler'>}
>>> HANDLERS["json"]().parse('{"x": 1}')
{'x': 1}

📝 關於輸出：在 REPL 或模組裡實際執行時，類別的 repr 會帶模組前綴，例如 <class '__main__.JSONHandler'>。為求簡潔，本章（以及後面 metaclass 的 RegistryMeta.registry、Plugin.registry 等輸出）一律省略 __main__. 這類模組前綴；逐字執行時看到前綴是正常的。

在 Java 你會用 @Component 加上 classpath scanning，或手動 map.put("json", new JSONHandler()) 在某個設定類別裡。Python 的類別裝飾器讓「定義即註冊」，登記的程式碼就寫在類別旁邊，一眼可見。

類別裝飾器也能增強類別——加方法、加屬性、改寫 __init__。而你其實天天在用一個重量級的類別裝飾器：

python 
from dataclasses import dataclass

@dataclass            # @dataclass 就是個類別裝飾器！
class Point:
    x: int
    y: int

@dataclass（第 7 章）收到 Point，讀取它的型別標注，動態生成 __init__、__repr__、__eq__ 等方法塞回類別裡，再回傳。它不是什麼編譯器魔法，就是一個你也寫得出來的類別裝飾器。

🔁 Java 對照：@dataclass ≈ Lombok 的 @Data，但機制相反。Lombok 是編譯期 annotation processor，在編譯階段偷偷生成位元組碼；@dataclass 是執行期的普通函式，在類別定義被執行時生成方法。Lombok 需要 IDE 外掛才看得到生成的方法，@dataclass 生成的方法執行期就真實存在，inspect.getsource 之外的工具一律看得到。

`__getattr__` / `__setattr__`：攔截屬性存取

元程式設計的下一層是攔截「屬性存取」本身。第 8 章用 __getattr__ 做過自動委派，這裡把它跟兄弟方法講清楚——它們是 Python 動態屬性的核心。

`__getattr__` vs `__getattribute__`

兩個名字很像，行為差很多：

  • __getattribute__：每一次屬性存取都呼叫它，無論屬性存不存在。它是屬性查找的總入口，預設實作負責去 __dict__、類別、descriptor 裡找。極少需要覆寫，覆寫錯了整個物件就壞了。
  • __getattr__：只有在正常查找失敗時才呼叫（找不到屬性、即將 AttributeError 之前）。這才是你 99% 想用的那個——做委派、惰性屬性、動態欄位。
python 
class Dynamic:
    def __init__(self) -> None:
        self.real = 42

    def __getattr__(self, name: str) -> str:
        # 只有 self.real 以外、找不到的屬性才會進來
        # 注意：這裡標 -> str 只是因為本範例剛好都回字串；
        # 真實的 __getattr__ 回傳型別取決於用途（委派、惰性屬性常是任意型別），
        # 簽章並非固定要回 str。
        return f"<動態產生的 {name}>"
python 
>>> d = Dynamic()
>>> d.real           # 真實存在，不會呼叫 __getattr__
42
>>> d.anything       # 找不到 → 觸發 __getattr__
'<動態產生的 anything>'
>>> d.foo_bar
'<動態產生的 foo_bar>'

💡 Pythonic 重點：要做「找不到時的後備（fallback）」用 __getattr__；要攔截「全部」屬性存取（罕見，例如 proxy）才用 __getattribute__。把 __getattr__ 理解成「屬性查找的 catch」就對了——只有 miss 才進來。

`__setattr__`：攔截所有指派，小心無窮遞迴

__setattr__ 攔截每一次 obj.x = value。拿來做唯讀物件或欄位驗證很方便，但藏著一個 Java 工程師必踩的陷阱：

python 
class ReadOnly:
    def __init__(self, **kwargs: object) -> None:
        # 不能直接寫 self.x = ...，否則又會觸發 __setattr__ → 無窮遞迴
        for key, value in kwargs.items():
            object.__setattr__(self, key, value)   # 繞過自己的 __setattr__

    def __setattr__(self, name: str, value: object) -> None:
        raise AttributeError(f"{type(self).__name__} 是唯讀的，不能設定 {name!r}")
python 
>>> p = ReadOnly(x=1, y=2)
>>> p.x
1
>>> p.x = 99
Traceback (most recent call last):
    ...
AttributeError: ReadOnly 是唯讀的，不能設定 'x'

關鍵在 __init__ 裡：不能寫 self.key = value，因為那會再次觸發 __setattr__，而我們的 __setattr__ 又要設值……要嘛無窮遞迴、要嘛（像這裡）直接拋例外。解法是繞過去——用 object.__setattr__(self, key, value)（或 super().__setattr__(...)），或直接操作 self.__dict__[key] = value。

下面是更實用的「驗證型」版本，攔截指派並檢查：

python 
class Account:
    def __setattr__(self, name: str, value: object) -> None:
        if name == "balance":
            if not isinstance(value, (int, float)):
                raise TypeError("balance 必須是數字")
            if value < 0:
                raise ValueError("balance 不能為負")
        super().__setattr__(name, value)        # 驗證通過，交給預設行為真的存進去
python 
>>> a = Account()
>>> a.balance = 100
>>> a.balance = -5
Traceback (most recent call last):
    ...
ValueError: balance 不能為負

⚠️ 陷阱：在 __setattr__ 內部設定實例屬性，一定要走 super().__setattr__() 或 self.__dict__，絕不能寫 self.name = value，否則無窮遞迴。同理 __getattribute__ 內部取屬性要用 super().__getattribute__()。這是覆寫這些 dunder 的鐵律。

不過，當你發現自己在 __setattr__ 裡用一堆 if name == ... 分流驗證時，那是個訊號：你真正想要的是 descriptor。

Descriptor 協定：`@property` 的底層

Descriptor（描述器）是 Python 物件模型裡最被低估、卻最強大的機制。一句話：descriptor 是一個「把屬性存取的控制權，從物件交給另一個物件」的協定。@property、classmethod、staticmethod、甚至「一般方法為什麼會自動綁定 self」——全都是 descriptor 在背後撐著。

一個物件只要實作了以下任一方法，它就是 descriptor：

  • __get__(self, obj, objtype)：讀取時呼叫
  • __set__(self, obj, value)：指派時呼叫
  • __set_name__(self, owner, name)：類別定義時，自動告訴 descriptor「它被綁到哪個屬性名」

把「非負整數欄位」這個常見驗證需求寫成 descriptor：

python 
class NonNegative:
    def __set_name__(self, owner: type, name: str) -> None:
        # 類別建立時自動呼叫，name 是這個 descriptor 被指派到的屬性名
        self._name = f"_{name}"

    def __get__(self, obj: object, objtype: type | None = None) -> int:
        if obj is None:                         # 從類別（而非實例）存取時
            return self                          # type: ignore[return-value]
        return obj.__dict__.get(self._name, 0)

    def __set__(self, obj: object, value: int) -> None:
        if not isinstance(value, int):
            raise TypeError(f"{self._name[1:]} 必須是 int")
        if value < 0:
            raise ValueError(f"{self._name[1:]} 不能為負，收到 {value}")
        obj.__dict__[self._name] = value         # 存進實例的 __dict__


class Order:
    quantity = NonNegative()        # 一個 descriptor 實例，當成類別屬性
    price = NonNegative()           # 可以重複用，零樣板

    def __init__(self, quantity: int, price: int) -> None:
        self.quantity = quantity    # 觸發 NonNegative.__set__
        self.price = price
python 
>>> o = Order(quantity=3, price=100)
>>> o.quantity           # 觸發 NonNegative.__get__
3
>>> o.quantity = 5       # 通過驗證
>>> o.quantity = -1      # 觸發 __set__ 的驗證
Traceback (most recent call last):
    ...
ValueError: quantity 不能為負，收到 -1

注意這套寫法的威力：驗證邏輯寫一次，quantity 和 price 兩個欄位重複用。對比前面 __setattr__ 裡 if name == ... 的分流，descriptor 把「每個欄位的存取控制」抽成了可重用元件——這正是 @property 做不到的（每個 property 要寫一組 getter/setter）。__set_name__（Python 3.6+）讓 descriptor 自動知道自己叫什麼名字，省掉手動傳名字的樣板。

⚠️ 陷阱：上面 NonNegative.__set__ 用 isinstance(value, int) 驗型別，前面 Account.__setattr__ 用 isinstance(value, (int, float))，兩處都有同一個盲點——Python 的 bool 是 int 的子類別，所以 isinstance(True, int) 為 True。實測 Order(quantity=True, price=100) 會把 True 當成合法數量收下、a.balance = True 也會通過（True 等於 1、False 等於 0）。這正是 Java 沒有的陷阱：Java 的 Boolean 不是 Integer 的子型別，兩者毫無繼承關係，boolean 也不會默默當成數字。若要嚴格只收「真正的整數」，得額外排除 bool，例如先擋掉 isinstance(value, bool)，或改用 type(value) is int 做精確型別比對。

python 
>>> isinstance(True, int)        # bool 是 int 的子類別！
True
>>> type(True) is int            # 但精確型別比對就擋得住
False

Data descriptor vs non-data descriptor

這裡有個影響屬性查找優先序的關鍵區別：

  • Data descriptor（資料描述器）：同時有 __set__ 或 __delete__。它的優先序高於實例 __dict__。
  • Non-data descriptor（非資料描述器）：只有 __get__。優先序低於實例 __dict__。

屬性查找（讀取 obj.x）的優先順序是：

  1. 型別的 data descriptor（最高）
  2. 實例的 __dict__
  3. 型別的 non-data descriptor（如一般方法、@cached_property）
  4. 觸發 __getattr__（最低，前面全 miss 才進來）

這解釋了為什麼 @property（是 data descriptor，有 __set__）總是攔得到存取，而一般方法（non-data descriptor，只有 __get__）會被同名的實例屬性蓋過。也解釋了 functools.cached_property 的把戲：它是 non-data descriptor，第一次 __get__ 算完就把結果寫進實例 __dict__，之後因為實例 __dict__ 優先序較高，就再也不會呼叫 descriptor——天然的快取。

@property 本身就是一個內建的 data descriptor 工廠。下面兩段完全等價：

python 
class Celsius:
    def __init__(self, temp: float) -> None:
        self._temp = temp

    @property                      # property 物件就是一個 data descriptor
    def temp(self) -> float:
        return self._temp

    @temp.setter
    def temp(self, value: float) -> None:
        if value < -273.15:
            raise ValueError("低於絕對零度")
        self._temp = value

💡 Pythonic 重點：單一欄位的計算 / 驗證屬性用 @property（簡單、直觀）；同一套規則要套到多個欄位（非負、字串長度上限、型別檢查……）就自己寫 descriptor，把驗證抽成可重用元件。看懂 descriptor，@property、classmethod、method binding 對你就不再是黑盒子。

🔁 Java 對照：Java 最接近的東西是「private 欄位 + getter/setter + 反射」，但沒有真正的對等物——Java 沒有「攔截某個欄位存取並轉交給另一個物件」的語言級協定。Kotlin 的 delegated property（by）在概念上最接近 descriptor。

Metaclass 簡介：類別也是物件

最後一層，也是最深、實務上最少用的一層。先接受一個事實：在 Python 裡，類別本身也是物件。既然是物件，它一定有自己的型別——那個型別就是 metaclass（元類別），預設是 type。

python 
>>> class Foo: ...
>>> type(Foo)            # Foo 這個「類別」的型別是 type
<class 'type'>
>>> type(42)             # 42 的型別是 int
<class 'int'>
>>> type(int)            # int 的型別也是 type
<class 'type'>

class Foo: ... 這個語法，骨子裡是呼叫 type 來建立一個類別物件。你甚至能跳過 class 語法、直接用 type(name, bases, namespace) 動態造類別：

python 
>>> Dog = type("Dog", (), {"speak": lambda self: "Woof"})
>>> Dog().speak()
'Woof'

Dog = type(...) 跟 class Dog: ... 建出來的東西一模一樣。metaclass 就是「製造類別的工廠」——你能自訂一個 metaclass，在類別被建立的瞬間介入（檢查、修改、註冊）。最小範例：建立子類別時自動註冊：

python 
class RegistryMeta(type):
    registry: dict[str, type] = {}

    def __new__(mcs, name: str, bases: tuple[type, ...], namespace: dict[str, object]):
        cls = super().__new__(mcs, name, bases, namespace)
        if bases:                          # 跳過基底類別自己，只註冊子類別
            RegistryMeta.registry[name] = cls
        return cls


class Plugin(metaclass=RegistryMeta):
    pass

class AudioPlugin(Plugin):
    pass

class VideoPlugin(Plugin):
    pass
python 
>>> RegistryMeta.registry
{'AudioPlugin': <class 'AudioPlugin'>, 'VideoPlugin': <class 'VideoPlugin'>}

每當有人 class XxxPlugin(Plugin)，metaclass 的 __new__ 就被呼叫、自動把新類別登記進 registry——子類別作者完全不必寫任何註冊程式碼。

但你幾乎永遠不該用它

metaclass 是 Python 最重的元程式設計武器，99% 的情境用不到。理由很實際：它難讀、難除錯、會跟多重繼承的 metaclass 衝突，而且絕大多數需求有更輕的替代品：

python 
# __init_subclass__：不寫 metaclass 就能在「子類別被建立時」介入（Python 3.6+）
class Plugin:
    registry: dict[str, type] = {}

    def __init_subclass__(cls, **kwargs: object) -> None:
        super().__init_subclass__(**kwargs)
        Plugin.registry[cls.__name__] = cls    # 同樣的自動註冊，但簡單太多

class AudioPlugin(Plugin):
    pass
python 
>>> Plugin.registry
{'AudioPlugin': <class 'AudioPlugin'>}

這個 __init_subclass__ 版本做到跟 metaclass 一樣的自動註冊，卻不需要任何 metaclass 知識。判準很簡單：

  • 要在類別建立後增強它 → 用類別裝飾器（如 @dataclass）。
  • 要在子類別被建立時介入（註冊、驗證子類別有沒有實作某方法）→ 用 __init_subclass__。
  • 上面兩個都解決不了，才考慮 metaclass——這種情境通常出現在你寫框架（ORM、序列化、API 框架）的時候。

🔁 Java 對照：metaclass + __new__ 在威力上接近「Java 反射 + 動態建立類別」，或 Lombok / annotation processor 那種「在類別生成階段改寫類別」的能力。但 Python 是純執行期、無編譯步驟。SQLAlchemy、Django ORM 的「定義欄位即定義資料表」就是 metaclass 的經典戰場——但那是框架作者的事，應用程式碼幾乎碰不到。

💡 Pythonic 重點：metaclass 要「知道它存在、看得懂別人的框架程式碼」，但自己動手前先問三次「__init_subclass__ 或類別裝飾器能不能解決？」。能，就別碰 metaclass。

本章小結

  1. 裝飾器就是「吃函式、回傳函式」的高階函式，@deco 完全等價於 f = deco(f)，只是語法糖。它在 def 執行的當下就套用，不是呼叫時。
  2. 裝飾器 vs Java annotation：目的相同（橫切關注點），機制相反——annotation 是被動中繼資料、要靠框架反射/織入；decorator 是主動函式、純語言特性、無需框架。
  3. 永遠加 functools.wraps：否則 wrapper 會抹掉原函式的 __name__、__doc__ 等身分，搞壞日誌與工具。@wraps 還留下 __wrapped__ 逃生門。
  4. 帶參數的裝飾器是三層巢狀：最外層吃裝飾器參數、中間層吃函式、最內層吃呼叫引數。@retry() 一定要加括號。
  5. 類別裝飾器收到類別、回傳（增強後的）類別，常用於自動註冊。@dataclass 本身就是類別裝飾器。
  6. __getattr__ 只在查找失敗時呼叫（做 fallback/委派），__getattribute__ 每次都攔截（罕用）。__setattr__ 攔截所有指派，內部設值務必走 super().__setattr__ 或 self.__dict__，否則無窮遞迴。
  7. descriptor（__get__/__set__/__set_name__）是 @property、classmethod、method binding 的底層。data descriptor（有 __set__）優先序高於實例 __dict__，non-data（只有 __get__）較低。要把驗證套到多個欄位就自己寫 descriptor。
  8. metaclass 是「類別的類別」（type），能在類別建立時介入，但 99% 用不到——多半 __init_subclass__ 或類別裝飾器就能取代，知道存在、看得懂框架碼即可。

練習

練習 1（拆語法糖） 不要執行，先寫出下面這段的「去語法糖」等價版本（用 = 賦值，不用 @），再執行驗證你的理解。說明 count_calls 是在「定義 hello 時」還是「第一次呼叫 hello 時」執行的。延伸思考：去語法糖後 hello 其實是 wrapper，而計數器 func.calls 掛在**原始 func（被包起來那個）**身上，不在 wrapper 上——試著存取 hello.calls，觀察為何會丟 AttributeError、計數器到底掛在哪個物件上。想讓 hello.calls 取得到，計數器該改掛哪裡（提示：改成 wrapper.calls，並補上 @wraps）？

python 
def count_calls(func):
    func.calls = 0
    def wrapper(*args, **kwargs):
        func.calls += 1
        return func(*args, **kwargs)
    return wrapper

@count_calls
def hello(): return "hi"

練習 2（修好遺失的身分） 拿掉某個你寫的裝飾器裡的 @wraps，用 your_func.__name__ 和 help(your_func) 觀察壞掉的樣子，再加回 @wraps 對比。額外題：用 your_func.__wrapped__ 拿回原函式並直接呼叫，繞過裝飾器邏輯。

練習 3（帶參數的裝飾器） 寫一個 @cache_for(seconds=60)：在指定秒數內，相同引數的呼叫直接回傳上次的結果（簡易 TTL 快取）。務必用三層巢狀結構，並加 @wraps。提示：用 time.monotonic() 記時間戳，用 dict 以引數為 key 存 (timestamp, result)。寫一個會 print 的慢函式驗證第二次呼叫沒有真的執行。

練習 4（descriptor 取代 property 樣板） 你有一個 Person，name 不能是空字串、age 必須是 0–150 的 int。先用兩組 @property 寫一版，感受重複的樣板；再寫一個帶參數的 descriptor（如 Validated(validator=...) 或 BoundedInt(low, high)），讓 Person 用宣告的方式套用驗證。比較兩版的程式碼量與可重用性。

練習 5（metaclass 的輕量替代） 用 __init_subclass__ 寫一個 Handler 基底類別：任何子類別若沒有定義 handle 方法，就在子類別被建立的當下（不是被呼叫時）拋出 TypeError。寫一個正確的子類別與一個忘了實作 handle 的子類別來驗證。說明為什麼這裡用 __init_subclass__ 比 metaclass 好，以及它跟 @abstractmethod（第 8 章，實例化時才報錯）在「何時報錯」上的差別。
