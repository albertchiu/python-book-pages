第 11 章：模組、套件與專案結構

在 Java，編譯期就把「我有哪些類別、它們屬於哪個 package」釘死了：package com.acme.billing; 寫在檔頭，class 名稱要跟檔名一致，classpath 在啟動時固定，JVM 載入類別是惰性、嚴格、無副作用的。Python 把這一整套搬到了執行期：import 是一條會跑的敘述，模組第一次被載入時會「從上到下執行整個檔案」，而且只執行一次、之後快取在記憶體裡。這個差異會顛覆你對「載入」的直覺——它解釋了循環匯入為何會炸、為何頂層別放重活、為何相對匯入在腳本裡會失效。本章用一個會貫穿全章的最小套件範例，把 Python 的模組系統與現代專案佈局講透。

module 與 package：執行期的命名空間

先把詞彙對齊。在 Python：

  • module（模組）：一個 .py 檔。檔名（去掉 .py）就是模組名。
  • package（套件）：一個「裝著模組的目錄」。傳統上目錄裡要有 __init__.py 才算 package（沒有的話是「namespace package」，後面會談）。
  • import：把另一個模組載入、執行、並在當前命名空間綁定一個名字。
Java Python 思維差異
package com.acme.billing;（檔頭宣告） 目錄結構就是套件名，不在檔案裡宣告 Python 的套件名來自它在檔案系統的相對位置
import com.acme.billing.Invoice; from acme.billing import invoice Java import 是編譯期符號解析；Python import 是執行期、會跑程式碼
classpath（JAR/目錄清單） sys.path（目錄/zip 清單） 兩者都是「去哪裡找」，但 sys.path 可在執行期被改
JVM 惰性載入、無副作用 模組第一次 import 時整檔執行 這是最大的心智轉換

🔁 Java 對照：Java 的 import 只是「打字省略」，不會載入任何東西，真正載入在第一次用到該類別時由 ClassLoader 完成。Python 的 import x 是一條敘述：找到 x、執行 x 的全部頂層程式碼、把模組物件綁到名字 x。

本章的範例套件

整章用這個最小但可運作的套件。先把目錄樹放在這裡，後面所有範例都對應它：

text 
billing-demo/
├── pyproject.toml
├── src/
│   └── billing/
│       ├── __init__.py
│       ├── money.py
│       ├── invoice.py
│       ├── tax.py
│       └── __main__.py
└── tests/
    └── test_invoice.py

money.py 是最底層、不依賴套件內其他模組的工具：

python 
# src/billing/money.py
from decimal import Decimal

def to_cents(amount: Decimal) -> int:
    """把金額（元）轉成分，避免浮點誤差。"""
    # to_integral_value() 預設採目前 Decimal context 的 ROUND_HALF_EVEN（銀行家捨入）；
    # 若計費需特定規則，應顯式指定，例如 .to_integral_value(rounding=ROUND_HALF_UP)。
    return int((amount * 100).to_integral_value())

def format_money(cents: int) -> str:
    return f"${cents / 100:.2f}"

import 機制：sys.path、sys.modules 與只載入一次

import billing.money 背後發生三件事：

  1. 尋找（find）：直譯器依序問每個「finder」：我能定位 billing 嗎？最常見的 finder 走訪 sys.path 裡的每個目錄找 billing/。
  2. 載入（load）：找到後，建立一個 module 物件，從上到下執行該檔的所有頂層程式碼，把產生的名字放進 module 的命名空間。
  3. 綁定（bind）：在你目前的命名空間綁定名字。import billing.money 綁的是 billing；from billing import money 綁的是 money。

sys.path 就是 Python 的 classpath。它從哪裡來？

python 
>>> import sys
>>> sys.path[:3]
['', '/usr/lib/python3.14', '/usr/lib/python3.14/lib-dynload']

開頭那個 sys.path[0] 很關鍵，但它的字面值會因啟動方式而不同：在 REPL、python -c、python -m 時它是空字串 ''（代表「目前工作目錄」，就是上面 REPL 範例看到的）；而你直接 python foo.py 時，它是 foo.py 所在目錄的絕對路徑（例如 /private/tmp），不是 ''。兩者語意相近——都讓你能 import 同目錄下的檔案——但字面不同，別誤以為 '' 跟「腳本目錄」是同一回事。重點是：你直接 python foo.py 時，foo.py 所在的目錄會被加進 sys.path 最前面。這是很多「在我電腦上跑得起來、裝起來卻 import 不到」問題的根源，後面 src layout 會回頭解決它。

模組只載入一次：sys.modules 快取

第一次 import 一個模組，Python 執行它的頂層程式碼，然後把 module 物件存進 sys.modules 這個 dict。之後所有 import 同一模組，都直接從 sys.modules 拿快取，不再重跑檔案。

python 
# src/billing/tax.py（以下頂層 print 純為示範 sys.modules 快取，正式範例請移除）
print("tax.py 正在被載入")          # ← 故意的副作用，看完本節就刪掉

TAX_RATE = 0.05

def with_tax(cents: int) -> int:
    return round(cents * (1 + TAX_RATE))

注意這行頂層 print 正是 18 行前批評的「頂層放動作」陷阱——它只是用來觀察快取行為。看完這一節就把它刪掉，否則之後凡是 import billing（會經 __init__.py→invoice→tax）都會先印出這行。本章後續所有執行框（python -m billing、uv run billing 等）都假設這行 debug print 已移除。

python 
>>> import billing.tax
tax.py 正在被載入
>>> import billing.tax            # 第二次：什麼都不印，因為走快取
>>> import sys
>>> 'billing.tax' in sys.modules
True
>>> sys.modules['billing.tax'].TAX_RATE
0.05

這帶來兩個資深工程師必須內化的後果：

⚠️ 陷阱：import 有副作用，而且副作用只發生一次。 如果你在模組頂層放了「開資料庫連線」「讀環境變數」「註冊全域 handler」這類重活，它會在第一次 import 時就跑（可能是在你意想不到的時機，例如測試蒐集階段），而且你無法靠「再 import 一次」重來。頂層只放「定義」（函式、類別、常數），把「動作」收進函式或 if __name__ == "__main__"。

💡 Pythonic 重點：sys.modules 是單例（singleton）模式的天然實作。你想要「整個程序共用一份設定物件」，最簡單的做法就是把它放在一個模組頂層——模組本身就是進程級單例，import 它就拿到同一份。比 Java 那套 private static INSTANCE + double-checked locking 乾淨太多，而且執行緒安全由 import 鎖保證。

`import x` vs `from x import y`：綁定的是不同名字

這兩種寫法載入的是同一個模組物件（sys.modules 只有一份），差別只在「你的命名空間裡多了什麼名字」：

python 
import billing.money               # 命名空間多了 billing；用 billing.money.to_cents(...)
from billing import money          # 命名空間多了 money；用 money.to_cents(...)
from billing.money import to_cents # 命名空間多了 to_cents；直接用 to_cents(...)
import billing.money as m          # 別名；用 m.to_cents(...)

關鍵理解：from billing.money import to_cents 之後，to_cents 是一個指向函式物件的新名字，跟 billing.money.to_cents 是兩個獨立綁定，只是現在指向同一個物件。這在 monkeypatch（測試替身）時是個經典陷阱：

⚠️ 陷阱：from os import getpid 之後，你在測試裡 patch os.getpid 不會影響已經 from-import 進來的那個 getpid 名字——它早在 import 當下就綁死了原本的函式物件。要 patch 得對，請 patch「使用端模組裡的那個名字」（例如 mymodule.getpid），或一開始就用 import os + os.getpid()。這跟第 3 章「名字只是綁定」一脈相承。

避免用 from module import *：它把模組所有公開名字倒進當前命名空間，污染嚴重又讓人看不出名字從哪來。唯一能稍微控制它的是 __all__（下一節）。

`__init__.py`、`__all__` 與相對 vs 絕對匯入

`__init__.py` 的角色

__init__.py 是「這個目錄被 import 為 package 時會執行的檔案」。它做三件事：

  1. 標記這個目錄是個傳統 package。
  2. 它的頂層程式碼在「package 第一次被 import」時執行——import billing 跑的就是 billing/__init__.py。
  3. 它定義 package 的門面（facade）：把深層模組的東西「拉上來」變成 package 級公開 API。
python 
# src/billing/__init__.py
"""billing：一個示範用的計費套件。"""
from billing.money import format_money, to_cents
from billing.invoice import Invoice

__all__ = ["Invoice", "format_money", "to_cents"]
__version__ = "0.1.0"

這樣使用者能寫 from billing import Invoice，不必知道它其實住在 billing/invoice.py。你可以隨意重構內部模組結構，只要 __init__.py 的門面不變，下游就不受影響——這跟 Java 用 package-info、或刻意只 public 暴露門面類別、其餘設 package-private 是同一個封裝思維。

__all__ 是一個字串清單，控制兩件事：

  • from billing import * 只會匯入 __all__ 列出的名字。
  • 它是給讀者與工具看的「公開 API 宣告」（很多 linter、文件工具拿它當公開介面的依據）。

💡 Pythonic 重點：Python 沒有 public/private 關鍵字，封裝靠慣例：底線開頭（_helper、_InternalThing）表示「內部用，別碰」，工具與同事都會尊重。__all__ 則是 package 層級的「這些是我保證的公開 API」。把這兩者當成你的 access modifier。

⚠️ 陷阱：別把 __init__.py 寫成重活。因為 import billing.anything 都會先觸發 billing/__init__.py 執行，如果你在門面裡 import 了一堆昂貴的子模組，使用者「只想用一個小工具」也得付全部 import 成本。大型套件常刻意讓 __init__.py 保持輕薄，或用延遲匯入（__getattr__ on module，PEP 562）按需載入。

絕對匯入 vs 相對匯入

套件內部模組互相引用，有兩種寫法：

python 
# src/billing/invoice.py

# 絕對匯入（absolute import）：從套件根算起的完整路徑
from billing.money import to_cents, format_money
from billing.tax import with_tax

# 相對匯入（relative import）：用點號表示「相對於我所在的套件」
from .money import to_cents, format_money     # . = 我所在的 package（billing）
from .tax import with_tax
# from ..otherpkg import thing                # .. = 上一層 package

. 代表「目前模組所在的 package」，.. 代表上一層，以此類推。兩者載入的是同一個模組，差別在可讀性與可移植性的取捨：

絕對匯入 相對匯入
寫法 from billing.money import to_cents from .money import to_cents
改套件名時 要全部改 不用改（相對的，跟著走）
可讀性 一眼看出來源 簡潔，但要知道「我在哪」
官方建議 PEP 8 偏好（清楚） 套件內、深層巢狀時方便

把 invoice.py 補完（這裡用絕對匯入，與門面一致）：

python 
# src/billing/invoice.py
from dataclasses import dataclass, field
from decimal import Decimal

from billing.money import to_cents, format_money
from billing.tax import with_tax

@dataclass
class Invoice:
    customer: str
    _lines: list[int] = field(default_factory=list)   # 各行金額（分）

    def add_line(self, amount: Decimal) -> None:
        self._lines.append(to_cents(amount))

    def total_cents(self, *, taxed: bool = True) -> int:
        subtotal = sum(self._lines)
        return with_tax(subtotal) if taxed else subtotal

    def __str__(self) -> str:
        return f"Invoice({self.customer}, {format_money(self.total_cents())})"

為什麼相對匯入不能用在「被當腳本直接執行」的檔案

這是 Java 工程師一定會踩、而且訊息很難懂的坑。相對匯入靠的是模組的 __package__／__name__ 來判斷「我相對於誰」。當你 python src/billing/invoice.py 直接執行它：

text 
$ python src/billing/invoice.py
ImportError: attempted relative import with no known parent package

原因：被直接執行的檔案，它的 __name__ 會變成 "__main__"，__package__ 是空的——直譯器根本不知道它屬於哪個 package，於是 .money 的「.」無從解析。相對匯入只在「這個檔案是被當作某個 package 的一員 import 進來」時才成立。

正確的執行方式是用 -m，讓 Python 把它當套件成員載入（下一節詳述）：

text 
$ python -m billing            # 把 billing 當套件執行，相對匯入正常運作

⚠️ 陷阱：看到 attempted relative import with no known parent package，幾乎都是「你直接 python 某個套件內檔案.py 了」。解法不是把相對改絕對硬上，而是改用 python -m pkg.module 執行，或在開發階段把套件 pip install -e .（editable install）讓它真的在 sys.path 上。

`if __name__ == "__main__"`：可執行又可被 import

Java 的進入點是 public static void main(String[] args)，由 JVM 在啟動指定的類別上呼叫。Python 沒有「特殊的 main 方法」，它用一個慣用判斷式：

python 
def main() -> None:
    inv = Invoice("ACME")
    inv.add_line(Decimal("19.99"))
    inv.add_line(Decimal("5.00"))
    print(inv)

if __name__ == "__main__":
    main()

機制很單純：每個模組都有一個 __name__ 變數。

  • 當這個檔案是被直接執行的（python foo.py 或 python -m foo），__name__ 是 "__main__"。
  • 當它是被 import 的，__name__ 是它的模組名（例如 "billing.invoice"）。
python 
>>> import billing.invoice
>>> billing.invoice.__name__
'billing.invoice'

所以 if __name__ == "__main__": 的意思是「只有當我是被當主程式跑時，才執行這段」。這讓同一個檔案既能被 import（當函式庫）、又能被直接執行（當腳本），而 import 它的人不會意外觸發那段示範/啟動程式碼。

🔁 Java 對照：if __name__ == "__main__" ≈ public static void main，但更靈活——任何模組都能有自己的「main」，方便邊寫邊測。對照 Java 你得另開一個 class 或寫 JUnit 才能跑一小段。

`python -m` 與 `__main__.py`

python -m billing.invoice 跟 python src/billing/invoice.py 看似一樣，其實天差地別：

  • python src/billing/invoice.py：把那個檔案當腳本跑，所在目錄塞進 sys.path，__package__ 為空，相對匯入會炸。
  • python -m billing.invoice：把 billing.invoice 當套件成員載入並執行，__package__ 正確設好，相對匯入正常，且當前目錄（含 src 在路徑上時）的套件解析一致。

-m 是「以模組方式執行」的正規做法，等同 Java 的 java -cp ... com.acme.Main——你給的是邏輯名稱，不是檔案路徑。

若想讓整個套件可被執行（python -m billing），就放一個 __main__.py 在套件目錄：

python 
# src/billing/__main__.py
from decimal import Decimal
from billing.invoice import Invoice

def main() -> None:
    inv = Invoice("ACME")
    inv.add_line(Decimal("19.99"))
    inv.add_line(Decimal("5.00"))
    print(inv)

if __name__ == "__main__":
    main()
text 
$ python -m billing
Invoice(ACME, $26.24)

pip、venv、json.tool、http.server 都是這樣設計的（你應該打過 python -m http.server）。這是讓套件「可像 CLI 一樣被叫起來」的標準入口。稍後 pyproject.toml 的 [project.scripts] 會給它一個更漂亮的指令名。

標準專案結構：src layout

你已經用過 Maven/Gradle 的 src/main/java、src/test/java。Python 社群現在的共識佈局是 src layout，動機與 Maven 把原始碼跟測試分目錄一樣，但解決的是 Python 特有的一個陰險問題。

text 
billing-demo/
├── pyproject.toml
├── src/
│   └── billing/
│       ├── __init__.py
│       ├── money.py
│       ├── invoice.py
│       ├── tax.py
│       └── __main__.py
└── tests/
    └── test_invoice.py

為什麼 src layout 優於 flat layout

「flat layout」是把 billing/ 直接放在專案根目錄（跟 pyproject.toml、tests/ 同層）。問題出在前面講過的：當前工作目錄會被加進 sys.path。於是你在專案根跑測試時，import billing 抓到的是「目錄裡的原始碼」，而不是「你已經 pip install 進 venv 的那一份套件」。

這聽起來不痛不癢，但它會讓你測到一份根本沒被正確打包的程式碼：假設某個檔案沒被列進打包設定，flat layout 下測試照樣 import 得到（因為它直接讀原始碼目錄），於是測試全綠——使用者裝起來卻 ModuleNotFoundError。

src layout 把 billing/ 藏進 src/。src/ 本身不在 sys.path 上，所以唯一能 import 到 billing 的方式，就是先把它（用 editable install）裝進環境。這強迫你的測試跑在「跟使用者一樣的安裝路徑」上，打包遺漏會立刻在測試階段爆出來。

💡 Pythonic 重點：src layout 的精神是「測你會發佈的東西，而不是測你硬碟上的原始碼目錄」。它把「能 import = 已正確安裝」變成不變量，等同 Maven 強制 mvn package 後測 artifact 的精神。新專案一律用 src layout，不要用 flat。

editable install：邊改邊跑

開發時你不想每改一行就重裝。**editable install（可編輯安裝）**把套件以「指向原始碼」的方式裝進 venv：import 時走的是你的 src/billing/，改檔即時生效，但 sys.path 解析又是「已安裝套件」的正規路徑。

text 
$ uv venv                       # 建立虛擬環境（詳見第 1 章）
$ uv pip install -e .           # editable install：把當前專案以可編輯方式裝入

或用 uv 的專案工作流，一個指令把環境、相依、editable install 一次搞定：

text 
$ uv sync                       # 依 pyproject.toml 建環境、裝相依、editable 裝本專案

裝好後，無論在哪個目錄、用 python -m billing 或在 tests/ 裡 import billing，抓到的都是同一份、且是「已安裝」語意的套件。

🔁 Java 對照：editable install ≈ 把模組以 source 形式掛進 reactor build，而不是每次都 mvn install 到本機 repo。你改 src/ 立刻反映，不必重打包。

`pyproject.toml`：完整解析與發佈到 PyPI

pyproject.toml 是現代 Python 專案的單一設定檔，地位等同 pom.xml／build.gradle，但用 TOML、且涵蓋「打包 + 工具設定」兩件事。完整看一遍：

toml 
# pyproject.toml

[build-system]
# 對照 Maven 的 build plugin：用什麼後端把專案打包成 wheel/sdist
requires = ["hatchling"]
build-backend = "hatchling.build"

[project]
# 對照 Maven 座標：name≈artifactId、version≈version（groupId 在 PyPI 沒有，name 全域唯一）
name = "billing-demo"
version = "0.1.0"
description = "示範用的計費套件"
readme = "README.md"
requires-python = ">=3.12"
authors = [{ name = "Albert Chiu", email = "chiu.albert@gmail.com" }]
license = "MIT"

# 執行期相依（對照 Maven <dependencies> 的 compile scope）
dependencies = [
    "httpx>=0.27",
]

[project.optional-dependencies]
# 對照 Maven optional/extra：使用者用 pip install "billing-demo[dev]" 才會裝
dev = ["pytest>=8", "mypy>=1.10"]

[project.scripts]
# 進入點：裝套件後會在 PATH 產生一個 `billing` 指令

幾個 Java 工程師要對齊的觀念：

  • [build-system] 宣告「用什麼後端打包」（hatchling、setuptools、flit、pdm…），對照你選 maven-compiler-plugin 還是別的。requires 是「打包時」要裝的工具，跟你專案的執行期相依無關。
  • [project] 是 PEP 621 標準化的 metadata。name 在 PyPI 是全域唯一（沒有 Java 那種 groupId 命名空間），所以好名字要先搶。version 沒有 SNAPSHOT 概念，但有 PEP 440 的 1.0.0a1、1.0.0.dev3 等預發佈標記。
  • dependencies 用版本約束字串（httpx>=0.27）。Python 沒有 Maven 那種 compile/test/provided scope，靠 optional-dependencies（extras）區分「核心相依」與「開發/額外相依」。
  • [project.scripts] 是 entry point：裝套件後 billing 變成可在終端機直接打的指令，等同 Maven 的 appassembler 或 Spring Boot 的可執行 jar，但更輕。注意 entry point 只做「import 那個模組、呼叫指定的 callable」兩件事——billing = "billing.__main__:main" 會 import billing.__main__ 並呼叫它的 main()，但不會執行 __main__.py 裡的 if __name__ == "__main__": 區塊（因為被 import 時該模組的 __name__ 是 'billing.__main__'，不是 '__main__'），這正與前面 if __name__ 主題呼應。若覺得拿 __main__.py 當 import 目標不夠乾淨，更慣用的做法是另設 cli.py 放 main，再把 entry point 指成 billing = "billing.cli:main"。

循環匯入：成因與三種解法

兩個模組互相 import，是 Java 工程師最容易低估的雷。在 Java，A 與 B 互相 import 完全沒問題——編譯器一次看到所有符號，循環依賴頂多是「設計味道」。在 Python，import 是執行期、由上而下逐行跑的，循環會讓某個模組「只初始化到一半」就被對方拿去用，於是炸出 ImportError 或 AttributeError。

成因：部分初始化的模組

製造一個循環。invoice.py 想用 report.py 的格式化函式，而 report.py 又想用 invoice.py 的 Invoice：

python 
# src/billing/report.py  —— 這會出錯
from billing.invoice import Invoice          # ← 頂層 import invoice

def render(inv: Invoice) -> str:
    return f"REPORT: {inv}"
python 
# src/billing/invoice.py（加上對 report 的頂層 import）—— 這會出錯
from billing.report import render            # ← 頂層 import report

# ...Invoice 定義...
text 
$ python -m billing
ImportError: cannot import name 'Invoice' from partially initialized module
'billing.invoice' (most likely due to a circular import)

逐步追因：載入 invoice →執行到 from billing.report import render →跳去載入 report →report 頂層執行 from billing.invoice import Invoice →此時 invoice 已在 sys.modules（避免無限遞迴），但它還沒跑到定義 Invoice 那行，命名空間裡還沒有 Invoice →取不到 →ImportError。關鍵字是錯誤訊息裡的 partially initialized module。

⚠️ 陷阱：循環匯入是否報錯，取決於「誰先被 import、卡在哪一行、要的名字此刻定義了沒」。同樣的循環，把頂層 import 換成 import billing.report（綁模組而非綁名字）有時就不炸——因為它不要求對方「此刻已有某個名字」，只要模組物件存在即可，等真的用到 report.render() 時對方早就初始化完了。但別靠這種運氣，循環本身就是該消除的設計問題。

解法一：延後 import 進函式

把 import 從模組頂層移進「真正用到的函式內部」。函式體在「被呼叫時」才執行，那時兩個模組通常都初始化完了：

python 
# src/billing/report.py
def render(inv) -> str:
    from billing.invoice import Invoice      # 延後到呼叫時才 import，循環解除
    assert isinstance(inv, Invoice)
    return f"REPORT: {inv}"

這是最快的止血法，代價是把依賴關係藏進函式、import 成本攤到第一次呼叫。

解法二：抽共用到第三模組（最根本）

循環常代表「兩個模組有一塊共用概念被放錯地方」。把它抽到雙方都依賴、但不反向依賴的底層模組（單向 DAG）：

text 
   money.py   ← 最底層，不依賴套件內任何模組
      ↑
   invoice.py ← 依賴 money、tax
      ↑
   report.py  ← 依賴 invoice（單向）

讓依賴變成單向有向無環圖：report → invoice → money。report 用 Invoice，但 invoice 絕不反過來 import report。這跟你在 Java 重構循環依賴的做法完全一樣——抽介面、調整方向、打破環。

解法三：type-only import 用 `if TYPE_CHECKING`

很多循環其實只是為了「型別提示」而 import，執行期根本不需要那個名字。typing.TYPE_CHECKING 是一個常數：靜態檢查器（mypy/pyright）眼中為 True，執行期永遠為 False。把純型別用的 import 包進去，執行期就不會真的 import，循環消失，型別檢查照常：

python 
# src/billing/report.py
from __future__ import annotations          # 讓註解延後求值，型別寫成字串不會在執行期解析

from typing import TYPE_CHECKING

if TYPE_CHECKING:
    from billing.invoice import Invoice     # 只有 mypy 看得到；執行期不 import

def render(inv: Invoice) -> str:            # 註解此刻是字串，不觸發真正 import
    return f"REPORT: {inv}"

from __future__ import annotations（或本就用 Python 3.12 的字串註解習慣）讓 Invoice 在函式簽章裡只是字串，不會在執行期被求值，因此即使執行期沒 import 它也不報錯，而 mypy 仍能在 TYPE_CHECKING 分支看到真正的 Invoice 做檢查。

💡 Pythonic 重點：if TYPE_CHECKING: 是處理「只為型別而生的循環」的標準解法，也是讓重型相依（只在型別簽章出現的大套件）不被執行期 import 的手段——靜態檢查拿到全部資訊，執行期保持輕巧。這正是第 9 章「型別提示不在執行期強制」的延伸應用。

跑起來：測試與執行整個範例

把測試補上，用 uv run 在隔離環境執行（套件已 editable 裝入）：

python 
# tests/test_invoice.py
from decimal import Decimal
from billing.invoice import Invoice          # 因為 editable install，import 抓到的是已安裝套件

def test_total_with_tax() -> None:
    inv = Invoice("ACME")
    inv.add_line(Decimal("19.99"))
    inv.add_line(Decimal("5.00"))
    # 小計 2499 分；含 5% 稅 → round(2499 * 1.05) = 2624
    assert inv.total_cents() == 2624

def test_total_without_tax() -> None:
    inv = Invoice("ACME")
    inv.add_line(Decimal("10.00"))
    assert inv.total_cents(taxed=False) == 1000
text 
$ uv sync --extra dev                      # 建環境、裝相依、裝 dev extra（pytest/mypy）、editable 裝 billing-demo
$ uv run pytest                            # 在隔離環境跑測試
$ uv run python -m billing                 # 用模組方式執行套件
Invoice(ACME, $26.24)
$ uv run billing                           # 用 [project.scripts] 的 entry point 指令
Invoice(ACME, $26.24)

⚠️ 陷阱：純 uv sync 不會安裝任何 optional-dependencies（extras）。pytest 住在 [project.optional-dependencies] 的 dev extra 裡，所以要跑測試得加 --extra dev（或 --all-extras），否則 uv run pytest 會直接報 Failed to spawn: pytest。這對 Java 工程師是個重要對照差異：Maven 的 test scope 相依在跑 mvn test 時自動可用，但 Python 的 extra 預設不裝，得明確指定。（若想要「開發相依預設就裝」的語意，可改用 PEP 735 的 [dependency-groups] 定義 dev group——uv sync 預設會裝 dev group。）

不想建立完整專案、只想快速試一個第三方套件，uv run --with 可以一次性帶入相依執行（不污染環境）：

text 
$ uv run --with rich python -c "from rich import print; print('[bold green]hello[/]')"

整條鏈路串起來了：模組是檔案、套件是目錄＋__init__.py、import 在執行期跑且只跑一次、src layout 保證「能 import = 已安裝」、pyproject.toml 描述座標與相依與入口、循環靠單向依賴打破。

本章小結

  • module 是 .py、package 是含 __init__.py 的目錄；import 依 sys.path（≈ classpath）尋找，但 sys.path 可在執行期改，且當前/腳本目錄會被自動加入最前面。
  • import 是會跑的敘述：第一次載入時整檔由上而下執行、結果快取在 sys.modules，之後 import 都走快取、不重跑。頂層只放定義，別放重活——副作用只發生一次且時機難料。
  • import x 與 from x import y 載入同一模組，差別在你命名空間綁了哪個名字；from 綁進來的名字在 import 當下就定死，是 monkeypatch 的經典陷阱。
  • __init__.py 是 package 的門面：用它＋__all__ 暴露穩定的公開 API，底線開頭與 __all__ 是 Python 沒有 public/private 時的封裝慣例。
  • 相對匯入（.module、..pkg）不能用在被直接執行的腳本，因為 __name__ 變 "__main__"、__package__ 為空；改用 python -m pkg.module 執行。
  • if __name__ == "__main__" ≈ Java 的 main，讓同一檔案可 import 可執行；python -m 以模組語意執行、__main__.py 讓整個套件可被 python -m pkg 叫起。
  • src layout 優於 flat layout：把套件藏進 src/ 強迫「唯有 editable install 後才 import 得到」，避免測到未正確打包的原始碼；新專案一律用它。
  • pyproject.toml 是 Python 的 pom：[build-system] 選打包後端、[project] 放座標/相依/scripts、[tool.*] 收工具設定；uv build 產 wheel/sdist、uv publish 推 PyPI。
  • 循環匯入因「import 即時執行頂層」而炸出 partially initialized module；三種解法：延後 import 進函式、抽共用到單向依賴的第三模組（最根本）、純型別 import 用 if TYPE_CHECKING。

練習

練習 1（觀察 sys.modules 快取） 在 tax.py 頂層放一行 print("loading tax")，然後在 REPL 連續 import billing.tax 三次。觀察那行只印一次，並用 sys.modules['billing.tax'] 取出同一個模組物件、確認三次 import ... as t 拿到的 t is sys.modules['billing.tax'] 為 True。提示：模組是進程級單例。

練習 2（修好相對匯入的腳本） 把 invoice.py 的絕對匯入改成相對匯入（from .money import ...），然後分別用 python src/billing/invoice.py 和 python -m billing.invoice 執行，記錄各自的結果與錯誤訊息。解釋為什麼前者報 attempted relative import with no known parent package。

練習 3（製造並修好循環匯入） 依本章 report.py ↔ invoice.py 的例子刻意造一個循環，重現 partially initialized module 錯誤。接著分別用「延後 import 進函式」與「if TYPE_CHECKING」兩種方式修好，並說明各自的取捨（執行期成本 vs 只能用於純型別場景）。提示：用 from __future__ import annotations。

練習 4（src layout 與 entry point） 依本章目錄樹建立 billing-demo，寫好 pyproject.toml（含 [project.scripts] 的 billing 入口），跑 uv sync --extra dev（記得 pytest 在 dev extra，純 uv sync 不會裝它）後依序執行 uv run python -m billing、uv run billing、uv run pytest，確認三者都成功。再故意把 src/billing/__init__.py 刪掉一行門面 import，觀察哪個指令先壞、錯誤訊息是什麼。

練習 5（打包並 dry-run 發佈） 對 billing-demo 跑 uv build，檢視 dist/ 下產出的 .whl 與 .tar.gz，用 unzip -l dist/*.whl 看 wheel 裡實際包了哪些檔案，確認 billing/*.py 都在內——注意 wheel 內的路徑是 billing/__init__.py、billing/money.py…，已被 build backend 去掉 src/ 前綴，因為 src/ 只是開發佈局、不是套件命名空間的一部分，不會進到發佈成品。思考：若 flat layout 下漏掉打包設定，為什麼測試仍會通過、但使用者裝起來會 ModuleNotFoundError？（對照 Maven 測 artifact 而非原始碼目錄的精神。）
