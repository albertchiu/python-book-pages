第 13 章：測試

測試你不陌生——JUnit、Mockito、Hamcrest、覆蓋率報表，這套工程紀律你早就內化了。但 Python 的測試文化會顛覆兩個直覺：第一，主流框架 pytest 用裸 assert，沒有 assertEquals/assertThat 那一大家子斷言方法，卻能給你比 JUnit 更漂亮的失敗訊息；第二，依賴注入不靠註解與容器，而靠一個叫 fixture 的函式機制，外加一個讓「測試案例本身變參數」的 parametrize。這一章把你的 JUnit/Mockito 經驗平移過來，再補上 Python 特有的兩件利器：property-based testing（hypothesis） 與 docstring 即測試（doctest）。

pytest 基礎：對照 JUnit 5

Python 標準庫內建 unittest（PyUnit），它是 JUnit 的直系後裔——class TestX(unittest.TestCase)、self.assertEqual(...)、setUp/tearDown，連 camelCase 都照抄。但 Python 社群的事實標準是第三方的 pytest，因為它把 JUnit 風格的冗長一掃而空。我們直接學 pytest。

測試的長相：函式，不是方法

JUnit 5 裡測試是類別中的方法：

java 
class PricingTest {
    @Test
    void subtotalSumsItems() {
        int result = Pricing.subtotal(items);
        Assertions.assertEquals(18000, result);
    }
}

pytest 的測試是模組層級的普通函式，靠命名慣例被發現——檔名 test_*.py（或 *_test.py）、函式 test_*，不需要任何註解：

python 
# test_pricing.py
from pricing import subtotal

def test_subtotal_sums_items():
    result = subtotal(items)
    assert result == 18000          # 就是 Python 內建的 assert，沒有 assertEquals

沒有 @Test、沒有繼承 TestCase、沒有 public。pytest 掃描檔案、找到 test_ 開頭的函式就當測試跑。你也可以把測試放進 class TestPricing: 裡（純粹為了分組，類別名要 Test 開頭、且不能有 __init__），但多數 Python 專案連類別都省了。

🔁 Java 對照：JUnit 用 @Test 註解標記測試；pytest 用「test_ 命名慣例」收集測試。慣例優於設定，這是貫穿 Python 工具鏈的風格。

為什麼可以用裸 `assert`：assert rewriting

Java 工程師看到 assert result == 18000 會皺眉：JUnit 不鼓勵用 Java 的 assert（它預設被 -ea 關掉，而且失敗訊息只有一句 AssertionError），所以才有 assertEquals(expected, actual) 這套 API 來產生有意義的訊息。

pytest 用了一個巧妙的手法繞過這個限制：assert 重寫（assertion rewriting）。pytest 在 import 測試模組時，會改寫它的 AST（抽象語法樹），把每個 assert 拆解、在失敗時注入兩邊運算元的值。於是裸 assert 也能給出豐富的 diff：

python 
def test_show_rich_message():
    actual = {"a": 1, "b": 2}
    expected = {"a": 1, "b": 3}
    assert actual == expected
text 
>       assert actual == expected
E       AssertionError: assert {'a': 1, 'b': 2} == {'a': 1, 'b': 3}
E         
E         Omitting 1 identical items, use -vv to show
E         Differing items:
E         {'b': 2} != {'b': 3}
E         Use -v to get more diff

它自動算出 dict 的差異、標出不同的 key。對 list、set、字串、dataclass 也都有客製化的 diff 輸出。這就是為什麼 pytest 不需要 assertEquals/assertThat——重寫機制讓最直覺的 assert a == b、assert x in coll、assert s.startswith("...") 全都附帶好讀的失敗訊息。

JUnit 5 pytest 說明
assertEquals(a, b) assert a == b
assertTrue(x) / assertFalse(x) assert x / assert not x
assertNull(x) assert x is None 用 is，不是 ==
assertThrows(E.class, () -> ...) with pytest.raises(E): ... 見後面
assertIterableEquals assert a == b list 比較直接用 ==
assertEquals(0.3, x, 1e-9) assert x == pytest.approx(0.3) 浮點數容差

浮點數那行值得單獨記住——它對應 JUnit 的 delta 參數：

python 
>>> from pytest import approx
>>> 0.1 + 0.2 == 0.3
False
>>> 0.1 + 0.2 == approx(0.3)
True

執行方式

bash 
uv run --with pytest pytest                 # 跑當前目錄下所有測試
uv run --with pytest pytest test_pricing.py # 跑單一檔案
uv run --with pytest pytest -k discount     # 只跑名字含 discount 的測試
uv run --with pytest pytest -q              # 安靜模式；-v 反之更囉嗦
uv run --with pytest pytest -x              # 第一個失敗就停（fail fast）
uv run --with pytest pytest --lf            # 只重跑上次失敗的（last failed）

-k 與 --lf 是日常神器：-k 像 JUnit 的 tag 過濾但更靈活（支援 and/or/not 運算式比對測試名），--lf 讓你修 bug 時只反覆跑那幾條紅燈的測試。

fixture：依賴注入，但更輕

JUnit 5 用 @BeforeEach/@AfterEach 準備與清理、用 extension（如 @ExtendWith）或建構子注入相依。pytest 把這些統一成一個機制：fixture。

基本款：fixture 是「會被注入的函式」

宣告一個 @pytest.fixture，它的回傳值會被注入到任何「參數名稱跟它同名」的測試裡：

python 
import pytest
from pricing import Item, subtotal

@pytest.fixture
def cart() -> list[Item]:
    return [
        Item("apple", price=3000, qty=2),
        Item("book", price=12000, qty=1),
    ]

def test_subtotal(cart: list[Item]) -> None:   # 參數名 cart 對上 fixture cart
    assert subtotal(cart) == 18000

pytest 看到 test_subtotal 有個參數叫 cart，就去找名為 cart 的 fixture、呼叫它、把結果餵進來。這是按名稱的依賴注入——像 Spring 的 @Autowired 加上 by-name 解析，但完全不需要容器或註解掃描，靠的是函式參數名。

🔁 Java 對照：@BeforeEach 建立的物件你存進欄位、測試方法讀欄位；fixture 則是把那個物件當參數注入。fixture 比 @BeforeEach 更模組化——每個測試只「點名」自己要的 fixture，不會被一個共用的 setup 塞一堆用不到的東西。

setup/teardown：yield fixture

要做清理（關連線、刪暫存檔），用 yield 把 fixture 拆成「之前」與「之後」兩半。yield 前是 setup、yield 出去的值被注入、控制權回來後（yield 之後）是 teardown——無論測試成功或失敗都會執行，等同 try/finally：

python 
@pytest.fixture
def conn():
    c = FakeConn()              # setup
    c.log.append("opened")
    yield c                     # 測試在此執行，拿到的就是 c
    c.close()                   # teardown：yield 之後一定會跑

這比 JUnit 把 setup 和 teardown 拆成兩個分開的方法（@BeforeEach/@AfterEach）更內聚——資源的取得與釋放寫在同一個函式裡，視覺上就在一起，不會漏掉配對。

scope：控制 fixture 多久建立一次

預設每個測試都會重新執行 fixture（scope="function"）。對昂貴的資源（DB 連線、啟動容器），可以放寬 scope：

python 
@pytest.fixture(scope="session")     # 整輪測試只建立一次
def db_engine():
    engine = create_engine(...)
    yield engine
    engine.dispose()
scope 多久建立一次 約略對應 JUnit
function（預設） 每個測試函式 @BeforeEach
class 每個測試類別 （JUnit 無直接對應）
module 每個 .py 檔 ——
package 每個套件 ——
session 整次 pytest 執行 @BeforeAll（但跨整個 suite）

⚠️ 陷阱：scope="session" 的 fixture 被所有測試共用，測試之間若改動它的狀態就會互相污染。共用的東西要嘛唯讀、要嘛每個測試自己負責回復原狀。這跟 JUnit @BeforeAll（static）的陷阱一樣——共享狀態是測試不穩定（flaky）的頭號來源。

conftest.py：跨檔案共享 fixture

把 fixture 放進 conftest.py，同目錄（與子目錄）下的所有測試檔都能直接使用它，不需 import。pytest 會自動發現 conftest.py。這是放共用 fixture、共用設定、自訂 marker 的地方——可以想成「測試專用的、按目錄階層生效的 @TestConfiguration」。

python 
# conftest.py
import pytest

@pytest.fixture
def cart():                  # 任何子目錄的 test_*.py 都能直接用 cart，免 import
    return [...]

內建 fixture：免費的工具箱

pytest 內建一批 fixture，直接當參數要就有，省掉你自己造輪子：

python 
def test_writes_file(tmp_path):                 # tmp_path 是一個專屬此測試的暫存目錄（pathlib.Path）
    f = tmp_path / "data.txt"
    f.write_text("hello", encoding="utf-8")
    assert f.read_text(encoding="utf-8") == "hello"

def test_captures_stdout(capsys):               # capsys 攔截 stdout/stderr
    print("Hello, Albert")
    captured = capsys.readouterr()
    assert captured.out == "Hello, Albert\n"

def test_patches_env(monkeypatch):              # monkeypatch 安全地改環境，測試結束自動還原
    monkeypatch.setenv("TIMEOUT", "5")
    monkeypatch.setattr("mymod.CONST", 42)
    assert read_timeout() == 5
  • tmp_path：每個測試獨立的暫存目錄，pytest 會保留最近幾輪再回收。對照你在 Java 用 @TempDir。
  • capsys：攔截標準輸出/錯誤（capfd 攔的是檔案描述子層級，連子行程都抓）。對照 JUnit 重導 System.out。
  • monkeypatch：在測試範圍內改屬性、環境變數、sys.path、dict 項目，測試結束自動復原。比手動存舊值再還原安全得多。下一節的 mock 是它的重砲版。

參數化與 marker

parametrize：一份邏輯，多組資料

JUnit 5 的 @ParameterizedTest + @CsvSource 你用過。pytest 的 @pytest.mark.parametrize 是同一個概念，但更輕、更 Python：

python 
@pytest.mark.parametrize(
    "amount, percent, expected",
    [
        (10000, 0, 10000),
        (10000, 10, 9000),
        (10000, 100, 0),
        (9999, 10, 9000),       # 整數除法：9999 * 10 // 100 = 999，所以 9999 - 999 = 9000
    ],
)
def test_apply_discount(amount, percent, expected):
    assert apply_discount(amount, percent) == expected

每一組 tuple 會展開成一個獨立的測試案例——上面這個就是 4 個測試，任何一組失敗，其他照跑、報表也會精準指出是哪一組（pytest 會把參數值印進測試 ID，像 test_apply_discount[9999-10-9000]）。對照 JUnit：

java 
@ParameterizedTest
@CsvSource({"10000, 0, 10000", "10000, 10, 9000", "10000, 100, 0"})
void testApplyDiscount(int amount, int percent, int expected) {
    assertEquals(expected, Pricing.applyDiscount(amount, percent));
}

差別在 Python 的版本直接傳 Python 物件（不是 CSV 字串再解析），所以你能塞 list、dict、自訂物件當參數，型別也不會在 CSV 來回轉換中走樣。要疊加多維組合就堆兩個 parametrize（笛卡兒積，等同 JUnit 的多 source 組合）。

marker：給測試貼標籤

marker 是 pytest 版的 tag/annotation。內建幾個最常用：

python 
import sys
import pytest

@pytest.mark.skip(reason="尚未實作")
def test_not_ready(): ...

@pytest.mark.skipif(sys.platform == "win32", reason="只在 POSIX 跑")
def test_posix_only():
    import os
    assert os.sep == "/"

@pytest.mark.xfail(reason="已知 rounding bug，尚未修")
def test_known_bug():
    assert round(2.675, 2) == 2.68      # 其實是 2.67（IEEE 754 浮點數），xfail 預期它失敗
  • skip / skipif：對應 JUnit 5 的 @Disabled / @EnabledOnOs / assumeTrue(...)。skipif 是有條件跳過。
  • xfail（expected to fail）：預期會失敗。它失敗了報表記為 xfailed（不算紅燈），萬一意外通過則記為 xpassed（提醒你 bug 可能修好了，該拿掉這個標記）。JUnit 沒有直接對應——這是把「已知缺陷」誠實寫進測試、又不讓 build 變紅的好工具。

自訂 marker 用來分類測試（之後可用 -m 過濾）：

python 
@pytest.mark.slow
def test_big_import(): ...
bash 
uv run --with pytest pytest -m "not slow"    # 跳過所有標 slow 的測試

記得在 pyproject.toml 註冊自訂 marker，否則 pytest 會發警告（嚴格設定下會報錯）：

toml 
[tool.pytest.ini_options]
markers = [
    "slow: 跑得慢的測試（用 -m 'not slow' 排除）",
]

pytest.raises：測例外

對應 JUnit 5 的 assertThrows，但語法是 context manager：

python 
def test_apply_discount_rejects_bad_percent():
    with pytest.raises(ValueError, match=r"percent must be in"):
        apply_discount(10000, 150)

match= 用正規表示式比對例外訊息（是 re.search，不必整段比對）。要進一步斷言例外物件本身，用 as：

python 
with pytest.raises(ValueError) as excinfo:
    apply_discount(10000, 150)
assert "150" in str(excinfo.value)

💡 Pythonic 重點：pytest.raises 的區塊裡只放預期會丟例外的那一行，別把無關的程式碼也包進去——否則例外可能是別處丟的，測試就失去意義。這對應你在 Java 只把單一 statement 放進 assertThrows 的 lambda。

unittest.mock：對照 Mockito

標準庫的 unittest.mock（不需第三方）就是 Python 的 Mockito。核心是 MagicMock 與 patch。

MagicMock：自動長出任何屬性的替身

MagicMock 對照 Mockito 的 mock(SomeClass.class)，但更激進——它自動回應任何屬性存取與方法呼叫，回傳的還是 MagicMock：

python 
>>> from unittest.mock import MagicMock
>>> m = MagicMock()
>>> m.anything.you.access            # 沒定義也不報錯，自動生出一個子 mock
<MagicMock name='mock.anything.you.access' id='...'>
>>> m.any_method(1, 2)               # 任何方法都能呼叫
<MagicMock name='mock.any_method()' id='...'>

設定回傳值用 return_value（對應 Mockito when(...).thenReturn(...)）、設定多次呼叫的不同回傳或丟例外用 side_effect：

python 
repo = MagicMock()
repo.find_user.return_value = {"id": 1, "name": "Albert"}   # when(repo.findUser(any)).thenReturn(...)

user = repo.find_user(1)
assert user["name"] == "Albert"

repo.find_user.assert_called_once_with(1)     # verify(repo).findUser(1)
Mockito unittest.mock 說明
mock(Repo.class) MagicMock() 建立替身
when(m.f()).thenReturn(x) m.f.return_value = x 設定回傳
when(m.f()).thenThrow(e) m.f.side_effect = e 設定丟例外
when(m.f()).thenReturn(a, b) m.f.side_effect = [a, b] 連續回傳
verify(m).f(1) m.f.assert_called_once_with(1) 驗證呼叫
verify(m, never()).f() m.f.assert_not_called() 驗證未被呼叫
verify(m, times(2)).f() m.f.call_count == 2 驗證次數

side_effect 比 Mockito 的 thenThrow/thenReturn 更靈活——它可以是例外（會被丟出）、可迭代物件（每次呼叫回傳下一個），或一個函式（用呼叫參數動態算回傳）：

python 
m = MagicMock()
m.next.side_effect = [10, 20, 30]             # 連續三次呼叫分別回 10, 20, 30
assert [m.next(), m.next(), m.next()] == [10, 20, 30]

m.fetch.side_effect = ConnectionError("down") # 一呼叫就丟例外

patch：在「使用處」替換

真正的重點是 patch——它在測試期間把某個名稱換成 mock，結束後自動還原。對照 Mockito 的 @Mock + @InjectMocks 或 mockStatic。假設我們的被測模組會去打外部 API：

python 
# rates.py
def fetch_rate(currency: str) -> float:
    ...                       # 真的會連網

def total_in_usd(amount_twd: int, currency: str = "TWD") -> float:
    rate = fetch_rate(currency)        # 我們想在測試裡 mock 掉這個
    return round(amount_twd * rate, 2)
python 
# test_rates.py
from unittest.mock import patch
import pytest
import rates

@patch("rates.fetch_rate")                     # 注意：patch 的是 rates.fetch_rate
def test_total_in_usd(mock_fetch):
    mock_fetch.return_value = 0.031
    result = rates.total_in_usd(100000, "TWD")
    assert result == 3100.0
    mock_fetch.assert_called_once_with("TWD")

@patch("rates.fetch_rate")
def test_network_failure(mock_fetch):
    mock_fetch.side_effect = ConnectionError("network down")
    with pytest.raises(ConnectionError):
        rates.total_in_usd(100000)

被 @patch 裝飾的測試會多收一個參數（那個 mock 物件）——多個 @patch 疊加時，注入順序由內到外（最靠近函式的 patch 是第一個參數），這個順序常讓人搞錯，記成「由下往上對應由左往右」。

python 
@patch("rates.b")              # 上面的 → 對應後面的參數
@patch("rates.a")              # ⬅ 最靠近 def → 對應第一個參數
def test(mock_a, mock_b):      # mock_a 是 rates.a、mock_b 是 rates.b
    ...

由下往上數：@patch("rates.a") 緊貼 def，所以它對應簽名裡的第一個 mock 參數 mock_a；再往上的 @patch("rates.b") 對應第二個 mock_b。

where to patch：本章最關鍵的一條規則

python 
@patch("rates.fetch_rate")        # ✅ 對：patch 它「被使用」的地方

注意我們 patch 的是 "rates.fetch_rate"，不是 fetch_rate 定義在哪個模組。這是初學者最常踩的坑。規則是：

patch the name where it is looked up, not where it is defined. （在「名稱被查找的地方」打 patch，不是在「它被定義的地方」。）

為什麼？因為 Python 的 import 是綁定名稱。當 rates.py 寫 from external import fetch_rate，它在 rates 模組的命名空間裡建立了一個叫 fetch_rate 的名字，指向那個函式物件。你要替換的是 rates 裡的那個綁定，去 patch 原始定義處的 external.fetch_rate 根本攔不到——rates 早就持有自己的參考了（這就是第 11 章講的 import 綁定機制在測試裡的直接後果）。

python 
# 如果 rates.py 是 `from external import fetch_rate`
@patch("rates.fetch_rate")        # ✅ patch 使用處
@patch("external.fetch_rate")     # ❌ 攔不到，rates 已綁定了自己的參考

# 如果 rates.py 是 `import external` 然後呼叫 external.fetch_rate()
@patch("external.fetch_rate")     # ✅ 這時要 patch 定義處，因為查找是經過 external 模組

⚠️ 陷阱：「patch 哪裡」取決於被測模組怎麼 import，不是函式定義在哪。from x import f（直接綁名）→ patch 被測模組.f；import x 後用 x.f()（每次經過模組查找）→ patch x.f。搞錯方向時的症狀是「mock 沒生效、真的去連網了」。這條規則在 Mockito 不存在，因為 Java 的方法呼叫是動態分派到物件，沒有「import 綁定名稱」這回事。

patch 也能當 context manager（with patch(...) as m:）或直接用 monkeypatch.setattr（pytest 風格，免裝飾器）。三種選一種，團隊統一即可。

coverage、hypothesis、doctest、tox/nox

coverage.py：覆蓋率與它的侷限

對照 JaCoCo。透過 pytest-cov 外掛整合進 pytest：

bash 
uv run --with pytest --with pytest-cov pytest --cov=pricing --cov-report=term-missing

以下為示意輸出，實際的 Stmts 行數與覆蓋率會依你自己的 pricing.py 而定：

text 
Name         Stmts   Miss  Cover   Missing
------------------------------------------
pricing.py      12      0   100%
------------------------------------------
TOTAL           12      0   100%

--cov-report=term-missing 會列出沒被覆蓋的行號（Missing 欄），這是它最有用的地方——直接告訴你哪幾行從沒被執行。也能輸出 html（互動報表）或 xml（餵給 CI/SonarQube）。在 pyproject.toml 設門檻：

toml 
[tool.coverage.report]
fail_under = 85          # 覆蓋率低於 85% 就讓 build 失敗
show_missing = true

⚠️ 陷阱：高覆蓋率 ≠ 測得好。覆蓋率只證明那行「被執行過」，不證明它「被驗證過」。一個沒有任何 assert、只是把所有函式呼叫一遍的測試也能拿 100%——這在 Java/JaCoCo 也一樣。把覆蓋率當「找出完全沒被測到的程式碼」的工具（那是真訊號），別把它當「測試品質」的指標。語句覆蓋（statement）尤其寬鬆，分支覆蓋（branch，--cov-branch）嚴一點但仍不保證斷言充分。

hypothesis：property-based testing

這是 Java 工程師最該帶走的新工具。對照 jqwik（或 Java 版 QuickCheck）。傳統測試（example-based）你手挑幾組輸入；property-based testing 你只描述「對任何合法輸入都該成立的性質」，框架去生成大量隨機輸入來打你的程式：

python 
from hypothesis import given, strategies as st
from pricing import apply_discount

@given(
    amount=st.integers(min_value=0, max_value=10_000_000),
    percent=st.integers(min_value=0, max_value=100),
)
def test_discount_never_exceeds_amount(amount, percent):
    discounted = apply_discount(amount, percent)
    assert 0 <= discounted <= amount        # 性質：折後價永遠在 [0, 原價] 之間

@given 配上 strategies（st.integers、st.text、st.lists、st.dictionaries……，對應 jqwik 的 Arbitraries）描述輸入空間，hypothesis 每跑一次測試就生一批案例（預設 100 個）丟進去。@given 與 pytest 完全相容——直接當普通 pytest 測試跑。

最殺的功能是 shrinking（縮小反例）：找到反例後，hypothesis 不會把那個又長又亂的隨機輸入丟給你，而是自動把它縮小到最簡形式，方便你 debug。看一個能真正抓出 bug 的 round-trip（往返）範例——「序列化再反序列化應該還原」這類性質，是 property testing 的經典戰場：

python 
# slugify.py（有 bug 的實作）
def to_slug(title: str) -> str:
    return title.strip().lower().replace(" ", "-")   # bug：連續空白變連續 '-'

def from_slug(slug: str) -> str:
    return slug.replace("-", " ")
python 
# test_roundtrip.py
from hypothesis import given, strategies as st
from slugify import to_slug, from_slug

words = st.text(alphabet="abc ", min_size=0, max_size=20)

@given(words)
def test_slug_roundtrip(title):
    normalized = " ".join(title.lower().split())     # 我們對「正規化後」的往返期望
    assert from_slug(to_slug(title)) == normalized

跑下去，hypothesis 找到反例並縮到最小：

text 
title = 'a  a'

>       assert from_slug(to_slug(title)) == normalized
E       AssertionError: assert 'a  a' == 'a a'
E         Falsifying example: test_slug_roundtrip(title='a  a')

它沒丟給你一個 20 字元的亂碼，而是縮到 'a a'（兩個空白）——一眼就看出 bug：連續空白被轉成連續 -、還原時又變回連續空白，跟正規化後的單一空白對不上。這種「我沒想到的邊界輸入」正是手寫範例測試會漏、property testing 會抓的。hypothesis 還會把找到的反例存進 .hypothesis/ 快取，下次優先重跑（迴歸保護）。

💡 Pythonic 重點：寫 property test 的訣竅是找「不變式（invariant）」而非「具體答案」。好用的性質模式：往返（decode(encode(x)) == x）、等冪（f(f(x)) == f(x)）、與簡單參考實作比對（my_sort(x) == sorted(x)）、輸出的結構性質（len(result) <= len(input)）。一條 property test 抵得上幾十條手寫案例，且會找到你想不到的邊界。

doctest：docstring 即測試

doctest 是標準庫工具：把 docstring 裡長得像 REPL 互動（>>>）的片段當測試執行，比對實際輸出與寫下的輸出。對照——Java 沒有真正等價物（JavaDoc 的 {@code} 不會被執行）。

python 
# mathy.py
def factorial(n: int) -> int:
    """回傳 n 的階乘。

    >>> factorial(0)
    1
    >>> factorial(5)
    120
    """
    result = 1
    for i in range(2, n + 1):
        result *= i
    return result
bash 
python3 -m doctest mathy.py -v               # 用標準庫跑
uv run --with pytest pytest --doctest-modules mathy.py   # 讓 pytest 一起收集 doctest

doctest 的價值在讓文件不會腐爛——範例輸出對不上就紅燈，逼你維護文件的正確性。但它對輸出格式很挑剔（多一個空白、dict 順序、物件的 repr 位址都會讓它失敗），所以只拿它測「輸出穩定、適合當文件示範」的小函式，別把它當主力測試框架。複雜邏輯與邊界仍交給 pytest。

tox / nox：跨多版本測試

對照 CI build matrix 與 Maven profiles。你的函式庫要支援 Python 3.11/3.12/3.13，就得在每個版本各跑一次測試——tox（設定檔驅動）與 nox（用 Python 寫設定）就是做這件事的：每個目標版本建一個隔離的虛擬環境、裝好相依、跑測試。

ini 
# tox.ini
[tox]
env_list = py311, py312, py313

[testenv]
deps = pytest
commands = pytest {posargs}
python 
# noxfile.py（nox 用 Python 寫，更靈活）
import nox

@nox.session(python=["3.11", "3.12", "3.13"])
def tests(session):
    session.install("pytest", "hypothesis")
    session.run("pytest")

nox -s tests 或 tox 會對每個版本各跑一輪。CI 裡你通常還會用 GitHub Actions 的 strategy.matrix 在不同 OS × 版本上平行跑——tox/nox 管「多 Python 版本」、CI matrix 管「多 OS／多維度」，兩者常一起用。

💡 Pythonic 重點：函式庫（要被別人裝、需支援多版本）值得 tox/nox；公司內部的單一部署應用（你只跑一個鎖定的版本）通常不需要——直接在 CI 釘死那個版本即可。別為了「看起來完整」而引入用不到的 matrix 複雜度。

本章小結

  1. pytest 用函式 + 命名慣例 + 裸 assert：test_*.py 裡的 test_* 函式自動被收集，無 @Test、無 TestCase。assert rewriting 讓裸 assert a == b 也有豐富 diff，所以不需要 assertEquals 那套 API。
  2. fixture 是按名稱注入的依賴：宣告 @pytest.fixture，測試用同名參數「點名」即可注入。yield 版把 setup/teardown 寫在一起（yield 後一定執行）；scope 控制建立頻率，但共享 scope 小心狀態污染。
  3. conftest.py 與內建 fixture 省你造輪子：tmp_path、capsys、monkeypatch 直接要就有；conftest.py 按目錄階層共享 fixture、免 import。
  4. parametrize 把資料展開成多個獨立案例，傳的是 Python 物件而非 CSV；marker 貼標籤（skip/skipif/xfail、自訂 marker + -m 過濾）；pytest.raises 測例外、match= 用 regex。
  5. unittest.mock = Python 版 Mockito：MagicMock 自動長屬性、return_value/side_effect 設行為、assert_called_once_with 驗證。
  6. where to patch 是 mock 的命門：patch「名稱被查找的地方」而非「定義處」——from x import f → patch 被測模組.f；import x 用 x.f() → patch x.f。搞錯就 mock 不生效。
  7. 覆蓋率是「找沒測到的碼」的工具，不是品質指標——高覆蓋率不代表斷言充分。
  8. hypothesis（property-based）會抓你想不到的邊界：用 @given + strategies 描述性質（往返、等冪、不變式），框架生成大量輸入並自動 shrink 反例。doctest 讓文件不腐爛、tox/nox 跨多版本——各用在對的地方。

練習

練習 1（JUnit → pytest 平移） 把這段 JUnit 風格的測試改寫成 pytest：一個 @BeforeEach 建立空 ShoppingCart、兩個測試方法（加一個商品後 size 為 1、加重複商品後 quantity 累加）。要求用 fixture 取代 @BeforeEach、用裸 assert。提示：fixture 回傳那個 cart，兩個測試函式都把它當參數。

練習 2（parametrize + raises） 為 apply_discount(amount, percent) 寫一條 parametrize 測試，涵蓋 percent 為 0、50、100 三組正常值，再寫一條獨立測試用 pytest.raises(ValueError, match=...) 驗證 percent=-1 與 percent=101 都會被拒絕。觀察失敗時 pytest 報的測試 ID 長什麼樣。

練習 3（where to patch） 寫一個 weather.py，裡面 from http_client import get_json 並用它實作 current_temp(city)。寫測試把 get_json mock 掉、回傳假資料，斷言 current_temp 算出正確結果。先故意 patch "http_client.get_json"（定義處）跑一次、看它為什麼攔不到，再改成 patch "weather.get_json"（使用處）。用一句話解釋差異。

練習 4（property-based：抓真 bug） 寫一個 parse_duration("1h30m") -> int（回傳總秒數）與反向的 format_duration(seconds) -> str，故意在某個邊界留 bug（例如剛好 0 秒或超過 60 分鐘）。用 hypothesis 寫 round-trip 性質 parse_duration(format_duration(s)) == s（s 用 st.integers(min_value=0, max_value=86400)），讓它幫你縮小出反例。提示：把反例貼出來，看 shrinking 縮到多小。

練習 5（覆蓋率的謊言） 為練習 2 的 apply_discount 寫一個沒有任何 assert、只呼叫函式的「假測試」，用 --cov 觀察它照樣能拿高覆蓋率。再補上真正的斷言。用這個對比解釋為什麼「覆蓋率達標」不能當驗收標準，以及它真正該被用來回答什麼問題。
