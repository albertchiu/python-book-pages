第 3 章：函式——Python 的一等公民

在 Java 裡，「函式」從來不是獨立存在的東西——它得依附在類別上，叫做方法（method）。你想把一段行為當資料傳遞，得先發明一個介面、塞進匿名類別，直到 Java 8 才有 lambda 與 method reference 把這件事變得勉強優雅。Python 的世界完全相反：函式從第一天起就是一等物件（first-class object），跟 int、str、list 平起平坐。這個差異看似小，卻會徹底改變你組織程式碼的方式——許多你習慣用設計模式（design pattern）解決的問題，在 Python 裡只是「把函式放進變數」這麼簡單。本章還會帶你踩兩個 Java 工程師幾乎 100% 會踩的坑：可變預設參數與 closure 的延遲綁定。

函式是一等物件

「一等物件」的意思很具體：函式可以被賦值給變數、放進資料結構、當參數傳入、當回傳值傳出。在 Python 裡，def 並不是宣告語法，而是一個會在執行期建立函式物件、並把它綁定到某個名字的陳述式。

python 
>>> def greet(name):
...     """say hi"""
...     return f"Hi {name}"
...
>>> type(greet)
<class 'function'>
>>> greet.__name__
'greet'
>>> greet.__doc__
'say hi'
>>> say_hello = greet      # 把函式賦值給另一個名字
>>> say_hello("Albert")
'Hi Albert'

greet 只是一個指向函式物件的名字（name），跟 x = 5 裡的 x 在語意上沒有本質差別（變數即名字綁定的觀念詳見第 2 章）。它甚至有屬性、有型別，是個如假包換的物件。

Java 的演進史，就是在補這一課

java 
// Java 8 之前：要傳遞行為，得發明介面 + 匿名類別
Comparator<String> byLength = new Comparator<String>() {
    public int compare(String a, String b) {
        return a.length() - b.length();
    }
};

// Java 8+：lambda 與 method reference 終於把它變短
Comparator<String> byLen = (a, b) -> a.length() - b.length();
Function<String, Integer> len = String::length;

Java 的 lambda 與 :: 本質上仍是「介面的單一方法實作」的語法糖——byLen 的型別是 Comparator<String>，是個物件，不是「函式本身」。Python 不需要這層包裝，函式就是函式。

python 
# Python：函式天生就能傳遞，不需要介面
def by_length(s: str) -> int:
    return len(s)

words = ["banana", "apple", "cherry"]
print(sorted(words, key=by_length))   # ['apple', 'banana', 'cherry']

🔁 Java 對照：Java 的 Function<T,R>／Comparator<T> 是「把函式包進物件」；Python 的函式本來就是物件，省掉了那層 @FunctionalInterface。

把函式放進 dict：dispatch table 取代 strategy pattern

Java 工程師遇到「根據某個 key 選擇不同行為」時，本能反應是 strategy pattern：定義介面、寫一堆實作類別、用 Map<String, Strategy> 註冊。Python 直接把函式塞進 dict 就好——這叫做 dispatch table（分派表）。

python 
from collections.abc import Callable

def add(a: float, b: float) -> float:
    return a + b

def sub(a: float, b: float) -> float:
    return a - b

def mul(a: float, b: float) -> float:
    return a * b

# dispatch table：key 對應到「行為本身」
# 型別標注用 Callable[[float, float], float]，而不是小寫內建的 callable
# （callable 是 builtin 函式、不是型別，mypy 會報 valid-type 錯誤）
OPS: dict[str, Callable[[float, float], float]] = {
    "+": add,
    "-": sub,
    "*": mul,
}

def calculate(op: str, a: float, b: float) -> float:
    handler = OPS[op]      # 取出函式
    return handler(a, b)   # 呼叫它

print(calculate("+", 3, 4))   # 7
print(calculate("*", 3, 4))   # 12

對照 Java 的 strategy pattern：

java 
interface Operation { double apply(double a, double b); }

Map<String, Operation> ops = Map.of(
    "+", (a, b) -> a + b,
    "-", (a, b) -> a - b
);
double result = ops.get("+").apply(3, 4);  // 7

兩者邏輯一致，但 Python 版本沒有介面、沒有匿名類別、沒有樣板程式碼。當「策略」只是一個純函式時，Python 的寫法直接消滅了整個 pattern 的儀式感。

💡 Pythonic 重點：當你想用 strategy / command pattern 時，先問自己「這個策略只是一個函式嗎？」如果是，dict + 函式就夠了，不需要類別階層。

參數機制

Python 的參數系統比 Java 靈活得多，但也因此有更多細節要掌握。核心觀念：Python 沒有方法多載（overloading）。Java 靠多個同名、不同簽章的方法來提供彈性；Python 靠的是預設值、可變參數與關鍵字參數，把所有彈性塞進一個函式定義裡。

位置參數、關鍵字參數、預設值

python 
def connect(host: str, port: int = 5432, timeout: float = 30.0) -> str:
    return f"{host}:{port} (timeout={timeout})"

print(connect("db.local"))                       # db.local:5432 (timeout=30.0)
print(connect("db.local", 6379))                 # 用位置傳 port
print(connect("db.local", timeout=5.0))          # 用關鍵字跳過 port
print(connect(port=3306, host="db.local"))       # 全用關鍵字，順序隨意

任何參數都可以用「位置」或「關鍵字」兩種方式傳遞。關鍵字傳參在 Java 沒有對應物（Java 只能靠位置），它讓呼叫端更具自我說明性——connect("db.local", timeout=5.0) 比 connect("db.local", 5432, 5.0) 清楚得多。

🔁 Java 對照：Java 用多載 connect(String)、connect(String, int)、connect(String, int, double) 提供彈性；Python 一個帶預設值的函式就涵蓋全部組合。

`*args` 與 `**kwargs`：可變參數

python 
def report(*args, **kwargs):
    return args, kwargs

print(report(1, 2, x=3, y=4))
# ((1, 2), {'x': 3, 'y': 4})
  • *args 把多餘的位置參數收進一個 tuple（對照 Java 的 varargs Object... args）。
  • **kwargs 把多餘的關鍵字參數收進一個 dict——這個 Java 完全沒有對應概念。

反向操作叫做「解包（unpacking）」：在呼叫端用 * 把序列攤成位置參數、用 ** 把 dict 攤成關鍵字參數。

python 
def show(a: int, b: int) -> int:
    return a + b

nums = [3, 4]
config = {"a": 3, "b": 4}
print(show(*nums))     # 等同 show(3, 4) -> 7
print(show(**config))  # 等同 show(a=3, b=4) -> 7

這是寫「轉發型」函式（裝飾器、wrapper、middleware）的關鍵工具，第 10 章會大量用到。

keyword-only 與 positional-only 參數

* 與 / 這兩個分隔符讓你精確控制每個參數怎麼被傳。

  • 放在 * 之後的參數是 keyword-only：呼叫端只能用關鍵字傳。
  • 放在 / 之前的參數是 positional-only（PEP 570，Python 3.8+）：呼叫端只能用位置傳。
python 
def f(pos_only, /, normal, *, kw_only):
    return pos_only, normal, kw_only

print(f(1, 2, kw_only=3))          # (1, 2, 3)
print(f(1, normal=2, kw_only=3))   # (1, 2, 3)，normal 兩種都可

# 沿用上面的 f：pos_only 不能用關鍵字傳，會拋 TypeError
try:
    f(pos_only=1, normal=2, kw_only=3)
except TypeError as e:
    print(e)
    # f() got some positional-only arguments passed as keyword arguments: 'pos_only'

為什麼要這樣設計？keyword-only 強迫呼叫端寫出參數名，避免一長串布林旗標讓人看不懂：create_user(name, admin=True, active=False) 遠比 create_user(name, True, False) 安全。positional-only 則讓你保留把參數改名的自由（因為呼叫端不依賴名字），標準函式庫許多 C 實作的函式都用它。

💡 Pythonic 重點：對於語意不明顯的布林或選項參數，把它放到 * 之後變成 keyword-only。這能讓呼叫端自我說明，是一種輕量的 API 防呆。

Python 沒有多載——這是特性不是缺陷

python 
def f(x): ...
def f(x, y): ...   # 沒有報錯，但第二個定義直接覆蓋第一個！

在 Python 裡寫兩個同名函式，後者只是重新綁定了名字，第一個就此消失。想表達「不同參數數量的同名行為」，請用預設值或 *args。若真要根據型別分派，標準函式庫提供 functools.singledispatch（基於第一個參數的型別），但那是進階手法，先建立「一個函式涵蓋所有情況」的思維才是正道。

預設參數的可變物件陷阱

這是 Java 工程師踩坑機率最高的一題，務必看透。

先看這段「看起來很正常」的程式碼：

python 
def add_item(item, items=[]):   # 這會出現意料外的行為
    items.append(item)
    return items

你的 Java 直覺會說：「每次呼叫沒傳 items 時，就給一個新的空 list 嘛。」錯。實際行為是：

python 
>>> add_item("a")
['a']
>>> add_item("b")     # 你以為會是 ['b']
['a', 'b']            # 結果是這樣！
>>> add_item("c")
['a', 'b', 'c']

為什麼？預設值在「定義時」只算一次

關鍵在於：預設值是在函式「定義」的那一刻求值一次，而不是每次呼叫時重新求值。 那個 [] 在 def 執行時就被建立成一個list 物件，並存進函式物件的 __defaults__。之後每次呼叫沒傳 items，用的都是同一個 list。

你可以直接看到這個共用的物件：

python 
>>> def g(items=[]):
...     items.append(1)
...     return items
...
>>> g.__defaults__       # 預設值就掛在這裡，是個物件
([],)
>>> g()
[1]
>>> g.__defaults__       # 同一個 list 被改了
([1],)

對照 Java：Java 根本沒有預設參數這個語言特性（你得靠多載），所以也不會有這個坑——但這正是為什麼 Java 工程師第一次看到會完全沒有防備。

標準解法：用 `None` 當哨兵

python 
def add_item(item, items: list | None = None) -> list:
    if items is None:        # 每次呼叫都判斷，需要時才建新 list
        items = []
    items.append(item)
    return items

print(add_item("a"))   # ['a']
print(add_item("b"))   # ['b']  ← 正確了

None 是不可變的單例，當預設值再安全不過。if items is None 把「建立新 list」這個動作從「定義時」移到「呼叫時」，問題迎刃而解。

⚠️ 陷阱：永遠不要用可變物件（list、dict、set）當預設參數值。 改用 None 當哨兵，在函式內判斷後再建立。這條規則沒有例外，linter（如 ruff 的 B006）會自動抓出來。

為什麼這個設計其實合理？

你可能覺得這設計很反直覺，但它其實是 Python「函式是物件、預設值是物件的一部分」這個一致性原則的自然結果。預設值跟函式物件一起活著，自然只算一次。理解這點的關鍵是 Python 的**可變性（mutability）**觀念：問題的根源不是「預設值只算一次」，而是「你拿一個可變物件去 append」。如果預設值是不可變的（像 None、0、"x"、tuple），就算共用同一個也不會有任何問題，因為你無法就地修改它。可變 vs 不可變的完整討論留到第 4 章，但這個坑正是它的最佳前導範例。

lambda 的限制

Python 有 lambda，但它跟 Java 的 lambda 不是同一個量級的工具。

Python 的 lambda 只能是「單一運算式（expression）」，不能包含任何陳述式（statement）：不能賦值、不能 if/else 區塊、不能迴圈、不能 try、不能 return（運算式的值就是回傳值）。

python 
square = lambda x: x * x
print(square(5))   # 25

# 三元運算子（是運算式）可以
classify = lambda n: "even" if n % 2 == 0 else "odd"
print(classify(4))   # even

對照 Java 的 lambda——它可以有完整的程式區塊：

java 
// Java lambda 可以塞一整個 block
Function<Integer, String> f = (n) -> {
    String result;
    if (n % 2 == 0) result = "even";
    else result = "odd";
    return result;   // 可以有陳述式、可以 return
};

Python 的 lambda 故意被設計得這麼弱，是一種語言哲學的表態：Guido 認為如果你的邏輯複雜到需要多行，那它就值得有一個名字。硬塞進 lambda 只會犧牲可讀性。

python 
# 反例：不要這樣寫
process = lambda data: [x * 2 for x in data if x > 0]   # 能跑，但何必？

# 正例：給它一個名字
def process(data: list[int]) -> list[int]:
    return [x * 2 for x in data if x > 0]

lambda 的正確用途：即用即丟的 key 函式

lambda 真正發光的地方是當「臨時的、一行的、用完即丟」的回呼，最經典的就是 sorted、min、max、filter 的 key 參數：

python 
words = ["banana", "apple", "cherry"]

# 按長度排序
print(sorted(words, key=lambda w: len(w)))      # 但其實 key=len 更好

# 按最後一個字母排序（這種就適合 lambda）
print(sorted(words, key=lambda w: w[-1]))       # ['banana', 'apple', 'cherry']

# 按 dict 的某個欄位排序
people = [{"name": "Bob", "age": 30}, {"name": "Alice", "age": 25}]
print(sorted(people, key=lambda p: p["age"]))   # Alice 先

💡 Pythonic 重點：lambda 限定在「一行就講完、且只用一次」的場景。需要重用、需要文件字串、或邏輯超過一個運算式，就寫具名函式。能用 key=len、key=str.lower 這類現成函式時，連 lambda 都別寫。comprehension 能取代的 map/filter+lambda（詳見第 5 章）也優先用 comprehension。

closure 與 nonlocal

巢狀函式（nested function）可以捕捉外層函式的區域變數，形成 closure（閉包）。這跟 Java 的 lambda 捕捉變數很像，但有一個關鍵差異。

基本 closure

python 
def make_multiplier(factor: int):
    def multiply(x: int) -> int:
        return x * factor      # 捕捉外層的 factor
    return multiply

double = make_multiplier(2)
triple = make_multiplier(3)
print(double(10))   # 20
print(triple(10))   # 30

factor 在 make_multiplier 回傳後本該消失，但 multiply 仍記得它。你甚至能透過函式物件看到被捕捉的變數：

python 
>>> double.__closure__[0].cell_contents
2

修改外層變數需要 `nonlocal`

預設情況下，內層函式只能讀取外層變數。一旦你在內層對它賦值，Python 會把它當成一個新的區域變數，而不是修改外層的那個。要明確表達「我要改外層的」，得用 nonlocal：

python 
def make_counter():
    count = 0
    def increment() -> int:
        nonlocal count       # 沒有這行，下一行的 count += 1 會報 UnboundLocalError
        count += 1
        return count
    return increment

c = make_counter()
print(c(), c(), c())   # 1 2 3

如果拿掉 nonlocal count，Python 看到 count += 1（等同 count = count + 1，含賦值）就認定 count 是 increment 的區域變數，於是在「讀取尚未賦值的區域變數」時拋出 UnboundLocalError。

🔁 Java 對照：Java 的 lambda/匿名類別只能捕捉 effectively final 的變數——你根本不能修改被捕捉的變數，編譯器直接擋。Python 預設也是「只讀」，但給了你 nonlocal（改外層函式變數）和 global（改模組層級變數）這兩個逃生門。Java 工程師習慣的「包一個陣列 int[] count = {0} 來繞過 final 限制」的 hack，在 Python 不需要——直接 nonlocal。

late binding 陷阱：closure 綁的是「變數」不是「值」

這是本章第二個高機率坑，跟可變預設參數一樣致命。

考慮這段「製造一串函式」的程式碼：

python 
funcs = [lambda: i for i in range(3)]
print([f() for f in funcs])   # 你以為是 [0, 1, 2]
python 
>>> funcs = [lambda: i for i in range(3)]
>>> [f() for f in funcs]
[2, 2, 2]                      # 居然全是 2！

為什麼？ closure 捕捉的是變數 i 本身，而不是建立 lambda 那一刻 i 的值。所有三個 lambda 共享同一個 i。等到你呼叫它們時（迴圈早已跑完），i 的值停在最後一個 2，於是三個都回傳 2。這叫「延遲綁定（late binding）」——變數值是在呼叫時才查找，不是在定義時鎖定。

你的 Java 直覺再次出賣你：Java 要求被捕捉變數 effectively final，所以這種「捕捉一個會變的迴圈變數」的程式碼根本編譯不過，等於 Java 編譯器幫你擋掉了這個錯誤。Python 不擋，就讓你踩。

解法：用預設參數在「定義時」立刻綁定值。 還記得前面說的嗎？預設值在定義時求值一次——這個特性在這裡反而成了救星：

python 
>>> funcs = [lambda i=i: i for i in range(3)]   # i=i 把當下的值綁進去
>>> [f() for f in funcs]
[0, 1, 2]                                        # 正確！

lambda i=i: i 的意思是：建立 lambda 的當下，把外層 i 的值抓進來當作參數 i 的預設值。每個 lambda 都有自己獨立的、已經定值的 i。

⚠️ 陷阱：在迴圈裡建立 closure（lambda 或巢狀函式）並捕捉迴圈變數時，捕捉到的是變數而非當下的值，等呼叫時才查找——你會拿到迴圈結束後的最終值。標準解法是用預設參數 lambda x=x: ... 在定義時就鎖定值。這在「迴圈裡註冊一堆 callback」的場景特別容易中招。

型別提示（type hints）

身為 Java 工程師，你大概一路看到這裡都在懷念那個「編譯器幫我擋型別錯誤」的安全感。好消息：Python 從 3.5 起有了型別提示（type hints，PEP 484），讓你把那份安全感找回來一大半。

基本語法

python 
def greet(name: str, times: int = 1) -> str:
    return f"Hi {name}! " * times

def find_user(uid: int) -> str | None:    # X | None 等同 Optional[str]
    return None
  • 參數標注：name: str、times: int
  • 回傳標注：-> str
  • 可空型別：str | None（Python 3.10+ 的寫法，取代舊的 Optional[str]）

容器型別直接用內建泛型（Python 3.9+），不要再 from typing import List：

python 
def total(prices: list[float]) -> float:
    return sum(prices, 0.0)   # 給 start=0.0，空 list 也回傳 float（sum([]) 會是 int 0）

def index_by_id(users: list[dict[str, int]]) -> dict[int, dict[str, int]]:
    return {u["id"]: u for u in users}

🔁 Java 對照：Map<String, Integer> 對應 dict[str, int]；List<Double> 對應 list[float]；Optional<T> 對應 T | None。對照速查表見附錄 A。

函式型別：`Callable`

要標注「一個接受 int、回傳 str 的函式」這種高階參數，用 Callable——這正是 Java 的 Function<Integer, String> 的對應物：

python 
from collections.abc import Callable

def apply(fn: Callable[[int], int], value: int) -> int:
    return fn(value)

print(apply(lambda x: x + 1, 10))   # 11

Callable[[int, str], bool] 表示「接受 (int, str)、回傳 bool 的可呼叫物件」。前面那個 dispatch table 的型別正是這樣寫成 dict[str, Callable[[float, float], float]]。

🔁 Java 對照：Callable 同時存在於 collections.abc 與 typing，兩者皆可用且 mypy --strict 都不報錯。在 Python 3.9+，官方建議優先用 from collections.abc import Callable，typing.Callable 是偏舊的別名——這跟本章前面「不要再 from typing import List、改用內建泛型」是同一個現代化主張。

關鍵差異：執行期「不」強制

這是你最需要扭轉的觀念。Python 的型別提示在執行期完全不被強制。 它們只是註解（annotation），存在 __annotations__ 裡，給「人」和「工具」看，直譯器執行時根本不檢查。

python 
>>> def double(x: int) -> int:
...     return x + x
...
>>> double("ab")       # 標注是 int，但傳 str 也照跑不誤
'abab'
>>> double.__annotations__
{'x': <class 'int'>, 'return': <class 'int'>}

對照 Java：Java 的型別是編譯期強制的，double("ab") 連編譯都過不了。Python 把這個檢查工作外包給靜態分析工具：

  • mypy 或 pyright：在 CI 或編輯器裡做靜態型別檢查（相當於 Java 編譯器的型別檢查階段）。
  • 編輯器（VS Code + Pylance、PyCharm）即時標紅、提供自動完成。

換句話說，型別提示 + mypy = 把 Java 編譯器的型別檢查，以**可選（opt-in）**的方式裝回 Python。你可以漸進式導入（gradual typing）：先標關鍵介面，再逐步覆蓋全專案。

💡 Pythonic 重點：型別提示是寫給工具與人看的契約，不是執行期防護。在團隊專案、公開 API、複雜邏輯裡務必標注並接上 mypy/pyright；快速腳本可以省略。把它當成「免費的文件 + 編輯器自動完成 + CI 防呆」，而不是「執行期驗證」——後者要靠 pydantic 之類的工具（詳見第 9 章）。

關於泛型（TypeVar、Generic）、Protocol（結構型別）、pydantic 執行期驗證等進階主題，第 9 章會專門深入。本章你只要建立一個觀念：型別提示讓你在動態語言裡找回一大部分靜態安全感，但它的執法者是工具，不是直譯器。

Docstring：給人看的文件字串

型別提示是寫給工具的契約；docstring（文件字串）則是寫給人的文件。兩者一搭一唱：型別提示說「這個參數是 str」，docstring 說「這個 str 代表什麼、為什麼、什麼情況會出錯」。

身為 Java 工程師，你寫過無數 JavaDoc。但這裡有一個本質差異要先扭轉：JavaDoc 是註解，docstring 是物件。

java 
/** Return a greeting for the given name. */   // 註解：編譯後（預設）就消失了
String greet(String name) { return "Hi " + name; }

JavaDoc 的 /** ... */ 是一段註解，編譯成 bytecode 後預設不保留，只能靠 javadoc 工具在原始碼階段抽出來產生 HTML。Python 的 docstring 不是註解，而是函式（或模組、類別）的第一個陳述式——一個字串字面值，直譯器會把它存進 __doc__ 屬性，在執行期就能讀到。

python 
>>> def greet(name: str) -> str:
...     """Return a greeting for the given name."""
...     return f"Hi {name}"
...
>>> greet.__doc__
'Return a greeting for the given name.'
>>> help(greet)        # docstring 就是 help() 的原料
Help on function greet in module __main__:

greet(name: str) -> str
    Return a greeting for the given name.

help()、IDE 的滑鼠懸停提示、Sphinx 自動產文件、甚至 doctest（第 13 章）能把 docstring 當測試跑——全都靠這個執行期就在的字串。這是 JavaDoc 給不了的：它在執行期早已不存在。

🔁 Java 對照：JavaDoc 是原始碼註解，編譯後消失，靠工具離線抽取；Python 的 docstring 是活在執行期的 str 物件（func.__doc__），help() 與工具直接讀它。對應慣例文件是 PEP 257。

它必須是「第一個陳述式」

docstring 之所以是 docstring，純粹因為它的位置：函式體（或模組、類別）的第一個陳述式且是字串字面值。位置一錯，它就只是一個被丟棄的字串，__doc__ 會是 None。

python 
>>> def f():
...     x = 1
...     "我以為這是 docstring"   # 不是！前面已經有 x = 1 了
...     return x
...
>>> f.__doc__
>>> # None

⚠️ 陷阱：docstring 必須是函式/類別/模組的第一個陳述式，且是字串字面值。放在其他陳述式之後不算；用 f-string（f"""..."""）也不算（那是運算式，不是字串字面值），__doc__ 都會是 None。

PEP 257 的格式慣例

PEP 257 訂下幾條約定，工具與同事都預期你遵守：

  1. 一律用三引號 """，就算只有一行也是——這樣日後要擴成多行不必改引號，也能塞進 " 而不必跳脫。
  2. 動詞用祈使句（imperative mood）：寫 Return the sum（回傳……），而不是 Returns the sum 或 This function returns...。這是 Python 社群慣例，讀起來像在對函式下指令。
  3. 單行 docstring：摘要 + 句號，全部擠在一行的三引號內。
  4. 多行 docstring：第一行是摘要句，接一個空行，再寫詳細說明，最後結尾的 """ 自成一行。
python 
def add(a: float, b: float) -> float:
    """Return the sum of a and b."""        # 單行：祈使句 + 句號
    return a + b


def fetch(url: str, *, timeout: float = 30.0, retries: int = 3) -> str:
    """Fetch a URL and return its body as text.

    Issues a GET request and retries on transient failures. The
    response body is decoded with the charset from the headers.

    Args:
        url: The absolute URL to fetch.
        timeout: Per-attempt timeout in seconds.
        retries: How many times to retry on a 5xx response.

    Returns:
        The decoded response body.

    Raises:
        ValueError: If url is not an absolute URL.
        TimeoutError: If every attempt exceeds timeout.
    """
    ...

縮排不必擔心：help() 與 inspect.getdoc() 會自動把整段 docstring 的共同縮排去掉，所以你照函式縮排層級寫即可。

三種結構化風格：Google／NumPy／reST

上面 fetch 用的是 Google style（Args:／Returns:／Raises: 區塊），是目前最常見、最好讀的一種。另外兩種你會在野外遇到：

python 
# NumPy style：欄位用底線分隔，科學運算生態（numpy/pandas/scipy）的主流
def fetch_numpy(url, timeout=30.0):
    """Fetch a URL and return its body.

    Parameters
    ----------
    url : str
        The absolute URL to fetch.
    timeout : float, optional
        Per-attempt timeout in seconds.

    Returns
    -------
    str
        The decoded response body.
    """

# reStructuredText / Sphinx style：原生的 :param:／:returns: 欄位
def fetch_rest(url, timeout=30.0):
    """Fetch a URL and return its body.

    :param url: The absolute URL to fetch.
    :param timeout: Per-attempt timeout in seconds.
    :returns: The decoded response body.
    """

三種都能被 Sphinx 解析（Google／NumPy 需 napoleon 擴充）。重點不是選哪一種，而是整個專案統一一種。 給 Java 工程師的建議：從 Google style 起手，它最接近你習慣的 JavaDoc @param／@return／@throws 心智模型，又最少符號雜訊。

別在 docstring 裡重抄型別

JavaDoc 時代你可能習慣在 @param 寫上型別。在 Python，型別交給型別提示，docstring 只講語意。下面是反例與正解：

python 
# 反例：型別已經在簽章裡了，docstring 重抄一遍只會雙重維護、容易不同步
def charge(amount: float) -> bool:
    """Charge the card.

    Args:
        amount (float): the amount.    # ← (float) 與型別提示重複；"the amount" 等於沒說明
    Returns:
        bool: True or False.            # ← 沒講 True/False 各代表什麼
    """

# 正解：型別讓提示講，docstring 講「為什麼、單位、邊界、回傳的意義」
def charge(amount: float) -> bool:
    """Charge the card for `amount` and return whether it succeeded.

    Args:
        amount: Charge value in the account's currency; must be > 0.

    Returns:
        True if the gateway accepted the charge, False if it was
        declined. Network errors raise instead of returning False.
    """

💡 Pythonic 重點：公開的模組、類別、函式都該有 docstring（私有 helper 可省）。摘要句用祈使句、單行就講完核心用途；多行才補 Args/Returns/Raises。不要重抄型別提示已表達的資訊——docstring 的價值在「型別說不出口的語意」：單位、邊界條件、副作用、回傳值的意義、會丟哪些例外。

docstring 放在哪裡

不只函式，模組（檔案第一行）與類別也吃同一套規則：

python 
"""Payment processing utilities.

This module wraps the billing gateway and exposes a single
`charge()` entry point.
"""                                   # ← 模組 docstring：檔案的第一個陳述式

class Stack:
    """A last-in, first-out collection."""   # ← 類別 docstring

    def push(self, item: int) -> None:
        """Add `item` to the top of the stack."""   # ← 方法 docstring
        ...

Stack.__doc__、Stack.push.__doc__ 都能在執行期讀到。模組 docstring 則存在 模組名.__doc__，也是 python -m pydoc 模組名 與 Sphinx 產文件的入口。

💡 Pythonic 重點：docstring 不是「有空再補」的裝飾，而是 Python 把文件做成執行期一等公民的設計——help()、IDE、Sphinx、doctest 全靠它。養成「寫完簽章先補一句祈使句摘要」的肌肉記憶，就像你寫 Java 方法會反射性補 JavaDoc 一樣。

本章小結

  1. 函式是一等物件：可賦值、可放進容器、可當參數與回傳值。def 是執行期建立函式物件並綁名字的陳述式，不是宣告。
  2. dispatch table 取代 strategy pattern：當策略只是純函式時，把函式塞進 dict 就好，不需要介面與類別階層。
  3. Python 沒有多載：用預設值、*args/**kwargs、keyword-only（* 後）、positional-only（/ 前）把所有彈性塞進一個函式定義。
  4. 可變預設參數是大坑：預設值在「定義時」只算一次並被所有呼叫共用。永遠用 None 當哨兵，函式內判斷後再建可變物件。
  5. lambda 只能是單一運算式：不能有陳述式。它只適合「一行、即用即丟」的 key 函式；複雜邏輯給它一個名字，或用 comprehension。
  6. closure 預設只讀外層變數：要修改得用 nonlocal（改外層函式變數）或 global（改模組變數），不必像 Java 那樣用陣列 hack。
  7. late binding 陷阱：迴圈中建立的 closure 捕捉的是「變數」而非當下的值，呼叫時才查找。用 lambda x=x: ... 在定義時鎖定值。
  8. 型別提示給工具看，不在執行期強制：搭配 mypy/pyright 把 Java 編譯器的型別檢查以可選方式裝回來，享受漸進式型別的彈性。
  9. docstring 是給人看的執行期文件：函式/類別/模組的第一個字串字面值，存進 __doc__，是 help()、IDE、Sphinx、doctest 的原料（不像 JavaDoc 編譯後消失）。遵 PEP 257：三引號、祈使句摘要；用 Google/NumPy/reST 任一風格但全專案統一；別重抄型別提示已表達的型別，只講語意。

練習

練習 1（dispatch table） 你有一段 Java 風格的 switch／if-else，依 command 字串（"start"、"stop"、"restart"）呼叫不同方法。把它改寫成 Python 的 dispatch table，要求：用 dict 把字串對應到函式，並提供一個 dispatch(command) 函式；當 command 不存在時回傳預設行為（提示：dict.get(command, default_fn)）。

練習 2（修掉可變預設參數） 下面的函式有兩個獨立的 bug，請各用一句話說明成因並修正：

python 
def append_log(msg, log=[]):
    log.append(msg)
    return log

（提示：成因在「預設值何時求值」；解法用 None 哨兵。額外挑戰：解釋為什麼把預設值改成 tuple () 雖然不會累積，但 log.append 會直接報錯。）

練習 3（late binding） 以下程式碼想建立三個「分別印出 0、1、2」的函式，但實際全印 2。請說明原因，並提供兩種以上的修法（提示：一種用預設參數綁定；想想還能不能用 functools.partial）。

python 
printers = [lambda: print(i) for i in range(3)]
for p in printers:
    p()

練習 4（keyword-only API 設計） 設計一個 create_account(name, *, is_admin=False, send_email=True)。說明為什麼把 is_admin、send_email 設計成 keyword-only 比讓它們能用位置傳更安全，並寫出一個會被 Python 擋下來的錯誤呼叫範例。

練習 5（型別提示 + Callable） 把練習 1 的 dispatch table 加上完整型別提示，包含 dict 的型別（用 Callable[...]）與 dispatch 函式的簽章。接著故意傳一個型別不符的參數，觀察「程式照常執行」這件事，再（若已裝）用 mypy 跑一次，比較執行期與靜態檢查的差別。

練習 6（docstring 與 PEP 257） 為練習 4 的 create_account 補一段 Google style 的多行 docstring：祈使句摘要、Args/Returns/Raises 區塊。要求（a）不要在 Args 重抄型別（型別交給型別提示），只寫語意與邊界；（b）寫完後在 REPL 用 help(create_account) 與 create_account.__doc__ 各看一次，說明兩者差在哪（提示：inspect.getdoc 與縮排）；（c）把摘要句故意改成第二個陳述式（前面加一行 x = 1），再看 __doc__，解釋為什麼變成 None。
