第 5 章：Pythonic 慣用法（Idiomatic Python）

你已經會用 Python 寫出能跑的程式碼了。但「能跑」和「Pythonic」是兩回事。一個 Java 工程師剛轉 Python 時，最容易寫出的就是「用 Python 語法表達 Java 思維」的程式碼——for i in range(len(items))、先 if key in d 再取值、滿坑滿谷的 .stream().map().filter().collect() 等價物。這些都能跑，但 Python 老手看了會皺眉。本章的目標只有一個：把你腦中那套「Java 直譯」的反射，替換成 Python 社群真正在用的慣用法。學會這些，你的程式碼會更短、更快、也更不容易出錯。

5.1 comprehension：取代 Stream 樣板的核心武器

如果只能學一個 Pythonic 慣用法，那就是 comprehension（綜合運算式，或譯「推導式」）。它是 Python 處理「從一個序列產生另一個序列」的標準答案，地位等同於 Java 8 之後的 Stream API——但語法上更直接。

5.1.1 list comprehension：map + filter 的合體

先看一個典型任務：取出一串數字裡的偶數，各自平方。

java 
// Java：Stream API
List<Integer> nums = List.of(1, 2, 3, 4, 5, 6);
List<Integer> result = nums.stream()
        .filter(n -> n % 2 == 0)
        .map(n -> n * n)
        .collect(Collectors.toList());
// [4, 16, 36]
python 
# Python：list comprehension
nums = [1, 2, 3, 4, 5, 6]
result = [n * n for n in nums if n % 2 == 0]
# [4, 16, 36]

對比一下 Java 工程師最初會寫的「直譯版本」：

python 
# ❌ Java 式直譯：能跑，但不 Pythonic
result = []
for n in nums:
    if n % 2 == 0:
        result.append(n * n)

comprehension 把這三行壓成一行，而且讀法跟英文一樣自然：[運算式 for 變數 in 來源 if 條件]。注意它的順序——先寫輸出運算式，再寫迴圈，最後寫過濾。這跟 Stream 的「資料流由左往右」順序相反，是 Java 工程師要適應的第一個點。

🔁 Java 對照：.stream().filter(...).map(...).collect(toList()) ≈ [map運算式 for x in 來源 if filter條件]。comprehension 不需要 .stream() 開頭，也不需要 .collect() 收尾——容器型別由外層括號決定。

Stream API comprehension 說明
.map(f) [f(x) for x in xs] 轉換
.filter(p) [x for x in xs if p(x)] 過濾
.map().filter() [f(x) for x in xs if p(x)] 組合
Collectors.toList() 外層 [ ] 收集成 list
Collectors.toSet() 外層 { } 收集成 set
Collectors.toMap() {k: v for ...} 收集成 dict

5.1.2 dict 與 set comprehension

換掉外層括號，就換掉容器型別。這是 comprehension 最優雅的地方：

python 
# dict comprehension：對照 Collectors.toMap
chars = "abc"
ords = {c: ord(c) for c in chars}
# {'a': 97, 'b': 98, 'c': 99}

# set comprehension：自動去重
words = ["hi", "yo", "sup", "ok"]
lengths = {len(w) for w in words}
# {2, 3}
java 
// Java 等價：toMap 的樣板冗長得多
Map<Character, Integer> ords = "abc".chars()
        .boxed()
        .collect(Collectors.toMap(c -> (char) c.intValue(), c -> c));

dict comprehension 反轉一個對應表是經典招式：

python 
>>> en_to_zh = {"cat": "貓", "dog": "狗"}
>>> zh_to_en = {v: k for k, v in en_to_zh.items()}
>>> zh_to_en
{'貓': 'cat', '狗': 'dog'}

⚠️ 陷阱：{} 是空 dict，不是空 set。空 set 只能用 set() 建立。但 set comprehension {x for x in ...} 因為有 for，直譯器分得出來，沒問題。

5.1.3 多重 for 與條件運算式

comprehension 支援多個 for（等同巢狀迴圈）與三元條件運算式：

python 
# 多重 for：笛卡兒積，順序與巢狀 for 一致
pairs = [(x, y) for x in range(1, 4) for y in range(1, 4)]
# [(1,1), (1,2), (1,3), (2,1), ...]

# 條件運算式在「運算式位置」：取代 map 中的三元
labels = ["even" if n % 2 == 0 else "odd" for n in range(4)]
# ['even', 'odd', 'even', 'odd']

關鍵區別要記牢：

  • if 放在最後（過濾用）→ 決定某個元素要不要被收進來。
  • ... if cond else ... 放在最前面的運算式位置（轉換用）→ 每個元素都產出，只是值不同。

兩者可以並存：

python 
# 先過濾掉 0，剩下的標記正負
result = ["pos" if n > 0 else "neg" for n in [-2, 0, 3, -1] if n != 0]
# ['neg', 'pos', 'neg']

5.1.4 何時用、何時別硬塞

comprehension 強大，但不是萬靈丹。我給你一條工程上的判準：

💡 Pythonic 重點：一個 comprehension 應該能一眼讀懂。一旦出現兩層以上的 for 加上條件，或運算式本身需要中間變數、副作用、例外處理，就請改回普通迴圈。可讀性永遠優先於「塞進一行」。

python 
# ❌ 過度炫技：三層巢狀，沒人看得懂
matrix = [[1, 2], [3, 4], [5, 6]]
flat = [x for row in matrix for x in row if x % 2 == 0 if x > 2]

# ✅ 簡單的攤平還能接受
flat = [x for row in matrix for x in row]
# [1, 2, 3, 4, 5, 6]

# ❌ 帶副作用是反模式：comprehension 是用來「產生值」，不是跑迴圈
[print(x) for x in range(3)]   # 別這樣，會產生一個沒人要的 [None, None, None]

# ✅ 純粹為了副作用就用迴圈
for x in range(3):
    print(x)

Java 的 Stream 也有同樣的戒律：別在 forEach 裡藏複雜邏輯，別在 map 裡製造副作用。原則相通。

5.2 generator expression 與惰性求值

list comprehension 會立刻把所有元素算出來、塞進記憶體。但很多時候你只是要「一個一個處理」，不需要整份結果同時存在。這時候就輪到 generator expression（產生器運算式）登場。

5.2.1 語法只差一個括號，行為天差地別

python 
# list comprehension：用 [ ]，立刻算完全部，回傳 list
squares_list = [x * x for x in range(1_000_000)]

# generator expression：用 ( )，惰性求值，回傳 generator 物件
squares_gen = (x * x for x in range(1_000_000))
python 
>>> import sys
>>> sys.getsizeof([x for x in range(10000)])   # list：實打實佔記憶體
85176
>>> sys.getsizeof((x for x in range(10000)))    # generator：固定大小
200

generator 不會預先算好任何東西，它只記得「下一步要做什麼」，每次跟它要一個元素，它才算一個：

python 
>>> g = (x * x for x in range(5))
>>> g
<generator object <genexpr> at 0x107751cb0>
>>> next(g)
0
>>> next(g)
1

🔁 Java 對照：generator expression 對應 Stream 的惰性特性——中間操作（map/filter）不會立刻執行，要等終端操作（collect/forEach）才觸發。差別是 Java 的 Stream 用過即丟（不能重複迭代），Python 的 generator 也是一樣——迭代到底就空了。

python 
>>> g = (x for x in range(3))
>>> list(g)
[0, 1, 2]
>>> list(g)   # 第二次：空的，generator 已耗盡
[]

5.2.2 何時省下來的不只是記憶體

當下游操作會「短路」時，惰性求值連 CPU 都省下。比方說只想知道「有沒有任何一筆超過門檻」：

python 
# 假設 read_huge_log() 是會吐出上億行的來源
# generator：只算到第一個符合條件就停，後面根本沒碰
has_error = any(line.startswith("ERROR") for line in read_huge_log())

# ❌ 若先用 list comprehension，會先把上億行全部讀進記憶體再判斷
has_error = any([line.startswith("ERROR") for line in read_huge_log()])

第二種寫法不只浪費記憶體，還喪失了 any 的短路優勢。

generator 還能表達無限序列，這是 list 永遠做不到的：

python 
def naturals():
    n = 0
    while True:
        yield n
        n += 1

# 配合惰性消費：只取前 5 個，不會無限執行
import itertools
first_five = list(itertools.islice(naturals(), 5))
# [0, 1, 2, 3, 4]

這裡用到的 yield 與 itertools，正是第 6 章「迭代器與產生器」的主角，本章先讓你建立「Python 的序列預設可以是惰性的」這個直覺。

💡 Pythonic 重點：當 generator 直接當作函式的唯一引數時，可以省掉一層括號：sum(x * x for x in nums) 而不必寫 sum((x * x for x in nums))。這是很常見的慣用寫法。

5.3 內建高階工具：別再寫 C 風格迴圈

Python 的內建函式（builtins）提供了一整套「跟 index 說再見」的工具。Java 工程師寫迴圈時還帶著 for (int i = 0; ...) 的肌肉記憶，這裡要徹底改掉。

5.3.1 enumerate：要 index 也要 element

java 
// Java
for (int i = 0; i < words.size(); i++) {
    System.out.println(i + ": " + words.get(i));
}
python 
# ❌ Java 式直譯：range(len(...)) 是 Python 程式碼壞味道的招牌
for i in range(len(words)):
    print(f"{i}: {words[i]}")

# ✅ Pythonic：enumerate 同時給你 index 與 element
for i, word in enumerate(words):
    print(f"{i}: {word}")

enumerate 還能指定起始值（例如做 1-based 編號）：

python 
>>> for rank, name in enumerate(["金", "銀", "銅"], start=1):
...     print(rank, name)
1 金
2 銀
3 銅

5.3.2 zip：平行迭代（Java 沒有的對應）

要同時走訪兩個（以上）序列，Java 得用 index 手動對齊；Python 用 zip 一次搞定：

python 
names = ["Alice", "Bob", "Carol"]
scores = [90, 85, 95]

for name, score in zip(names, scores):
    print(f"{name}: {score}")

zip 配 dict comprehension，是把兩個平行 list 併成 dict 的標準招式：

python 
>>> dict(zip(["a", "b", "c"], [1, 2, 3]))
{'a': 1, 'b': 2, 'c': 3}

⚠️ 陷阱：zip 以最短的序列為準，多的會被默默丟掉。Python 3.10+ 加了 zip(..., strict=True)，長度不一時直接拋 ValueError——處理「理應等長」的資料時請務必加上，避免靜默吃掉資料。

python 
>>> list(zip([1, 2, 3], ["a", "b"]))           # 默默截斷
[(1, 'a'), (2, 'b')]
>>> list(zip([1, 2, 3], ["a", "b"], strict=True))  # 這會出錯
ValueError: zip() argument 2 is shorter than argument 1

5.3.3 any / all：短路的全稱與存在判斷

java 
// Java Stream
boolean hasNeg = nums.stream().anyMatch(n -> n < 0);
boolean allPos = nums.stream().allMatch(n -> n > 0);
python 
# Python：搭配 generator expression，同樣短路
has_neg = any(n < 0 for n in nums)
all_pos = all(n > 0 for n in nums)

any/all 對應 anyMatch/allMatch，且同樣具備短路語意：any 遇到第一個 True 就回傳、all 遇到第一個 False 就回傳。注意空序列的邊界行為（與數學定義一致，跟 Java 相同）：

python 
>>> any([])   # 沒有任何 True
False
>>> all([])   # 空集合上「全部成立」恆為真（vacuous truth）
True

5.3.4 sorted 與 key：Comparator 的輕量替身

Java 的排序靠 Comparator；Python 靠一個 key 函式——傳入「如何從元素取出排序依據」，而不是「如何兩兩比較」。這種「Schwartzian transform」風格更不容易寫錯（不會有 compare 回傳值正負搞反的問題）。

java 
// Java：Comparator
List<String> sorted = words.stream()
        .sorted(Comparator.comparingInt(String::length))
        .collect(Collectors.toList());
python 
# Python：key 函式只說「拿什麼來比」
>>> sorted("the quick brown fox".split(), key=len)
['the', 'fox', 'quick', 'brown']

多層排序（先比這個、再比那個）在 Java 要串 thenComparing；Python 直接回傳一個 tuple，按元素逐一比較：

python 
people = [("bob", 30), ("alice", 30), ("carol", 25)]

# 先按年齡（升冪），同齡再按名字（字母序）
result = sorted(people, key=lambda p: (p[1], p[0]))
# [('carol', 25), ('alice', 30), ('bob', 30)]

降冪用 reverse=True。若要「某欄升冪、某欄降冪」混排，對數值欄位取負號是常見技巧：

python 
# 年齡降冪、同齡名字升冪
sorted(people, key=lambda p: (-p[1], p[0]))

🔁 Java 對照：Comparator.comparing(A).thenComparing(B) ≈ key=lambda x: (x.A, x.B)。Python 的 tuple 天生支援字典序比較，所以多鍵排序不需要任何額外 API。

reversed、min/max 也吃同樣的 key：

python 
>>> longest = max(words, key=len)            # 最長的字
>>> youngest = min(people, key=lambda p: p[1])  # 最年輕的人
>>> list(reversed(range(3)))                  # 反向迭代，不複製整份資料
[2, 1, 0]
Java Python 說明
Comparator.comparing(f) sorted(xs, key=f) 單鍵排序
.thenComparing(g) key=lambda x: (f(x), g(x)) 多鍵排序
.reversed() reverse=True 降冪
Collections.max(xs, cmp) max(xs, key=f) 取極值
stream().anyMatch(p) any(p(x) for x in xs) 存在判斷
stream().allMatch(p) all(p(x) for x in xs) 全稱判斷

5.4 EAFP vs LBYL：先做再說，出錯再接

這是 Java 與 Python 文化差異最深的一處。Java 受 checked exception 影響，養成了「先檢查、再行動（Look Before You Leap，LBYL）」的習慣。Python 社群信奉的卻是相反的「先做、錯了再道歉（Easier to Ask Forgiveness than Permission，EAFP）」。

5.4.1 兩種風格的對照

任務：從 dict 取值，不存在就給預設。

python 
# LBYL（Java 式思維）：先檢查 key 在不在
if "name" in config:
    name = config["name"]
else:
    name = "anonymous"

# EAFP（Pythonic）：直接取，沒有就接住例外
try:
    name = config["name"]
except KeyError:
    name = "anonymous"

當然，這個特定例子用 config.get("name", "anonymous") 最乾淨——但重點是理解兩種哲學。再看一個無法用 .get 簡化的例子：解析一個可能格式不對的字串。

python 
# LBYL：要把「合法整數」的規則重新實作一遍，又臭又長又不完整
def parse_or_default(s, default=0):
    if s.lstrip("-").isdigit():   # 還沒處理 '+', 空白, 底線分隔...
        return int(s)
    return default

# EAFP：讓 int() 自己決定什麼是合法的，它本來就最懂
def parse_or_default(s, default=0):
    try:
        return int(s)
    except ValueError:
        return default

💡 Pythonic 重點：EAFP 不是「鼓勵濫用例外」，而是「不要重複驗證一件函式本身已經會驗證的事」。LBYL 的檢查往往無法窮舉所有失敗情況，還會跟真正的操作邏輯重複，形成兩處要同步維護的真相來源。

5.4.2 並行下的關鍵理由：避免 race condition

EAFP 還有一個 Java 工程師會立刻 buy-in 的硬道理：LBYL 的「檢查」與「行動」之間有空窗，在並行環境下是 TOCTOU（Time-of-check to time-of-use）race condition。

python 
# ❌ LBYL：檢查與使用之間，別的 thread 可能刪掉這個 key
if key in shared_dict:
    # ← 這個瞬間，另一個 thread 可能執行 del shared_dict[key]
    value = shared_dict[key]   # 可能炸 KeyError

# ✅ EAFP：取值與失敗判斷合併成單一 try，沒有檢查與使用之間的空窗
#   （CPython 預設 build 下單次 __getitem__ 亦受 GIL 保護；free-threaded
#    無 GIL build 則需自行加鎖，才能保證複合操作的原子性）
try:
    value = shared_dict[key]
except KeyError:
    value = None

檔案系統操作也一樣經典：if os.path.exists(path): open(path) 之間檔案可能被刪。EAFP 把「檢查」和「使用」合而為一，從根本上消滅這個空窗。

⚠️ 陷阱：EAFP 的 except 子句要精準。只接住你預期的例外型別（KeyError、ValueError…），別寫成 except Exception 或裸 except:——那會把 KeyboardInterrupt、程式 bug、拼錯的屬性名全部吞掉，變成最難 debug 的那種程式碼。例外處理的細節（exception chaining、else/finally）留到第 12 章。

5.5 duck typing：會叫的就是鴨子

「如果牠走路像鴨子、叫起來像鴨子，那牠就是鴨子。」

Java 是名義型別（nominal typing）：一個物件能不能傳進某方法，取決於它有沒有 implements 那個介面、是不是那個類別的子型。Python 是鴨子型別（duck typing）：只看物件有沒有那個行為，不看它的型別名稱或繼承關係。

5.5.1 對照：介面 vs 行為

java 
// Java：呼叫端要求一個「具名」的型別契約
interface Readable {
    String read();
}
String firstLine(Readable source) {
    return source.read().split("\n")[0];
}
// 只有 implements Readable 的類別才能傳進來
python 
# Python：函式不在意你是什麼型別，只在意你有沒有 .read()
def first_line(source):
    return source.read().split("\n")[0]

這個 first_line 可以吃任何有 .read() 方法的物件——檔案、io.StringIO、io.BytesIO、網路 response、你自己寫的類別……它們之間沒有任何共同基底類別或介面：

python 
>>> import io
>>> first_line(io.StringIO("hello\nworld"))
'hello'

>>> class FakeFile:
...     def read(self):
...         return "fake line 1\nfake line 2"
>>> first_line(FakeFile())   # 跟 StringIO 八竿子打不著，照樣能用
'fake line 1'

在 Java 你得讓 FakeFile 去 implements Readable（或繼承某個共同基底），編譯器才放行。Python 不用——這也是為什麼 Python 寫測試替身（test double）/ mock 特別輕鬆：你只要做一個「長得像」的物件就好，不必繼承任何東西。

🔁 Java 對照：duck typing ≈ 「結構型別（structural typing）」。Java 直到有了 var 和某些泛型場景才稍微沾邊，但本質上仍是名義型別。最接近的類比反而是 Go 的隱式 interface 實作。

💡 Pythonic 重點：duck typing 讓 API 天然解耦——函式只依賴「需要的行為」，而非「具體的型別」。代價是這個契約只存在於文件與執行期，編譯器不幫你檢查。Python 3.8+ 用 typing.Protocol 把「結構型別」變成可被 mypy 靜態檢查的東西，等於「有型別安全的鴨子型別」——這是第 8 章的重頭戲。

別忘了，這正是為什麼第 3 章強調「函式是一等公民」、為什麼 key= 可以吃任何 callable：Python 全程靠行為驅動，不靠型別宣告。

5.6 with 與 context manager：try-with-resources 的進階版

資源（檔案、連線、鎖）用完要釋放，這你太熟了。Java 7 的 try-with-resources 解決了手動 finally 的繁瑣；Python 的對應物是 with 陳述式，而且機制更通用——它能管理的不只是「資源」，而是任何「需要成對進入/離開」的狀態。

5.6.1 基本對照

java 
// Java：try-with-resources，要求 AutoCloseable
try (var reader = Files.newBufferedReader(path)) {
    return reader.readLine();
}   // reader.close() 自動呼叫，即使丟例外
python 
# Python：with 陳述式
with open(path) as f:
    return f.readline()
# f.close() 自動呼叫，即使丟例外
python 
# ❌ Java 工程師偶爾會寫的「直譯版」：手動 try/finally，沒必要
f = open(path)
try:
    return f.readline()
finally:
    f.close()

with 保證 __exit__ 一定執行——不管區塊正常結束、return、還是拋例外。一次管理多個資源時，用逗號分隔（取代 Java 的 ;）：

python 
with open("in.txt") as src, open("out.txt", "w") as dst:
    dst.write(src.read())

5.6.2 協定：`__enter__` 與 `__exit__`

任何實作了 __enter__ 和 __exit__ 的物件都能用在 with，這就是 context manager 協定——對應 Java 的 AutoCloseable，但更強：__enter__ 可以回傳值（給 as 用），__exit__ 收到例外資訊、還能決定要不要吞掉例外。

寫一個計時器，示範「進入時記錄、離開時結算」這種非資源用途：

python 
import time

class Timer:
    def __init__(self, label):
        self.label = label

    def __enter__(self):
        self.start = time.perf_counter()
        return self            # 這個回傳值會綁到 as 後面的變數

    def __exit__(self, exc_type, exc_val, exc_tb):
        self.elapsed = time.perf_counter() - self.start
        print(f"[{self.label}] {self.elapsed:.4f}s")
        return False           # 回傳 False（或 None）：不吞例外，讓它繼續往外拋
python 
>>> with Timer("計算") as t:
...     total = sum(range(1_000_000))
[計算] 0.0039s          # 實際耗時因機器而異
>>> t.elapsed     # 離開後仍可讀取，與上面印出的是同一個值
0.0039...

__exit__ 的三個參數在正常結束時都是 None；發生例外時則帶著例外的型別、值與 traceback。return True 會「吞掉」例外（謹慎使用），return False 讓例外照常傳播。

5.6.3 contextlib.contextmanager：用 generator 偷懶

寫一整個類別只為了 __enter__/__exit__ 太囉嗦。contextlib.contextmanager 讓你用一個 generator 函式搞定——yield 之前是「進入」邏輯、yield 之後是「離開」邏輯，yield 出去的值就是 as 拿到的東西。

python 
from contextlib import contextmanager
import time

@contextmanager
def timer(label):
    start = time.perf_counter()
    try:
        yield                          # 控制權交給 with 區塊
    finally:
        elapsed = time.perf_counter() - start
        print(f"[{label}] 花費 {elapsed:.4f}s")
python 
>>> with timer("計算"):
...     sum(range(1_000_000))
[計算] 花費 0.0040s

那個 try/finally 很重要：它保證即使 with 區塊內拋例外，離開邏輯照樣執行。再看一個「暫時切換狀態、結束自動還原」的實用範例：

python 
import os
from contextlib import contextmanager

@contextmanager
def temporary_env(key, value):
    """暫時設定環境變數，離開時還原（無論成功或失敗）"""
    original = os.environ.get(key)
    os.environ[key] = value
    try:
        yield
    finally:
        if original is None:
            os.environ.pop(key, None)
        else:
            os.environ[key] = original
python 
>>> os.environ.get("MODE")          # 一開始沒有
>>> with temporary_env("MODE", "test"):
...     print(os.environ["MODE"])
test
>>> os.environ.get("MODE")          # 離開後自動還原成 None

💡 Pythonic 重點：context manager 不只是「關檔案」。任何「進入時做 A、離開時做 B」的成對動作——加鎖/解鎖、開啟/關閉交易、切換/還原設定、靜音/恢復日誌——都該包成 context manager。這比 Java 的 try-with-resources 用途廣得多，因為它脫離了「資源」這個狹義概念。標準庫的 threading.Lock、open、sqlite3.connect 全都是 context manager。

5.7 海象運算子 := 與 f-string

最後兩個語法糖，都是讓「Java 直譯寫法」變短的利器。

5.7.1 walrus operator `:=`：賦值兼運算式

Python 的 = 是陳述式（statement），不回傳值——所以你不能在 if、while 的條件裡賦值（這也避免了 Java/C 裡 if (x = 5) 誤寫成賦值的經典 bug）。但有時你真的需要「賦值並馬上用」，3.8 引入的海象運算子 :=（長得像海象的眼睛和獠牙）就是為此而生。

最常見的場景是讀取迴圈。對照 Java 的慣用寫法：

java 
// Java：在 while 條件裡賦值並判斷
String line;
while ((line = reader.readLine()) != null) {
    process(line);
}
python 
# ❌ 沒有 walrus 時，得把讀取寫兩次或用 break，很彆扭
while True:
    line = f.readline()
    if not line:
        break
    process(line)

# ✅ 有了 walrus：賦值兼判斷，一行到位，神似 Java
while (line := f.readline()):
    process(line)

walrus 在 comprehension 裡也很有用——當「過濾條件」需要用到「一個算起來很貴的中間值」，又不想算兩次時：

python 
# 計算每個值的平方，只保留平方大於 10 的——但只想算一次平方
data = [1, 2, 3, 4, 5, 6]
result = [y for x in data if (y := x * x) > 10]
# [16, 25, 36]

這裡 y := x * x 把平方算出來綁到 y，過濾用 y、輸出也用 y，只算一次。沒有 walrus 的話你得在運算式和條件裡各算一次 x * x。

⚠️ 陷阱：別過度使用 :=。它的價值在「避免重複計算」或「簡化讀取迴圈」。如果只是普通賦值，用一般的 = 在前一行，可讀性更好。把它當成精準的手術刀，不是日常菜刀。

5.7.2 f-string：格式化的最終答案

字串格式化請永遠用 f-string（formatted string literal，3.6+）。忘掉 % 和 .format()——它們是過時寫法。

java 
// Java：String.format 或文字區塊（text block）
String msg = String.format("使用者 %s 餘額 %.2f", name, balance);
python 
# Python：f-string，變數直接內嵌
msg = f"使用者 {name} 餘額 {balance:.2f}"

f-string 在 {} 裡可以放任何運算式，並用 : 後的格式規格（format spec）控制呈現：

python 
>>> import math
>>> f"{math.pi:.2f}"          # 小數兩位
'3.14'
>>> f"{1234567:,}"            # 千分位分隔
'1,234,567'
>>> f"{255:08b}"             # 8 位、補零、二進位
'11111111'
>>> f"{42:>6}"               # 靠右對齊，寬度 6
'    42'
>>> f"{3.14159:>10.2f}"      # 對齊 + 精度組合
'      3.14'
>>> name = "alice"
>>> f"{name.upper()}"        # 直接放運算式（這裡是方法呼叫）
'ALICE'

格式規格速查：</>/^ 是左/右/置中對齊，0 是補零，.2f 是浮點兩位，, 是千分位，% 是百分比，b/o/x 是二/八/十六進位。

3.8+ 還有一個 debug 神器——在運算式後面加 =，會把「運算式原文 + 值」一起印出來，丟掉一堆 print("x =", x) 的樣板：

python 
>>> x = 42
>>> f"{x=}"
'x=42'
>>> price, qty = 19.99, 3
>>> print(f"{price * qty=:.2f}")
price * qty=59.97
Java Python f-string 說明
String.format("%.2f", x) f"{x:.2f}" 小數兩位
String.format("%,d", n) f"{n:,}" 千分位
String.format("%05d", n) f"{n:05}" 補零
String.format("%-10s", s) f"{s:<10}" 左對齊
text block """...""" f-string + 三引號 多行內嵌

💡 Pythonic 重點：f-string 不只更短，還更快——格式化在編譯期就解析好了，執行期效能優於 .format() 與 %。再加上變數內嵌讓「字串模板與資料」緊鄰，可讀性遠勝把參數拖在後面的 String.format。

本章小結

  1. comprehension 是 Stream API 的直接替身：[f(x) for x in xs if p(x)] 一行表達 map+filter，外層括號決定 list/set/dict，不需 .stream()...collect() 樣板。但巢狀過深或帶副作用時，請退回普通迴圈。
  2. ( ) 是惰性的 generator expression：省記憶體、配合 any/sum/islice 還能短路省 CPU，是處理大資料與無限序列的基礎（詳見第 6 章）。
  3. 跟 range(len(...)) 說再見：enumerate 取 index、zip 平行迭代（記得 strict=True）、any/all 做短路判斷、sorted(key=...) 取代 Comparator（多鍵就回傳 tuple）。
  4. EAFP 優於 LBYL：先做、出錯再用精準的 except 接住，避免重複驗證，也消除並行下「檢查與使用之間」的 race condition。
  5. duck typing 看行為不看型別：函式只依賴「有沒有那個方法」，天然解耦、好測試；Protocol 能補回靜態檢查（第 8 章）。
  6. with 比 try-with-resources 更通用：靠 __enter__/__exit__ 協定管理任何「成對進入/離開」的狀態，contextlib.contextmanager 讓你用一個 generator 加 try/finally 快速自訂。
  7. := 賦值兼運算式：簡化讀取迴圈與「避免重複計算」的 comprehension，但別濫用。
  8. f-string 是格式化的唯一正解：內嵌運算式、豐富的對齊/補零/進位格式規格，加上 {x=} debug 語法，效能與可讀性雙贏。
