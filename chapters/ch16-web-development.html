第 16 章：Web 開發

你寫 Web 後端的肌肉記憶大概是這樣的：開一個 @RestController、灑一堆 @GetMapping／@PostMapping、用 @Valid 加 @NotNull／@Size 拼出驗證、靠 springdoc 生 Swagger、再用 JPA repository 接資料庫——整套 Spring Boot 的儀式你閉著眼睛都會。Python 這邊的主流是 FastAPI，它做的事情幾乎一樣，但底層哲學顛覆了兩個你習以為常的直覺：第一，「型別提示」不再只是給 IDE 看的裝飾，而是驅動執行期驗證、序列化、自動文件的單一真實來源（single source of truth）；第二，依賴注入不靠容器掃描的魔法，而是明確寫在函式參數裡的函式呼叫。這章我們用一個貫穿全章的 items CRUD API，把 FastAPI、pydantic、SQLAlchemy、Depends 串成一個能實際跑、能被測試的服務。

本章所有範例都可用 uv run --with 'fastapi[standard]' --with httpx --with sqlalchemy --with pydantic-settings python xxx.py 實際執行；FastAPI 範例一律用 TestClient 在記憶體內發請求，不必真的啟動伺服器；資料庫用 in-memory SQLite，可實跑。

FastAPI 入門：型別提示驅動的殺手鏈

Java 怎麼做 → Python 怎麼做

在 Spring Boot，一個帶驗證的 endpoint 大概長這樣：

java 
// Java：Spring Boot + Bean Validation
@RestController
@RequestMapping("/items")
public class ItemController {

    @PostMapping
    @ResponseStatus(HttpStatus.CREATED)
    public ItemOut create(@Valid @RequestBody ItemIn item) {
        return service.create(item);
    }

    @GetMapping("/{id}")
    public ItemOut get(@PathVariable int id) { ... }

    @GetMapping
    public List<ItemOut> list(@RequestParam(defaultValue = "10") int limit) { ... }
}

// DTO 上掛 annotation 才有驗證
public record ItemIn(@NotBlank @Size(max = 50) String name,
                     @Positive double price,
                     List<String> tags) {}

FastAPI 的對應寫法：

python 
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel, Field

app = FastAPI(title="Items API")  # ASGI app 物件，等同 Spring Boot 的應用入口

class ItemIn(BaseModel):                      # request body 的 schema（承接第 9 章）
    name: str = Field(min_length=1, max_length=50)
    price: float = Field(gt=0)
    tags: list[str] = []

class ItemOut(ItemIn):
    id: int

_db: dict[int, ItemOut] = {}
_next_id = 1

@app.post("/items", response_model=ItemOut, status_code=201)
def create_item(item: ItemIn) -> ItemOut:     # 參數型別是 ItemIn → 自動當 request body
    global _next_id
    out = ItemOut(id=_next_id, **item.model_dump())
    _db[_next_id] = out
    _next_id += 1
    return out

@app.get("/items/{item_id}", response_model=ItemOut)
def get_item(item_id: int) -> ItemOut:        # item_id: int → path 參數自動轉型
    if item_id not in _db:
        raise HTTPException(status_code=404, detail="Item not found")
    return _db[item_id]

@app.get("/items")
def list_items(limit: int = 10) -> list[ItemOut]:  # 有預設值 → query 參數 ?limit=
    return list(_db.values())[:limit]

對照表把映射關係講清楚：

Spring Boot FastAPI 說明
@RestController 類別 app = FastAPI() + 函式 FastAPI 路由掛在函式上，不需要類別
@PostMapping("/items") @app.post("/items") decorator 即路由註冊（詳見第 10 章）
@PathVariable int id 路徑 {item_id} + 參數 item_id: int 名字對上即綁定，型別即轉型規則
@RequestParam int limit 函式參數有預設值 limit: int = 10 有預設值 → query；無 → 必填 query
@RequestBody ItemIn 參數型別是 BaseModel 子類 pydantic model 自動當 body
@Valid + @NotBlank/@Positive Field(min_length=..., gt=...) 驗證內建在 model，不需另一層 annotation
@ResponseStatus(CREATED) status_code=201
springdoc-openapi 內建，零設定 /docs（Swagger UI）、/openapi.json

怎麼判斷一個參數是 path、query 還是 body？

FastAPI 的規則是看型別與位置，比 Spring 的 annotation 少很多儀式：

  • 名字出現在路徑樣板裡（{item_id}）→ path 參數。
  • 型別是 pydantic BaseModel 子類 → request body。
  • 其餘的簡單型別（int/str/bool...）→ query 參數；有預設值就是選填，沒有就是必填。

這套「慣例優於設定」的推斷，讓你不必像 Java 那樣每個參數都掛一個 @PathVariable／@RequestParam／@RequestBody。

那條殺手鏈：型別提示 → 自動驗證 → 自動文件

這是 FastAPI 真正顛覆 Java 直覺的地方。在 Spring，「DTO 定義」「驗證規則」「OpenAPI schema」是三件分開維護、靠 annotation 黏在一起的事；在 FastAPI，它們是同一個 pydantic model 的三種投影。你寫一次 ItemIn，就同時得到：

  1. 執行期解析與驗證：JSON → ItemIn 物件，型別不符、缺欄位、違反 Field 約束時自動回 422 Unprocessable Entity，附結構化錯誤。
  2. 回應序列化：response_model 控制輸出 schema，多餘欄位自動濾掉（資安上很重要，避免不小心把密碼欄位回給前端）。
  3. OpenAPI 文件：/openapi.json 與 /docs 自動生成，欄位型別、約束、範例全都有。

用 TestClient 實際跑一遍（這是 starlette 包的記憶體內 client，不啟動真正的 server，等同 Spring 的 MockMvc／WebTestClient）。以下輸出皆為實測：

python 
from fastapi.testclient import TestClient

client = TestClient(app)

r = client.post("/items", json={"name": "keyboard", "price": 89.9, "tags": ["gear"]})
# r.status_code == 201
# r.json() == {'name': 'keyboard', 'price': 89.9, 'tags': ['gear'], 'id': 1}

r = client.get("/items/1")
# 200, {'name': 'keyboard', 'price': 89.9, 'tags': ['gear'], 'id': 1}

r = client.get("/items/999")
# 404, {'detail': 'Item not found'}

驗證失敗的 422 回應長這樣（name 空字串違反 min_length、price 負數違反 gt）：

python 
r = client.post("/items", json={"name": "", "price": -5})
# r.status_code == 422
# r.json()["detail"] ==
# [
#   {"type": "string_too_short", "loc": ["body", "name"],
#    "msg": "String should have at least 1 character",
#    "input": "", "ctx": {"min_length": 1}},
#   {"type": "greater_than", "loc": ["body", "price"],
#    "msg": "Input should be greater than 0",
#    "input": -5, "ctx": {"gt": 0.0}}
# ]

loc 精確指出錯在哪個欄位（["body", "name"]），這比 Spring 預設那團 MethodArgumentNotValidException 訊息結構化得多，前端可以直接拿來標紅哪一格。

🔁 Java 對照：Spring 要 @Valid + Bean Validation annotation + springdoc 三套東西協作，才湊出「驗證 + 文件」；FastAPI 一個 pydantic model 全包。型別提示在 Java 是編譯期檢查，在 FastAPI 是執行期行為的定義來源。

💡 Pythonic 重點：把 request schema（ItemIn）和回應 schema（ItemOut）、以及資料庫 model（後面的 ItemRow）分成三個類別，不要共用一個。輸入不該含 id、回應不該含內部欄位、DB model 綁著 ORM——這三件事的形狀本來就不同。Spring 用 DTO/Entity 分離也是同一個道理，只是 FastAPI 讓你更自然地各寫一個。

⚠️ 陷阱：別把 ORM model 直接當 response_model 還順手回傳——若該物件帶有 lazy-load 的關聯欄位，序列化時可能觸發額外查詢甚至在 session 關閉後爆 DetachedInstanceError。永遠用一個乾淨的 pydantic 回應 model（搭配後面會講的 from_attributes=True）把要回的欄位明確列出來。

依賴注入：`Depends` 對照 Spring DI

函式式 DI，沒有容器掃描的魔法

Spring 的 DI 是「容器在啟動時掃 classpath、找 @Component/@Service/@Repository、建好 bean、用 @Autowired 或建構子注入」。整個過程是隱式的、靠反射與 classpath 掃描。FastAPI 的 Depends 走另一條路：依賴就是一個普通函式，你在 endpoint 的參數裡明確宣告「我需要這個函式的回傳值」，FastAPI 在每次請求時呼叫它、把結果塞進來。

java 
// Java：Spring 建構子注入
@Service
public class ItemService {
    private final ItemRepository repo;
    public ItemService(ItemRepository repo) { this.repo = repo; }  // 容器自動注入
}
python 
from typing import Annotated
from fastapi import Depends
from pydantic_settings import BaseSettings

# 最小設定物件（完整版見本章末「設定管理」一節，這裡先給本節用得到的兩個欄位）
class Settings(BaseSettings):
    app_name: str = "Items API"
    admin_token: str = "secret"

def get_settings() -> Settings:           # 依賴 = 一個普通函式
    return Settings()

@app.get("/admin/info")
def admin_info(settings: Annotated[Settings, Depends(get_settings)]) -> dict:
    return {"app": settings.app_name}     # settings 由 FastAPI 呼叫 get_settings() 注入

Annotated[Settings, Depends(get_settings)] 是現代寫法（也可以寫成舊式的 settings: Settings = Depends(get_settings)）。差別在於 Annotated 版可以被重複使用、且型別檢查器看得懂。

依賴可以巢狀：一個依賴函式本身也能宣告 Depends。下面 require_admin 同時依賴 HTTP header 與 get_settings，FastAPI 會自動建出整棵依賴樹：

python 
from fastapi import Header, HTTPException

def require_admin(
    x_token: Annotated[str, Header()],
    settings: Annotated[Settings, Depends(get_settings)],
) -> str:
    if x_token != settings.admin_token:
        raise HTTPException(401, "invalid token")
    return "admin"

@app.get("/admin/info")
def admin_info(
    user: Annotated[str, Depends(require_admin)],
    settings: Annotated[Settings, Depends(get_settings)],
) -> dict:
    return {"user": user, "app": settings.app_name}

實測（admin_token 預設 "secret"）：

python 
client.get("/admin/info").status_code                              # 422（缺 x-token header）
client.get("/admin/info", headers={"x-token": "nope"}).status_code # 401
client.get("/admin/info", headers={"x-token": "secret"}).json()
# {'user': 'admin', 'app': 'Items API'}
Spring DI FastAPI Depends 思維差異
@Autowired / 建構子注入 Depends(fn) 寫在參數 FastAPI 是顯式的，看函式簽章就知道依賴什麼
容器掃描 classpath 無容器，依賴就是被呼叫的函式 沒有「bean 找不到」的啟動期魔法錯誤
單例 scope（預設） 每請求呼叫一次（可加 cache） FastAPI 依賴預設 per-request，相同依賴在一次請求內會快取
@Transactional 切面 yield 依賴做 setup/teardown 見下

💡 Pythonic 重點：同一個依賴在一次請求內只會被呼叫一次，結果會被快取重複使用。所以上例 get_settings 即使被 require_admin 和 admin_info 都依賴，一次請求也只執行一次。要關掉這個快取行為（少見）可用 Depends(fn, use_cache=False)。

`yield` 依賴：setup / teardown 與 try-with-resources

需要「用前準備、用後清理」的資源（DB session、檔案、鎖），用 yield 依賴。yield 之前是 setup，yield 出去的值就是注入的東西，請求處理完（含發生例外）回來執行 yield 之後的 teardown。這正是 Python 版的 try-with-resources，也對應 Spring 的 @Transactional 切面或 bean 的 lifecycle callback。

python 
from collections.abc import Iterator
from sqlalchemy.orm import Session

def get_session() -> Iterator[Session]:
    session = SessionLocal()       # setup：開 session
    try:
        yield session              # 注入給 endpoint
        session.commit()           # 沒出錯 → 提交（像 @Transactional 正常結束）
    except Exception:
        session.rollback()         # 出錯 → 回滾
        raise
    finally:
        session.close()            # teardown：一定關閉

這段邏輯——「正常就 commit、例外就 rollback、無論如何都 close」——就是你在 Java 裡靠 @Transactional + 連線池替你做的事，只是這裡完全顯式、看得見每一步。

`dependency_overrides`：測試時的依賴替換

這是 Depends 設計上最甜的一點。FastAPI app 有一個 dependency_overrides 字典，可以把任何依賴換掉——測試時把 get_session 換成指向測試資料庫的版本，正式碼一行都不用改（承接第 13 章的測試替身概念）。

python 
def override_session() -> Iterator[Session]:
    s = TestSession()              # 指向獨立的測試 DB
    try:
        yield s
        s.commit()
    finally:
        s.close()

app.dependency_overrides[get_session] = override_session
# ... 跑測試 ...
app.dependency_overrides.clear()   # 收尾還原

🔁 Java 對照：app.dependency_overrides[real] = fake ≈ Spring Test 的 @MockBean／@TestConfiguration 覆寫 bean。差別是 FastAPI 這個就是一個普通 dict，沒有任何反射魔法——你看得懂它每一步在做什麼。

ORM：SQLAlchemy 2.0 對照 JPA/Hibernate

三個核心物件：Engine、Session、Model

SQLAlchemy 是 Python 的事實標準 ORM，2.0 版本的宣告式 model 用 Mapped[...] + mapped_column(...)，型別提示直接驅動欄位定義（再次呼應第 9 章），讀起來跟 dataclass 很像。對照關係：

JPA / Hibernate SQLAlchemy 2.0 說明
EntityManagerFactory Engine（create_engine） 連線工廠 / 連線池，全程式一個就好
EntityManager Session 一個工作單元（unit of work）
@Entity 類別 class X(Base) + Mapped 欄位 宣告式 model
persistence context identity map 同一 session 內同主鍵 = 同一物件
em.find(User, id) session.get(User, id) 主鍵查詢，先查 identity map
JPQL SELECT u FROM User u WHERE... select(User).where(...) 型別安全的查詢建構
em.persist(x) / em.merge session.add(x) 加入待寫入
@Transactional / tx.commit() session.commit() 提交
Spring Data JpaRepository 自己包一層或用 select SQLAlchemy 不內建 repository，但很好包
Flyway / Liquibase Alembic migration 工具，下面一句帶過

宣告 model 與基本 CRUD（in-memory SQLite 實跑）

python 
from sqlalchemy import create_engine, select, String, Float
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column, Session

class Base(DeclarativeBase):
    pass

class Item(Base):
    __tablename__ = "items"
    id: Mapped[int] = mapped_column(primary_key=True)        # 主鍵，自動遞增
    name: Mapped[str] = mapped_column(String(50))
    price: Mapped[float] = mapped_column(Float)
    description: Mapped[str | None] = mapped_column(default=None)  # 可空 → 對應 NULL

    def __repr__(self) -> str:
        return f"Item(id={self.id!r}, name={self.name!r}, price={self.price!r})"

engine = create_engine("sqlite:///:memory:")
Base.metadata.create_all(engine)     # 依 model 建表（DDL）。正式環境用 Alembic，不要靠這個

注意 Mapped[str] vs Mapped[str | None]——型別提示的可空性直接決定欄位是否 NOT NULL，不需要像 JPA 那樣寫 @Column(nullable = false)。這就是 2.0 風格的精髓：型別即 schema。

完整 CRUD，以下輸出皆為實測：

python 
with Session(engine) as session:
    session.add_all([
        Item(name="keyboard", price=89.9),
        Item(name="mouse", price=25.0, description="wireless"),
    ])
    session.commit()

    # READ：主鍵查詢
    item = session.get(Item, 1)
    print(item)                        # Item(id=1, name='keyboard', price=89.9)

    # identity map：同一 session 內、同主鍵 → 同一個 Python 物件
    same = session.get(Item, 1)
    print(item is same)                # True ← 不是相等，是同一個物件！

    # QUERY：select + where + order_by
    rows = session.execute(
        select(Item).where(Item.price > 30).order_by(Item.price)
    ).scalars().all()
    print(rows)                        # [Item(id=1, name='keyboard', price=89.9)]

    # 取單一列
    m = session.execute(
        select(Item).where(Item.name == "mouse")
    ).scalar_one()
    print(m.description)               # wireless

    # UPDATE：改屬性 + commit，靠髒檢查（dirty tracking）自動發 UPDATE
    item.price = 99.0
    session.commit()
    print(session.get(Item, 1).price)  # 99.0

幾個關鍵 API 對 Java 工程師的提醒：

  • session.execute(select(...)) 回傳 Result，不能直接迭代成 model。要拿 ORM 物件得接 .scalars()（解開單欄結果），再 .all() / .first() / .one()。這層 Result 包裝是 2.0 統一 Core 與 ORM 查詢介面的結果，一開始會覺得囉嗦，但它讓「查多欄」「查整個物件」「查聚合」走同一套 API。
  • scalar_one() = 「我預期剛好一列」，多於或少於一列就拋例外，對應 JPA 的 getSingleResult()。要「找不到回 None」用 scalar_one_or_none()。
  • UPDATE 不需要呼叫任何 save：改了 model 屬性，session 的髒檢查會在 commit（或 flush）時自動發 UPDATE。這就是 Hibernate 的 dirty checking，行為一致。

Session = 工作單元 + identity map

item is same 回傳 True 不是巧合，是 identity map 的核心保證：在同一個 session（= Hibernate 的 persistence context）裡，同一主鍵的列只會有一個 Python 物件代表它。這帶來兩個你從 Hibernate 已經知道的後果：

  1. 快取：第二次 session.get(Item, 1) 不會再查資料庫，直接回 identity map 裡的物件。
  2. 一致性：你在程式各處拿到的「id=1 的 item」是同一個物件，改了一處到處生效。

session 是短命的——一個請求、一個工作單元、用完即丟，這也是為什麼上面 get_session 依賴每次請求開新 session。把一個 session 當成 application-scope 長期共用是反模式（會累積物件、執行緒不安全），就跟你不會把 Hibernate 的 EntityManager 設成單例一樣。

📌 Alembic 一句帶過：Base.metadata.create_all() 只適合測試與 demo。正式環境的 schema 演進用 Alembic（alembic revision --autogenerate 比對 model 與資料庫差異生 migration script，alembic upgrade head 套用），就是 Python 版的 Flyway/Liquibase，有版本號、可上下遷移、進 git 控管。

把它們串起來：可測試的 items CRUD 服務

現在用一個檔案把 FastAPI + pydantic + SQLAlchemy + Depends 全部接起來，並用 TestClient 證明它能跑。這段是本章的核心交付物，可直接 uv run 執行。

python 
from collections.abc import Iterator
from fastapi import FastAPI, Depends, HTTPException
from fastapi.testclient import TestClient
from pydantic import BaseModel, Field, ConfigDict
from sqlalchemy import create_engine, select, String, Float, StaticPool
from sqlalchemy.orm import (
    DeclarativeBase, Mapped, mapped_column, Session, sessionmaker,
)

# ---------- ORM model（資料庫的形狀）----------
class Base(DeclarativeBase):
    pass

class ItemRow(Base):
    __tablename__ = "items"
    id: Mapped[int] = mapped_column(primary_key=True)
    name: Mapped[str] = mapped_column(String(50))
    price: Mapped[float] = mapped_column(Float)

# ---------- pydantic schema（API 的形狀）----------
class ItemCreate(BaseModel):                    # 輸入：沒有 id
    name: str = Field(min_length=1, max_length=50)
    price: float = Field(gt=0)

class ItemRead(BaseModel):                      # 輸出：有 id
    model_config = ConfigDict(from_attributes=True)  # 允許 model_validate 從 ORM 物件屬性讀（見下「陷阱二」）
    id: int
    name: str
    price: float

# ---------- DB setup ----------
engine = create_engine(
    "sqlite:///:memory:",
    connect_args={"check_same_thread": False},  # TestClient 在別的 thread 跑 handler
    poolclass=StaticPool,                        # 讓整個 app 共用同一條 in-memory 連線
)
SessionLocal = sessionmaker(bind=engine, expire_on_commit=False)
Base.metadata.create_all(engine)

def get_session() -> Iterator[Session]:          # yield 依賴
    session = SessionLocal()
    try:
        yield session
        session.commit()
    except Exception:
        session.rollback()
        raise
    finally:
        session.close()

# ---------- 路由 ----------
app = FastAPI(title="Items API")

@app.post("/items", response_model=ItemRead, status_code=201)
def create_item(payload: ItemCreate,
                session: Session = Depends(get_session)) -> ItemRow:
    row = ItemRow(name=payload.name, price=payload.price)
    session.add(row)
    session.flush()        # flush 取得自動產生的 id（還沒 commit）
    return row             # 回 ORM 物件，由 response_model 序列化成 ItemRead（見下「陷阱二」）

@app.get("/items/{item_id}", response_model=ItemRead)
def read_item(item_id: int, session: Session = Depends(get_session)) -> ItemRow:
    row = session.get(ItemRow, item_id)
    if row is None:
        raise HTTPException(404, "Item not found")
    return row

@app.get("/items", response_model=list[ItemRead])
def list_items(session: Session = Depends(get_session)) -> list[ItemRow]:
    return list(session.execute(select(ItemRow)).scalars().all())

接上一段 TestClient 測試，這就是「至少一段證明它能跑」的測試（承接第 13 章，這寫法可以直接搬進 pytest）：

python 
def test_items_crud() -> None:
    client = TestClient(app)

    r = client.post("/items", json={"name": "keyboard", "price": 89.9})
    assert r.status_code == 201
    assert r.json() == {"id": 1, "name": "keyboard", "price": 89.9}

    assert client.get("/items/1").json() == {"id": 1, "name": "keyboard", "price": 89.9}
    assert client.get("/items").json() == [{"id": 1, "name": "keyboard", "price": 89.9}]
    assert client.get("/items/999").status_code == 404

    r = client.post("/items", json={"name": "x", "price": 0})   # price 違反 gt=0
    assert r.status_code == 422

實測全部通過。這裡有兩個會在 Java 工程師身上扎一刀的陷阱，值得展開：

⚠️ 陷阱一：in-memory SQLite 的「no such table」。sqlite:///:memory: 預設用 SingletonThreadPool——每條執行緒一條連線，而每條連線是獨立的記憶體資料庫。create_all(engine) 在主執行緒那條連線上建了表；但 TestClient 跑 sync def handler 時，FastAPI 會把它丟到 threadpool 的另一條執行緒執行，那條執行緒拿到自己的新 in-memory 連線、上面沒有那張表，於是 INSERT 炸 no such table: items。解法是 poolclass=StaticPool，強制整個 app（不分執行緒）共用同一條 in-memory 連線。正式環境換成真資料庫（PostgreSQL）就沒這問題，但寫測試時這個坑幾乎人人踩過。

⚠️ 陷阱二：from_attributes 與 expire_on_commit 的真正作用。ItemRead 的 model_config = ConfigDict(from_attributes=True) 讓 pydantic 能從一個物件的屬性（而不是 dict）建立 schema。它的「必要」情境是你直接呼叫 ItemRead.model_validate(orm_obj)：沒開 from_attributes 會丟 ValidationError。這對應 Spring 的「Entity → DTO 映射」，只是 pydantic 用一個開關搞定，不用 MapStruct。要澄清一個常見誤解：FastAPI 透過 response_model 回傳 ORM 物件時，即使不開 from_attributes 通常也能序列化成功（FastAPI 的序列化器本來就能讀物件屬性）——但開著它是更明確、推薦的作法，也讓你能無痛改用 model_validate。

expire_on_commit=False 同理是穩健起見的建議設定，不是本範例的必要條件。為什麼本範例不設它也沒事？因為在 FastAPI/TestClient 下，事件順序是「handler return → 回應序列化讀屬性 → yield 依賴 teardown 的 commit → close」——序列化發生在 commit/close 之前，屬性都還活著，所以即使用預設 expire_on_commit=True、本範例仍回 201 且 body 正確。那它什麼時候會咬人？一旦你改成在 handler 內就 commit、之後再讀 ORM 屬性（或把物件交給背景任務處理），預設 commit 會 expire 屬性、session 關閉後再讀就爆 DetachedInstanceError——這時 expire_on_commit=False 才是救命設定。

💡 Pythonic 重點：三個類別各司其職——ItemRow 是資料庫的形狀、ItemCreate 是輸入的形狀、ItemRead 是輸出的形狀。不要為了少打字把它們合併。形狀分離換來的是：改 DB 欄位不會意外洩漏給 API、API 驗證規則跟資料庫約束解耦。

ASGI vs WSGI：對照 Servlet 容器

兩種介面，兩種世界觀

Python web 框架與 server 之間有一層標準介面，類似 Java 的 Servlet 規範把「web 框架」與「Servlet 容器（Tomcat/Jetty）」解耦。Python 有兩代這樣的介面：

  • WSGI（Web Server Gateway Interface）：同步介面。一個請求 = 一次函式呼叫，回傳前那條執行緒被佔住。對應傳統的 Servlet API + thread-per-request 的 Tomcat。Flask、傳統 Django 是 WSGI 框架。
  • ASGI（Asynchronous Server Gateway Interface）：非同步介面（承接第 14 章的 asyncio）。支援 async/await、長連線、WebSocket、伺服器推送（SSE）。FastAPI / Starlette 是 ASGI 框架。
WSGI ASGI
介面性質 同步 callable async callable（協程）
Java 類比 傳統 Servlet + Tomcat（thread-per-request） Servlet 3.1+ 非同步 / 反應式（WebFlux）
並行模型 一請求佔一條執行緒 事件迴圈，單執行緒處理大量並行連線
WebSocket / SSE 不支援 原生支援
代表框架 Flask、傳統 Django FastAPI、Starlette、Django(3.0+ async)
對應 server gunicorn（sync worker）、uWSGI uvicorn、hypercorn、gunicorn+uvicorn worker

關鍵釐清：`async def` 不會讓 CPU bound 變快

這是第 14 章 GIL 那一章的直接延伸，也是最常被誤解的點。FastAPI 同時支援 def（同步）與 async def（非同步）的 endpoint，行為不同：

python 
@app.get("/sync")
def sync_handler() -> dict:           # 同步 handler
    return {"mode": "在 threadpool 執行緒裡跑，不阻塞事件迴圈"}

@app.get("/async")
async def async_handler() -> dict:    # 非同步 handler
    await asyncio.sleep(0)            # await 點才會把控制權交回事件迴圈
    return {"mode": "在事件迴圈裡跑"}

FastAPI 的處理規則很聰明：def（同步）handler 會被丟到 threadpool 執行，所以你寫阻塞式的 SQLAlchemy（同步）也不會卡住整個事件迴圈；async def handler 直接在事件迴圈裡跑，適合內部全是 await（非阻塞 I/O）的情境。

但要記牢這條：async 解決的是 I/O 並行（在等網路/磁碟時讓出 CPU 給別的請求），不是 CPU 平行。 在 async def 裡塞一段純 CPU 的重計算（一個沒有 await 的 for 迴圈跑十秒），它會獨佔事件迴圈、阻塞所有其他請求——比同步版還糟（同步版至少在 threadpool 裡，不擋事件迴圈）。GIL 也仍在：單一進程內 Python bytecode 還是序列執行。CPU bound 的工作該丟進程池或獨立 worker（第 14 章 multiprocessing），不是包個 async 就會變快。

⚠️ 陷阱：在 async def endpoint 裡呼叫阻塞式函式（同步的 requests.get、同步 DB driver、time.sleep），會卡住整個事件迴圈、拖垮所有並行請求。要嘛改用 def（讓它去 threadpool），要嘛換成 async 版（httpx.AsyncClient、async DB driver、asyncio.sleep）。混用是新手最常見的效能事故。

🔁 Java 對照：WSGI ≈ thread-per-request 的傳統 Servlet/Tomcat；ASGI ≈ Servlet 3.1 非同步 I/O 或 Spring WebFlux 的反應式模型。差別是 Python 的 async 受 GIL 限制，平行計算仍要靠多進程，不像 JVM 的 thread 能直接吃滿多核。

部署：uvicorn / gunicorn / Docker

從開發到生產

TestClient 只用於測試。要真的對外服務，你需要一個 ASGI server 來跑 app。

開發 / 單機用 uvicorn（純 ASGI server，基於 uvloop + httptools，很快）：

bash 
# 假設 app 在 main.py 的 app 變數
uvicorn main:app --reload          # --reload：改檔自動重啟，等同 spring-boot-devtools
uvicorn main:app --host 0.0.0.0 --port 8000

main:app 的格式是 模組:變數名，對應 Java 的 主類別。--reload 只在開發用，正式環境要關掉。

生產要跑多個 worker 進程繞過 GIL（呼應第 14 章：多進程才有真平行）。兩種主流做法：

bash 
# 做法 A：uvicorn 自己開多 worker
uvicorn main:app --host 0.0.0.0 --port 8000 --workers 4

# 做法 B：gunicorn 當進程管理器，每個 worker 用 uvicorn 的 worker class
gunicorn main:app -w 4 -k uvicorn.workers.UvicornWorker --bind 0.0.0.0:8000

gunicorn 做法在生產更常見，因為 gunicorn 是成熟的進程管理器：worker 掛掉自動重啟、graceful reload、優雅關閉。每個 worker 是獨立進程（各有自己的 GIL 與事件迴圈），-w 4 就是四個進程吃四顆核——這正是 Java 不需要操心（JVM 一個進程多 thread 吃滿核）但 CPython 必須面對的部署現實。worker 數的經驗值常見是 2 * CPU 核數 + 1，但 I/O bound 的 async 服務可以更激進，要實測。

Spring Boot FastAPI 部署 說明
內嵌 Tomcat（一進程多 thread） uvicorn / gunicorn（多進程） CPython 靠多進程吃多核
java -jar app.jar uvicorn main:app 啟動命令
application.yml 環境變數 + pydantic-settings 設定來源

設定管理：pydantic-settings 對照 `@ConfigurationProperties`

正式服務的設定（DB URL、密鑰、debug 開關）走環境變數，用 pydantic-settings 綁成型別化物件——這就是 Python 版的 @ConfigurationProperties，連「型別轉換 + 驗證」都一起做了：

python 
from pydantic_settings import BaseSettings, SettingsConfigDict

class Settings(BaseSettings):
    model_config = SettingsConfigDict(env_prefix="APP_")  # 只讀 APP_ 開頭的環境變數
    database_url: str = "sqlite:///:memory:"
    debug: bool = False

實測：環境變數會自動覆寫預設值，且型別自動轉換（字串 "true" → bool True）：

python 
Settings().model_dump()
# 無環境變數時：{'database_url': 'sqlite:///:memory:', 'debug': False}

# 設了 APP_DEBUG=true、APP_DATABASE_URL=postgresql://localhost/prod 後：
Settings().model_dump()
# {'database_url': 'postgresql://localhost/prod', 'debug': True}

把 get_settings 包成依賴（@lru_cache 讓它變單例），就能在任何 endpoint 注入設定，測試時用 dependency_overrides 換掉。

python 
from functools import lru_cache

@lru_cache
def get_settings() -> Settings:    # lru_cache → 整個進程只建一次（像 Spring 單例 bean）
    return Settings()

🔁 Java 對照：BaseSettings 子類 ≈ @ConfigurationProperties(prefix = "app") 的 record；env_prefix="APP_" ≈ prefix = "app"；pydantic 的型別驗證 ≈ Spring 的 relaxed binding + JSR-303 驗證。最大差別：Spring 預設讀 application.yml，雲原生風格的 FastAPI 服務預設讀環境變數（十二要素 app 的主張），更貼近容器化部署。

Dockerfile

最後把它裝箱。對照 Spring Boot 的「fat jar + FROM eclipse-temurin + java -jar」，FastAPI 的 Dockerfile 一樣精簡。這裡用 uv 管理依賴（第 1、11 章），多階段思維上把依賴安裝與程式碼分層以利快取：

dockerfile 
FROM python:3.12-slim

WORKDIR /app

# 先複製依賴清單，利用 Docker layer cache：程式碼變了也不必重裝套件
COPY pyproject.toml uv.lock ./
RUN pip install --no-cache-dir uv && uv sync --frozen --no-dev

COPY . .

EXPOSE 8000
# 生產用 gunicorn 管多個 uvicorn worker
# 前提：gunicorn 與 uvicorn[standard] 都要列進 pyproject.toml 的依賴，否則 uv run 找不到指令
CMD ["uv", "run", "gunicorn", "main:app", \
     "-w", "4", "-k", "uvicorn.workers.UvicornWorker", \
     "--bind", "0.0.0.0:8000"]

💡 關於 worker 寫法：gunicorn -k uvicorn.workers.UvicornWorker 是長年慣用、目前仍可用的寫法。但截至 2025，官方已逐漸推薦兩種更現代的替代：用獨立的 uvicorn-worker 套件（gunicorn -k uvicorn_worker.UvicornWorker），或乾脆不要 gunicorn、直接 uvicorn main:app --workers 4 讓 uvicorn 自己管多進程。三者都能達到「多進程繞過 GIL」的目的，新專案可優先考慮後兩者。

對照 Spring Boot 的 Dockerfile：

dockerfile 
# Java 對照
FROM eclipse-temurin:21-jre
WORKDIR /app
COPY target/app.jar app.jar
EXPOSE 8080
CMD ["java", "-jar", "app.jar"]

兩者結構幾乎一樣——基底映像、複製產物、開埠、啟動命令。差別只在 Python 沒有「打包成單一 jar」的步驟，而是把原始碼 + 依賴環境一起裝箱，啟動時由 ASGI server 載入。設定全走環境變數（docker run -e APP_DATABASE_URL=...），由 pydantic-settings 接收。

本章小結

  • FastAPI 的核心殺手鏈是「型別提示 → 自動驗證 → 自動文件」：一個 pydantic model 同時是執行期驗證規則、序列化 schema、OpenAPI 文件來源。Java 要 @Valid + Bean Validation + springdoc 三套協作才湊得出來。
  • 路由參數靠慣例推斷：名字在路徑樣板裡 = path；型別是 BaseModel = body；其餘簡單型別 = query（有預設值即選填）。比 Spring 每參數掛 annotation 清爽。
  • 把 request / response / DB model 分成三個類別：輸入沒 id、輸出有 id、ORM 綁資料庫，形狀本就不同。輸出 schema 建議開 from_attributes=True——它讓 model_validate(orm) 能直接從物件屬性建模，也是回傳 ORM 物件時更明確的作法。
  • Depends 是顯式的函式式 DI：依賴就是被呼叫的函式，沒有容器掃描魔法；yield 依賴做 setup/teardown（Python 版 try-with-resources / @Transactional）；dependency_overrides 是測試替換依賴的利器。
  • SQLAlchemy 2.0 用 Mapped[...] + mapped_column(...)，型別即 schema（str | None → 可空）。session = 工作單元 + identity map（同 session 同主鍵 = 同物件），對應 Hibernate persistence context；UPDATE 靠髒檢查自動發。
  • ASGI（async，FastAPI）vs WSGI（同步，Flask） 對應反應式 vs thread-per-request；但 async 解決 I/O 並行不解決 CPU 平行，GIL 仍在，CPU bound 別包 async。
  • 生產部署用多進程繞過 GIL：gunicorn 管多個 uvicorn worker（每進程一顆核），設定走環境變數 + pydantic-settings（≈ @ConfigurationProperties），裝進精簡 Dockerfile。
  • in-memory SQLite 測試記得 StaticPool，否則 sessionmaker 拿到別條連線、看不到表（no such table）。

練習

  1. 加一個 PUT 與 DELETE：在 items 服務補上 @app.put("/items/{item_id}")（接 ItemCreate、更新後回 ItemRead）與 @app.delete("/items/{item_id}")（成功回 204）。提示：更新用 session.get 拿到 ORM 物件後直接改屬性、靠髒檢查 commit；找不到一律回 404。寫 TestClient 測試覆蓋「更新存在 / 更新不存在 / 刪除後再查回 404」。

  2. 加分頁與過濾：把 list_items 改成接 query 參數 skip: int = 0、limit: int = 10、name_contains: str | None = None，用 select(...).where(ItemRow.name.contains(...)).offset(skip).limit(limit) 實作。觀察 OpenAPI /docs 是否自動把這些 query 參數列出來。

  3. 用 dependency_overrides 寫一個乾淨的 pytest fixture：建一個指向獨立 in-memory DB（記得 StaticPool）的 get_session 覆寫，包成 pytest fixture，每個測試拿到全新空白 DB。提示：fixture 裡 app.dependency_overrides[get_session] = ...，yield client，收尾 app.dependency_overrides.clear()（承接第 13 章）。

  4. 驗證錯誤的自訂回應：用 @app.exception_handler(RequestValidationError) 攔截 422，把 pydantic 的 detail 陣列轉成你團隊前端想要的格式（例如 {"field": "name", "error": "..."} 的扁平列表）。提示：handler 收到的 exc.errors() 就是那個結構化錯誤陣列。

  5. 設定驅動的環境切換：用 pydantic-settings 讓服務根據 APP_DATABASE_URL 環境變數切換資料庫——沒設時用 in-memory SQLite，設了就連那個 URL。用 get_settings 依賴 + @lru_cache 注入，並寫一個測試證明用 dependency_overrides 換掉 settings 後，服務會連到測試指定的 DB。
