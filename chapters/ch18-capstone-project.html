第 18 章：總整專案（Capstone）

前十七章把工具一件件交到你手上：型別系統、Pythonic 寫法、類別與協定、pydantic、SQLAlchemy、FastAPI、click、pytest、ruff、mypy。但工具不是專案。一個資深 Java 工程師交付的是「一個結構清楚、有測試、CI 全綠、可被 pip install 的東西」，不是一堆能跑的 script。這一章我們把所有零件組成一個小而完整的服務——一個任務清單（task list），同時提供 REST API 與 CLI 兩個入口，共用同一份核心邏輯，並通過 ruff + mypy strict + coverage 三道品質門檻。重點不在功能多，而在分層、邊界、與工程紀律如何在 Python 裡落地。本章所有核心程式碼（API、service、CLI、測試）都已實際跑通，你可以照著做出一模一樣的成品。

專案規格與架構

需求很小，刻意如此：一個任務有 title 與 status（todo / doing / done），支援增刪改查與依狀態篩選。麻雀雖小，但它逼你面對每個真實專案都會遇到的決策：資料模型放哪？驗證在哪一層？API 跟 CLI 怎麼共用邏輯？怎麼測？

src layout 目錄樹

承接第 11 章，我們用 src layout——原始碼放進 src/，強迫測試走「已安裝的套件」路徑，避免「在專案根目錄剛好 import 得到、打包後卻壞掉」的假陽性。

分層：對照 Spring 的 Controller / Service / Repository

你在 Spring 裡的反射動作是把程式碼切成三層。這個專案完全照搬這個心智模型，只是把 Java 的物件圖換成 Python 的模組：

Spring 本專案 職責
@RestController main.py（FastAPI）/ cli.py（click） 入口層：解析輸入、呼叫 service、組裝回應
@Service service.py 商業邏輯，不認識 HTTP、不認識 click
@Repository / JPA models.py + db.py（SQLAlchemy） 持久層
DTO / Request body schemas.py（pydantic） API 邊界的請求/回應契約

💡 Pythonic 重點：分層不是靠 @Service 註解或 Spring 容器，而是靠模組相依方向。service.py 只 import models 與 schemas，不 import fastapi 或 click。這條規則用肉眼就能稽核：core 不該知道誰在呼叫它。這正是「同一份邏輯、兩種介面」能成立的原因。

🔁 Java 對照：Spring 用依賴注入容器把三層接起來；Python 直接用函式參數傳遞 Session——service 的每個函式第一個參數都是 session: Session。這就是「窮人的依賴注入」，而且它顯式、可測、零魔法。

型別與資料模型三分

第 9 章立過一條規矩：信任邊界用 pydantic 做執行期驗證，邊界內用 dataclass / 普通物件傳遞。在這個專案裡，這條規矩具體化成三種模型，各司其職，不要混用：

模型 工具 用在哪 為什麼
ORM model SQLAlchemy DeclarativeBase 持久層 對應資料表、管理 identity 與 session
API schema pydantic BaseModel HTTP 請求/回應 不信任外部輸入，要驗證、要序列化
（內部 DTO） dataclass / 直接傳 ORM 物件 service 內部 邊界內已可信，不需重複驗證

新手常犯的錯是把 SQLAlchemy model 直接丟去當 API 回應，或把 pydantic model 拿去存資料庫。三分法讓每一層的契約獨立演進——資料庫加個 updated_at 欄位不會逼 API 改版。

models.py：持久層（SQLAlchemy 2.0）

SQLAlchemy 2.0 的 Mapped[...] 風格跟 JPA 的 annotated entity 神似，差別是型別資訊由 Mapped[] 帶出、mapped_column() 補欄位細節。StrEnum（3.11+）讓 enum 值在序列化時自然變成字串。

python 
# src/tasks/models.py
from __future__ import annotations

import enum
from datetime import UTC, datetime

from sqlalchemy import DateTime, Enum, String
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column


class Base(DeclarativeBase):
    pass


class Status(enum.StrEnum):
    TODO = "todo"
    DOING = "doing"
    DONE = "done"


def _now() -> datetime:
    return datetime.now(UTC)


class Task(Base):
    __tablename__ = "tasks"

    id: Mapped[int] = mapped_column(primary_key=True)
    title: Mapped[str] = mapped_column(String(200))
    status: Mapped[Status] = mapped_column(Enum(Status), default=Status.TODO)
    created_at: Mapped[datetime] = mapped_column(DateTime(timezone=True), default=_now)

🔁 Java 對照：class Task(Base) ≈ @Entity class Task；mapped_column(primary_key=True) ≈ @Id @GeneratedValue。但 SQLAlchemy 沒有 JPA 那層 EntityManager 的隱式快取魔法——session 邊界與 flush 時機你要自己掌握，比 Hibernate 更顯式（也更少驚喜）。

schemas.py：API 邊界（pydantic）

注意三個 schema 的分工：TaskCreate 是「能建立的最小集合」（只收 title，id 由 DB 配、status 預設 todo）；TaskUpdate 全部欄位 optional（PATCH 語意）；TaskOut 是回應契約，from_attributes=True 讓它能直接從 ORM 物件讀屬性。

python 
# src/tasks/schemas.py
from __future__ import annotations

from datetime import datetime

from pydantic import BaseModel, ConfigDict, Field

from tasks.models import Status


class TaskCreate(BaseModel):
    title: str = Field(min_length=1, max_length=200)


class TaskUpdate(BaseModel):
    title: str | None = Field(default=None, min_length=1, max_length=200)
    status: Status | None = None


class TaskOut(BaseModel):
    model_config = ConfigDict(from_attributes=True)

    id: int
    title: str
    status: Status
    created_at: datetime

Field(min_length=1) 不是裝飾——它是契約。空字串 title 會在進入 service 前就被 pydantic 擋下並回 422，service 層因此完全不必寫 if not title: raise ... 這種防禦碼。這是「驗證集中在邊界」帶來的乾淨。

service.py：核心邏輯（無框架相依）

這是整個專案的心臟，也是兩個入口共用的那份「同一套邏輯」。每個函式收一個 Session、回 ORM 物件，找不到就丟自訂例外 TaskNotFound——它不知道 HTTP 404，也不知道 click 的錯誤碼，那是入口層的事。

python 
# src/tasks/service.py
from __future__ import annotations

from sqlalchemy import select
from sqlalchemy.orm import Session

from tasks.models import Status, Task
from tasks.schemas import TaskCreate, TaskUpdate


class TaskNotFound(Exception):
    def __init__(self, task_id: int) -> None:
        super().__init__(f"task {task_id} not found")
        self.task_id = task_id


def create_task(session: Session, data: TaskCreate) -> Task:
    task = Task(title=data.title, status=Status.TODO)
    session.add(task)
    session.commit()
    session.refresh(task)
    return task


def get_task(session: Session, task_id: int) -> Task:
    task = session.get(Task, task_id)
    if task is None:
        raise TaskNotFound(task_id)
    return task


def list_tasks(session: Session, status: Status | None = None) -> list[Task]:
    stmt = select(Task).order_by(Task.id)
    if status is not None:
        stmt = stmt.where(Task.status == status)
    return list(session.scalars(stmt))


def update_task(session: Session, task_id: int, data: TaskUpdate) -> Task:
    task = get_task(session, task_id)
    if data.title is not None:
        task.title = data.title
    if data.status is not None:
        task.status = data.status
    session.commit()
    session.refresh(task)
    return task


def delete_task(session: Session, task_id: int) -> None:
    task = get_task(session, task_id)
    session.delete(task)
    session.commit()

💡 Pythonic 重點：自訂例外 TaskNotFound 是分層的關鍵。service 用「丟例外」表達失敗（第 12 章 EAFP），讓每個入口層自己決定怎麼翻譯——HTTP 翻成 404、CLI 翻成非零 exit code。如果 service 直接 raise HTTPException(404)，它就被 FastAPI 綁死，CLI 再也用不了它。

db.py：engine 與 session 工廠

python 
# src/tasks/db.py
from __future__ import annotations

from sqlalchemy import Engine, create_engine
from sqlalchemy.orm import Session, sessionmaker

from tasks.models import Base

DEFAULT_URL = "sqlite:///tasks.db"


def make_engine(url: str = DEFAULT_URL) -> Engine:
    engine = create_engine(url)
    Base.metadata.create_all(engine)
    return engine


def make_session_factory(url: str = DEFAULT_URL) -> sessionmaker[Session]:
    return sessionmaker(bind=make_engine(url))

把「怎麼建 engine」抽成工廠，是讓測試能塞 in-memory DB 的關鍵——production 用 sqlite:///tasks.db，測試用 sqlite:///:memory:，service 與入口層對此一無所知。

API 與 CLI 兩個入口

現在來證明「一份核心、兩種介面」的價值：同樣呼叫 service.create_task，一個透過 HTTP、一個透過命令列。

FastAPI app（main.py）

承接第 16 章，FastAPI 用 Depends 做依賴注入。這裡的關鍵是 get_db 這個 generator dependency——它 yield 一個 session、請求結束後在 finally 關閉，行為等同 Spring 的 request-scoped EntityManager，但機制是 Python 的 generator + context（第 6、12 章）。Annotated[Session, Depends(get_db)] 是 FastAPI 推薦的注入寫法，可重用又對 mypy 友善。

python 
# src/tasks/main.py
from __future__ import annotations

from collections.abc import Iterator
from typing import Annotated

from fastapi import Depends, FastAPI, HTTPException
from sqlalchemy.orm import Session, sessionmaker

from tasks.db import make_session_factory
from tasks.models import Status
from tasks.schemas import TaskCreate, TaskOut, TaskUpdate
from tasks.service import (
    TaskNotFound,
    create_task,
    delete_task,
    get_task,
    list_tasks,
    update_task,
)

app = FastAPI(title="Tasks API")

_factory: sessionmaker[Session] = make_session_factory()


def get_db() -> Iterator[Session]:
    session = _factory()
    try:
        yield session
    finally:
        session.close()


DB = Annotated[Session, Depends(get_db)]


@app.post("/tasks", response_model=TaskOut, status_code=201)
def post_task(payload: TaskCreate, db: DB) -> object:
    return create_task(db, payload)


@app.get("/tasks", response_model=list[TaskOut])
def get_tasks(db: DB, status: Status | None = None) -> object:
    return list_tasks(db, status)


@app.get("/tasks/{task_id}", response_model=TaskOut)
def get_one(task_id: int, db: DB) -> object:
    try:
        return get_task(db, task_id)
    except TaskNotFound as exc:
        raise HTTPException(status_code=404, detail=str(exc)) from exc


@app.patch("/tasks/{task_id}", response_model=TaskOut)
def patch_task(task_id: int, payload: TaskUpdate, db: DB) -> object:
    try:
        return update_task(db, task_id, payload)
    except TaskNotFound as exc:
        raise HTTPException(status_code=404, detail=str(exc)) from exc


@app.delete("/tasks/{task_id}", status_code=204)
def remove_task(task_id: int, db: DB) -> None:
    try:
        delete_task(db, task_id)
    except TaskNotFound as exc:
        raise HTTPException(status_code=404, detail=str(exc)) from exc

入口層的工作只剩「翻譯」：把 TaskNotFound 翻成 HTTPException(404)（並用 raise ... from exc 保留例外鏈，第 12 章）。商業邏輯一行都沒有。response_model=TaskOut 讓 FastAPI 自動把 ORM 物件序列化成回應 schema、產生 OpenAPI 文件、並過濾掉不該外洩的欄位。

click CLI（cli.py）

承接第 15 章，CLI 用 click（對照 picocli）。@click.group() 建出子指令樹（tasks add / tasks list / tasks done），ctx.obj 是跨指令傳遞的上下文——我們把 session 工廠塞進去，讓每個子指令都拿得到。注意 if ctx.obj is None 這個小設計：production 走 --db 建工廠，測試則用 runner.invoke(main, ..., obj=factory) 直接注入 in-memory 工廠（下一節會用到）。

python 
# src/tasks/cli.py
from __future__ import annotations

import click

from tasks.db import make_session_factory
from tasks.models import Status
from tasks.schemas import TaskCreate, TaskUpdate
from tasks.service import (
    TaskNotFound,
    create_task,
    delete_task,
    list_tasks,
    update_task,
)


@click.group()
@click.option("--db", "db_url", default="sqlite:///tasks.db", help="SQLAlchemy URL")
@click.pass_context
def main(ctx: click.Context, db_url: str) -> None:
    """tasks — 命令列任務管理。"""
    # 測試時可用 runner.invoke(main, ..., obj=factory) 注入 in-memory factory
    if ctx.obj is None:
        ctx.obj = make_session_factory(db_url)


@main.command()
@click.argument("title")
@click.pass_obj
def add(factory: object, title: str) -> None:
    with factory() as session:  # type: ignore[operator]
        task = create_task(session, TaskCreate(title=title))
        click.echo(f"#{task.id} {task.title} [{task.status.value}]")


@main.command(name="list")
@click.option("--status", type=click.Choice([s.value for s in Status]), default=None)
@click.pass_obj
def list_cmd(factory: object, status: str | None) -> None:
    with factory() as session:  # type: ignore[operator]
        wanted = Status(status) if status else None
        for task in list_tasks(session, wanted):
            click.echo(f"#{task.id} {task.title} [{task.status.value}]")


@main.command()
@click.argument("task_id", type=int)
@click.pass_obj
def done(factory: object, task_id: int) -> None:
    with factory() as session:  # type: ignore[operator]
        try:
            task = update_task(session, task_id, TaskUpdate(status=Status.DONE))
        except TaskNotFound as exc:
            raise click.ClickException(str(exc)) from exc
        click.echo(f"#{task.id} {task.title} [{task.status.value}]")


@main.command()
@click.argument("task_id", type=int)
@click.pass_obj
def rm(factory: object, task_id: int) -> None:
    with factory() as session:  # type: ignore[operator]
        try:
            delete_task(session, task_id)
        except TaskNotFound as exc:
            raise click.ClickException(str(exc)) from exc
        click.echo(f"deleted #{task_id}")


if __name__ == "__main__":
    main()

#2 跑測試 [todo]

CLI 的 done 指令在底層呼叫的是和 HTTP PATCH 完全相同的 service.update_task。這就是分層的紅利：商業規則只實作一次。

⚠️ 陷阱：click 的子指令是 callback 函式，@click.pass_obj 把 ctx.obj 當第一個參數注入。Java 工程師容易想用全域變數或 module-level singleton 來傳 session——別這麼做，那會讓測試無法注入替身。用 ctx.obj 傳上下文，保持可測。

測試策略

承接第 13 章，分兩種測試：service 層的單元測試（純邏輯，最多最快）與入口層的整合測試（API / CLI 各一套）。所有測試都跑在 in-memory sqlite 上，零外部相依、毫秒級。

conftest.py：in-memory DB 與 dependency override

這個 conftest.py 是測試基礎建設的核心，三個地方值得細看：

python 
# tests/conftest.py
from __future__ import annotations

from collections.abc import Iterator

import pytest
from fastapi.testclient import TestClient
from sqlalchemy import create_engine
from sqlalchemy.orm import Session, sessionmaker
from sqlalchemy.pool import StaticPool

from tasks.main import app, get_db
from tasks.models import Base


@pytest.fixture
def session_factory() -> sessionmaker[Session]:
    engine = create_engine(
        "sqlite:///:memory:",
        connect_args={"check_same_thread": False},
        poolclass=StaticPool,
    )
    Base.metadata.create_all(engine)
    return sessionmaker(bind=engine)


@pytest.fixture
def session(session_factory: sessionmaker[Session]) -> Iterator[Session]:
    with session_factory() as s:
        yield s


@pytest.fixture
def client(session_factory: sessionmaker[Session]) -> Iterator[TestClient]:
    def override_get_db() -> Iterator[Session]:
        with session_factory() as s:
            yield s

    app.dependency_overrides[get_db] = override_get_db
    yield TestClient(app)
    app.dependency_overrides.clear()

⚠️ 陷阱：sqlite:///:memory: 的記憶體資料庫是綁在連線（connection）上的——連線一關，資料就沒了。SQLAlchemy 預設的連線池每次 checkout 可能給你不同連線，於是 create_all 建的表在下一次查詢時「消失」（no such table: tasks）。解法是 poolclass=StaticPool 強制所有 session 共用同一條連線，check_same_thread=False 則允許 TestClient 跨執行緒使用它。要精確一點說：這個「no such table」只在連線池真的發出不同連線時才會浮現——例如 TestClient 把 sync endpoint 丟進 threadpool、跨執行緒 checkout 另一條連線時；若整個測試只在單一連線上順序重用，未必每次都觸發。但 in-memory sqlite 測試一律用 StaticPool 是穩當的標準建議，不必賭它哪次會踩到。這是經典坑，我在寫這章時也實際踩到了。

client fixture 用 app.dependency_overrides[get_db] = ... 把 production 的 get_db 換成測試版——這就是 FastAPI 的依賴覆寫機制，等同 Spring Test 的 @MockBean，但更輕：一個 dict 賦值而已。測試結束 clear() 還原，避免污染其他測試。

# tests/test_service.py
from __future__ import annotations

import pytest
from sqlalchemy.orm import Session

from tasks.models import Status
from tasks.schemas import TaskCreate, TaskUpdate
from tasks.service import (
    TaskNotFound,
    create_task,
    delete_task,
    get_task,
    list_tasks,
    update_task,
)


def test_create_defaults_to_todo(session: Session) -> None:
    task = create_task(session, TaskCreate(title="買牛奶"))
    assert task.id is not None
    assert task.status is Status.TODO


def test_update_status(session: Session) -> None:
    task = create_task(session, TaskCreate(title="寫測試"))
    updated = update_task(session, task.id, TaskUpdate(status=Status.DONE))
    assert updated.status is Status.DONE


def test_list_filters_by_status(session: Session) -> None:
    a = create_task(session, TaskCreate(title="a"))
    create_task(session, TaskCreate(title="b"))
    update_task(session, a.id, TaskUpdate(status=Status.DONE))
    done = list_tasks(session, Status.DONE)
    assert [t.title for t in done] == ["a"]


def test_delete_removes_task(session: Session) -> None:
    task = create_task(session, TaskCreate(title="待刪"))
    delete_task(session, task.id)
    with pytest.raises(TaskNotFound):
        get_task(session, task.id)


def test_get_missing_raises(session: Session) -> None:
    with pytest.raises(TaskNotFound):
        get_task(session, 999)

service 測試不碰 HTTP、不碰 CLI，純粹餵 session 與 schema、驗結果。with pytest.raises(TaskNotFound) 對照 JUnit 的 assertThrows。

API 整合測試（FastAPI TestClient）

TestClient 用 in-process 的方式打 app，不需要真的啟動 server（底層走 ASGI transport），所以又快又穩。

python 
# tests/test_api.py
from __future__ import annotations

from fastapi.testclient import TestClient


def test_create_and_get(client: TestClient) -> None:
    resp = client.post("/tasks", json={"title": "買咖啡"})
    assert resp.status_code == 201
    task_id = resp.json()["id"]

    got = client.get(f"/tasks/{task_id}")
    assert got.status_code == 200
    assert got.json()["title"] == "買咖啡"
    assert got.json()["status"] == "todo"


def test_validation_rejects_empty_title(client: TestClient) -> None:
    resp = client.post("/tasks", json={"title": ""})
    assert resp.status_code == 422


def test_patch_status(client: TestClient) -> None:
    task_id = client.post("/tasks", json={"title": "x"}).json()["id"]
    resp = client.patch(f"/tasks/{task_id}", json={"status": "done"})
    assert resp.status_code == 200
    assert resp.json()["status"] == "done"


def test_missing_returns_404(client: TestClient) -> None:
    assert client.get("/tasks/424242").status_code == 404

test_validation_rejects_empty_title 證明了第二節的論點：空 title 在 pydantic 邊界就被擋下回 422，這個驗證我們一行都沒寫，是 Field(min_length=1) 自動產生的。

CLI 整合測試（click CliRunner）

click 提供 CliRunner 在 process 內執行指令、攔截輸出與 exit code，對照 picocli 的測試 API。我們透過 obj=session_factory 注入 in-memory 工廠。

python 
# tests/test_cli.py
from __future__ import annotations

from click.testing import CliRunner
from sqlalchemy.orm import Session, sessionmaker

from tasks.cli import main


def test_add_and_list(session_factory: sessionmaker[Session]) -> None:
    runner = CliRunner()
    add = runner.invoke(main, ["add", "買牛奶"], obj=session_factory)
    assert add.exit_code == 0
    assert "#1 買牛奶 [todo]" in add.output

    listed = runner.invoke(main, ["list"], obj=session_factory)
    assert "#1 買牛奶 [todo]" in listed.output


def test_rm_deletes(session_factory: sessionmaker[Session]) -> None:
    runner = CliRunner()
    runner.invoke(main, ["add", "待刪"], obj=session_factory)
    removed = runner.invoke(main, ["rm", "1"], obj=session_factory)
    assert removed.exit_code == 0
    assert "deleted #1" in removed.output

    listed = runner.invoke(main, ["list"], obj=session_factory)
    assert "待刪" not in listed.output


def test_done_missing_errors(session_factory: sessionmaker[Session]) -> None:
    runner = CliRunner()
    result = runner.invoke(main, ["done", "999"], obj=session_factory)
    assert result.exit_code != 0
    assert "not found" in result.output

工程品質門檻：ruff + mypy + coverage 全綠

Java 的品質門檻你熟得很：Checkstyle 管風格、SpotBugs 抓 bug、JaCoCo 卡覆蓋率、編譯器擋型別。Python 把這些換成更少、更快的工具，全部設定收斂在一個 pyproject.toml。

pyproject.toml：單一設定中心

toml 
[project]
name = "tasks"
version = "0.1.0"
requires-python = ">=3.12"
dependencies = [
    "fastapi>=0.115",
    "sqlalchemy>=2.0",
    "pydantic>=2.7",
    "click>=8.1",
]

[project.scripts]
tasks = "tasks.cli:main"

[dependency-groups]
dev = [
    "pytest>=8",
    "pytest-cov",
    "ruff",
    "mypy",
    "httpx",
]

[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

[tool.ruff]
line-length = 100
src = ["src", "tests"]

[tool.ruff.lint]
select = ["E", "F", "I", "UP", "B", "SIM"]

[tool.mypy]
strict = true
mypy_path = "src"

[tool.pytest.ini_options]
pythonpath = ["src"]
addopts = "--cov=tasks --cov-report=term-missing --cov-fail-under=85"

注意 [dependency-groups] 的 dev 群組——pytest、pytest-cov、ruff、mypy 這些品管工具只在開發/CI 時需要，不該混進 [project].dependencies 跟著 wheel 散佈出去（對照 Maven 的 <scope>test</scope>）。本機與 CI 用 uv sync --dev 一次裝齊，之後所有 uv run ruff/uv run mypy/uv run pytest 才有可執行檔可用。httpx 是 FastAPI TestClient 的相依，也歸在這裡。

Java 工具 本專案 說明
Checkstyle + SpotBugs ruff lint（E/F/B/SIM） 風格 + 常見 bug，毫秒級
Spotless / google-java-format ruff format 格式化，取代 black
javac 型別檢查 mypy strict 第 9 章，最嚴格的漸進式型別
JaCoCo 門檻 pytest-cov --cov-fail-under=85 低於 85% 直接 fail

select 裡的 UP 規則特別值得一提：它會把舊寫法升級成現代語法。我在寫這章時，datetime.now(timezone.utc) 就被 UP017 揪出來，建議改成 datetime.now(UTC)——這正是讓程式碼跟上 3.12+ 的免費守門員。

mypy strict 過了 11 個檔案（含測試）這件事不平凡：它逼你給每個函式回傳型別、不准 implicit Any、不准漏掉 Optional。寫這章時它就抓到 make_engine 漏標回傳型別——一個 Java 工程師絕不會放過的等級。

pre-commit hook

對照 Git 的 pre-commit、或 Maven 的 validate phase，把門檻前移到 commit 當下：

yaml 
# .pre-commit-config.yaml
repos:
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.8.0
    hooks:
      - id: ruff          # lint
        args: [--fix]
      - id: ruff-format   # format
  - repo: https://github.com/pre-commit/mirrors-mypy
    rev: v1.13.0
    hooks:
      - id: mypy
        additional_dependencies: [pydantic, sqlalchemy]

pre-commit install 後，每次 git commit 自動跑 ruff + mypy，不過不讓你 commit。本機就擋下九成的 CI 紅燈。

GitHub Actions CI

對照 Jenkins pipeline 或 Maven 的 GitHub Actions，這份 workflow 把四道關卡串成 matrix——lint、type-check、test、build，跨 3.12 / 3.13 兩個版本：

yaml 
# .github/workflows/ci.yml
name: CI
on: [push, pull_request]

jobs:
  quality:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        python-version: ["3.12", "3.13"]
    steps:
      - uses: actions/checkout@v4
      - uses: astral-sh/setup-uv@v5
        with:
          python-version: ${{ matrix.python-version }}
      - run: uv sync --dev
      - name: Lint
        run: uv run ruff check .
      - name: Format check
        run: uv run ruff format --check .
      - name: Type check
        run: uv run mypy src tests
      - name: Test
        run: uv run pytest
      - name: Build
        run: uv build

setup-uv 取代了 Java 的 setup-java + 快取設定；uv sync 比 Maven 的 dependency resolution 快一個數量級。matrix 跨版本對照你在 Java 跨 JDK 17/21 測試的習慣。

pyproject.toml 裡的 [project.scripts] 是把 CLI 變成可安裝指令的關鍵——它對照 Maven 的 shade plugin + manifest Main-Class，但更乾淨。tasks = "tasks.cli:main" 這行讓 wheel 裡產生一個 console script entry point：

console 
$ unzip -p dist/tasks-0.1.0-py3-none-any.whl '*/entry_points.txt'
[console_scripts]
tasks = tasks.cli:main

任何人 pip install tasks-0.1.0-py3-none-any.whl 之後，shell 裡就多了一個 tasks 指令，直接 tasks add "..." 即可——不必 python -m。版本號統一由 [project].version 管理（也可用 hatch-vcs 從 git tag 自動推導）。要發佈到 PyPI，一句 uv publish（或 twine upload dist/*）即可，配合 PyPI 的 trusted publishing 還能免 token。

本章小結

  1. 分層靠相依方向，不靠註解：service.py 不 import fastapi/click，才能讓「一份核心、兩種入口」成立。這是整個專案最重要的設計決策。
  2. 型別三分：pydantic 守 API 邊界（驗證 + 序列化）、SQLAlchemy model 管持久、邊界內傳 ORM 物件或 dataclass。不要混用。
  3. 驗證集中在邊界：Field(min_length=1) 讓空 title 在進 service 前就回 422，core 不必寫防禦碼。
  4. service 用例外表達失敗：丟自訂 TaskNotFound，由各入口層翻譯成 HTTP 404 或 CLI 非零 exit code，保留例外鏈。
  5. 測試靠注入替身：FastAPI 用 dependency_overrides、click 用 obj=、DB 用 in-memory sqlite + StaticPool。記得 in-memory sqlite 必須用 StaticPool 共用連線，否則「no such table」。
  6. 三道門檻全綠：ruff（lint+format）、mypy strict、pytest-cov（--cov-fail-under），全收斂在 pyproject.toml，由 pre-commit 與 GitHub Actions matrix 自動把關。
  7. 打包靠 entry point：[project.scripts] 把 CLI 變成可安裝指令，uv build 出 wheel/sdist，uv publish 上 PyPI。
  8. 務實的覆蓋率：核心邏輯逼近 100%、入口層容忍缺口，整體守住 85%。不要為了數字寫沒意義的測試。

練習

  1. 加一層快取：在 service.list_tasks 前面加一個記憶體快取（functools.lru_cache 不適用於有 session 參數的情況，想想為什麼），並寫測試證明 create_task 後快取會失效。提示：快取要以「是否有寫入」為失效依據，思考它該放在 service 層還是入口層。
  2. 補 PUT 與分頁：替 API 加 PUT /tasks/{id}（全量取代，對照 PATCH 的部分更新語意差異）與 GET /tasks?limit=&offset= 分頁。CLI 的 list 也加 --limit。確認新程式碼讓 coverage 不掉到 85% 以下。
  3. 把例外翻譯抽成 exception handler：FastAPI 支援 @app.exception_handler(TaskNotFound)，把 main.py 裡重複的 try/except → 404 收斂成一個全域 handler。比較前後的 endpoint 程式碼，體會「橫切關注點」在 FastAPI 怎麼處理（對照 Spring 的 @ControllerAdvice）。
  4. 換成 PostgreSQL（選做）：把 DEFAULT_URL 改成 postgresql+psycopg://...，用 testcontainers 或 docker 跑整合測試，in-memory sqlite 留給單元測試。觀察哪些程式碼需要改（提示：理論上只有設定，service/schema 一行都不該動——這就是分層的回報）。
  5. CI 加一道安全掃描：在 GitHub Actions workflow 加上 uv run pip-audit（或 bandit）這道關卡，對照 Java 的 OWASP dependency-check。讓它在有已知漏洞相依時 fail。
