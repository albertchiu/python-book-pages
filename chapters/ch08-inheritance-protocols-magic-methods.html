第 8 章：繼承、協定與魔術方法

繼承你太熟了，但 Python 在這塊有兩個 Java 沒有的領域要重新學：一是多重繼承與它背後的 MRO（方法解析順序），Java 用「單一繼承 + 介面」繞開的問題，Python 正面迎擊；二是結構型別（structural typing）的 Protocol，讓你能享受 duck typing 的彈性又拿回靜態檢查。再加上 Python 真正的超能力——魔術方法（magic / dunder methods），讓你的物件能用 len()、[]、+、() 這些內建語法。本章也會講為什麼在 Python 裡「組合優於繼承」這句老話更容易落實。

繼承與 `super()`

基本繼承跟 Java 幾乎一樣，差別在語法細節：

python 
class Animal:
    def __init__(self, name: str) -> None:
        self.name = name

    def speak(self) -> str:
        return "..."

class Dog(Animal):                       # class Dog(Animal) ≈ class Dog extends Animal
    def __init__(self, name: str, breed: str) -> None:
        super().__init__(name)           # 呼叫父類別的 __init__
        self.breed = breed

    def speak(self) -> str:              # override，不需要 @Override 標注
        return "Woof"

幾個對照重點：

  • super().__init__(name) 對應 Java 的 super(name)。但 Python 3 的 super() 不用帶參數（Python 2 要寫 super(Dog, self)，現在看到那種寫法就知道是舊碼）。
  • override 不需要 @Override。Python 沒有這個標注；子類別定義同名方法就覆蓋了。代價是「拼錯方法名變成新方法而非覆蓋」這種錯誤，編譯器不會幫你抓——這正是型別檢查工具（第 9 章）的價值之一。
  • 沒有 final。任何類別都能被繼承、任何方法都能被覆蓋。

🔁 Java 對照：super() 在單一繼承下就等於 Java 的 super。但下一節你會看到，在多重繼承裡 super() 的意義會超出「父類別」——它其實是「MRO 中的下一個」。

多重繼承與 MRO

Java 嚴禁多重類別繼承（只能 extends 一個類別，但能 implements 多個介面），就是為了躲開「菱形繼承」（diamond problem）的歧義。Python 允許多重繼承，並用 MRO（Method Resolution Order，方法解析順序） 明確定義「該找哪個父類別」。

MRO 由 C3 線性化演算法算出，存在 __mro__：

python 
class A:
    def greet(self) -> None:
        print("A")

class B(A):
    def greet(self) -> None:
        print("B")
        super().greet()

class C(A):
    def greet(self) -> None:
        print("C")
        super().greet()

class D(B, C):                  # 菱形：D 繼承 B、C，兩者都繼承 A
    def greet(self) -> None:
        print("D")
        super().greet()
python 
>>> [cls.__name__ for cls in D.__mro__]
['D', 'B', 'C', 'A', 'object']
>>> D().greet()
D
B
C
A

停下來看這個輸出的驚人之處：B.greet 裡的 super().greet() 呼叫的是 C.greet，不是 A.greet——即使 B 的程式碼裡 C 連影子都沒有。這就是 Java 工程師最容易誤解的地方：

⚠️ 陷阱：super() 不是「我的父類別」，而是「MRO 裡排在我後面的下一個類別」。在多重繼承中，super() 的目標取決於最終實例的 MRO，而不是定義 super() 那行所在的類別。把 super() 讀成「parent」會在菱形繼承裡完全猜錯。

這套機制叫協作式多重繼承（cooperative multiple inheritance）：每個類別都呼叫 super()，MRO 保證每個類別只被走訪一次（上面 A 只印一次，不是 B、C 各帶出一次）。要讓它正確運作，協作鏈上的方法簽章要相容（這也是為什麼多重繼承容易出錯、要謹慎使用）。

「簽章要相容」在 __init__ 上最容易翻車：MRO 上每個類別都要能把參數往下一棒傳，但它們各自需要的參數又不同。標準解法是每個協作式 __init__ 都收 **kwargs，取走自己要的、其餘原封不動往下傳：

python 
from typing import Any

class Base:
    def __init__(self, **kwargs: Any) -> None:
        super().__init__()                       # 鏈尾：object.__init__ 不收多餘參數，到此吃乾淨

class SerializableMixin:
    def __init__(self, *, fmt: str = "json", **kwargs: Any) -> None:
        self.fmt = fmt
        super().__init__(**kwargs)               # 取走 fmt，其餘往 MRO 下一棒傳

class TimestampedMixin:
    def __init__(self, *, created_at: str = "now", **kwargs: Any) -> None:
        self.created_at = created_at
        super().__init__(**kwargs)

class Document(SerializableMixin, TimestampedMixin, Base):
    def __init__(self, title: str, **kwargs: Any) -> None:
        self.title = title
        super().__init__(**kwargs)
python 
>>> d = Document("報告", fmt="yaml", created_at="2026-06-16")
>>> d.title, d.fmt, d.created_at
('報告', 'yaml', '2026-06-16')
>>> [c.__name__ for c in Document.__mro__]
['Document', 'SerializableMixin', 'TimestampedMixin', 'Base', 'object']

關鍵紀律：(1) mixin 用僅限關鍵字參數（* 後的 fmt、created_at）避免位置參數對位混亂；(2) 每一棒都 super().__init__(**kwargs) 把剩下的傳下去；(3) 鏈尾要有一個吃乾淨的終點（這裡 Base.__init__ 只呼叫無參數的 super().__init__()），否則多餘的 kwargs 傳到 object.__init__ 會 TypeError。這套寫法不容易維護，所以實務上能用組合就別走這條路——只有當你真的需要可疊加的有狀態 mixin 時才這麼寫。

💡 Pythonic 重點：多重繼承威力大但複雜。實務上最常見的「乾淨」用法是 mixin（後面會講）——用小而專一、不帶狀態的類別疊加能力。一般的「is-a」階層維持單一繼承，能用組合就別硬用多重繼承。

抽象類別：`abc.ABC` 與 `@abstractmethod`

對照 Java 的 abstract class：定義介面契約、強迫子類別實作、自己不能被實例化。Python 用 abc 模組：

python 
from abc import ABC, abstractmethod

class Shape(ABC):                        # 繼承 ABC 使它成為抽象基底類別
    @abstractmethod
    def area(self) -> float:
        ...                              # 沒有實作，子類別必須提供

    def describe(self) -> str:           # 抽象類別可以有具體方法
        return f"這個形狀面積是 {self.area():.2f}"

class Circle(Shape):
    def __init__(self, radius: float) -> None:
        self.radius = radius

    def area(self) -> float:             # 必須實作 area，否則 Circle 也是抽象的
        return 3.141592653589793 * self.radius ** 2
python 
>>> Shape()                              # 不能實例化抽象類別
Traceback (most recent call last):
    ...
TypeError: Can't instantiate abstract class Shape without an implementation for abstract method 'area'
>>> c = Circle(5)
>>> c.describe()
'這個形狀面積是 78.54'

跟 Java abstract 行為一致：少實作一個抽象方法，連子類別都不能被實例化。差別只在 Python 是「執行期」在你嘗試 Circle() 時才報錯，不是編譯期。

`Protocol`（結構型別）vs ABC（名義型別）

這是 Python 型別世界裡跟 Java 差最多、也最值得學的觀念。先把兩種「型別相容」的哲學講清楚：

  • 名義型別（nominal typing）：型別相容看「名字／宣告」。Java 的 class Dog implements Comparable 必須明寫 implements，編譯器才認。ABC 也是名義的——你得 class Circle(Shape) 或用 Shape.register(...)。
  • 結構型別（structural typing）：型別相容看「結構／有沒有對的方法」。只要長得對就算數，不需要宣告任何繼承關係。這正是 duck typing（第 5 章）的靜態版本。

Python 用 typing.Protocol 提供結構型別：

python 
from typing import Protocol

class Readable(Protocol):                # 定義「結構」：有 read() -> str 就算
    def read(self) -> str: ...

def dump(source: Readable) -> None:      # 只要傳進來的東西有 read()，型別檢查就過
    print(source.read())

class File:                              # 注意：File 沒有繼承 Readable！
    def read(self) -> str:
        return "file content"

class Socket:                            # 同樣沒繼承，但結構符合
    def read(self) -> str:
        return "socket bytes"

dump(File())        # ✅ mypy 接受
dump(Socket())      # ✅ mypy 接受

File 與 Socket 完全沒有提到 Readable，但因為它們都有 read(self) -> str，mypy 就認定它們符合 Readable。這在 Java 是做不到的——Java 的介面是名義的，File 一定要 implements Readable 編譯器才讓你傳。

🔁 Java 對照：Protocol ≈ 「不用 implements 的 interface」，概念上接近 Go 的 interface 或 TypeScript 的 structural typing。它讓你能事後為「你無法修改的第三方類別」定義契約——只要它結構符合就能用，不必去改它的繼承宣告。

需要在執行期用 isinstance 檢查結構時，加上 @runtime_checkable：

python 
from typing import Protocol, runtime_checkable

@runtime_checkable
class Readable(Protocol):
    def read(self) -> str: ...

print(isinstance(File(), Readable))      # True（執行期只檢查「有沒有 read」，不檢查簽章）

⚠️ 陷阱：@runtime_checkable 的 isinstance 不是 Java instanceof 的等價物。它只看「有沒有叫這個名字的方法」，不檢查參數、不檢查回傳型別，更不保證簽章相容——所以一個 read() 簽章其實不對的物件照樣會讓 isinstance(..., Readable) 回 True，等你真的呼叫才因簽章不符炸掉。Java 工程師習慣 instanceof 是可靠的型別判斷，這裡反直覺。真正的安全保證來自靜態 mypy 檢查（它會比對簽章）；runtime_checkable 只是執行期的粗略守門，別拿它當型別正確性的依據。此外它對「資料屬性型 Protocol」的支援也有限，主要設計是給「方法集合」用的。要「描述一個函式需要的能力、讓任何結構符合的物件都能傳進來（含第三方類別）」，用 Protocol。Protocol 是把 duck typing 的彈性與靜態檢查的安全感結合的關鍵工具。

魔術方法（dunder methods）

魔術方法（magic methods，因前後雙底線又叫 dunder methods）是 Python 的運算子重載與協定機制：實作對應的 dunder，你的物件就能用內建語法。第 7 章的 __init__、__repr__、__eq__、__hash__ 都是。這裡再補幾個高頻的，並做成一個完整範例。

你想用的語法 要實作的 dunder Java 對照
len(obj) __len__ size()（但 Java 不能讓 length 變語法）
obj[key] __getitem__ / __setitem__ get(i) / set(i, v)，無索引語法
obj() __call__ Function.apply / Runnable.run
a + b __add__ 無運算子重載
a < b __lt__ Comparable.compareTo
x in obj __contains__ contains(x)
for x in obj __iter__ Iterable.iterator（詳見第 6 章）

一個能用「+、len()、索引、==、repr」的二維向量：

python 
from __future__ import annotations

class Vector:
    def __init__(self, *components: float) -> None:
        self._values = list(components)

    def __repr__(self) -> str:
        inner = ", ".join(str(v) for v in self._values)
        return f"Vector({inner})"

    def __len__(self) -> int:                       # len(v)
        return len(self._values)

    def __getitem__(self, index: int) -> float:     # v[0]
        return self._values[index]

    def __eq__(self, other: object) -> bool:        # v1 == v2
        if not isinstance(other, Vector):
            return NotImplemented
        return self._values == other._values

    def __add__(self, other: Vector) -> Vector:     # v1 + v2
        if len(self) != len(other):
            raise ValueError("維度不一致")
        return Vector(*(a + b for a, b in zip(self._values, other._values)))
python 
>>> v1 = Vector(1, 2, 3)
>>> v2 = Vector(10, 20, 30)
>>> v1 + v2                  # __add__
Vector(11, 22, 33)
>>> len(v1)                  # __len__
3
>>> v1[2]                    # __getitem__
3
>>> v1 == Vector(1, 2, 3)    # __eq__
True

在 Java 你只能寫 v1.add(v2)、v1.get(2)、v1.size()——語言不准你重載 + 或 []。Python 讓自訂型別能融入內建語法，數值類別、容器類別寫起來特別自然。

⚠️ 陷阱：運算子重載要克制。+ 對向量、矩陣、金額這種「數學上本來就有加法語意」的型別很自然；但別硬把 + 拿去表達「把使用者加進群組」這種沒有直覺加法語意的操作，那會讓讀者更難懂。語意清楚才重載。

⚠️ 陷阱：上面的 Vector 只定義了 __eq__ 而沒定義 __hash__，於是 Python 自動把它的 __hash__ 設成 None，實例變成不可雜湊——hash(Vector(1, 2)) 會拋 TypeError: unhashable type: 'Vector'，也就不能當 dict 的 key 或放進 set。這是第 7 章 __eq__/__hash__ 規則的直接後果，但運算子重載一寫完最容易踩到（你正想把向量丟進 set 去重）。若需要可雜湊，依第 7 章規則一併定義 __hash__（例如 return hash(tuple(self._values))），前提是物件不可變——可變物件當 key 會在內容改變後找不回來。

💡 Pythonic 重點：只實作 __getitem__、__len__，你的物件就已經能被索引、能取長度、甚至能被舊式 for 迴圈迭代。要完整融入 Python 的容器生態，可以繼承 collections.abc 裡的 ABC（如 Sequence、Mapping）——實作少數幾個 dunder，就免費獲得一整組混入方法（如 __contains__、index、count）。

繼承 `collections.abc.Sequence`：少實作幾個、免費拿一整組

上面那句「免費獲得一整組混入方法」值得實際看一次。collections.abc.Sequence 是個 ABC：你只要實作 __getitem__ 與 __len__ 兩個抽象方法，它就用這兩個當基礎，自動補出 __contains__、__iter__、__reversed__、index、count：

python 
from __future__ import annotations
from collections.abc import Sequence
from typing import overload

class CharGrid(Sequence[str]):                 # 繼承 Sequence，型別參數是元素型別
    def __init__(self, *rows: str) -> None:
        self._rows = list(rows)

    def __len__(self) -> int:                  # 自己實作 1
        return len(self._rows)

    @overload
    def __getitem__(self, index: int) -> str: ...
    @overload
    def __getitem__(self, index: slice) -> Sequence[str]: ...
    def __getitem__(self, index: int | slice) -> str | Sequence[str]:  # 自己實作 2
        return self._rows[index]
python 
>>> g = CharGrid("abc", "def", "abc")
>>> "abc" in g              # __contains__：免費
True
>>> g.index("def")          # index：免費
1
>>> g.count("abc")          # count：免費
2
>>> list(reversed(g))       # __reversed__：免費
['abc', 'def', 'abc']
>>> [row for row in g]      # __iter__：免費（用 __getitem__ 逐一取，第 6 章的舊式迭代協定）
['abc', 'def', 'abc']

🔁 Java 對照：這正是 Java 的 AbstractList——你 extends AbstractList 後只要實作 get(int) 與 size()，就免費得到 contains、indexOf、iterator() 等。兩邊都是同一個 OO 設計：模板方法模式（template method pattern），父類別用「少數幾個抽象掛鉤」推導出「一整組具體行為」。差別在 Python 的版本同時是個結構面更鬆的工具：Sequence 也支援 register() 與 isinstance 檢查，且它生出的方法直接讓 in、reversed()、for 這些語法生效，而 Java 仍是方法呼叫。

⚠️ 陷阱：注意上面 __getitem__ 為什麼非用 @overload 不可。在 mypy --strict 下，子類別的 __getitem__ 簽章必須與 Sequence 的相容——而 Sequence.__getitem__ 是多載的：傳 int 回單一元素（str）、傳 slice 回子序列（Sequence[str]）。若只寫 def __getitem__(self, index: int) -> str，mypy 會判 override 不相容（Signature of "__getitem__" incompatible with supertype "typing.Sequence"）；就算改成單一函式 def __getitem__(self, index: int | slice) -> str | Sequence[str] 也還是不相容——因為那等於宣稱「傳 int 進來也可能回 Sequence[str]」，比父類別承諾的 int -> str 更寬。唯一乾淨過關（mypy 與 pyright 皆然）的寫法是用 @overload 把兩種情況分開精確標注。這提醒你：繼承一個有型別的 ABC，等於承諾履行它完整的契約（含切片），不是只實作你眼前想用的那一半。

`NotImplemented` 與反向運算：`__radd__`

前面 __eq__ 在不認得對方型別時 return NotImplemented，__add__ 遇到維度不符卻是 raise。為什麼一個回特殊值、一個拋例外？因為 NotImplemented 是運算子協定的核心機制：當左運算元的 __add__ 回 NotImplemented，Python 不會立刻放棄，而是改試右運算元的反向方法 __radd__（reflected add）。這正是 1 + vector 或與 numpy 互動時「誰被呼叫」的答案。

python 
from __future__ import annotations

class Vector:
    def __init__(self, *components: float) -> None:
        self._values = list(components)

    def __repr__(self) -> str:
        inner = ", ".join(str(v) for v in self._values)
        return f"Vector({inner})"

    def __add__(self, other: object) -> Vector:
        if isinstance(other, Vector):
            return Vector(*(a + b for a, b in zip(self._values, other._values)))
        if isinstance(other, (int, float)):              # 向量 + 純量
            return Vector(*(a + other for a in self._values))
        return NotImplemented                            # 不認得，交還給 Python

    def __radd__(self, other: object) -> Vector:         # 純量 + 向量 走到這
        return self.__add__(other)
python 
>>> v = Vector(1, 2, 3)
>>> v + 10                  # Vector.__add__
Vector(11, 12, 13)
>>> 10 + v                  # int.__add__(10, v) 回 NotImplemented → 改試 Vector.__radd__
Vector(11, 12, 13)

10 + v 時 Python 先問左邊的 int：(10).__add__(v)，int 不認得 Vector 而回 NotImplemented，Python 才轉去呼叫 v.__radd__(10)。

💡 Pythonic 重點：運算子相關 dunder 對「不認得的型別」要 return NotImplemented（不是 raise，也不是回 False）。回 NotImplemented 等於告訴 Python「我不會處理，換對方試試」——__eq__、__add__、__lt__ 全走這套協作協定。直接 raise 會切斷反向方法的機會；維度不符那種「型別對但值不合法」的情況才該 raise。

比較運算子與 `functools.total_ordering`

Java 實作 Comparable 只要寫一個 compareTo，<、<=、>、>= 全由它推得（雖然你還是得自己用 compareTo(o) < 0 拼）。Python 沒有單一的 compareTo，比較分散在 __lt__、__le__、__gt__、__ge__ 四個 dunder。手寫四個很煩，functools.total_ordering 幫你補齊：只要定義 __eq__ 與 __lt__，其餘四種比較自動生成。

python 
from __future__ import annotations
import functools

@functools.total_ordering
class Version:
    def __init__(self, major: int, minor: int) -> None:
        self.major, self.minor = major, minor

    def __eq__(self, other: object) -> bool:
        if not isinstance(other, Version):
            return NotImplemented
        return (self.major, self.minor) == (other.major, other.minor)

    def __lt__(self, other: Version) -> bool:    # 具體類別比較，參數標自身型別即可
        return (self.major, self.minor) < (other.major, other.minor)

（這裡 __lt__ 的參數標成具體的 Version 是對的——這是「自己跟自己比」的具體類別。但若哪天你要把 __lt__ 寫進一個當作泛型上界的 Protocol（「任何可比較的東西」），參數就得改標 Any 而非具體型別，否則 int 這類內建型別會被擋下。那是另一個脈絡，第 9 章談有界泛型時會詳細解釋為什麼。）

python 
>>> Version(1, 2) < Version(1, 5)
True
>>> Version(1, 2) >= Version(1, 5)     # total_ordering 由 __eq__ + __lt__ 自動補出
False

🔁 Java 對照：Comparable.compareTo 是「一個方法定義全序」；total_ordering 則是「定義 __eq__ + __lt__ 兩個方法，裝飾器幫你展開成六種運算子」。差別在 Python 直接讓你用 <、>= 這些運算子語法比較物件，而 Java 永遠是 a.compareTo(b) < 0。代價：total_ordering 生成的方法效能略低於手寫，極熱路徑才需要全部手刻。

mixin 模式

mixin 是「用多重繼承疊加能力」的乾淨用法：一個 mixin 是小而專一、通常不帶自己的 __init__ 狀態的類別，只提供一組方法，被「混入」其他類別。慣例命名以 Mixin 結尾。

python 
import json
from typing import Any

class AsDictMixin:
    def as_dict(self) -> dict[str, Any]:
        return self.__dict__               # 假設子類別的狀態都在實例 __dict__

class JSONMixin:
    def to_json(self) -> str:
        # 依賴「混入的對象也提供了 as_dict()」——這是 mixin 之間的隱性契約
        return json.dumps(self.as_dict())  # type: ignore[attr-defined]

class User(AsDictMixin, JSONMixin):        # 混入兩種能力
    def __init__(self, name: str, age: int) -> None:
        self.name = name
        self.age = age
python 
>>> u = User("Albert", 42)
>>> u.to_json()
'{"name": "Albert", "age": 42}'

User 自己只管「我有什麼資料」，序列化能力靠 mixin 疊上去。對照 Java：Java 8 之後 interface 的 default method 提供了類似「混入行為」的能力（介面帶實作），是最接近 mixin 的對應物——但 Java 的 default method 不能持有狀態，而 Python mixin 可以（雖然慣例上盡量不要）。

💡 Pythonic 重點：mixin 要遵守幾條紀律：(1) 名字以 Mixin 結尾、職責單一；(2) 盡量別有自己的 __init__；(3) 清楚記載它對「被混入對象」的隱性要求（上例 JSONMixin 要求對方有 as_dict()）。違反這些，多重繼承的複雜度就會反咬你。

組合優於繼承在 Python 中的實踐

「組合優於繼承」（composition over inheritance）是 OOP 共識，但 Python 的動態特性讓委派（delegation）特別好寫，所以更容易落實。基本款就是「持有一個物件、把工作轉發給它」：

python 
class Engine:
    def start(self) -> str:
        return "引擎啟動"

class Car:
    def __init__(self) -> None:
        self._engine = Engine()          # 持有，而非繼承

    def start(self) -> str:
        return self._engine.start()      # 委派

當要轉發的方法很多、不想一個個手寫時，Python 可以用 __getattr__ 做自動委派——它在「正常屬性查找失敗」時被呼叫：

python 
from typing import Any

class Logger:
    def info(self, msg: str) -> None:
        print(f"[INFO] {msg}")
    def error(self, msg: str) -> None:
        print(f"[ERROR] {msg}")

class Service:
    def __init__(self) -> None:
        self._logger = Logger()

    def __getattr__(self, name: str) -> Any:    # 委派目標型別不定，標 Any 通過 mypy --strict
        # 只有當 Service 自己找不到該屬性時才會進來，自動轉發給 _logger
        return getattr(self._logger, name)
python 
>>> s = Service()
>>> s.info("started")        # Service 沒有 info，__getattr__ 轉發給 _logger
[INFO] started
>>> s.error("oops")
[ERROR] oops

這種「不繼承 Logger，卻能用它所有方法」的彈性，在 Java 要嘛逐一手寫轉發方法、要嘛用動態代理（dynamic proxy）才能達成。

💡 Pythonic 重點：判準跟 Java 一樣——「is-a」用繼承，「has-a / 用其能力」用組合。Python 額外的好處是：(1) duck typing 讓你不需要為了「能被當某型別用」而硬去繼承（有對的方法就行）；(2) __getattr__ 讓委派幾乎零樣板。所以在 Python 裡，遇到「想繼承只是為了重用幾個方法」時，更該停下來想想組合。

本章小結

  1. super() 在 Python 3 不帶參數，override 不需 @Override，沒有 final——彈性更大，但拼錯方法名這類錯誤要靠型別檢查工具補。
  2. 多重繼承靠 MRO（C3 線性化）解決菱形問題。super() 的真正意義是「MRO 中的下一個」，不是「我的父類別」——這是多重繼承最大的認知陷阱。
  3. abc.ABC + @abstractmethod 對應 Java abstract class：定義契約、強迫實作、抽象類別不能實例化（執行期報錯）。
  4. ABC 是名義型別、Protocol 是結構型別。ABC 要明寫繼承；Protocol 只看「有沒有對的方法」，能事後為第三方類別定契約——Java 介面做不到。@runtime_checkable 的 isinstance 只查方法名是否存在、不查簽章，不是 Java instanceof 的等價物，安全保證來自靜態 mypy。
  5. 魔術方法讓自訂物件融入內建語法：__len__、__getitem__、__call__、__add__ 等對應 len()、[]、()、+。Java 沒有運算子重載。重載要語意清楚、克制使用。運算子 dunder 對不認得的型別要 return NotImplemented（讓 Python 改試對方的反向方法如 __radd__），比較運算子可用 functools.total_ordering 由 __eq__ + __lt__ 自動補齊；另記得自訂 __eq__ 而不定 __hash__ 會讓實例變不可雜湊。
  6. mixin 是多重繼承的乾淨用法：小而專一、命名以 Mixin 結尾、職責單一，最接近 Java 8 介面的 default method（但 mixin 可帶狀態）。
  7. 組合優於繼承在 Python 更易落實：__getattr__ 讓自動委派幾乎零樣板，duck typing 讓你不必為「能被當某型別用」而強行繼承。

練習

練習 1（看穿 MRO） 不要執行，先用紙筆推出下面 D().m() 的輸出與 D.__mro__，再執行驗證。說明 B.m 裡的 super().m() 為什麼會跑到 C.m。

python 
class A:
    def m(self): print("A")
class B(A):
    def m(self): print("B"); super().m()
class C(A):
    def m(self): print("C"); super().m()
class D(B, C):
    def m(self): print("D"); super().m()

練習 2（ABC 強制契約） 用 abc.ABC 定義一個 Repository 抽象類別，含抽象方法 get(id: int) 與 save(item)，並提供一個具體方法 get_or_raise(id)（找不到就 raise）。寫一個只實作了 get 卻忘了 save 的子類別，觀察實例化時的錯誤訊息，並修好它。

練習 3（ABC 換成 Protocol） 把練習 2 的 Repository 改寫成 Protocol，並寫一個函式 backup(repo: Repository)。接著寫一個沒有繼承 Repository 但結構符合的 InMemoryRepo，把它傳進 backup。說明這跟練習 2 的 ABC 版在「相容性如何判定」上的本質差別。

練習 4（魔術方法做一個區間型別） 實作一個 Range(start, end)，支援：len(r)（長度）、x in r（__contains__，判斷是否落在區間）、r == Range(...)（__eq__）、repr(r)（顯示成 Range(1, 5)）。寫幾行驗證每個語法。

練習 5（組合 + 自動委派） 你有一個第三方的 dict，想做一個 TrackedDict：每次 __setitem__ 都印出「set key=...」，但其餘所有 dict 操作（len、in、get、keys…）照常運作。用「持有一個 dict + __getattr__ 委派」實作，而不是繼承 dict。說明為什麼這裡用組合比繼承 dict 更安全（提示：繼承內建型別時，內部方法不一定會回頭呼叫你覆寫的 __setitem__）。
