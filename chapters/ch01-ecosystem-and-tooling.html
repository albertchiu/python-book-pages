第 1 章：Python 生態系與開發環境

身為資深 Java 工程師，你對「編譯、依賴管理、IDE、建置工具」這條工具鏈早已駕輕就熟——javac、Maven、Gradle、JUnit 各司其職，邊界清晰。進到 Python，你會發現規則重新洗牌：有人說「Python 是直譯語言所以不用編譯」（其實不對）、依賴管理的歷史包袱讓初學者一頭霧水（pip、venv、poetry、conda 到底用哪個）、而型別檢查竟然是「選用的」。本章要做的，是把 Python 的執行模型與現代工具鏈一次講清楚，並用你熟悉的 JVM 與 Maven/Gradle 當座標，讓你在第一天就建立正確的心智模型，避免踩進那些「Java 沒有、Python 才有」的坑。

1.1 執行模型：CPython、bytecode 與虛擬機

Java 怎麼做

Java 的執行流程你閉著眼睛都能畫出來：

text 
.java  --javac-->  .class (bytecode)  --classloader-->  JVM 解譯 + JIT 編譯 --> 機器碼

關鍵是編譯與執行是兩個明確分開的步驟。你先跑 javac，把原始碼轉成平台無關的 .class bytecode；執行期 JVM（HotSpot）載入 bytecode，先解譯，熱點程式碼再由 JIT 編譯成原生機器碼。

Python 怎麼做

很多人說「Python 是直譯語言（interpreted language）」，這句話會誤導你。真相是：Python 也會編譯成 bytecode，只是這個步驟是隱形且即時的。

text 
.py  --compile-->  bytecode (.pyc / __pycache__)  -->  PVM (Python Virtual Machine) 逐條執行

當你執行 python foo.py，CPython（Python 的官方參考實作，用 C 寫成）做了這些事：

  1. 詞法/語法分析：把 .py 解析成抽象語法樹（AST）。
  2. 編譯：把 AST 編譯成 bytecode（一串虛擬機指令）。
  3. 執行：由 PVM（一個基於堆疊的虛擬機，本質是一個巨大的 C 語言 evaluation loop）逐條解譯 bytecode。

你可以親眼看到 bytecode。dis 模組就是 Python 版的 javap -c：

python 
>>> import dis
>>> def add(a, b):
...     return a + b
...
>>> dis.dis(add)
  1           0 RESUME                   0

  2           2 LOAD_FAST                0 (a)
              4 LOAD_FAST                1 (b)
              6 BINARY_OP                0 (+)
             10 RETURN_VALUE

（以上為 Python 3.12 的輸出。最左欄是原始碼行號，第二欄是指令在 bytecode 中的位元組偏移量，接著才是 opcode 與運算元；LOAD_FAST 0 (a) 的 0 是區域變數 a 的索引。）

注意：實際 opcode 名稱會隨 CPython 版本演進。例如 Python 3.14 引入 LOAD_FAST_BORROW 之類的 fused/借用指令，會把上面兩條 LOAD_FAST 融合成單一條 LOAD_FAST_BORROW_LOAD_FAST_BORROW 1 (a, b)；Python 3.13 則是 LOAD_FAST_LOAD_FAST 1 (a, b)。重點不是記住指令，而是理解「Python 原始碼確實被編譯成了 bytecode」。

`__pycache__` 與 `.pyc`：被快取的 bytecode

那 .class 在哪？對應物是 .pyc，但只有被 import 的模組才會把編譯結果快取下來。你執行的主腳本（python foo.py 的 foo.py）不會留下 .pyc，但它 import 的模組會：

python 
>>> import py_compile
>>> py_compile.compile("greet.py")   # 手動觸發編譯
>>> import os
>>> os.listdir("__pycache__")
['greet.cpython-314.pyc']

.pyc 檔名帶有實作與版本標記（cpython-314），這是刻意的：不同版本的 bytecode 格式不相容，所以要隔離快取。.pyc 純粹是編譯快取——下次 import 時若原始碼沒變（比對 mtime 或 hash），就直接載入 bytecode 跳過編譯，加快啟動。它不是給你拿去獨立散佈執行的產物（這點跟 .class 很不一樣）。

💡 Pythonic 重點：__pycache__ 是自動產生的快取，不需要納入版控。所有 Python 專案的 .gitignore 第一條幾乎都是 __pycache__/。

🔁 Java 對照：.pyc ≈ .class，但前者是隱形快取、版本綁定、不對外散佈；後者是你建置流程的一級產物。

「直譯」與「編譯」的誤解，一次講清

面向 Java Python (CPython)
編譯步驟 顯式（javac） 隱式（import / 執行時自動）
中間產物 .class bytecode .pyc bytecode（快取）
執行引擎 JVM（堆疊式 VM） PVM（堆疊式 VM）
JIT HotSpot 內建 JIT CPython 無 JIT（3.13 起有實驗性 JIT）
啟動成本 JVM 暖機較重 直接跑，啟動快

所以更精確的說法是：Python 是「編譯成 bytecode 後在虛擬機上解譯執行」的語言——在這點上它和 Java 的模型驚人地相似。最大的差別是：CPython 預設沒有 JIT，bytecode 就是一條一條解譯，不會像 HotSpot 那樣把熱點編成機器碼。這正是 CPython 在純 CPU 密集運算上比 Java 慢的根本原因。

其他實作：CPython 為何是事實標準

Python 是一份語言規格，可以有多種實作，就像 Java 有 HotSpot、OpenJ9、GraalVM：

實作 對照 特性
CPython HotSpot（事實標準） 官方參考實作，C 寫成，生態系預設目標
PyPy GraalVM / HotSpot JIT 帶 JIT 的實作，純 Python 數值運算可快數倍
Jython 跑在 JVM 上的腳本語言 編譯成 JVM bytecode，可呼叫 Java 函式庫（但停在 Python 2.7，已式微）
GraalPy GraalVM 多語言 Oracle 的 GraalVM 上的 Python，可與 Java 互通

為什麼 CPython 是事實標準？因為生態系。NumPy、pandas、PyTorch 這類效能關鍵套件大量使用 C 擴充（CPython 的 C API），這些 C 擴充在 PyPy/Jython 上往往無法直接運作或需要相容層。對 99% 的工作，「Python」就等於「CPython」。本書所有範例都以 CPython 3.12+ 為準。

⚠️ 陷阱：別急著為了效能跳去 PyPy。PyPy 對長時間運行的純 Python 數值程式很有效，但若你的瓶頸在 NumPy（底層已是 C/Fortran），PyPy 幫不上忙，反而可能因相容性問題踩雷。先量測再優化。

關於 GIL（Global Interpreter Lock）：CPython 有一把全域鎖，同一時刻只有一個執行緒能執行 Python bytecode——這是 Java 工程師最大的認知衝擊之一（你習慣的 Thread 能真正並行跑 CPU 工作，Python 的 threading 在 CPU 密集任務上不行）。這牽涉並行模型的根本選擇，我們留到第 14 章專門深入，這裡先記住「它存在、它會影響你選 threading 還是 multiprocessing」即可。

1.2 虛擬環境（venv）：一個 Java 工程師沒有的痛點

為什麼需要隔離

先理解 Java 為什麼沒有這個問題。在 Maven/Gradle 世界，依賴是每專案獨立的：jar 放在本地 ~/.m2 倉庫，但每個專案的 classpath 由它自己的 pom.xml/build.gradle 決定。專案 A 用 Guava 31、專案 B 用 Guava 32，彼此井水不犯河水——因為依賴解析是專案層級的。

Python 預設不是這樣。pip install requests 預設裝到全域 site-packages，全機所有 Python 程式共用。後果：

  • 專案 A 需要 django==3.2，專案 B 需要 django==4.2——全域只能裝一個版本，直接衝突。
  • 你裝的套件污染了系統 Python，某天系統工具因為版本被你升級而壞掉。

這就是「相依地獄（dependency hell）」。虛擬環境（virtual environment, venv） 就是解法：為每個專案建立一個獨立的、隔離的 Python 環境與其專屬的 site-packages。

⚠️ 陷阱：現代發行版（含 Homebrew 安裝的 Python）多依 PEP 668 把系統 Python 標記為 externally-managed（在其標準函式庫目錄放一個 EXTERNALLY-MANAGED 檔）。此時直接對系統 Python 跑 pip install 會被擋下，並提示你改用 venv 或 pipx。這從另一個角度強化了 venv 的必要性——連 Python 自己都不希望你污染全域。

🔁 Java 對照：venv 提供的隔離 ≈ Maven/Gradle「每專案獨立 classpath」這件你本來就免費享有的事。差別是 Java 自動做到，Python 要你顯式建立並啟用一個 venv。

建立與使用

bash 
# 在專案根目錄建立虛擬環境（慣例命名 .venv）
python3 -m venv .venv

# 啟用（macOS / Linux）
source .venv/bin/activate

# 啟用（Windows PowerShell）
.venv\Scripts\Activate.ps1

# 啟用後，提示字元前會出現 (.venv)，此時 python / pip 都指向這個環境
(.venv) $ which python
/path/to/project/.venv/bin/python

# 停用
(.venv) $ deactivate

`.venv` 裡面有什麼

text 
.venv/
├── bin/              # 可執行檔（python、pip、以及套件提供的 CLI）
│   ├── python -> ...  # 指向直譯器的符號連結
│   ├── pip
│   └── activate      # 啟用腳本（修改 PATH 與 PROMPT）
├── lib/
│   └── python3.12/
│       └── site-packages/   # 這個環境專屬的套件就裝在這
├── include/
└── pyvenv.cfg        # 設定檔，記錄基底直譯器路徑等

「啟用」做的事其實很單純：把 .venv/bin 插到 PATH 最前面，於是 python 與 pip 都解析到 venv 內的版本。pip install 自然就裝進這個環境的 site-packages，與全域隔離。

💡 Pythonic 重點：.venv 是環境而非設定，絕對不要進版控（.gitignore 它）。你版控的是「如何重建環境」的描述（pyproject.toml / lock 檔），而不是環境本身——就像你版控 pom.xml 而非 ~/.m2。

⚠️ 陷阱：很多新手忘了啟用 venv 就 pip install，結果裝到了全域、或裝到另一個專案的環境。養成「進專案先確認 (.venv) 出現」的習慣。後面介紹的 uv 會幫你免去手動啟用的麻煩。

1.3 套件與專案設定：從 pip 到 pyproject.toml

pip 與 requirements.txt 的不足

pip 是 Python 的套件安裝器（≈ 你直接呼叫 Maven 的依賴下載功能）。最傳統的依賴宣告方式是 requirements.txt：

text 
# requirements.txt
requests==2.32.3
flask>=3.0
bash 
pip install -r requirements.txt
pip freeze > requirements.txt   # 把當前環境凍結成清單

這套老方法有幾個 Java 工程師一眼能看穿的問題：

  1. 沒有專案元資料：requirements.txt 只是一串套件，沒有專案名稱、版本、進入點、建置設定。它不是 pom.xml，比較像光禿禿的 dependency 區段。
  2. 直接依賴與間接依賴混在一起：pip freeze 把所有安裝的東西（含遞移依賴）攤平列出，你看不出哪些是你真正想要的、哪些是被拖進來的。對照 Maven 你能清楚區分 direct 與 transitive dependency。
  3. 不可重現（non-reproducible）：flask>=3.0 這種範圍宣告，今天解析到 3.0.3、下個月可能解析到 3.1.0，導致「在我機器上能跑」。

現代標準：pyproject.toml（PEP 621）

pyproject.toml 是現在 Python 專案設定的單一事實來源（single source of truth），地位等同 pom.xml / build.gradle。PEP 621 標準化了 [project] 區段：

toml 
[project]
name = "greeter"
version = "0.1.0"
description = "A friendly greeting service"
readme = "README.md"
requires-python = ">=3.12"
dependencies = [
    "httpx>=0.28",
    "rich>=13.0",
]

[project.optional-dependencies]
dev = ["ruff", "mypy", "pytest"]

[project.scripts]
greeter = "greeter.cli:main"   # 安裝後可在終端機直接打 greeter，≈ Maven 的可執行 jar 進入點

[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

對照表：

pom.xml / build.gradle pyproject.toml 說明
<groupId>/<artifactId> [project] name 專案座標
<version> [project] version 版本
<dependencies> [project] dependencies 直接依賴
<scope>test</scope> [project.optional-dependencies] dev 開發/測試依賴
Maven 外掛 build 設定 [build-system] 建置後端
<java.version> requires-python 執行環境要求

💡 Pythonic 重點：pyproject.toml 不只放依賴，還是 ruff、mypy、pytest 等工具的共用設定檔（用 [tool.ruff]、[tool.mypy] 區段）。一個檔案統管整個工具鏈——比 Java 散落在多個 plugin 設定中乾淨。

lock 檔：可重現建置

pyproject.toml 宣告的是意圖（「我要 httpx 0.28 以上」），但要做到可重現建置（reproducible build），你需要鎖定確切解析出來的整棵依賴樹與雜湊值。這就是 lock 檔的角色，對照觀念是 Gradle 的 gradle.lockfile 或 npm 的 package-lock.json。

差別在於：pyproject.toml 進版控且人類可讀可編輯；lock 檔進版控但由工具產生、不該手改。團隊成員與 CI 用 lock 檔安裝，就能保證大家拿到位元級相同的依賴集合，徹底消滅「在我機器上能跑」。

過去 Python 在 lock 檔這塊一片混亂（各家工具各搞一套格式），這正是下一節主角 uv 帶來的最大價值之一。

1.4 uv：現代一體化工具

過去 Python 工具鏈是「拼裝車」：venv 管環境、pip 裝套件、pip-tools/poetry 管 lock、pyenv 管 Python 版本……每個工具一個指令、一套心智模型。uv（用 Rust 寫成）把這些全部整合進一個指令，而且極快——依賴解析與安裝常比 pip 快一到兩個數量級。

🔁 Java 對照：uv 之於 Python，就像 Gradle wrapper 給你的「一個入口管全部」體驗：建環境、解依賴、裝套件、跑程式、鎖版本，全在一個工具裡——只是它快到讓你懷疑人生。

安裝（macOS/Linux）：

bash 
curl -LsSf https://astral.sh/uv/install.sh | sh

核心指令：

uv 指令 作用 Maven/Gradle 類比
uv init 初始化新專案（建 pyproject.toml、git、範本） mvn archetype:generate / gradle init
uv add <pkg> 加依賴並安裝、同步更新 lock 編輯 pom.xml + 下載
uv remove <pkg> 移除依賴 反向操作
uv sync 依 lock 檔重建環境 mvn dependency:resolve（鎖定版本）
uv run <cmd> 在專案環境中執行命令（自動確保環境同步） gradle run / ./gradlew <task>
uv lock 重新解析並寫出 lock 檔 更新 lockfile
uv python install 3.12 安裝指定 Python 版本 ≈ 管理 toolchain（pyenv 的活）

uv 還有個殺手級體驗：你幾乎不用手動啟用 venv。uv run 會自動建立並使用 .venv、確保依賴與 lock 一致，然後在那個環境裡跑你的命令。.python-version 檔則記錄專案要用哪個 Python 版本，uv 會自動下載安裝對應版本——連 pyenv 都省了。

1.5 工具鏈：ruff 與 mypy

ruff：linter + formatter 一體（且快上百倍）

Java 端你大概用過一整套：Checkstyle（風格）、SpotBugs/PMD（找 bug）、google-java-format / Spotless（排版）。每個工具獨立設定、各跑各的。

ruff（同樣 Rust 寫成）把 linter 與 formatter 合而為一，涵蓋數百條規則（吸收了 Flake8、isort、pyupgrade、pydocstyle 等老工具的能力），而且快到誇張——大型專案的檢查常在毫秒級完成。

bash 
uv run ruff check .          # 檢查（≈ Checkstyle + SpotBugs + isort）
uv run ruff check --fix .    # 自動修可修的問題
uv run ruff format .         # 排版（≈ google-java-format）

實際抓出問題的樣子（對一段刻意寫爛的程式碼）：

text 
$ ruff check bad.py
F401 [*] `os` imported but unused
 --> bad.py:1:8
  |
1 | import os
  |        ^^
2 |
3 | def f():
  |
help: Remove unused import: `os`

F841 Local variable `y` is assigned to but never used
 --> bad.py:4:5
  |
3 | def f():
4 |     y = 1
  |     ^
5 |     return 2
  |
help: Remove assignment to unused variable `y`

Found 2 errors.
[*] 1 fixable with the `--fix` option (1 hidden fix can be enabled with the `--unsafe-fixes` option).

每個診斷下方會貼出對應的程式碼片段（含 | 與指向問題位置的 ^）與 help: 修法建議；結尾的 [*] 標示有多少問題能用 --fix 自動修好。

設定就寫在 pyproject.toml：

toml 
[tool.ruff]
line-length = 100
target-version = "py312"

[tool.ruff.lint]
select = ["E", "F", "I", "UP"]   # 啟用 pycodestyle/pyflakes/isort/pyupgrade 規則集

🔁 Java 對照：ruff check ≈ Checkstyle + SpotBugs + import 排序，ruff format ≈ google-java-format。差別是它們合一、設定集中、且快上百倍。

mypy / pyright：靜態型別檢查（選用、漸進式）

這是 Java 工程師最需要調整心態的地方。Java 的型別檢查是強制的——javac 不過你就無法產出 .class。Python 的型別檢查是選用且漸進的（gradual typing）：你可以加型別提示（type hints），也可以不加；直譯器執行時不會強制它們，型別提示純粹是給靜態檢查工具與 IDE 看的標註。

python 
def greet(name: str) -> str:
    return "Hello, " + name

greet(42)   # 直譯器照跑不誤……直到 "Hello, " + 42 在執行期才 TypeError

把 mypy（或 Microsoft 的 pyright，VS Code Pylance 的引擎）接上去，這個錯誤在寫程式當下就被抓到：

text 
$ mypy typed.py
typed.py:4: error: Argument 1 to "greet" has incompatible type "int"; expected "str"  [arg-type]
Found 1 error in 1 file (checked 1 source file)
面向 Java（javac） Python（mypy/pyright）
是否強制 是，不過編譯不出檔 否，獨立工具、選用
執行期效果 型別保證 提示在執行期被忽略（無保證）
採用方式 全有或全無 漸進式，可逐檔逐函式加
典型用法 編譯流程內建 CI 內或 pre-commit 跑

💡 Pythonic 重點：別把型別提示當負擔，把它當成你帶進 Python 的「工程紀律」。在共享的函式庫、模組邊界、資料模型上加好型別 + mypy/pyright，你就把 Python 的安全網拉到接近 Java 的水準。型別系統的深入（泛型、Protocol、TypeVar）見第 9 章。

⚠️ 陷阱：型別提示不會在執行期驗證。greet(42) 不會因為標了 name: str 就在進入函式時報錯——它照樣執行，直到實際的型別不相容運算才炸。要執行期驗證請用 pydantic（第 9 章）。

IDE：VS Code 與 PyCharm

  • VS Code + Python/Pylance 擴充：輕量、免費、生態活躍。Pylance 底層是 pyright，提供即時型別檢查與自動補全。搭配 ruff 擴充可存檔自動排版。多數現代 Python 開發者的首選。
  • PyCharm（JetBrains）：若你來自 IntelliJ IDEA，這是最無痛的過渡——重構、除錯、測試整合一應俱全，介面與快捷鍵幾乎一致。Community 版免費足以應付多數需求。

兩者都能讀懂 pyproject.toml 與 .venv，自動把專案環境接上。

1.6 REPL 與互動式開發：Java 工程師相對陌生的工作流

Java 直到 9 才有 jshell，而且用的人不多——你的肌肉記憶是「改 code → 編譯 → 跑」的批次循環。Python 的核心工作流之一是 REPL（Read-Eval-Print Loop）：開一個直譯器，逐行輸入、立即看結果，探索式地試 API、驗證想法。

python 
$ python3
>>> import datetime
>>> datetime.date.today()
datetime.date(2026, 6, 14)
>>> [x**2 for x in range(5)]
[0, 1, 4, 9, 16]
>>> help(str.split)   # 直接在 REPL 查文件

這種「想到就試、試了就知道」的即時回饋，是 Python 探索性極強的原因。不確定某個函式回傳什麼？貼進 REPL 跑一下，比翻文件快。

進階選項：

  • IPython：增強版 REPL。tab 補全、語法高亮、? 查物件（str.split?）、%timeit 微基準、shell 整合（!ls）。互動開發的標配。
  • Jupyter Notebook/Lab：把 REPL 變成可儲存的「筆記本」，程式碼、輸出、圖表、文字混排。資料科學與教學的主流；對需要「邊算邊看圖」的探索工作極強大。
bash 
uv run --with ipython ipython     # 不污染專案，臨時拉 ipython 起來玩

💡 Pythonic 重點：把 REPL 當成你的第二雙手。寫複雜的 comprehension 或 regex 前，先在 REPL 拆小步驗證，遠比寫完整支程式再 debug 有效率。uv run --with <pkg> 能臨時引入工具而不動到專案依賴，很適合一次性探索。

🔁 Java 對照：REPL ≈ jshell，但 Python 的 REPL 是生態系的一級公民，從教學到生產除錯無所不在，遠比 jshell 在 Java 圈的存在感強。

1.7 完整走查：從零建立一個最小專案

把上面所有概念串起來。我們用 uv 從零做一個會打招呼的小工具，跑完依賴、lint、型別檢查、執行的完整循環。

步驟 1：初始化專案

bash 
$ uv init --name greeter demo-greeter
Initialized project `greeter` at /path/to/demo-greeter

$ cd demo-greeter
$ ls -A
.git  .gitignore  .python-version  main.py  pyproject.toml  README.md

uv init 一次給你：git 倉庫、.gitignore（已包含 .venv、__pycache__）、鎖定 Python 版本的 .python-version、範本 main.py、以及 pyproject.toml：

toml 
[project]
name = "greeter"
version = "0.1.0"
description = "Add your description here"
readme = "README.md"
requires-python = ">=3.12"
dependencies = []

步驟 2：寫一支小程式

把 main.py 改成用 httpx 抓資料、用型別提示寫清楚：

python 
# main.py
import httpx


def fetch_user_name(user_id: int) -> str:
    """從 JSONPlaceholder 取得使用者名稱。"""
    resp = httpx.get(f"https://jsonplaceholder.typicode.com/users/{user_id}")
    resp.raise_for_status()
    data: dict[str, object] = resp.json()
    return str(data["name"])


def main() -> None:
    name = fetch_user_name(1)
    print(f"Hello, {name}!")


if __name__ == "__main__":
    main()

（if __name__ == "__main__": 的完整意義見第 11 章；現在先理解成「這支檔案被直接執行時才跑 main()」，類似 Java 的 public static void main。）

注意這裡刻意寫 data: dict[str, object] 而非裸 dict：JSON 回應的值型別不一（name 是字串，但 id 是數字、address 是巢狀物件），用 object 如實標註；取出 data["name"] 後再用 str(...) 收斂回宣告的回傳型別 str。這個範本在 mypy 預設與 mypy --strict 下都是全綠的——可以安心當成你的個人起手式。

步驟 3：加依賴

bash 
$ uv add httpx
Using CPython 3.12.12
Creating virtual environment at: .venv
Resolved 8 packages in 336ms
Installed 7 packages in 14ms
 + httpx==0.28.1
 + ...（含遞移依賴 anyio、certifi、h11、httpcore、idna、typing-extensions）

注意 uv 自動完成了三件事：建立 .venv（你沒下任何 venv 指令）、解析並安裝整棵依賴樹、寫出 uv.lock。pyproject.toml 的 dependencies 也自動更新成 ["httpx>=0.28.1"]。uv.lock 釘死了每個套件的確切版本與雜湊——這就是可重現建置的保證，請進版控。

加開發工具到獨立的依賴群組（不會混進正式依賴）：

bash 
$ uv add --dev ruff mypy

步驟 4：ruff 與 mypy 把關

bash 
$ uv run ruff check .
All checks passed!

$ uv run ruff format .
1 file left unchanged

$ uv run mypy main.py
Success: no issues found in 1 source file

（uv init 產生的範本本來就是 ruff 格式化過的，加上我們上面照規範手寫的程式也合乎排版，所以 ruff format 回報 1 file left unchanged；只有當檔案實際被改動過排版時才會看到 1 file reformatted。）

刻意製造一個型別錯誤試試——把 fetch_user_name(1) 改成 fetch_user_name("1")（傳字串給標註為 int 的參數）：

text 
$ uv run mypy main.py
main.py:14: error: Argument 1 to "fetch_user_name" has incompatible type "str"; expected "int"  [arg-type]
Found 1 error in 1 file (checked 1 source file)

mypy 在執行前就攔下了這個 Java 工程師最在意的型別錯誤。改回來。

步驟 5：執行

bash 
$ uv run main.py
Hello, Leanne Graham!

uv run 在背後確保 .venv 與 lock 同步後才執行——你從頭到尾沒有手動啟用過 venv。

步驟 6：在另一台機器重現

同事 clone 你的 repo（含 pyproject.toml 與 uv.lock，但不含 .venv）後，只要：

bash 
$ uv sync     # 依 uv.lock 重建出位元級相同的環境
$ uv run main.py
Hello, Leanne Graham!

整個循環——初始化、寫程式、加依賴、lint、型別檢查、執行、重現——就是你未來每個 Python 專案的骨架。它在概念上完全對應你熟悉的 Maven/Gradle 工作流，只是更快、更集中、心智負擔更低。

本章小結

  1. Python 不是「純直譯」：CPython 會把 .py 編譯成 bytecode（快取於 __pycache__/*.pyc），再由 PVM 解譯執行——模型上與 Java 的 javac → .class → JVM 高度相似，最大差別是 CPython 預設無 JIT。
  2. CPython 是事實標準：PyPy（帶 JIT）、Jython、GraalPy 各有定位，但因為 C 擴充生態系，實務上「Python」≈「CPython」。GIL 的影響留待第 14 章。
  3. venv 解決 Java 工程師沒遇過的痛：Python 預設裝到全域 site-packages，必須用虛擬環境做每專案隔離（Java 的 per-project classpath 是免費的）。.venv 不進版控。
  4. pyproject.toml 是新標準：它是 pom.xml/build.gradle 的對應物，統管依賴與工具設定；requirements.txt 的攤平、不可重現問題由 lock 檔解決。
  5. uv 一統江湖：環境、依賴解析、lock、執行、Python 版本管理全整合，且極快。記住 uv init / uv add / uv run / uv sync 四個動詞就能走完全流程。
  6. ruff = linter + formatter 合一（≈ Checkstyle + SpotBugs + format，但快上百倍）；mypy/pyright 是選用、漸進式的型別檢查——把它當成你帶進 Python 的工程紀律。
  7. 型別提示執行期不強制：標了 name: str 不代表執行期會擋下 int，要靠 mypy/pyright 在 CI 把關，或用 pydantic 做執行期驗證（第 9 章）。
  8. REPL 是一級工作流：python / ipython / Jupyter 的「逐行試、立即看」探索模式，是比 jshell 普及得多的 Python 開發習慣。

練習

  1. 看穿 bytecode：在 REPL 用 dis.dis 反組譯一個含 if/else 的函式，找出對應的條件跳轉 opcode。再寫一個等價的三元運算式 x if cond else y（Python 沒有 ?:，見第 2 章），比較兩者 bytecode 差異。

    • 提示：import dis; dis.dis(your_func)，留意 POP_JUMP_IF_FALSE 之類的指令。

  2. 製造並理解相依衝突：建兩個 uv 專案，一個 uv add "httpx<0.27"、另一個 uv add "httpx>=0.28"，確認兩者的 .venv 互不影響。思考：若沒有 venv，這兩個版本要如何共存於同一台機器？

    • 提示：分別在各專案 uv run python -c "import httpx; print(httpx.__version__)"。

  3. 把 requirements.txt 升級成 pyproject.toml：給定一份 requirements.txt（含 requests、flask、pytest），用 uv init 起新專案，把正式依賴用 uv add、測試依賴用 uv add --dev 分別加入，產生 pyproject.toml 與 uv.lock。對照思考它與 Maven compile/test scope 的關係。

  4. 讓 mypy 抓到一個真實 bug：寫一個函式 def average(nums: list[int]) -> float，但內部故意回傳 sum(nums)（漏了除以長度，回傳 int）。先確認直譯器照跑不報錯，再用 uv run mypy 觀察它是否抓到回傳型別不符；接著修正並讓 mypy 通過。

    • 提示：int 是 float 的相容子型別，這題可能比你想的微妙——試試把回傳改成明顯錯誤的 str 看 mypy 反應。

  5. 建立你自己的專案範本：從零跑完 1.7 的完整走查（uv init → 寫程式 → uv add → ruff → mypy → uv run），在 pyproject.toml 加上 [tool.ruff] 與 [tool.mypy] 設定區段，讓 ruff check、ruff format --check、mypy 三者全綠。把它存成你個人的起手式範本。

    • 提示：[tool.mypy] 可加 strict = true 體驗 Java 等級的嚴格度。注意 strict 模式會收緊許多檢查：裸 dict 會觸發 type-arg 錯誤（要求你補上 dict[str, str] 之類的型別參數）、把 Any 當宣告型別回傳也會被擋。這正呼應本書「用 dict[str, str] 而非裸 dict」的現代型別提示規範。
