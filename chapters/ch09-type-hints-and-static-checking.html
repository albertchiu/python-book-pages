第 9 章：型別提示與靜態檢查（Java 工程師的舒適圈）

這一章是你的主場。第 3 章已經點過型別提示，這裡要把它變成完整的工程能力：用 typing 找回你習慣的型別詞彙、用泛型寫出型別安全的容器、用 mypy/pyright 把「編譯期型別檢查」以可選的方式裝回 Python、再用 pydantic 補上型別提示不做的那件事——執行期驗證。最後談「漸進式型別」（gradual typing）的取捨：不是型別標越多越好，而是標在刀口上。

`typing` 模組：找回靜態型別的詞彙

現代 Python（3.10+）的型別語法已經很接近你要的樣子。先建立詞彙對照：

你想表達 Python（現代寫法） Java 備註
整數串列 list[int] List<Integer> 用內建 list，不是 typing.List
字串對整數的對應 dict[str, int] Map<String, Integer>
可有可無 str | None Optional<String> Optional[str] 是同義舊寫法
多種型別之一 int | str （無直接對應） Union[int, str] 是舊寫法
函式型別 Callable[[int, str], bool] BiFunction<Integer,String,Boolean> 從 collections.abc 匯入
固定長度異質 tuple[int, str, float] （record / 多型別） 跟 tuple[int, ...]（任意長度）不同
放棄型別 Any （Object 但更鬆） 逃生出口，會關閉檢查
python 
def lookup(table: dict[str, int], key: str) -> int | None:
    return table.get(key)          # 找不到回 None，所以回傳型別是 int | None

幾個 Java 工程師要記住的轉換：

  • 不要用 typing.List／Dict／Optional／Union 的舊寫法。Python 3.9 起內建容器可直接當泛型（list[int]），3.10 起用 X | Y 取代 Union、X | None 取代 Optional。看到 from typing import List 就知道是舊碼。
  • str | None 不是 Java 的 Optional。它沒有 .map()/.orElse() 那套 API，就是「值或 None」。你靠型別縮小（narrowing）來安全使用它：
python 
def greet(name: str | None) -> str:
    if name is None:
        return "Hi, stranger"
    return name.upper()            # 這裡 mypy 已知 name 一定是 str（None 被上面排除了）

mypy 會做流程敏感的型別縮小（type narrowing）：if name is None: return 之後，它知道剩下的分支裡 name 必為 str，所以 name.upper() 合法。這是型別提示真正好用的地方——它不只是文件，工具會順著你的控制流推理。

泛型：`TypeVar`、`Generic`，與型別抹除的另一面

寫「對任何型別都成立、且能保留型別資訊」的函式或容器，就要泛型。Python 3.12（PEP 695）給了跟 Java 很像的簡潔語法：

python 
# Python 3.12+ 新語法：方括號宣告型別參數，不必先建 TypeVar
def first[T](items: list[T]) -> T:
    return items[0]

class Stack[T]:
    def __init__(self) -> None:
        self._items: list[T] = []

    def push(self, item: T) -> None:
        self._items.append(item)

    def pop(self) -> T:
        return self._items.pop()
python 
>>> s: Stack[int] = Stack()
>>> s.push(1)
>>> reveal_type(s.pop())     # mypy 會說：Revealed type is "int"

（小提醒：較新版本的 mypy 預設用短名顯示型別——int；很舊的版本或開啟特定設定才會印出完整的 builtins.int。這純粹是顯示差異，不影響檢查結果。）

對照 Java 的 class Stack<T> 與 <T> T first(List<T> items)，概念一模一樣。3.12 之前（或要相容舊版）的寫法是先建 TypeVar：

python 
from typing import TypeVar, Generic

T = TypeVar("T")

def first(items: list[T]) -> T:
    return items[0]

class Stack(Generic[T]):     # 舊寫法要明確繼承 Generic[T]
    ...

有界型別（bounded）也有對應——「T 必須具備某種能力」。下面要求 T 必須支援 < 比較，用一個 Protocol（第 8 章）當上界：

python 
from typing import Protocol, Any

class Comparable(Protocol):
    def __lt__(self, other: Any, /) -> bool: ...    # 注意：是 Any 不是 object

def largest[T: Comparable](items: list[T]) -> T:   # T 的上界是 Comparable
    biggest = items[0]
    for item in items[1:]:
        if biggest < item:                          # 因為有上界，mypy 知道 T 可以用 <
            biggest = item
    return biggest

⚠️ 陷阱：上面 __lt__ 的參數刻意標 Any 而非 object。原因很微妙：int.__lt__ 的參數在 typeshed 裡是 int（比 object 窄），而方法參數是逆變（contravariant）的——要符合「參數是 object」的協定，實作的參數必須更寬。寫成 other: object 會讓 mypy 認定連 int、str 都不滿足 Comparable，largest([3, 1, 4]) 直接被 --strict 擋下（Value of type variable "T" of "largest" cannot be "int"）。標 Any 才能讓內建型別順利通過。這正是「相信 mypy」的代價之一：協定的方向性要標對。

python 
>>> largest([3, 1, 4, 1, 5, 9])
9
>>> largest(["banana", "apple", "cherry"])
'cherry'

上界（T: Comparable）對應 Java 的 <T extends Comparable<T>>：沒有它，mypy 不知道 T 能不能用 <，會擋下 biggest < item。加了上界，它才放行——而且回傳型別仍精準保留為傳入的 T（傳 list[int] 回 int，傳 list[str] 回 str）。

型別抹除：Java 的痛，Python 的坦白

Java 的泛型是型別抹除（type erasure）：List<String> 在執行期只是 List，於是你不能 new T()、不能 new T[]、不能 obj instanceof List<String>，還會收到一堆 unchecked 警告。Java 在編譯期「假裝」有泛型，執行期再抹掉，這個落差就是摩擦的來源。

Python 走得更徹底——型別提示在執行期幾乎完全被忽略（連 PEP 695 的泛型也是純標注，不影響執行）：

python 
>>> s = Stack[int]()              # 用參數化型別建立實例
>>> type(s) is Stack              # 建立出來的就是普通 Stack，沒記住 int
True
>>> Stack[int].__origin__ is Stack  # 參數化只是外殼，被包住的原始型別仍是 Stack
True
>>> isinstance([1, 2], list)      # 你只能檢查到「原始型別」list，檢查不到「list[int]」
True
>>> isinstance([1, 2], list[int]) # 不能拿參數化泛型做 isinstance，直接報錯
Traceback (most recent call last):
    ...
TypeError: isinstance() argument 2 cannot be a parameterized generic

抹除的「另一面」是：Python 從不假裝型別參數在執行期存在，所以你不會像 Java 那樣以為有、結果被抹除咬一口。代價是型別提示給你零執行期保證——def f(x: int) 傳進 str 照跑不誤。靜態安全感完全來自 mypy/pyright（這一節），執行期安全則要另外靠 pydantic（後面那節）。把這兩件事分清楚，是 Python 型別世界的核心心法。

變異性（variance）：`list[int]` 不是 `list[object]`

這是 Java 工程師最容易踩的一個直覺陷阱。憑直覺，int 是 object 的子型別，所以 list[int] 應該也是 list[object] 的子型別吧？不是。 mypy 會直接擋下：

python 
def total_len(xs: list[object]) -> int:
    return len(xs)

nums: list[int] = [1, 2, 3]
total_len(nums)        # mypy 報錯：list 是不變的（invariant）
text 
error: Argument 1 to "total_len" has incompatible type "list[int]"; expected "list[object]"  [arg-type]
note: "list" is invariant -- see ...
note: Consider using "Sequence" instead, which is covariant

原因是 list 是可變的，所以型別系統把它定為不變（invariant）：若允許 list[int] 當成 list[object] 傳入，函式內部就能往裡塞一個 str（對 list[object] 而言合法），破壞了呼叫端 list[int] 的假設。把參數改成唯讀的 Sequence（從 collections.abc）就過了——Sequence 是協變（covariant），只能讀不能寫，沒有上述風險：

python 
from collections.abc import Sequence

def total_len(xs: Sequence[object]) -> int:    # 協變：list[int] 可以傳進來
    return len(xs)

🔁 Java 對照：Java 的陣列是協變的（Integer[] 可當 Object[]），但這個設計是個歷史錯誤——往 Object[] 塞 String 會在執行期丟 ArrayStoreException。Java 的泛型記取教訓改成不變：List<Integer> 不是 List<Object>，這跟 Python 的 list 一致。要表達協變，Java 用 List<? extends T>（只讀，對應 Python 的 Sequence[T]），要表達逆變用 List<? super T>（只寫）——就是 PECS（Producer Extends, Consumer Super）。Python 沒有 ? 萬用字元，而是讓型別本身帶 variance：可變的 list 不變、唯讀的 Sequence/Iterable 協變。思維差異：Java 把 variance 標在「使用點」（wildcard 寫在你用它的地方），Python 標在「定義點」（variance 是型別本身的屬性）；所以在 Python，你選 Sequence 還是 list 當參數型別，就同時決定了它的 variance。

（補一個逆變的實例：函式參數位置本身也是逆變的。Callable[[object], None] 是 Callable[[int], None] 的子型別——一個「什麼都能收」的回呼，當然能用在「只需要收 int」的地方；反過來不行。要一個 Callable[[int], None] 的位置，你不能塞 Callable[[str], None]，mypy 會擋。這跟前面 Comparable.__lt__ 標 Any 是同一個方向性問題的不同切面。）

`Self`：讓工廠方法的回傳型別跟著子類別收斂

第 7 章的 classmethod 工廠方法（@classmethod def margherita(cls) -> ?）刻意寫 return cls(...) 而非 return Pizza(...)，好讓子類別呼叫時拿到子類別實例——cls 參與多型。但那時回傳型別只能寫成字串前向參照 -> "Pizza"，這在型別層其實漏掉了多型：mypy 會把 DeepDishPizza.margherita() 推成 Pizza，而非實際拿到的 DeepDishPizza。

typing.Self（3.11+）正是補上這塊的工具——它代表「目前這個類別」，會跟著實際呼叫的子類別收斂：

python 
from typing import Self

class Pizza:
    def __init__(self, toppings: list[str]) -> None:
        self.toppings = toppings

    @classmethod
    def margherita(cls) -> Self:          # 回傳 Self，不再寫死 "Pizza"
        return cls(["mozzarella", "tomato"])

class DeepDishPizza(Pizza):
    pass
python 
>>> p = DeepDishPizza.margherita()
>>> reveal_type(p)        # mypy 會說：Revealed type is "DeepDishPizza"

把回傳改回 -> "Pizza"，同一行 reveal_type 就會印出 Pizza——靜態端丟失了子類別資訊。Self 讓型別系統終於跟上 cls 在執行期的多型行為。除了 classmethod 工廠，鏈式 API（builder.add(...).add(...)，每個方法 return self）也該把回傳標成 Self，子類別才能繼續鏈下去而不被收窄成基底型別。

🔁 Java 對照：Java 沒有 self-type。要讓 builder 的鏈式方法在子類別也回傳子類別型別，得動用遞迴泛型：abstract class Builder<T extends Builder<T>>，再讓每個方法 return (T) this——又囉嗦又要強制轉型。Python 一個 Self 就解決，因為它本來就是動態語言、cls／self 天生帶多型，型別系統只是把這個既有事實標出來而已。

`Protocol` 與結構型別檢查

第 8 章介紹過 Protocol（結構型別）。從型別系統的角度再補一點：它讓你能為「行為」而非「繼承關係」寫型別，這在做泛型抽象時特別有用。

python 
from typing import Protocol

class SupportsClose(Protocol):
    def close(self) -> None: ...

def use_then_close(resource: SupportsClose) -> None:
    # 任何有 close() 的東西都能傳進來：檔案、連線、你的自訂類別……
    # 完全不需要它們去繼承某個共同基底
    resource.close()

對照 Java：要表達「任何可關閉的東西」，Java 用 AutoCloseable 介面，而且每個型別都得明寫 implements AutoCloseable。Python 的 Protocol 是結構性的——第三方類別只要剛好有 close() 就自動相容，你不必、也無法去改它的繼承宣告。需要執行期 isinstance 檢查時，加 @runtime_checkable（見第 8 章）。

💡 Pythonic 重點：寫函式參數型別時，優先問「我需要對方具備什麼能力」，用 Protocol 描述那個最小能力集，而不是要求一個具體的類別或基底。這讓你的 API 更好測（傳個 fake 就行）、更不綁實作。

mypy / pyright 的導入與 CI 整合

型別提示要發揮價值，得有檢查器在 CI 與編輯器裡跑。兩大選擇：

  • mypy：型別檢查的事實標準，由 Python 官方人員維護。
  • pyright：微軟出品、用 TypeScript 寫、速度快，VS Code 的 Pylance 內建它。

設定寫在 pyproject.toml（對照 Maven 的 <configuration>）：

toml 
[tool.mypy]
python_version = "3.12"
strict = true                      # 一鍵開啟一整組嚴格檢查（強烈建議新專案就開）
warn_unused_ignores = true         # 抓出已經不需要的 # type: ignore
warn_redundant_casts = true

strict = true 會打開像 disallow_untyped_defs（不准有沒標型別的函式）、warn_return_any 等一整組規則，相當於 Java「把所有 compiler warning 當 error」。新專案建議一開始就開 strict；既有專案則漸進導入（下一節）。

CI 整合就是在流水線加一步（以 GitHub Actions 為例）：

yaml 
- name: Type check
  run: uv run mypy src/

讓 mypy 失敗就讓 build 紅燈，型別錯誤就跟測試失敗一樣擋下 PR——這就把 Java 編譯器的型別把關，以工具的形式裝回了 Python 的工作流。

💡 Pythonic 重點：debug 型別時善用 reveal_type(x)——它不是真函式，是給 mypy 看的「請告訴我你推出 x 是什麼型別」。執行前刪掉即可（mypy 不要求 import）。另一個是 reveal_locals()。這兩個是釐清「為什麼 mypy 覺得這裡型別不對」的利器。

`Final` 與 `Literal`：把 Java 的 `final` 與受限值裝回來

前兩章一再說「Python 沒有 final」（第 7、8 章）——指的是執行期：任何類別都能被繼承、任何名稱都能被重新賦值，直譯器不攔你。但靜態層可以把那份安全感裝回來。typing.Final 標的名稱，mypy 會擋下任何重新賦值：

python 
from typing import Final

MAX_RETRIES: Final = 3      # 不必寫 Final[int]，mypy 會從 3 推出 int

MAX_RETRIES = 5             # mypy 報錯：Cannot assign to final name "MAX_RETRIES"

跟 Java final int MAX_RETRIES = 3 對照：差別仍是老地方——Java 的 final 由編譯器強制，Python 的 Final 只在 mypy 眼裡有效，執行期照樣能改。但用途完全一樣：宣告常數的「這就是定值」意圖，並讓檢查器替你把關。Final 也能標類別屬性，甚至（搭配 @final 裝飾器）標記「這個類別／方法不准被繼承或覆寫」——這是把 Java final class／final method 的語意以靜態檢查形式找回來最直接的對應。

Literal 則處理「這個參數只能是這幾個值之一」。狀態碼、模式字串這類受限字串，與其用 str（什麼字串都放行）不如收緊成字面值集合：

python 
from typing import Literal

def open_file(path: str, mode: Literal["r", "w", "a"]) -> None:
    ...

open_file("data.txt", "r")     # OK
open_file("data.txt", "rw")    # mypy 報錯：expected "Literal['r', 'w', 'a']"

對照 Java：要限制「只能是這幾個值」，Java 往往開一個 enum。Literal 更輕——不必定義新型別，直接把允許的字面值列進簽章；mypy 會把傳錯的值當場標紅，等於免費得到 enum 的約束力。值多、或想附帶行為時才升級成真正的 enum（Python 的 enum.Enum）。

💡 Pythonic 重點：Final、Literal 跟前面的 Self 一樣，都是「Python 執行期沒有，但靜態層幫你補回」的工具——本章的核心命題正是用 mypy 把 Java 編譯器的把關一塊塊裝回來，而這些就是其中幾塊拼圖。

🔁 Java 對照：寫公開 API 時若「同名函式因參數型別不同、回傳型別也跟著不同」，用 @overload（從 typing）疊出多個型別簽章、再寫一個實作本體：@overload def f(x: int) -> int、@overload def f(x: str) -> list[str]，呼叫端傳 int mypy 推得 int、傳 str 推得 list[str]。表面像 Java 的方法多載（overloading），機制卻相反：Java 是真有多個方法、由編譯器選一個；Python 執行期只有一個函式，@overload 簽章純粹是給檢查器看的型別資訊，本體得自己用 isinstance 分流。屬進階工具，需要時再查即可。

pydantic：執行期驗證（對照 Bean Validation）

再強調一次關鍵事實：型別提示在執行期不做任何檢查。def f(age: int) 收到 "100" 不會報錯。當資料來自外部世界（HTTP request、JSON、設定檔、DB），你需要的是執行期驗證——這正是 pydantic 的領域，也是 FastAPI（第 16 章）的核心。

python 
from pydantic import BaseModel, Field, ValidationError

class User(BaseModel):
    name: str
    age: int = Field(ge=0, le=150)       # 約束：0 <= age <= 150
    email: str

pydantic 在建立物件那一刻就驗證、甚至轉型：

python 
>>> u = User(name="Albert", age=42, email="a@b.com")
>>> u.age
42
>>> User(name="Albert", age="42", email="a@b.com").age    # 字串 "42" 自動轉成 int
42
>>> User(name="Albert", age=200, email="a@b.com")         # 違反 le=150
Traceback (most recent call last):
    ...
pydantic_core._pydantic_core.ValidationError: 1 validation error for User
age
  Input should be less than or equal to 150 [type=less_than_equal, ...]

對照 Java 的 Bean Validation（@NotNull、@Min、@Max、@Email）+ Hibernate Validator：兩者都是「宣告式地對資料下約束、在執行期驗證」。差別在 pydantic 把型別轉換也一起做了（"42" → 42），且與 Python 型別提示無縫整合——你的型別標注同時就是驗證 schema。

把三層責任放在一起看，整個型別故事就完整了：

工具 何時作用 對照 Java 負責
型別提示 + mypy/pyright 開發／CI（靜態） javac 型別檢查 你自己的程式邏輯一致性
pydantic 執行期 Bean Validation 來自外部、不可信的資料

`TypedDict` 與 `cast`：不想付執行期成本時的兩個工具

pydantic 有執行期成本（驗證、轉型、建物件）。有時你只是想在 mypy 眼中讓一坨 dict 或一個 Any 有形狀，而不需要真的驗證——這時有兩個零成本工具：

TypedDict 描述「dict 應該長什麼形狀」，但不做任何執行期檢查，純粹是給檢查器看的：

python 
from typing import TypedDict

class UserDict(TypedDict):
    name: str
    age: int

def show(u: UserDict) -> str:
    return f"{u['name']} ({u['age']})"   # mypy 知道 u['age'] 是 int、打錯 key 會報錯

cast(T, x) 則是型別斷言：「我比 mypy 更清楚這是 T，別管推斷了」。它在執行期什麼都不做（連檢查都沒有），只改變 mypy 的看法。處理 json.loads() 回傳的 Any 時，這是一種把它收斂成具體形狀的輕量做法：

python 
from typing import cast
import json

data = json.loads('{"name": "Albert", "age": 42}')   # data 是 Any
user = cast(UserDict, data)                           # 斷言成 UserDict，零成本、不驗證
reveal_type(user["age"])                              # mypy 推得 int

🔁 Java 對照：cast(T, x) 對應 Java 的 (T) x 強制轉型——但只有一半。Java 的 (T) 會在執行期檢查，型別不符就丟 ClassCastException；Python 的 cast 在執行期完全不檢查，你斷言錯了也不會當場炸，錯誤會延後到後面某個操作。思維差異：cast 是把責任從工具拿回自己手上，斷言錯的後果你自負。所以該不該用 cast vs pydantic，本質是「要不要花執行期成本換取真正的保證」：邊界內、你確信形狀正確時用 TypedDict/cast（零成本）；資料來自外部、不可信時用 pydantic（有成本，但會真的擋下壞資料）。

💡 Pythonic 重點：內部程式碼（你自己呼叫自己）靠 mypy 就夠了，別讓 pydantic 的執行期開銷散落各處。把 pydantic 用在信任邊界上——API 入口、設定載入、外部資料解析。邊界內用普通 @dataclass（第 7 章）、邊界外用 pydantic BaseModel，是很常見的分工。

漸進式型別（gradual typing）的取捨

Python 的型別系統是漸進式的：你可以一行型別都不標（純動態），也可以標到 strict 全綠（接近靜態語言），或落在中間。Any 是這套系統的「逃生出口」——任何東西都相容 Any、Any 也相容任何東西，等於對那一塊關閉檢查。

python 
from typing import Any

def parse(raw: Any) -> dict[str, int]:
    raise NotImplementedError    # raw 是 Any：進來不檢查，但回傳值有型別，呼叫端仍受保護

導入策略與取捨：

  • 新專案：直接 strict = true，從第一天就維持全綠，成本最低。
  • 既有大型專案：漸進導入。先在 pyproject.toml 對少數模組開檢查，或用 # type: ignore[code] 暫時壓住特定錯誤（記得標明 error code，配合 warn_unused_ignores 之後就能清掉過期的）。優先標公開 API 邊界與複雜的領域邏輯——這裡型別的投資報酬率最高；一次性腳本可以完全不標。
  • 務必標明 ignore 的 error code：寫 # type: ignore[arg-type] 而不是裸的 # type: ignore，才不會連帶吞掉其他真正的錯誤。

⚠️ 陷阱：Any 會「傳染」。一個回傳 Any 的函式，其結果再拿去做任何運算往往還是 Any，靜態檢查在那條路徑上等於失效。json.loads() 的回傳就是 Any——拿到外部 JSON 後，盡早用 pydantic 或明確的型別斷言把它「收斂」成具體型別，別讓 Any 一路擴散到整個程式。

💡 Pythonic 重點：漸進式型別的精神不是「型別越多越好」，而是把型別投資在會回本的地方：團隊協作的公開介面、容易錯的複雜邏輯、外部資料的邊界。享受動態語言的彈性（快速原型、duck typing），同時用型別把關鍵介面釘死——這正是資深工程師能讓 Python 程式碼「比一般人寫得更穩」的地方。

本章小結

  1. 用現代型別語法：list[int]、dict[str, int]、X | None、X | Y，別再用 typing.List/Optional/Union 舊寫法。str | None 不是 Java Optional，靠型別縮小來安全使用。
  2. mypy 會做流程敏感的型別縮小：if x is None: return 之後它知道 x 是非 None 型別。型別提示不只是文件，工具會順著控制流推理。
  3. 泛型有 3.12 新語法：def first[T](...)、class Stack[T]，跟 Java <T> 神似；舊寫法用 TypeVar + Generic[T]。
  4. 型別抹除的另一面：Java 編譯期假裝有泛型、執行期抹掉造成摩擦；Python 坦白地讓提示在執行期幾乎完全無效——零執行期保證，靜態安全全靠檢查器。
  5. 變異性（variance）：list[int] 不是 list[object]（list 不變），改用唯讀的 Sequence（協變）才能收子型別容器——對應 Java 泛型的不變與 ? extends/? super（PECS）。cast/TypedDict 是零執行期成本的型別斷言／形狀描述，pydantic 才有真正的執行期驗證。
  6. Self／Final／Literal 把 Java 的把關裝回靜態層：Self 讓 classmethod 工廠／鏈式方法的回傳型別跟著子類別收斂（回扣第 7 章 cls 多型，免去 Java 的遞迴泛型）；Final 是靜態版 final（mypy 擋重新賦值）；Literal 收緊「只能是這幾個值」（輕量版 enum）。
  7. Protocol 為「能力」而非「繼承」寫型別：描述函式需要的最小能力集，API 更好測、更不綁實作。
  8. 三層責任要分清：mypy/pyright 管靜態（你的程式邏輯）、pydantic 管執行期（外部不可信資料、對照 Bean Validation）。邊界內用 @dataclass、邊界外用 pydantic。
  9. 漸進式型別是投資決策：新專案直接開 strict；既有專案漸進導入、優先標公開 API 與複雜邏輯。Any 是逃生出口但會傳染，盡早收斂外部資料。

練習

練習 1（型別縮小） 為下面函式加上型別提示，讓 mypy 在 strict 下全綠。config.get("port") 回傳 str | None，你要處理 None 的情況再轉成 int。寫完用 uv run --with mypy mypy 驗證，並故意把回傳改成可能回 None，觀察 mypy 怎麼抱怨。

python 
def get_port(config):
    port = config.get("port")
    return int(port)

練習 2（泛型容器） 用 Python 3.12 新語法寫一個泛型 Pair[A, B]（兩個可能不同型別的欄位 first: A、second: B），提供 swap() -> Pair[B, A]。用 reveal_type 驗證 Pair(1, "x").swap() 被推導成 Pair[str, int]。

練習 3（抹除實驗） 不查文件，先預測下面每行的結果再執行：list[int] is list[str]、isinstance([1], list[int])、isinstance([1], list)。用一句話解釋為什麼 Python 不讓你對 list[int] 做 isinstance。

練習 4（mypy vs pydantic 的分界） 寫一個 @dataclass 版的 Config 和一個 pydantic BaseModel 版的 Config，各塞一個 port: int。分別用 Config(port="8080")（字串）建立，觀察兩者行為的差異（一個照建立但 mypy 會在靜態時抱怨、一個在執行期轉型或報錯）。用這個差異說明「信任邊界」該用哪一個。

練習 5（馴服 Any） data = json.loads(raw) 的 data 是 Any。寫一個 pydantic model Event(name: str, count: int)，把 data 收斂成 Event，讓後續程式碼享有型別保護。再示範若不收斂、直接 data["count"] + 1，為什麼 mypy 幫不了你（提示：Any 的傳染性）。
