第 14 章：並行與非同步（Concurrency & Async）

這是全書與 Java 直覺差距最大的一章。你在 Java 裡建立的核心信念——「開幾條 thread 就吃幾顆核」「java.util.concurrent 是萬用工具箱」「並行 = 平行」——到了 CPython 全部要打上問號。原因只有一個名字：GIL（Global Interpreter Lock，全域直譯器鎖）。它不會讓你不能用多執行緒，但會徹底改變「什麼時候該用哪種並行模型」的決策。讀完這章，你要能在面對一個任務時，三秒內判斷它是 I/O bound 還是 CPU bound，然後選對 threading、multiprocessing、還是 asyncio——而不是反射性地 new Thread()。

GIL：理解 CPython 並行的第一原理

Java 的並行心智模型

在 JVM 上，thread 是「真的」作業系統執行緒，可以同時在多顆核心上跑 Java bytecode。你寫一個 CPU 密集的計算，丟給四條 thread，在四核機器上大約能快四倍（理想狀況）。synchronized、volatile、java.util.concurrent 這整套工具，存在的目的就是協調這些真正同時執行的執行緒。

java 
// Java：四條 thread 真的同時跑，CPU bound 也能平行加速
ExecutorService pool = Executors.newFixedThreadPool(4);
List<Future<Long>> futures = new ArrayList<>();
for (int i = 0; i < 4; i++) {
    futures.add(pool.submit(() -> heavyCompute()));  // 四顆核同時算
}

CPython 的真相

CPython 有一把全域鎖：同一時刻，只有一個 thread 能執行 Python bytecode。threading.Thread 開出來的確實是真正的 OS thread，但它們必須輪流持有 GIL 才能執行直譯器。所以對純 CPU 的 Python 計算，多開 thread 不會變快——它們在搶同一把鎖。

實測勝過千言萬語。下面四份等量的 CPU 工作，序列跑與用四條 thread 跑，時間幾乎一樣（在 Python 3.14、10 核機器上實測）。註解裡的秒數是某次實測的約略值，實際數字依機器而異，重點看的是兩者的相對比例，不是絕對秒數：

python 
import time
from concurrent.futures import ThreadPoolExecutor

def cpu_work(n: int) -> int:
    return sum(i * i for i in range(n))

N = 5_000_000

t0 = time.perf_counter()
[cpu_work(N) for _ in range(4)]
print(f"seq:     {time.perf_counter() - t0:.3f}s")   # seq:     ~0.5s

t0 = time.perf_counter()
with ThreadPoolExecutor(max_workers=4) as ex:
    list(ex.map(cpu_work, [N] * 4))
print(f"threads: {time.perf_counter() - t0:.3f}s")   # threads: ~0.5s ← 沒變快！

四條 thread 跟序列一樣慢。在 Java，你會預期接近四倍加速；在 CPython，GIL 把它們序列化了。

🔁 Java 對照：JVM 的 thread 能讓 CPU bound 工作吃滿多核；CPython 的 thread 在 CPU bound 工作上等於沒有平行——它們排隊搶 GIL。

但 GIL 在 I/O 等待時會放手

關鍵轉折：當一個 thread 進入阻塞式 I/O（讀檔、time.sleep、socket 等待、資料庫查詢），CPython 會釋放 GIL，讓別的 thread 趁這段空檔去執行。所以對 I/O bound 工作，threading 非常有效——大家大部分時間都在等網路/磁碟，根本不需要 CPU。

python 
import time
from concurrent.futures import ThreadPoolExecutor

def io_work(n: int) -> int:
    time.sleep(0.2)      # 模擬 I/O 等待，這期間 GIL 被釋放
    return n

t0 = time.perf_counter()
[io_work(i) for i in range(5)]
print(f"seq io:     {time.perf_counter() - t0:.3f}s")   # seq io:     ~1.0s

t0 = time.perf_counter()
with ThreadPoolExecutor(max_workers=5) as ex:
    list(ex.map(io_work, range(5)))
print(f"threads io: {time.perf_counter() - t0:.3f}s")   # threads io: ~0.2s ← 快五倍

同樣五條 thread，I/O bound 時間從約 1 秒掉到約 0.2 秒（一樣，數字依機器而異，看的是這個數量級的落差）。秘訣就是「等的時候把 GIL 讓出去」。標準函式庫和大多數 C 擴充（requests、資料庫驅動、檔案 I/O）都會在阻塞前主動釋放 GIL，所以這不是你要手動處理的事——但你要理解為什麼 threading 對 I/O 有效、對 CPU 無效。

💡 Pythonic 重點：判斷該不該用 threading，問一句話——「這個工作大部分時間在等東西，還是在算東西？」等東西（I/O bound）→ threading 有效。算東西（CPU bound）→ threading 沒用，要走 multiprocessing。

GIL 保護的是直譯器，不是你的程式

最後一個必須釐清的誤解：GIL 讓單一 bytecode 操作不被打斷，但它不會讓你的多步驟邏輯自動 thread-safe。counter += 1 看起來是一行，實際上是「讀值 → 加一 → 寫回」三個 bytecode，thread 可能在中間被切換走。

python 
import threading

counter = 0
lock = threading.Lock()

def inc() -> None:
    global counter
    for _ in range(100_000):
        with lock:          # 沒有這把鎖，最後的數字會少於 400000
            counter += 1

threads = [threading.Thread(target=inc) for _ in range(4)]
for t in threads:
    t.start()
for t in threads:
    t.join()
print(counter)              # 400000（有鎖才保證；拿掉 with lock 就會 race）

⚠️ 陷阱：「有 GIL 所以不用加鎖」是錯的。GIL 保護的是 CPython 直譯器的內部資料結構（reference count、物件記憶體），不是你的應用邏輯。counter += 1 這種 read-modify-write 在多 thread 下照樣 race，該加鎖就得加。Java 工程師在這點上反而有優勢——你本來就知道要加鎖，別因為聽說「Python 有 GIL」就把這個直覺丟掉。

threading：I/O bound 的主力

threading 模組的 API 跟 java.util.concurrent 高度對應，你會很快上手。重點在於用對場景（I/O bound），而不是 API 本身。

你要做的事 Java Python
建立並啟動執行緒 new Thread(r).start() threading.Thread(target=f).start()
等待結束 t.join() t.join()

Thread 與 Lock

python 
import threading

class BankAccount:
    def __init__(self, balance: int = 0) -> None:
        self._balance = balance
        self._lock = threading.Lock()

    def deposit(self, amount: int) -> None:
        with self._lock:           # 等同 Java 的 synchronized 區塊
            self._balance += amount

    @property
    def balance(self) -> int:
        return self._balance

with self._lock 就是 Java 的 synchronized (lock) { ... }：進入時取得鎖，離開（含例外）時自動釋放。這是 context manager（詳見第 12 章）的典型用途。

Queue：執行緒間溝通的首選

跟 Java 一樣，用佇列傳遞資料、而不是共享可變狀態，是最不容易出錯的並行模式。queue.Queue 內建鎖，本身就是 thread-safe 的，對應 Java 的 BlockingQueue。

python 
import threading
import queue

work_q: queue.Queue[int | None] = queue.Queue()

def producer() -> None:
    for i in range(3):
        work_q.put(i)
    work_q.put(None)               # sentinel：通知消費者收工

def consumer(results: list[int]) -> None:
    while True:
        item = work_q.get()
        if item is None:
            break
        results.append(item * 10)
        work_q.task_done()         # 搭配 work_q.join() 用；見下方說明

results: list[int] = []
t1 = threading.Thread(target=producer)
t2 = threading.Thread(target=consumer, args=(results,))
t1.start(); t2.start()
t1.join(); t2.join()
print(results)                     # [0, 10, 20]

這裡有兩種「等所有工作做完」的方式，別搞混：本例用 t.join()（等執行緒結束）收尾，所以 work_q.task_done() 其實不是必要的——它是搭配 work_q.join()（等佇列裡的項目全部被處理完）用的。若你不想靠 sentinel 與 thread join，而是想讓主執行緒「等佇列清空」再往下走，才需要消費端每處理一筆就 task_done()、主執行緒呼叫 work_q.join() 來等。本例保留 task_done() 是為了讓你看到它的存在，但它在這個以 thread join 收尾的版本裡沒有實際作用。

Event 對應一個「一次性/可重設的旗標」，常用來通知一組 thread 該停了，類似你在 Java 用 volatile boolean running 或 CountDownLatch 做的事。

💡 Pythonic 重點：手寫 Thread + Lock + Queue 能動，但在現代 Python 裡，多數情況你不該手刻——直接跳到下面的 concurrent.futures，讓執行緒池幫你管理生命週期。手寫 thread 留給需要長駐背景執行緒、或要精細控制的場景。

multiprocessing：CPU bound 的解法

既然 GIL 卡死了多 thread 的 CPU 平行，怎麼真的吃滿多核？答案是多進程（multiprocessing）：開多個獨立的 Python 進程，每個進程有自己的直譯器、自己的 GIL，於是它們能真正同時在不同核心上跑。這是 Java 工程師最陌生的一招——在 JVM 上你幾乎不會為了 CPU 平行而開多進程。

代價有三個，都要記住：

  1. 跨進程要序列化：進程間不共享記憶體，傳參數與回傳值都要用 pickle 序列化再傳。不可 pickle 的東西（lambda、開啟的檔案、某些閉包）傳不過去。
  2. 啟動成本高：開一個進程比開一條 thread 貴得多，資料量小的工作可能得不償失。
  3. 需要 __name__ 守衛：在預設使用 spawn 的平台（macOS、Windows）上，子進程會重新 import 你的主模組，沒有 if __name__ == "__main__": 保護就會無限遞迴開進程（承接第 11 章的模組執行模型）。
python 
import time
from concurrent.futures import ProcessPoolExecutor

def cpu_work(n: int) -> int:
    return sum(i * i for i in range(n))

N = 5_000_000

if __name__ == "__main__":          # ← 多進程的必備守衛，別省略
    t0 = time.perf_counter()
    with ProcessPoolExecutor(max_workers=4) as ex:
        list(ex.map(cpu_work, [N] * 4))
    print(f"procs: {time.perf_counter() - t0:.3f}s")

對前面那份 CPU 工作，多進程版本能真正分散到多核（雖然啟動與 pickle 成本讓它對小工作不見得划算，工作越重、攤平成本後加速越明顯）。

⚠️ 陷阱：把上面的 if __name__ == "__main__": 拿掉，在 macOS/Windows 上會出現 RuntimeError，甚至狂開進程把機器卡死。這不是龜毛——是 spawn 啟動方式的必然結果。記住：任何用到 multiprocessing 的腳本，進入點一定要有 __main__ 守衛。

🔁 Java 對照：Java 靠多 thread 就能 CPU 平行，幾乎不需要多進程；CPython 因為 GIL，CPU 平行的標準解是多進程——付出序列化與啟動成本，換真正的多核利用。

實務上還有第三條路常被忽略：讓 C 擴充去扛 CPU。numpy、scipy、pandas、Pillow 這些函式庫的熱點是 C/Fortran 寫的，計算時會釋放 GIL。所以「用 numpy 做向量運算」往往比「自己用 multiprocessing 切 Python 迴圈」又快又簡單。CPU bound 不一定要多進程——先問能不能交給釋放 GIL 的 C 擴充。

concurrent.futures：你該優先用的那一層

concurrent.futures 提供統一的高階介面，把「執行緒池」和「進程池」包成同一組 API。它直接對應 Java 的 ExecutorService / Future，而且換並行模型只要換一個類別名稱——這是它最大的價值。

Java Python
ExecutorService Executor（ThreadPoolExecutor / ProcessPoolExecutor）
pool.submit(task) executor.submit(fn, *args)
Future<T> concurrent.futures.Future
future.get() future.result()
invokeAll(...) executor.map(fn, iterable)
CompletableFuture 組合 as_completed(...) / wait(...)

下面這個範例會發出真實網路請求，需連網才能執行；印出的位元組數依當下各網站的內容而定，每次跑不見得一樣：

python 
from concurrent.futures import ThreadPoolExecutor, as_completed
import urllib.request

URLS = ["https://example.com", "https://www.python.org", "https://docs.python.org"]

def fetch(url: str) -> tuple[str, int]:
    with urllib.request.urlopen(url, timeout=10) as resp:
        return url, len(resp.read())

with ThreadPoolExecutor(max_workers=8) as ex:
    # submit：拿到 Future，之後再收結果（對照 CompletableFuture）
    future_to_url = {ex.submit(fetch, url): url for url in URLS}
    for future in as_completed(future_to_url):   # 誰先好誰先回
        url, size = future.result()              # 子執行緒的例外會在這裡重新拋出
        print(f"{url}: {size} bytes")

兩個要點：

  • map vs submit：executor.map(fn, iterable) 像並行版的 list(map(...))，保留輸入順序回傳結果，最簡單。要更細控制（哪個先完成、個別處理例外）就用 submit 拿 Future，搭配 as_completed。
  • 例外傳遞：worker 裡丟出的例外不會憑空消失，會被存在 Future 裡，等你呼叫 .result() 時在主執行緒重新拋出。這跟 Java 的 Future.get() 把例外包成 ExecutionException 一樣的精神——別忘了去 result()，否則例外會被默默吞掉。

切換到 CPU bound？把 ThreadPoolExecutor 換成 ProcessPoolExecutor，加上 __main__ 守衛，其餘程式碼幾乎不動：

python 
from concurrent.futures import ProcessPoolExecutor

if __name__ == "__main__":
    with ProcessPoolExecutor() as ex:        # 預設 worker 數 = CPU 核數
        results = list(ex.map(cpu_work, [N] * 8))

💡 Pythonic 重點：除非有特殊需求，先用 concurrent.futures，不要手寫 Thread / Process。它幫你管池子大小、回收、例外傳遞，而且 thread↔process 一鍵切換。這層就是你熟悉的 ExecutorService，照著用就對了。

asyncio：單執行緒的協作式並行

threading 用作業系統的搶佔式排程（preemptive scheduling）跑大量 I/O；asyncio 給你另一條路——單一執行緒、單一 GIL，靠協作式排程（cooperative scheduling）在一條 thread 裡並發處理上萬個 I/O。沒有 thread 切換成本、沒有鎖、記憶體佔用極省。代價是：你要把程式碼改寫成 async/await 的形式。

核心三件套：async def / await / event loop

python 
import asyncio

async def fetch(name: str, delay: float) -> str:   # 協程函式（coroutine function）
    print(f"start {name}")
    await asyncio.sleep(delay)                      # 在等待點交出控制權
    print(f"done {name}")
    return name

async def main() -> None:
    # gather：併發跑多個協程，全部完成後一起回傳（對照 CompletableFuture.allOf）
    results = await asyncio.gather(
        fetch("a", 0.3),
        fetch("b", 0.3),
        fetch("c", 0.3),
    )
    print(results)            # ['a', 'b', 'c']

asyncio.run(main())           # 啟動 event loop，跑到 main 結束

執行結果是三個 start 先一起印出來，等 0.3 秒後三個 done 才出現——總耗時約 0.3 秒而非 0.9 秒。它們並發執行：在 await asyncio.sleep 的等待點，協程把控制權交還給 event loop（事件迴圈），loop 趁機去跑別的協程。這就是「協作式」——每個協程在 await 處主動讓出，沒有 await 就不會被打斷。

asyncio.run 是進入點（對應你只在最外層呼叫一次的「啟動」）。await 是「在這裡暫停、等這個 awaitable 完成、把控制權還給 loop」。

TaskGroup（3.11+）：結構化併發

Python 3.11 引入 asyncio.TaskGroup，是目前最推薦的併發寫法，比裸 gather 更安全。它提供「結構化併發（structured concurrency）」：所有任務都綁在 async with 區塊內，區塊結束前保證全部完成或全部取消，而且任一任務失敗會自動取消其他任務。

python 
import asyncio

async def fetch(name: str, delay: float) -> str:
    await asyncio.sleep(delay)
    return name

async def main() -> None:
    async with asyncio.TaskGroup() as tg:
        t1 = tg.create_task(fetch("a", 0.3))
        t2 = tg.create_task(fetch("b", 0.3))
        t3 = tg.create_task(fetch("c", 0.3))
    # 離開 async with 時，三個 task 都已完成
    print(t1.result(), t2.result(), t3.result())   # a b c

asyncio.run(main())

當多個任務同時失敗，TaskGroup 會把它們的例外打包成 ExceptionGroup，用 Python 3.11 的 except* 語法接：

python 
import asyncio

async def boom(name: str) -> None:
    await asyncio.sleep(0.1)
    raise ValueError(name)

async def main() -> None:
    try:
        async with asyncio.TaskGroup() as tg:
            tg.create_task(boom("one"))
            tg.create_task(boom("two"))
    except* ValueError as eg:
        print(f"caught {len(eg.exceptions)} errors")   # caught 2 errors

asyncio.run(main())

「函式著色」：asyncio 最大的思維負擔

這是 Java 工程師（尤其用過 Java 21 virtual threads 的）最該注意的差異。在 asyncio 裡，函式分成兩種「顏色」：普通函式與 async 函式。async 函式只能被 async 函式 await，這個「顏色」會像傳染病一樣往上蔓延整個 call stack——一旦底層某個函式變 async，沿途呼叫它的全部都得變 async。這就是業界說的「function coloring（函式著色）」問題。

python 
# 錯誤示範：不能在普通函式裡 await，這會出錯（SyntaxError）
def caller():
    data = await fetch_data()   # await 只能出現在 async def 裡面

# 正確寫法：要 await，整條呼叫鏈都得是 async
async def caller():
    data = await fetch_data()

對比之下，Java 21 的 virtual threads 走的是完全相反的路線：你照常寫阻塞式程式碼（socket.read()、Thread.sleep()），JVM 在底層自動把阻塞的 virtual thread 從 carrier thread 上卸下，換別的上來跑。沒有 async/await 關鍵字、沒有函式著色、沒有兩種顏色的函式——舊的同步程式碼幾乎免改就能享受高併發。

Python asyncio Java 21 virtual threads
風格 顯式 async/await 隱式、看起來像阻塞同步碼
函式著色 有（async 會傳染） 無
排程 協作式（在 await 讓出） 由 JVM 自動掛載/卸載
既有同步碼 需改寫成 async 幾乎免改

Python 目前沒有等價於 virtual threads 的東西；asyncio 的顯式模型是它的設計選擇——你永遠看得到 await，就知道哪裡可能切換協程（這對推理併發正確性其實有好處），代價是著色傳染與生態系分裂（同一個函式庫常要出同步與 async 兩個版本）。

⚠️ 陷阱：絕對不要在協程裡呼叫阻塞式同步 I/O 或做 CPU 重活。time.sleep(5)、同步的 requests.get()、跑一個大迴圈——這些不會讓出控制權，會把整個 event loop 卡死，所有協程一起停擺。要等就用 await asyncio.sleep(5)；要用 HTTP client 就用 aiohttp/httpx 的 async 版本；非得跑阻塞或 CPU 的東西，用 await asyncio.to_thread(blocking_fn, ...) 丟到執行緒池。

python 
import asyncio

def blocking_cpu_or_io():     # 一個沒有 async 版本的阻塞函式
    import time; time.sleep(1)
    return 42

async def main() -> None:
    # 把阻塞呼叫丟到 thread，不卡住 event loop
    result = await asyncio.to_thread(blocking_cpu_or_io)
    print(result)             # 42

asyncio.run(main())

💡 Pythonic 重點：async ≠ 平行。asyncio 全程在一條 thread、一把 GIL 上跑，它擅長的是「同時等很多 I/O」，而不是「同時算很多東西」。CPU bound 塞進 event loop 只會更慢還卡死別人——那是 multiprocessing 的工作。asyncio 的甜蜜點是：大量、高併發的網路 I/O（網路服務、爬蟲、API 閘道、同時打上千個連線）。

決策樹：到底該用哪一個

把整章濃縮成一張可以貼在螢幕旁的表：

工作性質 規模 選擇 為什麼
I/O bound 少量～中等並發 ThreadPoolExecutor（concurrent.futures） 最簡單，等待時釋放 GIL，免改寫程式碼
I/O bound 大量、高併發（上千連線） asyncio 單執行緒省記憶體與切換成本，為海量 I/O 而生
CPU bound 任意 ProcessPoolExecutor / multiprocessing 多進程各自有 GIL，真正吃多核
CPU bound（數值） 任意 numpy / C 擴充 熱點在 C，計算時釋放 GIL，常比多進程更省事
混合（async 流程中有阻塞/CPU 段） — asyncio + to_thread / run_in_executor 把阻塞/CPU 部分卸載到池子，不卡 loop

判斷流程，三個問題：

  1. 是 CPU bound 還是 I/O bound？（在等東西還是在算東西？）
    • CPU bound → 走多進程（ProcessPoolExecutor），或交給 numpy/C 擴充。到此結束，別碰 threading/asyncio。
  2. I/O bound，那並發量多大？
    • 少量～中等（幾十、上百）→ ThreadPoolExecutor，最省腦。
    • 海量（上千、需要極致效率的網路服務）→ asyncio。
  3. 已經在 asyncio 裡，但需要跑阻塞/CPU 的東西？
    • 用 asyncio.to_thread()（阻塞 I/O）或 loop.run_in_executor(ProcessPoolExecutor(), ...)（CPU）卸載出去。

🔁 Java 對照：在 Java 你大致只需要「丟給 ExecutorService」一招打天下，因為 thread 又能 I/O 又能 CPU 平行。Python 把這件事拆成兩個維度（I/O vs CPU、少量 vs 海量），逼你先分類工作再選工具。多花的這點分類成本，正是 GIL 的存在稅。

free-threading（no-GIL）的現況

最後談談未來。GIL 不是不可動搖的——PEP 703 開啟了「自由執行緒（free-threading）」build，目標是讓 CPython 可以在編譯時關掉 GIL，讓多 thread 真正 CPU 平行，往 Java 的模型收斂。

狀態（截至本書撰寫，請以你的版本為準）：

  • Python 3.13 首次提供實驗性的 free-threaded build（俗稱 3.13t），需要特別編譯、預設不啟用。
  • Python 3.14 起 free-threaded build 進入官方支援階段（不再標記為實驗性）——依 PEP 779，free-threaded build 自 3.14 起轉為官方支援的第二階段（Phase II）；但仍非預設，生態系（C 擴充、第三方套件）對它的支援還在補齊中。
  • 你日常下載的標準 CPython 預設仍帶 GIL。要關掉得用 free-threaded build，並可能付出單執行緒效能的代價。

怎麼確認自己手上的直譯器？

python 
>>> import sys
>>> sys.version
'3.14.4 (main, Apr  7 2026, 13:13:20) [Clang 21.0.0 (clang-2100.0.123.102)]'
>>> sys._is_gil_enabled()        # 3.13+ 才有這個函式
True                             # True = GIL 還在；free-threaded build 會是 False

也可以從建置設定看：

python 
>>> import sysconfig
>>> sysconfig.get_config_var("Py_GIL_DISABLED")
0                                # 0 = 一般 build；1 = free-threaded build

命令列上 python3 -VV 會顯示完整 build 資訊；free-threaded build 通常會帶 free-threading 字樣或 t 後綴。

意義在哪？如果 free-threading 成熟、生態系跟上，那麼「CPython 多 thread 也能 CPU 平行」會成真，本章「CPU bound 必須走多進程」的鐵則就會鬆動——你可以像在 Java 那樣，用一個執行緒池同時處理 I/O 與 CPU 工作，不必再為了繞過 GIL 而忍受 multiprocessing 的序列化與啟動成本。Python 的並行模型會與你熟悉的 Java 模型收斂。

但要保守看待：這仍在演進中，預設 build 還是有 GIL，C 擴充相容性與單執行緒效能都還在權衡。在可預見的未來，假設你的生產環境帶 GIL 仍是安全的預設——也就是說，本章的決策樹現在依然完全適用。等你真的部署在 free-threaded build 上，前面那條「threading 不能 CPU 平行」才需要重新評估。

💡 Pythonic 重點：別把 free-threading 當成「現在就能依賴」的東西。它是方向，不是現狀。今天寫程式，仍照本章決策樹走；同時讓自己的並行碼本來就 thread-safe（該加鎖加鎖、優先用 Queue 傳資料）——這樣哪天搬到 no-GIL build，你的程式才不會因為失去「GIL 順手提供的序列化錯覺」而炸開。

本章小結

  • GIL 不等於「不能用多執行緒」。它讓同一時刻只有一個 thread 執行 Python bytecode，所以多 thread 對 CPU bound 無法平行加速；但 I/O 等待時 GIL 會釋放，故 threading 對 I/O bound 仍然有效。
  • GIL 保護的是直譯器內部狀態，不是你的程式。counter += 1 這種 read-modify-write 在多 thread 下照樣 race，該加鎖就加鎖——別丟掉 Java 帶來的鎖直覺。
  • CPU bound 走多進程：每個進程有自己的直譯器與 GIL，能真正吃多核，代價是 pickle 序列化、啟動成本、以及必備的 if __name__ == "__main__": 守衛。數值運算優先考慮交給釋放 GIL 的 numpy/C 擴充。
  • 優先用 concurrent.futures：ThreadPoolExecutor / ProcessPoolExecutor 是你熟悉的 ExecutorService，例外會在 .result() 重新拋出，I/O↔CPU 一鍵切換，不要手刻 Thread/Process。
  • asyncio 是單執行緒協作式並行，為海量並發 I/O 而生。async/await 顯式讓出控制權；注意「函式著色」會往上傳染，且 async ≠ 平行，CPU bound 與阻塞呼叫絕不能塞進 event loop。
  • 與 Java 21 virtual threads 的根本差異：asyncio 是顯式 await、有函式著色；virtual threads 是隱式、寫成阻塞同步碼、無著色。Python 目前沒有等價物。
  • 決策樹：CPU bound → 多進程/numpy；I/O 少量 → ThreadPoolExecutor；I/O 海量 → asyncio。先分類工作（I/O vs CPU、少量 vs 海量）再選工具。
  • free-threading（PEP 703）是方向不是現狀：3.13 實驗性、3.14 起官方支援但非預設（依 PEP 779 進入第二階段）。用 sys._is_gil_enabled() 確認。今天仍照決策樹走，但讓並行碼本來就 thread-safe，為 no-GIL 未來鋪路。

練習

  1. 分類與選型：給定四個任務——(a) 同時下載 2000 個網頁、(b) 對一張大圖做像素級濾鏡運算、(c) 同時查詢 8 個資料庫並彙整、(d) 計算一個 50 位數的質因數分解——分別判斷 I/O bound 還是 CPU bound，並從決策樹選出工具。提示：先問「在等還是在算」，CPU bound 別選 threading/asyncio。

  2. 證明 GIL：寫一段程式，對同一份純 Python 的 CPU 工作分別用「序列」「4 條 thread」「4 個 process」計時，印出三者耗時。觀察 thread 版為何沒比序列快、process 版為何能快。提示：用 concurrent.futures 的兩種 Executor 切換，process 版記得加 __main__ 守衛。

  3. 抓 race condition：把本章 counter += 1 的例子拿掉 with lock，跑 10 次看最終數字是否每次都不同、且小於預期。然後改用 queue.Queue 收集各 thread 的結果、最後在主執行緒加總，做到完全不需要鎖。提示：用 Queue 傳資料取代共享可變狀態，是並行最不易出錯的模式。

  4. 同步轉 async：把一段用 ThreadPoolExecutor + 同步 urllib 抓多個 URL 的程式，改寫成 asyncio + httpx（async client）版本，用 TaskGroup 併發、用 except* 處理可能的 ExceptionGroup。可用 uv run --with httpx python your_script.py 執行。提示：注意「函式著色」會讓 fetch 與 main 都得是 async def。

  5. 別卡住 event loop：寫一個 asyncio 程式，故意在某個協程裡呼叫 time.sleep(2)（阻塞），觀察其他協程是否也被卡住；接著改成 await asyncio.to_thread(time.sleep, 2)，確認其他協程恢復併發。提示：在協程裡放阻塞呼叫會凍結整個 loop，這是 asyncio 最常見的 bug。
