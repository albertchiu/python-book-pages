第 15 章：檔案、I/O 與標準函式庫巡禮

身為 Java 工程師，你習慣「任何稍微像樣的事都要拉一個函式庫」：日期要 java.time 或更早是 Joda、JSON 要 Jackson、CSV 要 Apache Commons、HTTP client 要 OkHttp。Python 的世界觀不太一樣——它的標準函式庫信奉「內建電池」（batteries included），檔案、JSON、CSV、SQLite、日期時區、正規表示式、子行程、CLI 解析全都在標準庫裡，pip install 一個都不用。這一章是一趟標準庫巡禮，目標是讓你在「想寫一段 shell script 或拉一個小工具」時，第一反應是「Python 標準庫就有」，而不是 npm/Maven 找輪子。

pathlib：物件式的路徑，告別字串拼接

Java 的 java.nio.file.Path / Files 已經把「路徑是物件、檔案操作是靜態方法」這套教會你了。Python 的舊世界是 os.path——一堆吃字串、回字串的函式（os.path.join、os.path.exists、os.path.splitext），你得自己拼字串、自己記哪個函式做什麼。現代 Python（3.4+，到 3.12 已非常成熟）用 pathlib.Path 取而代之，而且比 Java 更俐落。

java 
// Java：java.nio
Path base = Paths.get("/var/log");
Path file = base.resolve("app").resolve("server.log");
if (Files.exists(file) && Files.isRegularFile(file)) {
    String text = Files.readString(file, StandardCharsets.UTF_8);
}
python 
from pathlib import Path

base = Path("/var/log")
file = base / "app" / "server.log"          # / 運算子組路徑，跨平台
if file.exists() and file.is_file():
    text = file.read_text(encoding="utf-8")

/ 運算子是最 Pythonic 的亮點：Path.__truediv__ 被多載成「組路徑」，等價於 Java 的 .resolve(...)，但讀起來像在寫真正的路徑。它會自動用對的分隔符（Windows 上是 \），所以你永遠不該手動 "a/" + "b" 或 os.path.join。

🔁 Java 對照：Paths.get(a).resolve(b) ≈ Path(a) / b。Files.readString(p) ≈ p.read_text()，Files.writeString ≈ p.write_text()，Files.exists(p) ≈ p.exists()。差別是 Python 把這些變成 Path 的方法，不是 Files 的靜態工具方法。

常用 API 速查

Path 物件把「路徑解析」和「檔案系統操作」都掛在自己身上：

python 
>>> from pathlib import Path
>>> p = Path("/var/log/app/server.log")
>>> p.name            # 檔名（含副檔名）
'server.log'
>>> p.stem            # 去掉副檔名的檔名
'server'
>>> p.suffix          # 副檔名
'.log'
>>> p.parent          # 父目錄（也是 Path）
PosixPath('/var/log/app')
>>> p.parts           # 拆成元組
('/', 'var', 'log', 'app', 'server.log')
>>> p.with_suffix(".gz")   # 把 .log 換成 .gz，回傳新 Path（不可變）
PosixPath('/var/log/app/server.gz')
>>> p.with_name("error.log")
PosixPath('/var/log/app/error.log')

Path 是不可變的（immutable），with_suffix/with_name 都回傳新物件，這跟 Java 的 Path 一致。

注意 with_suffix 是替換最後一個 suffix，不是附加。Path("server.log").with_suffix(".gz") 得到 server.gz（.log 被換掉），不是 server.log.gz。若你要在現有副檔名後再加一層（server.log → server.log.gz），請用 p.with_name(p.name + ".gz") 或 Path(str(p) + ".gz")。

讀寫一行解決，不用開檔案、不用記得關：

python 
config = Path("config.toml")
config.write_text("debug = true\n", encoding="utf-8")   # 覆寫，自動開關檔
content = config.read_text(encoding="utf-8")            # 一次讀完
data = Path("blob.bin").read_bytes()                    # 二進位

glob 與走訪目錄

glob 是 pathlib 真正讓人上癮的地方——比 Java 的 Files.newDirectoryStream + glob pattern 直覺得多：

python 
project = Path("src")

# 當層所有 .py
for py in project.glob("*.py"):
    print(py)

# 遞迴所有層的 .py（** 代表任意層數）
for py in project.rglob("*.py"):       # 等同 glob("**/*.py")
    print(py.relative_to(project))

# iterdir 只列當層（檔案 + 子目錄），不遞迴
for child in project.iterdir():
    kind = "dir " if child.is_dir() else "file"
    print(kind, child.name)

glob/rglob/iterdir 回傳的是惰性迭代器（lazy iterator，不是一次算完的 list），不會一次把整個目錄載進記憶體，掃幾十萬個檔案也不會爆。（嚴格來說 iterdir 是 generator，glob/rglob 在不同版本是 map 或一般 iterator，效果相同——都是惰性、邊走邊算；惰性求值的概念詳見第 6 章。）

建目錄與刪除：

python 
out = Path("build/reports")
out.mkdir(parents=True, exist_ok=True)   # 等同 mkdir -p，已存在不報錯
(out / "old.txt").unlink(missing_ok=True)  # 刪檔，不存在也不報錯

💡 Pythonic 重點：新程式碼一律用 pathlib.Path，把 os.path 當成只在讀舊碼時才需要的歷史包袱。需要把 Path 傳給只吃字串的舊 API 時，str(path) 即可；反過來大多數標準庫（含 open）都直接吃 Path，因為它實作了 os.PathLike 協定。

⚠️ 陷阱：Path("/a/b") / "/c" 會得到 Path("/c")，不是 /a/b/c——/ 運算子右邊若是絕對路徑，會覆蓋左邊，跟 Java resolve 行為一致。組路徑時別讓使用者輸入帶開頭斜線的字串。

檔案讀寫與編碼：那個會在 CI 上炸掉的 encoding

當你需要逐行串流、控制緩衝、或處理大檔時，就回到底層的 open()。承接第 5 章，慣用法是搭配 with（context manager），等同 Java 的 try-with-resources，離開區塊自動關檔——即使中途拋例外。

java 
// Java try-with-resources
try (BufferedReader r = Files.newBufferedReader(path, StandardCharsets.UTF_8)) {
    String line;
    while ((line = r.readLine()) != null) {
        process(line);
    }
}
python 
with open("access.log", encoding="utf-8") as f:
    for line in f:               # 檔案物件本身就是逐行的迭代器，惰性、不吃記憶體
        process(line.rstrip("\n"))

直接 for line in f 是 Pythonic 的逐行讀法：檔案物件是迭代器，一次只讀一行，處理 10GB 的 log 也只佔一行的記憶體。不要用 f.readlines()（一次全載進 list）除非你真的要整份在記憶體裡。

編碼：最該記住的一條規則

永遠明確寫 encoding="utf-8"。 這是從 Java 帶過來最容易踩的雷。Java 的 new FileReader(f)（無 charset）和 Python 的 open(f)（無 encoding）犯的是同一個錯：用平台預設編碼。在你的 Mac 上是 UTF-8、跑得好好的，到了 CI 的某些環境或 Windows（預設可能是 cp950/cp1252）就會 UnicodeDecodeError 或讀出亂碼。

python 
# 不要這樣（依賴 locale，跨機器不可重現）
text = open("data.txt").read()            # 這在某些平台會出錯或亂碼

# 一律這樣
text = open("data.txt", encoding="utf-8").read()
行為 Java Python
不指定編碼 new FileReader（無 charset）用 Charset.defaultCharset() open() 用 locale.getpreferredencoding()
明確指定 Files.newBufferedReader(p, UTF_8) open(p, encoding="utf-8")
讀整份字串 Files.readString(p, UTF_8) Path(p).read_text(encoding="utf-8")

📌 版本註記：自 Java 18（JEP 400）起 file.encoding 預設改為 UTF-8，平台預設不再是 locale 相依的雷，所以「new FileReader 用平台預設」這個坑主要在 Java 18 之前的版本。但 new FileReader 無 charset 多載仍沿用 Charset.defaultCharset()，而 Java 11 起 FileReader 已新增可指定 Charset 的建構子——明確指定編碼在現代 Java 一樣是好習慣。

💡 Pythonic 重點：Python 核心團隊也認為這是設計缺陷。PEP 686 規劃讓 UTF-8 成為未來的預設編碼，而 3.10+ 起你可以加 -X warn_default_encoding（或設環境變數 PYTHONWARNDEFAULTENCODING=1）讓直譯器在你忘記寫 encoding= 時發警告。在那之前，手寫 encoding="utf-8" 不是可選項，是紀律。

文字 vs 二進位、newline 處理

模式字串控制文字/二進位與讀寫：

python 
with open("photo.jpg", "rb") as f:    # 'b' = binary，回傳 bytes，沒有編碼概念
    header = f.read(8)

with open("out.csv", "w", encoding="utf-8", newline="") as f:
    ...                                # 寫 CSV 時 newline="" 交給 csv 模組處理換行

文字模式（預設）有兩個轉換：解碼（bytes ↔ str）和通用換行（universal newlines）。讀檔時 \r\n、\r 都會被翻譯成 \n；寫檔時 \n 會被翻回平台換行。這通常很貼心，但寫 CSV 時要 newline="" 關掉它，否則 Windows 上會出現多餘的 \r（下一節詳述）。

⚠️ 陷阱：二進位模式（"rb"/"wb"）下傳 encoding= 會直接報錯——bytes 沒有編碼概念。文字模式回 str、二進位回 bytes，兩者不能混（這呼應 Java 的 Reader/Writer 對 InputStream/OutputStream 之分）。

json / csv / sqlite3：序列化與內建資料庫

json：標準庫就是你的 Jackson

java 
// Jackson
ObjectMapper mapper = new ObjectMapper();
String json = mapper.writeValueAsString(user);
User u = mapper.readValue(json, User.class);
python 
import json

data = {"name": "Ada", "age": 36, "tags": ["math", "cs"]}
text = json.dumps(data, ensure_ascii=False, indent=2)   # 序列化成字串
parsed = json.loads(text)                               # 反序列化回 dict

對照 Jackson 的關鍵差異：

  • json.dumps ↔ writeValueAsString，json.loads ↔ readValue。檔案版本是 json.dump(obj, file) / json.load(file)（少一個 s，s 代表 string）。
  • 沒有「綁定到類別」這回事。json.loads 回傳的是 dict/list/str/int/float/bool/None 的組合，型別標注是 Any（呼應第 9 章——靜態檢查器對它一無所知，存取 parsed["age"] 不會有型別保護）。要綁到具型別的物件，請接 dataclass 或 pydantic（第 9 章）。
  • ensure_ascii=False 很重要：預設 True 會把中文跳脫成 \uXXXX，加 False 才會輸出可讀的 UTF-8。
python 
>>> import json
>>> json.dumps({"city": "台北"})
'{"city": "\\u53f0\\u5317"}'
>>> json.dumps({"city": "台北"}, ensure_ascii=False)
'{"city": "台北"}'

複雜物件（如 datetime）不能直接序列化，要給 default= 回呼，或先轉成可序列化型別：

python 
from datetime import datetime
json.dumps({"now": datetime(2026, 6, 14, 12, 0)}, default=str)
# '{"now": "2026-06-14 12:00:00"}'

csv：別自己 split 逗號

CSV 看似「用逗號切就好」，但欄位內含逗號、含換行、含引號的跳脫規則會讓手寫 parser 死得很難看。用標準庫的 csv：

python 
import csv
from pathlib import Path

# 寫：newline="" 是官方文件要求，避免 Windows 上多餘的 \r
with open("people.csv", "w", encoding="utf-8", newline="") as f:
    writer = csv.writer(f)
    writer.writerow(["name", "city"])
    writer.writerow(["Ada", "台北, 信義區"])   # 含逗號，csv 會自動加引號

# 讀成 dict（第一列當表頭）——比 index 取值好讀太多
with open("people.csv", encoding="utf-8", newline="") as f:
    for row in csv.DictReader(f):
        print(row["name"], "->", row["city"])

csv.reader 回傳每列為 list[str]；csv.DictReader 用表頭當 key 回傳 dict，等同 Java 你會自己包的「header-to-value map」，但內建。注意 CSV 的值永遠是字串，數字要自己 int()/float()。

sqlite3：口袋裡的關聯式資料庫

這是標準庫最被低估的寶藏：一個零設定、單檔、完整 SQL 的關聯式資料庫，直接內建。對 Java 而言這像「JDBC + H2 嵌入式資料庫」整包送你，連 driver 都不用下載。Python 的 DB API 是 PEP 249（DB-API 2.0），所有資料庫驅動（PostgreSQL 的 psycopg、MySQL 的驅動等）都遵循同一套介面，學會 sqlite3 等於學會通用模式。

python 
import sqlite3

# 連線（":memory:" 是純記憶體 DB，測試最愛用；或給檔名持久化）
conn = sqlite3.connect("app.db")
conn.execute("""
    CREATE TABLE IF NOT EXISTS users (
        id   INTEGER PRIMARY KEY,
        name TEXT NOT NULL,
        age  INTEGER
    )
""")

# 參數化查詢——務必用 ? 佔位符，絕不字串拼接
name, age = "Ada", 36
conn.execute("INSERT INTO users (name, age) VALUES (?, ?)", (name, age))

# 查詢：cursor 是迭代器
for row in conn.execute("SELECT name, age FROM users WHERE age > ?", (30,)):
    print(row)            # ('Ada', 36) —— 預設每列是 tuple
conn.commit()
conn.close()

參數化查詢防 SQL injection——這跟 Java 的 PreparedStatement 是同一條鐵律，而且 Python 更容易因為 f-string 太方便而犯錯：

python 
# 這會出錯 —— 災難性的 SQL injection 漏洞，永遠不要這樣
conn.execute(f"SELECT * FROM users WHERE name = '{name}'")   # 絕對禁止

# 正確：值交給 driver 用 ? 佔位
conn.execute("SELECT * FROM users WHERE name = ?", (name,))

⚠️ 陷阱：佔位符用 ?（sqlite3 的 qmark 風格），用法跟 Java JDBC PreparedStatement 的 ? 幾乎一樣——位置對應、值另外用元組傳入，driver 負責跳脫。注意參數一定要包成元組，單一參數也要寫成 (name,)（逗號不能省，否則不是 tuple）。

連線物件可以當 context manager，但語意是交易控制（transaction），不是關閉連線——這點和 Java 不同，容易誤解：

python 
import sqlite3

conn = sqlite3.connect(":memory:")
conn.execute("CREATE TABLE t (x INTEGER)")

with conn:                              # 進入區塊不做事，離開時 commit；拋例外則 rollback
    conn.execute("INSERT INTO t VALUES (1)")
    conn.execute("INSERT INTO t VALUES (2)")
# 區塊正常結束 → 自動 commit

讓查詢結果像 dict 一樣用欄名取值，設 row_factory：

python 
conn.row_factory = sqlite3.Row
row = conn.execute("SELECT 1 AS x, 2 AS y").fetchone()
print(row["x"], row["y"])              # 用欄名取值，1 2

💡 Pythonic 重點：寫單元測試需要資料庫時，sqlite3.connect(":memory:") 是神器——零 I/O、測完即丟、不留檔案，比起在 Java 拉一個 Testcontainers 或嵌入式 H2 輕量太多。

datetime 與時區：naive vs aware 的世界

java.time 已經把日期時間的正確心智模型灌給你了：Instant、LocalDateTime、ZonedDateTime 的分野你很熟。Python 的 datetime 模組概念相近，但詞彙不同，而且有一個你必須內化的二分法：naive 與 aware。

python 
from datetime import datetime, date, timedelta, timezone
from zoneinfo import ZoneInfo            # 3.9+，直接讀作業系統的 IANA 時區資料庫

d = date(2026, 6, 14)                    # 只有日期，對應 LocalDate
dt = datetime(2026, 6, 14, 12, 30)       # naive datetime，無時區資訊
delta = timedelta(days=7, hours=3)       # 對應 Duration / Period
print(dt + delta)                        # 2026-06-21 15:30:00
Java Python 性質
LocalDate date 純日期
LocalDateTime naive datetime（無 tzinfo） 牆上時鐘，不知道自己在哪個時區
ZonedDateTime aware datetime（有 tzinfo） 綁定時區，是地球上一個明確時刻
Duration / Period timedelta 時間長度
ZoneId.of("Asia/Taipei") ZoneInfo("Asia/Taipei") IANA 時區

naive 是陷阱，aware 才安全

naive datetime（沒有 tzinfo）就像 Java 的 LocalDateTime：它是「牆上的 12:30」，但 12:30 在台北和倫敦是不同的瞬間。拿 naive datetime 做跨時區運算、存進資料庫、或比較先後，就是 bug 的溫床。

python 
from datetime import datetime, timezone
from zoneinfo import ZoneInfo

# 不要：naive now，不知道在哪個時區
naive = datetime.now()                   # 危險，避免

# 要：aware now，綁定時區
taipei = datetime.now(ZoneInfo("Asia/Taipei"))
utc_now = datetime.now(timezone.utc)     # 存資料庫/傳輸一律用 UTC

# aware datetime 可以安全地跨時區轉換
print(taipei.astimezone(ZoneInfo("Europe/London")))   # 同一瞬間，換個時鐘看

ISO 8601 解析與輸出（資料交換的通用格式）：

python 
>>> from datetime import datetime
>>> from zoneinfo import ZoneInfo
>>> datetime.fromisoformat("2026-06-14T12:30:00+08:00")   # +08:00 偏移格式各版本皆支援
datetime.datetime(2026, 6, 14, 12, 30, tzinfo=datetime.timezone(datetime.timedelta(seconds=28800)))
>>> datetime.fromisoformat("2026-06-14T12:30:00Z")        # Z 後綴（與更寬鬆的 ISO 變體）需 3.11+
datetime.datetime(2026, 6, 14, 12, 30, tzinfo=datetime.timezone.utc)
>>> # isoformat 會把 +08:00 偏移一併輸出（用固定建構的 aware datetime 示範，可重現）
>>> datetime(2026, 6, 14, 12, 30, 0, 123456, tzinfo=ZoneInfo("Asia/Taipei")).isoformat()
'2026-06-14T12:30:00.123456+08:00'

💡 Pythonic 重點：黃金法則跟 Java 一樣——內部一律用 UTC 的 aware datetime 儲存與運算，只在顯示給人看的最後一刻才 astimezone 轉成當地時區。datetime.now(timezone.utc) 取代 naive 的 datetime.now()，把 naive datetime 當成「尚未補全時區資訊」的半成品。

⚠️ 陷阱：naive 和 aware datetime 不能互相比較或相減，會拋 TypeError: can't subtract offset-naive and offset-aware datetimes。這其實是好事——它逼你在邊界把時區補齊，而不是讓兩個語意不同的時間默默運算出錯誤答案。

re：正規表示式

Java 的 Pattern/Matcher 你很熟，Python 的 re 模組概念一一對應，但 API 更扁平，而且有個語法層的好朋友：raw string。

java 
// Java
Pattern p = Pattern.compile("(\\d{4})-(\\d{2})-(\\d{2})");
Matcher m = p.matcher("date: 2026-06-14");
if (m.find()) {
    String year = m.group(1);
}
python 
import re

pat = re.compile(r"(\d{4})-(\d{2})-(\d{2})")   # r"" raw string，反斜線不用跳脫
m = pat.search("date: 2026-06-14")
if m:
    year = m.group(1)              # '2026'
    print(m.groups())              # ('2026', '06', '14')

💡 Pythonic 重點：正規表示式一律用 raw string r"..."。否則 \d 在普通字串裡 Python 會嘗試解讀跳脫序列（雖然 \d 剛好沒事，但 \b 就會變成退格字元），你會被迫寫成 Java 那種 "\\d" 雙反斜線地獄。r"\d" 直接就是你要的兩個字元。

match / search / findall / finditer

這幾個方法的差別是 Java 工程師常搞混的：

方法 行為 Java 對照
re.match(p, s) 只從字串開頭比對 matcher.lookingAt()
re.search(p, s) 在字串中找第一個符合處 matcher.find()（第一次）
re.fullmatch(p, s) 整個字串完全符合 matcher.matches()
re.findall(p, s) 回所有符合的字串 list 反覆 find() 收集
re.finditer(p, s) 回所有 match 物件的迭代器 反覆 find()（惰性）
python 
text = "errors: 12, warnings: 5, errors: 3"

re.findall(r"\d+", text)               # ['12', '5', '3']  ——只要值
for m in re.finditer(r"(\w+): (\d+)", text):
    print(m.group(1), m.group(2))      # 連同位置/群組都要時用 finditer

命名群組讓正則更可維護（對照 Java 的 (?<year>...) 完全一樣）：

python 
m = re.search(r"(?P<year>\d{4})-(?P<month>\d{2})", "2026-06")
m.group("year")        # '2026'
m.groupdict()          # {'year': '2026', 'month': '06'}

re.sub 做取代，可吃字串或回呼函式：

python 
re.sub(r"\d+", "#", "a1b22c333")                       # 'a#b#c#'
re.sub(r"\d+", lambda m: str(int(m.group()) * 2),
       "a1b22c333")                                    # 'a2b44c666'，每個數字 ×2

💡 Pythonic 重點：模組層的 re.search(pattern, s) 會自動快取已編譯的 pattern，偶爾用幾次不必手動 re.compile。但若同一個 pattern 在迴圈裡跑幾萬次，pat = re.compile(...) 一次、之後重用 pat.search(...)，意圖更清楚也省去快取查找——這跟你在 Java 把 Pattern.compile 提到迴圈外是同一個直覺。

subprocess：取代 shell script 的正路

要呼叫外部程式時，Java 你會用 ProcessBuilder。Python 的 subprocess.run 更精簡，但有幾個安全與正確性的預設你必須記住。

java 
// Java ProcessBuilder
Process p = new ProcessBuilder("git", "rev-parse", "HEAD")
        .redirectErrorStream(true).start();
String out = new String(p.getInputStream().readAllBytes());
p.waitFor();
python 
import subprocess

result = subprocess.run(
    ["git", "rev-parse", "HEAD"],   # 引數用 list，不是一整串字串
    capture_output=True,             # 抓 stdout/stderr
    text=True,                        # 解碼成 str（否則是 bytes）
    check=True,                       # 非零退出碼就拋 CalledProcessError
)
print(result.stdout.strip())         # 命令的標準輸出

四個關鍵參數，缺一個就少一道防護：

  • 引數傳 list（["git", "rev-parse", "HEAD"]），每個引數一個元素。這是安全的核心。
  • capture_output=True：把 stdout/stderr 收進 result.stdout/result.stderr。
  • text=True：以文字模式解碼（記得這也是編碼，可加 encoding="utf-8"）。不加就是 bytes。
  • check=True：行為等同 shell 的 set -e——子程式失敗（非零退出碼）就拋例外，而不是默默繼續。這跟你檢查 p.exitValue() 的紀律一樣，但自動化了。

⚠️ 陷阱：不要用 shell=True。它會把你的命令丟給 /bin/sh 解析，等於開了一個 command injection 大門：subprocess.run(f"cat {user_input}", shell=True) 裡只要 user_input 是 "; rm -rf ~" 就完蛋。傳 list of args（不開 shell）時，引數會原封不動交給程式、不經 shell 解析，使用者輸入裡的 ;、|、$() 都只是普通字元，注入無從發生。需要管線（pipe）時，用 Python 串接兩個 subprocess，不要靠 shell=True。

🔁 Java 對照：ProcessBuilder("a", "b")（vararg 形式，不經 shell）≈ subprocess.run(["a", "b"])。Java 也沒有「把整串字串丟給 shell」的預設——Python 的 shell=True 就是那個你在 Java 從沒有過、也不該想要的危險捷徑。

argparse / click：寫 CLI 工具

標準庫已經能讓你寫出有 --help、子命令、型別轉換的專業 CLI，這正是「用 Python 取代 shell script」的最後一塊拼圖。

argparse：標準庫，宣告式

argparse 內建，零依賴，宣告式定義參數，自動產生 --help 和錯誤訊息：

python 
# greet.py
import argparse

def main() -> None:
    parser = argparse.ArgumentParser(description="跟某人打招呼")
    parser.add_argument("name", help="對象名字")            # 位置引數
    parser.add_argument("-c", "--count", type=int, default=1,
                        help="重複次數")                    # 選項，自動轉 int
    parser.add_argument("--shout", action="store_true",
                        help="大聲喊")                       # 布林旗標
    args = parser.parse_args()

    msg = f"Hello, {args.name}!"
    if args.shout:
        msg = msg.upper()
    for _ in range(args.count):
        print(msg)

if __name__ == "__main__":
    main()
text 
$ python greet.py Ada --count 2 --shout
HELLO, ADA!
HELLO, ADA!
$ python greet.py --help
usage: greet.py [-h] [-c COUNT] [--shout] name
...

argparse 對應 Java 生態的 Apache Commons CLI 或 picocli，但內建。它支援子命令（add_subparsers，像 git commit / git push 那種風格），足以撐起一個正經的工具。

click：第三方，decorator 式

當 CLI 變大（巢狀子命令、彩色輸出、互動提示），社群標準是 click。它用 decorator 宣告（呼應第 10 章），把參數變成函式的引數，讀起來像在寫普通函式：

python 
# 用 uv 驗證：uv run --with click greet_click.py Ada --count 2 --shout
import click

@click.command()
@click.argument("name")
@click.option("--count", default=1, help="重複次數")
@click.option("--shout", is_flag=True, help="大聲喊")
def greet(name: str, count: int, shout: bool) -> None:
    """跟某人打招呼。"""
    msg = f"Hello, {name}!"
    if shout:
        msg = msg.upper()
    for _ in range(count):
        click.echo(msg)

if __name__ == "__main__":
    greet()
text 
$ uv run --with click greet_click.py Ada --count 2 --shout
HELLO, ADA!
HELLO, ADA!

click 的 decorator 把「參數宣告」和「使用該參數的函式」綁在一起，少了 args.name 這層手動取值；--help 直接取自 docstring 與 help=。子命令用 @click.group() 組起來，比 argparse 的 subparser 優雅。

💡 Pythonic 重點：選型原則——零依賴、簡單腳本用 argparse（標準庫，不增加部署負擔）；功能多、要分發給別人用的正式 CLI 用 click（或它的進化版 typer，把參數宣告交給型別提示，呼應第 9 章）。這跟 Java 在 picocli vs 手刻 args 之間的權衡一樣，只是 Python 的標準庫起點高很多。

本章小結

  1. pathlib.Path 取代 os.path：物件式 API、/ 運算子組路徑、.read_text()/.glob()/.iterdir() 等方法掛在 Path 上，比 Java 的 Files 靜態方法更順手；glob/iterdir 回產生器不吃記憶體。
  2. 編碼是紀律不是選項：open() 與 read_text() 一律明確寫 encoding="utf-8"，否則依賴平台預設會在別的機器上爆掉（同 Java new FileReader 之坑）；PEP 686 是未來解，現在靠手寫。
  3. 序列化用標準庫：json（dumps/loads，回傳是 Any，要 ensure_ascii=False）、csv（DictReader 別自己切逗號，寫檔記得 newline=""），都內建、不必拉 Jackson/Commons。
  4. sqlite3 是口袋裡的關聯式 DB：零設定、:memory: 模式測試神器、遵循 DB-API 2.0；參數化查詢用 ? 防 SQL injection（同 PreparedStatement），連線當 context manager 是控制交易、不是關閉。
  5. datetime 分 naive 與 aware：naive ≈ LocalDateTime（危險）、aware ≈ ZonedDateTime（安全）；用 datetime.now(timezone.utc) 與 zoneinfo.ZoneInfo，內部存 UTC、顯示才 astimezone，naive 與 aware 不能混算。
  6. re 一律用 raw string r"..."：match（開頭）/search（任意處）/fullmatch（全符合）/findall（回字串）/finditer（回 match 物件）分清楚；命名群組 (?P<name>...) 與 re.sub 的回呼形式提升可維護性。
  7. subprocess.run 傳 list of args、配 capture_output=True, text=True, check=True，永遠不要 shell=True（command injection 風險）；它是取代 shell script 的正路。
  8. CLI 用 argparse（標準庫）或 click（第三方 decorator 式）：Python 的「內建電池」讓你能把一段段 shell script 升級成有 --help、有型別轉換、有子命令的正經工具。

練習

  1. pathlib 改寫：給一個目錄路徑，找出底下（含所有子目錄）大於 1 MB 的 .log 檔，印出相對路徑與大小。請只用 pathlib，不要用 os.path。 提示：rglob("*.log") 配 .stat().st_size；relative_to(base) 取相對路徑。

  2. 編碼地雷排查：下面的程式在你機器上沒事，但同事在 Windows 上跑出 UnicodeDecodeError，為什麼？怎麼修？

    python 
    names = open("names.txt").read().splitlines()

    提示：問題在那個沒寫的參數。順便想想如果檔案很大，.read().splitlines() 還有什麼缺點。

  3. SQL injection 修補：把這段有漏洞的查詢改成安全的參數化版本，並說明為什麼 f-string 在這裡是大忌。

    python 
    def find_user(conn, username):
    return conn.execute(
        f"SELECT * FROM users WHERE name = '{username}'"
    ).fetchall()

    提示：? 佔位符 + 參數元組，注意單一參數的逗號。

  4. 時區運算：寫一個函式 meeting_times(utc_iso: str) -> dict[str, str]，輸入一個 UTC 的 ISO 時間字串，回傳這個瞬間在 Asia/Taipei、Europe/London、America/New_York 三地各是幾點（ISO 格式）。 提示：datetime.fromisoformat 解析、ZoneInfo 建時區、astimezone 轉換。先確認解析出來的是 aware datetime。

  5. shell script 升級：把這個 Bash 一行命令 git log --oneline -n 5 | wc -l 用 Python 重寫，不准用 shell=True，並用 check=True 處理「不在 git repo」的失敗情況。 提示：兩個 subprocess.run，或一個 subprocess.run 抓 git log 輸出後在 Python 裡數行數（更 Pythonic，免去管線）。
