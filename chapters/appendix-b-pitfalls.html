附錄 B：常見陷阱清單（Java 工程師專屬）

你已經讀完十八章，知道 Python 怎麼「正確地寫」。但移植十年的 Java 直覺時，有些坑是語法看起來合法、編譯器（mypy）也不一定攔得住、單元測試偶爾才紅一次的那種——它們源自綁定時機（binding time）、身分相等（identity vs equality）、求值時機（evaluation time）與 GIL 等機制差異。本附錄把這些高頻陷阱濃縮成可掃讀的速查表，每一條都附實際跑過的最小重現、機制解釋與正解。建議在 code review 時把結尾的檢查清單貼進 PR 範本。

慣例：標注「反模式」或「刻意示範錯誤」的程式碼請勿照抄；每條都有「正解」段落給你帶走。

B.1 綁定與求值時機

陷阱 1：可變預設參數 `def f(items=[])`

Java 沒有預設參數，你大概用多載（overload）或 Optional 模擬；移植時很自然寫成 def f(items=[])，以為「每次呼叫沒給就是一個新的空 list」。錯。

⚠️ 刻意示範錯誤：

python 
def append_to(item, items=[]):
    items.append(item)
    return items

print(append_to(1))
print(append_to(2))
print(append_to(3))

實際輸出：

text 
[1]
[1, 2]
[1, 2, 3]

為什麼：預設值物件在函式定義期（def 執行時）只建立一次，綁在 function 物件的 __defaults__ 上，被所有呼叫共用。這跟「import-time 副作用」「class body 求值」是同一個機制（程式碼由上而下執行，def 本身是一個敘述）。list 是可變的，於是每次 mutate 都累積在同一個物件上。

正解：用 None 哨兵（sentinel），在函式體內建立新物件：

python 
def append_to(item, items: list | None = None) -> list:
    if items is None:
        items = []
    items.append(item)
    return items

詳見第 3 章（可變預設參數、type hints）。

陷阱 2：預設值與型別提示在「定義期」求值

承上，不只 list 會中招。任何寫在預設值位置的運算式都在定義期求值一次，之後恆定不變——這常見於用「現在時間」「讀環境變數」「連資料庫」當預設值。

⚠️ 刻意示範錯誤：

python 
import time

def log(msg, ts=time.time()):
    return f"{ts}: {msg}"

print(log("first"))
time.sleep(1)
print(log("second"))   # ts 一樣！不是呼叫時的時間

實際輸出：

text 
1781482018.610712: first
1781482018.610712: second

為什麼：time.time() 在 def 執行的那一刻就被算成一個 float 並凍結。Java 的方法簽章沒有這種東西，所以沒有對應直覺。注意預設值（=time.time()）一律於定義期求值，這是核心論點、跨版本恆真。型別提示則要分版本看：在 Python 3.12/3.13，未加 from __future__ import annotations 時，標註運算式（如 def f(x: make())）也會在定義期求值——這也是為何昂貴或會循環匯入的型別常用 from __future__ import annotations 或字串延後求值。但自 Python 3.14 起（PEP 649/749），標註預設改為延後求值（deferred annotations），定義期不再執行標註運算式（連 def f(x: undefined_name) 都不會在定義期報錯），上述 __future__ 動機在 3.14 已大致被預設行為取代。

正解：同樣用 None 哨兵，把「要在呼叫時才算」的東西移進函式體：

python 
def log(msg: str, ts: float | None = None) -> str:
    if ts is None:
        ts = time.time()
    return f"{ts}: {msg}"

詳見第 3 章（type hints）、第 11 章（import-time 副作用）。

陷阱 3：closure 延遲綁定迴圈變數（lambda / comprehension in loop）

在 Java 你寫 lambda 捕捉迴圈變數時，編譯器強迫變數 effectively final，你會用 final int j = i; 複製一份。Python 沒這個限制，於是一不小心全部 lambda 共享同一個變數。

⚠️ 刻意示範錯誤：

python 
funcs = [lambda: i for i in range(3)]
print([f() for f in funcs])

實際輸出：

text 
[2, 2, 2]

為什麼：closure 捕捉的是變數本身（by reference），不是當下的值。三個 lambda 都指向同一個 i；等到實際呼叫 f() 時迴圈早已跑完，i 停在最後一個值 2。Java 的「effectively final」正是為了防這個坑而設的限制。

正解：用預設參數在「定義期」把當下的值快照下來（呼應陷阱 1/2 的同一機制，這次反過來利用它）：

python 
>>> funcs = [lambda i=i: i for i in range(3)]
>>> print([f() for f in funcs])
[0, 1, 2]

詳見第 3 章（closure、nonlocal）。

B.2 身分、相等與數值

陷阱 4：`is` 與 `==` 混用；`is` 只該用於 None / 哨兵

Java 的 == 比參考、.equals() 比內容，你被訓練成「物件一律用 .equals()」。到了 Python，== 才是內容比較（會呼叫 __eq__），is 是身分比較（記憶體同一物件，等同 Java 的 ==）。把兩者對調是常見手滑。

⚠️ 刻意示範錯誤與對照：

python 
class Weird:
    def __eq__(self, other):
        return True

w = Weird()
print(w == None)   # __eq__ 被呼叫，可能說謊
print(w is None)   # 身分比較，不會被騙

實際輸出：

text 
True
False

接著示範「值一樣 ≠ 同一物件」，但這一段必須在互動式 REPL逐行輸入才看得到，原因見陷阱 5：

python 
>>> a = "hello world this is long"
>>> b = "hello world this is long"
>>> a == b      # 值相等
True
>>> a is b      # 身分比較，跨編譯單元各自建立物件
False

為什麼：== 可被 __eq__ 覆寫，連跟 None 比都可能回 True；只有 is 保證問的是「同一個物件嗎」。判斷 None、True、False 或自訂哨兵時務必用 is，因為它們是單例（singleton）。反之，比較內容（字串、數字、容器）一律用 ==——上面 REPL 裡 a is b 回 False 證明「值一樣不代表同一物件」。注意這個 False 只在 REPL（各行獨立編譯）成立；若把同樣兩行寫進一個 .py 腳本，CPython 會在編譯期把兩個相同字面量 intern／摺疊成同一物件，a is b 反而印 True——這恰恰證明 is 的結果是實作細節、不可依賴（詳見陷阱 5）。

正解：

python 
if value is None: ...        # 對 None/哨兵用 is
if name == "admin": ...      # 對內容用 ==

詳見第 2 章（is vs ==、None、truthiness）。

陷阱 5：整數快取 `-5..256` 製造的 `is` 假象

接續上一條。新手常「測試 is 時剛好成功」而誤以為 is 可以拿來比數字。那是 CPython 的小整數快取在騙你。

⚠️ 刻意示範錯誤（在互動式 REPL逐行輸入）：

python 
>>> a = 256
>>> b = 256
>>> a is b      # 都指向快取的同一個物件
True
>>> c = 257
>>> d = 257
>>> c is d      # 超出快取範圍，且各行獨立編譯
False
>>> m = -6
>>> n = -6
>>> m is n      # 同理
False

為什麼：CPython 預先建立並快取 -5 到 256 的整數物件重複使用，所以這個範圍內 is 會「碰巧成立」。但 is 的結果不只受小整數快取影響，也受編譯期常數摺疊（constant folding）左右——在同一個編譯單元（同一個檔案、同一個函式）內寫兩次 257，CPython 會把它們摺疊成 co_consts 裡的同一個常數物件，使 c is d 變成 True。唯有跨編譯單元（互動 REPL 各行各自編譯、或 exec 不同字串）才看得到上面 False 的結果。所以這段刻意用 REPL 逐行呈現：若把同樣的程式寫進一個 .py 腳本，c is d 與 m is n 反而都會印出 True。這整套都是實作細節，不是語言保證，PyPy 或不同版本行為可能不同。注意上面用變數而非字面量——直接寫 256 is 256 會觸發 SyntaxWarning: "is" with 'int' literal。

正解：數值永遠用 ==，把 is 留給 None。

詳見第 2 章。

陷阱 6：浮點數相等與 `0.1 + 0.2`

BigDecimal 在 Java 是金額的標配，但日常 double 你大概也踩過精度。Python 的 float 同樣是 IEEE 754 雙精度，問題一模一樣，只是 REPL 直接把真相印給你看。

⚠️ 刻意示範錯誤：

python 
>>> 0.1 + 0.2
0.30000000000000004
>>> 0.1 + 0.2 == 0.3
False

為什麼：0.1、0.2、0.3 都無法用二進位浮點精確表示，相加後的捨入誤差讓 == 失敗。這與 Java 完全同源。

正解：金額用 decimal.Decimal（且用字串建構，別用 Decimal(0.1)）；一般近似比較用 math.isclose：

python 
>>> from decimal import Decimal
>>> Decimal("0.1") + Decimal("0.2") == Decimal("0.3")
True
>>> import math
>>> math.isclose(0.1 + 0.2, 0.3)
True

詳見第 2 章（數值）、第 13 章（近似比較 pytest.approx）；金額用 Decimal 的範例見第 11 章。

陷阱 7：整數除法 `/` vs `//`，且 `//` 是向下取整

Java 的 / 對兩個 int 是整數除法且向零取整（truncation）：-7 / 2 == -3。Python 的 / 一律回 float，整數除法要用 //，而且 // 是向下取整（floor），負數結果跟 Java 不同。

運算式 Java（int） Python 說明
7 / 2 3 3.5 Python / 永遠 true division 回 float
7 // 2 （無此運算子） 3 floor division
-7 // 2 -3（-7/2） -4 Java 向零、Python 向下
-7 % 3 -1 2 Python 餘數與除數同號

⚠️ 對照：

python 
>>> 7 / 2
3.5
>>> -7 // 2
-4
>>> -7 % 3
2

為什麼：Python 定義 a == (a // b) * b + (a % b)，並選擇讓 // 向負無窮取整、餘數與除數同號。Java 選擇向零取整、餘數與被除數同號。移植涉及負數的索引、雜湊、分頁計算時最容易出錯。

正解：要 Java 那種向零行為用 math.trunc(a / b) 或 int(a / b)：

python 
>>> import math
>>> math.trunc(-7 / 2)
-3

詳見第 2 章。

B.3 容器、別名與共享狀態

陷阱 8：別名與淺拷貝——賦值不複製、`[[0]*n]*m` 共享同一列

兩個同源的坑放一起看。其一，Java 物件參考賦值也是別名，這點你懂；但 Python 沒有原生型別的 value semantics，連看似輕量的 list/dict 賦值都只是多綁一個名字到同一物件。

⚠️ 刻意示範錯誤（賦值即別名）：

python 
>>> a = [1, 2, 3]
>>> b = a
>>> b.append(4)
>>> a
[1, 2, 3, 4]

為什麼：b = a 只是讓名字 b 綁到 a 指的同一個 list 物件（name binding，第 2 章）。透過任一名字 mutate 都會反映到另一個。

其二，要初始化二維陣列，Java 寫 new int[m][n] 各列獨立。Python 用 [[0]*n]*m 看起來等價，其實 *m 是把同一個內層 list 的參考複製 m 份。

⚠️ 刻意示範錯誤（*m 共享內層列）：

python 
>>> grid = [[0] * 3] * 2
>>> grid[0][0] = 9
>>> grid
[[9, 0, 0], [9, 0, 0]]
>>> grid[0] is grid[1]
True

為什麼：* 複製的是元素參考，不是深拷貝；外層兩個元素指向同一個內層 list，改一個就「全改」。copy.copy（淺拷貝）也只複製一層，巢狀容器仍共享：

⚠️ 對照淺 vs 深：

python 
>>> import copy
>>> original = [[1, 2], [3, 4]]
>>> shallow = copy.copy(original)
>>> shallow[0].append(99)
>>> original                       # 內層被一起改到
[[1, 2, 99], [3, 4]]

正解：要拷貝就明講——a.copy()、list(a)、a[:]（淺）或 copy.deepcopy（深）；每列要獨立用 comprehension；要完整獨立用 deepcopy：

python 
>>> grid = [[0] * 3 for _ in range(2)]
>>> grid[0][0] = 9
>>> grid
[[9, 0, 0], [0, 0, 0]]
>>> deep = copy.deepcopy(original)   # 巢狀也獨立

詳見第 4 章（切片、深淺拷貝）。

陷阱 9：可變類別屬性被所有實例共享（誤當 instance field）

這是 Java 工程師最痛的一條。你以為在 class body 裡寫 transactions = [] 等同 Java 的 instance field；其實它是 class attribute（類比 Java 的 static 欄位），被所有實例共享。

⚠️ 刻意示範錯誤：

python 
class Account:
    transactions = []           # 看似 instance field，實為 class attribute

    def add(self, x):
        self.transactions.append(x)

a, b = Account(), Account()
a.add(100)
b.add(200)
print(a.transactions)
print(b.transactions)
print(a.transactions is b.transactions)

實際輸出：

text 
[100, 200]
[100, 200]
True

為什麼：class body 的賦值建立的是類別層級屬性，只有一份。self.transactions.append(...) 透過實例讀到的其實是那個共享的 class attribute（讀取會沿 MRO 往上找），於是兩個實例 mutate 的是同一個 list。注意：若是 self.transactions = [...]（重新指派）才會建立 instance attribute 遮蔽掉它——但 .append 是 mutate，不會。

正解：實例狀態一律在 __init__ 裡用 self. 建立；或直接用 @dataclass 搭配 field(default_factory=list)：

python 
from dataclasses import dataclass, field

@dataclass
class Account:
    transactions: list[int] = field(default_factory=list)

詳見第 7 章（類別 vs 實例屬性、@dataclass）。

陷阱 10：迴圈中修改正在迭代的 list

Java 在迭代時改集合會直接丟 ConcurrentModificationException 救你。Python 不會報錯，而是默默跳過元素，給你一個錯誤結果。

⚠️ 刻意示範錯誤：

python 
nums = [1, 2, 4, 5]
for n in nums:
    if n % 2 == 0:
        nums.remove(n)
print(nums)

實際輸出：

text 
[1, 4, 5]

為什麼：for 在底層用整數索引推進迭代器。走到索引 1 讀到 2、判定為偶數並 remove 後，後面元素全部左移一格（4 從索引 2 移到索引 1），但迭代器的索引仍繼續往前推進到 2——於是直接跳過了剛遞補到索引 1 的 4，4 沒被檢查到就殘留下來。正確結果應是 [1, 5]，buggy 版卻是 [1, 4, 5]——兩個值明顯不同，漏刪的 4 就是 bug 的證據。連續相鄰的偶數最容易觸發這種漏刪。沒有例外保護，bug 會悄悄溜進正式環境。

正解：建立新 list（filter 思維），別原地刪：

python 
nums = [n for n in nums if n % 2 != 0]

若必須原地，迭代副本 for n in nums[:]: 或反向走訪。

詳見第 4 章（list 內部實作）、第 5 章（comprehension）。

陷阱 10.5：成員測試 `x in my_list` 是 O(n) 線性掃描

Java 你也許下意識用 List.contains 做查重；在小資料沒事，但 Python 的 x in some_list 同樣是逐一比對的 O(n) 線性掃描，把它放進迴圈就變成 O(n²)，大量資料下會從「慢」變成「卡死」。set／dict 的成員測試才是雜湊查找的 O(1)。

以下為實測（CPython 3.14，在 100 萬元素的容器各查 100 次最後一個元素）：

python 
import timeit
big = list(range(1_000_000))
t_list = timeit.timeit(lambda: 999_999 in big, number=100)
big_set = set(big)
t_set = timeit.timeit(lambda: 999_999 in big_set, number=100)
print(f"list: {t_list:.4f}s  set: {t_set:.6f}s")

實際輸出（數量級差異，絕對值因機器而異）：

text 
list: 0.5286s  set: 0.000004s

為什麼：list 沒有索引結構，in 必須從頭比對到命中或掃完；set／dict 用雜湊表，平均一次計算就定位。差距在這裡是十萬倍量級。同理，list.insert(0, ...)／list.pop(0) 在頭部操作要搬移整個底層陣列，也是 O(n)——需要雙端高頻進出時改用 collections.deque（兩端 O(1)，但隨機索引退化為 O(n)）。

正解：需要反覆查重／成員測試時，先把資料放進 set（或 dict 取鍵），別對 list 反覆 in：

python 
seen = set(existing_ids)        # 一次性 O(n) 建表
if new_id in seen:              # 之後每次查 O(1)
    ...

詳見第 4 章（容器複雜度、deque、set/dict）。

B.4 語法陷阱

陷阱 11：連鎖比較、`and`/`or` 回傳運算元、單元素 tuple

三個獨立但都會讓 Java 人愣住的語法細節：

(a) 連鎖比較：Python 支援 a < b < c，這是數學寫法不是兩個布林。Java 寫 1 < x < 10 根本編譯不過（boolean < int）。

python 
>>> x = 5
>>> 1 < x < 10
True
>>> 1 < x <= 5 < 100
True

中間運算元只求值一次，且具短路特性。

(b) and/or 回傳「運算元」而非 bool：Java 的 &&/|| 一定回 boolean；Python 回的是最後決定結果的那個運算元本體。

python 
>>> "a" or "b"
'a'
>>> "" or "fallback"
'fallback'
>>> 0 and 99
0
>>> 2 and 99
99

這常被用作 x = config or default 的慣用法，但若 config 是合法的 falsy 值（0、""、[]）就會誤判——這時該用 x if x is not None else default。

(c) 單元素 tuple 一定要逗號：括號不構成 tuple，逗號才構成。

python 
>>> type((1))
<class 'int'>
>>> type((1,))
<class 'tuple'>

漏掉逗號是「函式只想回一個元素的 tuple」「裝飾器/路由設定」最隱晦的 bug 來源。

詳見第 2 章（運算子）、第 5 章（慣用法、f-string、海象）。

B.5 例外與日誌

陷阱 12：裸 `except` 吞例外、`except` 子類順序寫反

Java 有 checked exception 與「子類在前、父類在後」的編譯期檢查；Python 兩者都沒有（第 12 章說明無 checked exception）。寫反順序或裸 except 不會被編譯器擋下。

⚠️ 刻意示範錯誤：

python 
try:
    raise ValueError("specific")
except Exception:
    print("攔到 Exception")
except ValueError:
    print("攔到 ValueError（永遠不會到這）")

實際輸出：

text 
攔到 Exception

為什麼：except 由上而下比對，第一個符合的就贏。ValueError 是 Exception 子類，父類寫在前面就把所有東西攔光，後面的子類分支變成死碼（Python 不會像 Java 那樣編譯期報「已被攔截」）。直譯器與 mypy 都不會攔，但 ruff／pylint 等 linter 可偵測到無法到達的 except（如 ruff 的 B014、pylint 的 W0703／bad-except-order）與過寬攔截，建議納入 CI。

另一半問題是裸 except: 會連 KeyboardInterrupt、SystemExit、MemoryError 都吞掉，讓程式無法被 Ctrl-C 中止、也掩蓋真正的 bug：

python 
def parse(s):
    try:
        return int(s)
    except:          # 反模式：吞掉一切
        return None

正解：子類在前、父類在後；只攔你能處理的最窄型別；保留例外鏈：

python 
try:
    value = int(s)
except ValueError as exc:
    raise ConfigError(f"無效數值: {s}") from exc

詳見第 12 章（例外階層、raise from、無 checked exception）。

陷阱 13：用 `print` 取代 `logging`；f-string 在 logging 的求值問題

Java 你早就用 SLF4J 的 log.debug("x={}", x) 參數化佔位，而非字串相加——正是為了等級不符時不付出字串組裝成本。移植到 Python 卻常退化成 logging.debug(f"...")，把這個好習慣丟了。

⚠️ 刻意示範錯誤（DEBUG 關閉時仍付出代價）：

python 
import logging
logging.basicConfig(level=logging.WARNING)
logger = logging.getLogger("demo")

calls = 0
def expensive():
    global calls
    calls += 1
    return "payload"

logger.debug(f"result={expensive()}")          # f-string：先求值
print("expensive 被呼叫次數:", calls)

實際輸出：

text 
expensive 被呼叫次數: 1

為什麼：f-string 在傳入 logger.debug 之前就被 Python 求值完畢，即使 DEBUG 等級不會輸出，expensive() 還是被呼叫、字串還是被組好。%-style 參數化則把格式化延後到確定要輸出時才做——所以等級不符時不會呼叫引數的 __str__／不展開格式：

python 
>>> logger.debug("result=%s", obj)   # obj.__str__() 在 DEBUG 關閉時不會被呼叫

但注意：%-style 只延後「格式化」，引數運算式本身（如 expensive()）仍會先求值。要連昂貴運算都跳過，就先檢查等級或包成惰性物件：

python 
if logger.isEnabledFor(logging.DEBUG):
    logger.debug("result=%s", expensive())

正解：用 logging 而非 print；訊息一律用 %s 佔位讓格式化惰性化；昂貴引數加 isEnabledFor 守衛。

還有一個對 Java 工程師同樣高頻的相關坑：在 except 區塊記錄錯誤時只寫 logger.error(str(e))（或 logger.error(f"{e}")）會丟失 traceback——對照你在 Java 寫的 log.error(e.getMessage()) 沒把 exception 物件傳進去，看到的就只有一句訊息、沒有堆疊。Python 要在 except 內用 logger.exception(msg)（或等價的 logger.error(msg, exc_info=True)），它會自動附上當前正在處理的例外與完整 traceback：

⚠️ 對照（同一個 ZeroDivisionError，記錄方式不同）：

python 
import logging
logging.basicConfig(level=logging.ERROR, format="%(levelname)s %(message)s")
logger = logging.getLogger("demo")

try:
    1 / 0
except ZeroDivisionError as e:
    logger.error(str(e))                  # 反模式：只剩一句，丟失 traceback

try:
    1 / 0
except ZeroDivisionError:
    logger.exception("division failed")   # 正解：自動帶 traceback

實際輸出：

text 
ERROR division by zero
ERROR division failed
Traceback (most recent call last):
  File "...", line 13, in <module>
    1 / 0
    ~~^~~
ZeroDivisionError: division by zero

logger.exception 只能在 except 區塊內呼叫（它讀取「當前正在處理的例外」）；等級固定為 ERROR。

詳見第 12 章（logging、warnings、exc_info）。

B.6 並行與 import

陷阱 14：以為 thread 能加速 CPU bound（GIL）

Java 的 thread 是真平行，CPU 密集任務開多執行緒就提速。Python 的 CPython 有 GIL（全域直譯器鎖，Global Interpreter Lock）：同一時刻只有一條 thread 在執行 Python bytecode。CPU 密集用 thread 等於沒開。

以下為實測（CPython 3.14，兩個各數 2000 萬次的迴圈）：

python 
import time
from concurrent.futures import ThreadPoolExecutor, ProcessPoolExecutor

def count(n):
    total = 0
    for _ in range(n):
        total += 1
    return total

N = 20_000_000

def main():
    t0 = time.perf_counter(); count(N); count(N)
    print(f"序列        : {time.perf_counter()-t0:.2f}s")
    t0 = time.perf_counter()
    with ThreadPoolExecutor(max_workers=2) as ex:
        list(ex.map(count, [N, N]))
    print(f"ThreadPool  : {time.perf_counter()-t0:.2f}s")
    t0 = time.perf_counter()
    with ProcessPoolExecutor(max_workers=2) as ex:
        list(ex.map(count, [N, N]))
    print(f"ProcessPool : {time.perf_counter()-t0:.2f}s")

if __name__ == "__main__":   # ProcessPool 在 spawn 平台必須有這個 guard
    main()

實際輸出：

text 
序列        : 0.43s
ThreadPool  : 0.43s
ProcessPool : 0.27s

為什麼：兩條 thread 搶同一把 GIL，CPU 密集時等於序列執行（0.43s vs 0.43s，毫無提速）；改用多行程（ProcessPoolExecutor）各自有獨立直譯器與 GIL，才真正吃到兩核心（0.27s）。注意 thread 對 I/O bound 仍有效，因為 I/O 等待時會釋放 GIL。Python 3.13+ 的 free-threading（無 GIL）建置正在改變這個局面，但預設建置仍受限。

正解：CPU 密集用 ProcessPoolExecutor（或 free-threading 直譯器）；I/O 密集用 thread 或 asyncio。決策樹見第 14 章。

詳見第 14 章（GIL、free-threading、concurrent.futures）。

陷阱 15：asyncio 裡呼叫阻塞式 I/O 卡死 event loop

async def 不會讓阻塞呼叫變成非阻塞。在 coroutine 裡呼叫 time.sleep、requests.get、同步 DB driver，會整個 event loop 卡住，所有其他 task 一起停擺——這是 async 新手最常見的效能塌陷。

⚠️ 對照（正確的 asyncio.sleep vs 阻塞的 time.sleep）：

python 
import asyncio, time

async def good(name, d):
    await asyncio.sleep(d)

async def bad(name, d):
    time.sleep(d)          # 阻塞！卡住 event loop

async def run(coro):
    await asyncio.gather(coro("A", 1), coro("B", 1))

t0 = time.perf_counter(); asyncio.run(run(good))
print(f"asyncio.sleep 總耗時 {time.perf_counter()-t0:.1f}s")
t0 = time.perf_counter(); asyncio.run(run(bad))
print(f"time.sleep    總耗時 {time.perf_counter()-t0:.1f}s")

實際輸出：

text 
asyncio.sleep 總耗時 1.0s
time.sleep    總耗時 2.0s

為什麼：兩個 task 各睡 1 秒。await asyncio.sleep 會把控制權交還 event loop，兩個 task 重疊執行 → 共 1 秒。time.sleep 不交還控制權，event loop 被它霸佔，第二個 task 只能等第一個跑完 → 共 2 秒，完全失去並發。await 關鍵字只能用於本來就會讓出控制權的 awaitable。

正解：用 async 版函式庫（asyncio.sleep、httpx.AsyncClient、async DB driver）；非用不可的同步阻塞呼叫丟到執行緒池：

python 
result = await asyncio.to_thread(blocking_io_call, arg)

詳見第 14 章（asyncio、async/await、決策樹）。

陷阱 16：import 副作用與循環匯入

Java 的 class 是惰性載入、import 只是編譯期的命名空間宣告，沒有執行語意。Python 的 import 會執行整個模組頂層程式碼一次（之後快取在 sys.modules）。把連線、讀檔、跑初始化寫在模組頂層，import 它就觸發副作用。

模組頂層的副作用只在第一次 import 時執行：

python 
# config.py
print("config.py 被匯入了")
DB_URL = "sqlite:///app.db"

# main.py
print("--- 第一次 import ---")
import config
print("--- 第二次 import（不會再執行頂層程式碼）---")
import config
print(config.DB_URL)

實際輸出：

text 
--- 第一次 import ---
config.py 被匯入了
--- 第二次 import（不會再執行頂層程式碼）---
sqlite:///app.db

當兩個模組在頂層互相 import 並在 import 時就讀對方的屬性，會炸成 AttributeError（模組還沒初始化完）：

⚠️ 刻意示範錯誤（a import b、b import a，且頂層互讀；以 python3 a.py 或先 import a 觸發）：

python 
# a.py
import b
VALUE_A = b.VALUE_B + 1   # 此時 b 才執行到 import a，VALUE_B 還不存在
# b.py
import a
VALUE_B = a.VALUE_A + 1

實際輸出：

text 
AttributeError: module 'b' has no attribute 'VALUE_B'

（3.14 起訊息尾端還會附上 (consider renaming ... if it has the same name as a library you intended to import) 提示，此處省略。）

為什麼：以 python3 a.py 觸發時，a.py 第一行 import b 去執行 b.py，b.py 第一行 import a 又回頭執行 a.py，a.py 第二行就想讀 b.VALUE_B——但 b 模組此刻只建立到「正在執行 import a」，VALUE_B 尚未賦值，於是 AttributeError。錯誤訊息取決於哪個模組先被匯入：若改成先 import b（或 python3 b.py），對稱地會報 module 'a' has no attribute 'VALUE_A'。Java 沒有「import 時執行頂層」的概念，所以這類循環依賴在 Java 通常無感。

正解：把副作用包進 if __name__ == "__main__": 或函式／main()，別放模組頂層；循環匯入則改成函式內延後 import、抽出共用模組打破環、或改用依賴注入（第 16 章 FastAPI Depends）。

詳見第 11 章（import 機制、循環匯入、__main__）。

陷阱 16.5：`requests.get(...)` 預設沒有 timeout，會無限等待

Java 的 HttpClient 至少有連線 timeout 的概念，你很容易帶著「總會逾時吧」的直覺移植過來。但 requests 的 get／post 預設 timeout=None，意思是永遠不逾時：對端不回應時，這條呼叫會無限期阻塞佔住一條 worker thread，足以拖垮整個 thread pool。這在正文被點名為生產環境頭號事故來源。

⚠️ 反模式（請勿照抄）：

python 
import requests
resp = requests.get("https://example.com/api")   # 對端掛住 → 永遠不回來

為什麼：timeout 預設 None，底層 socket 不設逾時，連線或讀取卡住就一直等。同步阻塞呼叫不會自己醒來，外層若沒有更高層的逾時保護，這條 thread 等於洩漏掉了。

正解：每一個 HTTP 請求都明確帶 timeout。單一數字同時當連線與讀取逾時；要分別控制就傳 (連線逾時, 讀取逾時) 的 tuple：

python 
import requests
# 連線與讀取都 10 秒
resp = requests.get("https://example.com/api", timeout=10)
# 連線 3.05 秒、讀取 27 秒（連線逾時略大於 3 的倍數是官方建議）
resp = requests.get("https://example.com/api", timeout=(3.05, 27))

逾時會丟 requests.exceptions.Timeout，搭配陷阱 12 的窄攔截處理。httpx 同理（且預設帶 5 秒逾時，但仍建議顯式設定）。

詳見第 17 章（HTTP 客戶端、requests、逾時與重試）。

陷阱 16.6：mock/patch 要打在「使用端的名字」而非定義處

精通 Mockito 的你移植單元測試時，最隱晦的坑是這個：unittest.mock.patch 要替換的是被測模組裡引用該名字的位置，不是函式原本的定義處。打錯位置時 mock「靜默失效」——沒有錯誤、測試照跑，但實際上去連了真的網路或真的時鐘。這條規則在 Mockito 不存在（Mockito 直接攔截物件／類別本身）。

⚠️ 反模式：被測模組 svc.py 寫了 from time import time，測試卻 patch("time.time")——mock 不生效，svc 用的還是真的時間。

機制是 import 綁定名字（呼應第 3、11 章）：from x import f 會在被測模組的命名空間建立一個指向 f 的綁定，被測程式碼呼叫的是那個本地名字，所以要 patch 的是 被測模組.f：

python 
# svc.py
from time import time          # svc 命名空間多了一個名字 svc.time
def make_id():
    return f"id-{int(time())}"

# test_svc.py —— 正解：patch 使用端 svc.time
from unittest.mock import patch
import svc
with patch("svc.time", return_value=1000.0):
    assert svc.make_id() == "id-1000"

判斷規則：

  • 被測模組寫 from x import f、呼叫 f() → patch "被測模組.f"。
  • 被測模組寫 import x、呼叫 x.f() → patch "x.f"（因為它每次都從 x 模組去查名字）。

正解：永遠 patch「被測程式碼實際查找名字的那個命名空間」，必要時把第二種寫法（import x; x.f()）當成讓 patch 更穩健的風格。

詳見第 11 章（import 綁定名字）、第 13 章（unittest.mock、patch 位置、from x import 的影響）。

B.7 自我檢查清單（Review Checklist）

把這份貼進 PR 範本，逐條掃：

  • 沒有可變預設參數：函式簽章看到 =[]、={}、=set()、=SomeMutable() 一律改成 =None + 函式體建立（陷阱 1）。
  • 預設值/型別提示無定義期副作用：預設值不寫 time.time()、讀環境、連線等會「凍結」的運算式（陷阱 2）。
  • 迴圈內的 lambda/closure 有快照變數：lambda x=x: ... 或別在迴圈裡建會捕捉迴圈變數的 closure（陷阱 3）。
  • is 只用於 None/單例；內容比較一律 ==，不對數字/字串用 is（陷阱 4、5）。
  • 浮點不用 ==：金額用 Decimal("...")，近似用 math.isclose（陷阱 6）。
  • 除法語意確認：要整數用 //，留意負數 floor vs Java truncation，餘數同號規則（陷阱 7）。
  • 拷貝是顯式的：賦值=別名；要複製用 .copy()/list()/deepcopy，沒寫 [[...]*n]*m（陷阱 8）。
  • 實例狀態在 __init__ 用 self. 建立，不在 class body 放可變物件當欄位；dataclass 用 default_factory（陷阱 9）。
  • 沒有邊迭代邊改容器：改用 comprehension 產生新容器或迭代副本（陷阱 10）。
  • 成員測試別對 list 反覆 in：查重／高頻成員測試改用 set/dict（O(1)），雙端高頻進出用 deque（陷阱 10.5）。
  • 語法細節：留意連鎖比較、and/or 回運算元（falsy 值用 is not None）、單元素 tuple 的逗號（陷阱 11）。
  • 例外攔得夠窄：無裸 except:；except 子類在前、父類在後；用 raise ... from（陷阱 12）。
  • 用 logging 非 print；訊息用 %s 佔位而非 f-string；昂貴引數加 isEnabledFor 守衛（陷阱 13）。
  • except 內記錯誤用 logger.exception(msg)（或 exc_info=True），別只記 str(e) 而丟失 traceback（陷阱 13）。
  • 並行模型對得上負載：CPU 密集用 process（或 free-threading），別指望 thread 加速（陷阱 14）。
  • async 內無阻塞呼叫：用 async 函式庫或 asyncio.to_thread，不在 coroutine 裡 time.sleep/同步 I/O（陷阱 15）。
  • 無 import-time 副作用與循環匯入：頂層程式碼放進 main()/if __name__ == "__main__":；循環依賴用延後 import 或重構打破（陷阱 16）。
  • 每個 HTTP 請求都帶 timeout：requests.get(..., timeout=10)（或 (連線, 讀取) tuple），絕不依賴預設的無限等待（陷阱 16.5）。
  • mock/patch 打在使用端的名字：from x import f → patch 被測模組.f；import x; x.f() → patch x.f（陷阱 16.6）。
