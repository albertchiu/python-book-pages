第 7 章：Python 的類別與物件

你已經是 OOP 老手——封裝、繼承、多型這些觀念不用人教。但 Python 的 class 雖然長得跟 Java 很像，骨子裡的哲學卻完全相反：它沒有 private、沒有 final、沒有編譯器幫你把關，封裝靠的是慣例與信任（Python 社群的名言是「我們都是成年人」we're all consenting adults）。同時它又給了你 Java 求之不得的工具——@property 讓你能先寫最簡單的公開屬性、需要時再無痛加上驗證；@dataclass 一行幹掉 record／Lombok 的所有樣板。本章帶你把 Java 的 class 直覺重新校準成 Python 的寫法。

class 語法、`__init__` 與 `self`

先看最小的對照。Java 的類別：

java 
public class Account {
    private String owner;
    private int balance;

    public Account(String owner, int balance) {
        this.owner = owner;
        this.balance = balance;
    }

    public void deposit(int amount) {
        this.balance += amount;
    }
}

Python 的等價物：

python 
class Account:
    def __init__(self, owner: str, balance: int = 0) -> None:
        self.owner = owner
        self.balance = balance

    def deposit(self, amount: int) -> None:
        self.balance += amount

差異一眼可見，但每一個都值得停下來想：

  • 沒有欄位宣告區。Java 要先宣告 private String owner;，Python 沒有這一段——實例屬性是在 __init__ 裡「第一次賦值」時才誕生的。self.owner = owner 同時是宣告也是賦值。
  • __init__ 不是建構子。它是初始化器（initializer）：物件其實是由 __new__ 先建立好、再交給 __init__ 填內容。99% 的時候你只需要 __init__，但要知道「建立」與「初始化」是兩個階段，這跟 Java 把 new 與建構子綁在一起不同。
  • self 是明寫的。Java 的 this 是隱形的；Python 規定每個實例方法的第一個參數就是實例本身，慣例命名 self。它不是關鍵字，只是慣例——但請永遠叫它 self。

🔁 Java 對照：obj.deposit(100) 在 Python 裡等價於 Account.deposit(obj, 100)——self 就是被點號左邊那個物件「自動帶入」的第一個參數。理解這點，後面的 @classmethod、@staticmethod 就不會難。

建立實例不需要 new：

python 
acc = Account("Albert", 100)   # 沒有 new 關鍵字
acc.deposit(50)
print(acc.balance)             # 150

方法就是「綁定了 self 的函式」

承接第 3 章「函式是一等物件」的觀念：當你透過實例存取方法，拿到的是一個bound method（綁定方法），它把 self 預先綁好了。

python 
>>> acc = Account("Albert")
>>> f = acc.deposit       # 取出方法但先不呼叫
>>> f                     # 它記得自己屬於哪個 acc
<bound method Account.deposit of <__main__.Account object at 0x...>>
>>> f(50)                 # 等同 acc.deposit(50)
>>> acc.balance
50

這在 Java 裡要等到 method reference（acc::deposit）才有對應物，而且 Java 的 method reference 是「介面的實作」，Python 的 bound method 就是個普通可呼叫物件。

沒有 access modifier：`_protected` 與 `__private` 的名稱改編

這是 Java 工程師最不習慣的一點：Python 沒有 public／protected／private。所有屬性與方法都是公開的。封裝靠兩個慣例：

寫法 意義 強制力
name 公開 API 無
_name（單底線） 「內部用，外人別碰」 純慣例，直譯器不管
__name（雙底線） 觸發名稱改編（name mangling） 半強制：改了名字，但仍可存取

單底線 _balance 就是一句口頭約定：「這是實作細節，你要碰我攔不住，但出事自負」。linter 與 IDE 會提醒，僅此而已。

雙底線 __balance 比較特別，它會觸發name mangling：在類別內寫 self.__balance，Python 會把它偷偷改名成 _ClassName__balance。

python 
class Account:
    def __init__(self) -> None:
        self.__balance = 100      # 實際存成 _Account__balance

acc = Account()
# print(acc.__balance)           # AttributeError！
print(acc._Account__balance)     # 100 —— 改了名字，但根本沒擋住你

⚠️ 陷阱：別把 __name 當成 Java 的 private。它不是為了防止存取，而是為了避免子類別不小心覆蓋父類別的屬性（兩個類別各自的 __cache 不會打架，因為被改成 _A__cache 和 _B__cache）。把它當 access control 用，只會讓你的程式更難測試、更難 debug。

💡 Pythonic 重點：預設用公開屬性。需要「告訴別人這是內部用」時用單底線 _。雙底線 __ 留給「真的會在繼承階層中撞名」的少數情況——實務上很少用到。封裝在 Python 裡是設計溝通，不是執行期警察。

類別屬性 vs 實例屬性

Java 的 static 欄位與實例欄位界線分明，宣告時就決定了。Python 兩者都是「在某個物件上的屬性」，差別在掛在哪裡：

python 
class Dog:
    species = "Canis familiaris"     # 類別屬性：所有實例共享

    def __init__(self, name: str) -> None:
        self.name = name             # 實例屬性：每個實例各自一份

存取時 Python 走一條查找鏈：先找實例自己的 __dict__，找不到再往類別找。

python 
>>> a = Dog("Rex")
>>> b = Dog("Momo")
>>> a.species            # 實例沒有，往類別找
'Canis familiaris'
>>> Dog.species = "Canis lupus"   # 改類別屬性，所有實例同步看到
>>> a.species, b.species
('Canis lupus', 'Canis lupus')

到這裡都很像 Java 的 static。但有一個微妙且關鍵的差異——透過實例賦值：

python 
>>> a.species = "Wolf"   # 不是改類別屬性！是在 a 上「新建」一個實例屬性
>>> a.species            # 實例的遮蔽（shadow）了類別的
'Wolf'
>>> b.species            # b 沒被動到
'Canis lupus'
>>> Dog.species          # 類別屬性也沒被動到
'Canis lupus'

在 Java 裡 a.species = "Wolf"（若 species 是 static）會修改那個唯一的 static 欄位（而且 IDE 會警告你別用實例存取 static）。在 Python 裡，指派給 instance.attr 永遠是在那個實例上建立／覆寫屬性，絕不會回頭改類別。這個「讀取會穿透、寫入不穿透」的不對稱，是 Java 工程師最常誤判的地方。

可變類別屬性的共享陷阱

這個不對稱碰上可變物件，就會變成跟第 3 章「可變預設參數」同源的經典坑：

python 
class Team:
    members: list[str] = []          # ⚠️ 所有實例共享同一個 list！

    def add(self, name: str) -> None:
        self.members.append(name)    # append 是「修改」，不是「指派」

a = Team()
b = Team()
a.add("Albert")
print(b.members)        # ['Albert'] —— b 也被影響了！

關鍵在 self.members.append(...) 是就地修改那個共享的 list，從頭到尾沒有對 self.members 做指派，所以查找鏈一路穿透到類別屬性，大家改的是同一個。正解是把可變狀態放進 __init__，讓每個實例各持一份：

python 
class Team:
    def __init__(self) -> None:
        self.members: list[str] = []   # ✅ 每個實例各自一個 list

💡 Pythonic 重點：類別屬性只放不可變、真正共享的常數或預設值（如設定、計數器型別的旗標）。任何「每個實例該獨立擁有」的可變狀態（list、dict、set），一律在 __init__ 裡建立。判準跟可變預設參數完全一樣。

`@property`：取代 getter/setter

Java 文化是「欄位一律 private + 配一組 getter/setter」，多半是為了「未來可能要加驗證或計算」而預先付出的樣板稅。Python 反過來：先把屬性公開，等真的需要邏輯時，再用 @property 把它升級成有 getter/setter 的屬性——而且呼叫端一個字都不用改。

先看沒有 property 的起手式，簡單到不行：

python 
class Circle:
    def __init__(self, radius: float) -> None:
        self.radius = radius        # 就是個公開屬性

c = Circle(5)
c.radius = 10                       # 直接存取，沒有 getRadius/setRadius

哪天需求變了——半徑不准是負數、而且想提供一個「算出來」的 area。在 Java 你得把 radius 改 private、加 getRadius/setRadius，然後所有寫過 c.radius 的呼叫端全部要改成方法呼叫。Python 不用：

python 
class Circle:
    def __init__(self, radius: float) -> None:
        self.radius = radius        # 注意：這行會「觸發」下面的 setter

    @property
    def radius(self) -> float:           # getter
        return self._radius

    @radius.setter
    def radius(self, value: float) -> None:   # setter：把驗證放這
        if value < 0:
            raise ValueError("半徑不能為負")
        self._radius = value

    @property
    def area(self) -> float:             # 唯讀的計算屬性，沒有 setter
        return 3.141592653589793 * self._radius ** 2

呼叫端完全不變，但行為升級了：

python 
>>> c = Circle(5)
>>> c.radius          # 看起來是讀屬性，其實呼叫了 getter
5
>>> c.area            # 看起來是讀屬性，其實是即時計算
78.53981633974483
>>> c.radius = -3     # 看起來是賦值，其實呼叫了 setter
Traceback (most recent call last):
    ...
ValueError: 半徑不能為負
>>> c.area = 100      # area 沒有 setter，唯讀
Traceback (most recent call last):
    ...
AttributeError: property 'area' of 'Circle' object has no setter

這就是 Python 封裝哲學的精髓：因為屬性存取可以「事後」變成方法呼叫，所以你不需要為了防患未然而預先寫 getter/setter。Java 那套「永遠先寫 getter/setter」的習慣，在 Python 裡是反模式（over-engineering）。

⚠️ 陷阱：@property 的 getter 裡如果寫 return self.radius，會無窮遞迴（它會再呼叫 getter 自己）。慣例是底層用單底線欄位 self._radius 存實際值，property 名字用公開的 radius。上面 __init__ 裡的 self.radius = radius 之所以能驗證，正是因為它走了 setter，而 setter 才把值放進 self._radius。

🔁 Java 對照：@property ≈ Java 的 getter/setter，但差別是呼叫端語法不變（仍是 c.radius，不是 c.getRadius()）。所以你能無痛地從「公開欄位」演進到「有邏輯的屬性」，不會破壞既有程式碼。

`@radius.deleter`：攔截 `del`

getter/setter 之外還有第三隻手 @radius.deleter，攔截 del c.radius。用得遠比前兩者少，但偶爾需要在屬性被刪時做收尾（清快取、釋放資源）：

python 
    @radius.deleter
    def radius(self) -> None:
        del self._radius        # 真正刪掉底層欄位
python 
>>> del c.radius      # 觸發 deleter
>>> c.radius          # 底層 _radius 已不在
Traceback (most recent call last):
    ...
AttributeError: 'Circle' object has no attribute '_radius'

沒定義 deleter 時，del c.radius 會拋 AttributeError: property 'radius' of 'Circle' object has no deleter，跟唯讀 property 被賦值的行為對稱。

💡 延伸：@property 本身是一個掛在類別上的描述器（descriptor）——Circle.radius 取出來會是個 property 物件，攔截實例存取的就是它的 __get__／__set__／__delete__。property 是 Python 內建、最常用的描述器；如果你想把同一套驗證或計算邏輯抽出來、套到多個欄位上而不重複寫 property，可以自訂描述器類別（實作 __get__／__set__／__set_name__）。機制細節（含 data／non-data descriptor 的優先序）見第 10 章「Descriptor 協定」一節。

`@classmethod` 與 `@staticmethod`

這兩個 decorator 對應 Java 的 static method，但 @classmethod 多了一個 Java 沒有的殺手級特性。

python 
class Pizza:
    def __init__(self, toppings: list[str]) -> None:
        self.toppings = toppings

    @classmethod
    def margherita(cls) -> "Pizza":          # 第一參數是 cls（類別本身）
        return cls(["mozzarella", "tomato"])

    @staticmethod
    def is_vegetarian(toppings: list[str]) -> bool:   # 沒有 self/cls
        return "pepperoni" not in toppings
  • @classmethod 的第一參數是 cls（類別本身，不是實例）。最常見的用途是替代建構子（alternative constructor）／工廠方法：Python 一個類別只能有一個 __init__，所以「用不同方式建立物件」就靠 classmethod。
  • @staticmethod 既不收 self 也不收 cls，純粹是「放在類別命名空間裡的普通函式」——邏輯上跟這個類別有關，但不需要碰實例或類別狀態。
python 
>>> p = Pizza.margherita()       # 不必先有實例就能呼叫
>>> p.toppings
['mozzarella', 'tomato']
>>> Pizza.is_vegetarian(["mozzarella"])
True

`cls` 的多型，才是重點

為什麼 classmethod 工廠要寫 return cls(...) 而不是 return Pizza(...)？因為 cls 會隨著實際呼叫的類別變動——子類別呼叫時，cls 就是子類別。

python 
class DeepDishPizza(Pizza):
    pass

>>> DeepDishPizza.margherita()
<__main__.DeepDishPizza object at 0x...>   # 拿到的是 DeepDishPizza，不是 Pizza！

如果 margherita 裡寫死 return Pizza(...)，子類別呼叫就會拿到錯誤的型別。這正是 Java static 工廠方法的痛點：static 方法不參與多型，DeepDishPizza.margherita() 在 Java 裡仍然回傳寫死的 Pizza。Python 的 cls 讓工廠方法天生對繼承友善。

💡 Pythonic 重點：要「換個方式建立物件」就用 @classmethod 配 cls(...)（例如 from_json、from_string、from_config）。要「一個跟類別相關但不碰狀態的工具函式」就用 @staticmethod——但其實很多時候直接寫成模組層級函式更 Pythonic，不必硬塞進類別。

`@dataclass`：對照 Java record / Lombok

寫一個只是用來「裝資料」的類別，手刻 __init__、__repr__、__eq__ 很煩。Java 的解法是 Lombok @Data 或 Java 16 的 record。Python 的解法是標準函式庫的 @dataclass：

python 
from dataclasses import dataclass

@dataclass
class Point:
    x: int
    y: int

這短短幾行，@dataclass 自動幫你生出 __init__、__repr__、__eq__：

python 
>>> p = Point(1, 2)
>>> p                       # 自動 __repr__
Point(x=1, y=2)
>>> p == Point(1, 2)        # 自動 __eq__：依「值」比較，不是依身份
True
>>> p.x
1

注意那些 x: int 是型別提示（第 3、9 章）——@dataclass 靠讀取類別的型別標注來決定有哪些欄位。沒標型別的就不算欄位。

常用選項，對照 record 的能力

python 
@dataclass(frozen=True, order=True, slots=True)
class Money:
    amount: int
    currency: str = "TWD"           # 有預設值的欄位（要放在無預設的後面）
選項 效果 Java 對照
frozen=True 不可變，建立後不能改欄位（且變成 hashable） record 的 final 欄位
order=True 自動生 <、<=、>、>=（依欄位順序比較） 自己實作 Comparable
slots=True（3.10+） 用 __slots__ 省記憶體、禁止亂加屬性 一般物件配置
kw_only=True（3.10+） 所有欄位變 keyword-only —
python 
>>> m = Money(100)
>>> m.amount = 200
Traceback (most recent call last):
    ...
dataclasses.FrozenInstanceError: cannot assign to field 'amount'
>>> sorted([Money(30), Money(10), Money(20)])    # order=True 才能排序
[Money(amount=10, currency='TWD'), Money(amount=20, currency='TWD'), Money(amount=30, currency='TWD')]

@dataclass(frozen=True) 幾乎就是 Java record 的對應：不可變的純資料載體，自動有 equals/hashCode/toString。差別是 dataclass 預設可變，要不可變得明寫 frozen=True；而且它更靈活（可選 order、slots、自訂方法）。

⚠️ 陷阱：slots=True 省記憶體、又能擋住打錯字的屬性名，但代價是實例不再有 __dict__——你無法臨時掛一個沒宣告的屬性。對 Java 工程師最容易中招的是測試：習慣在 mock／patch 時往實例塞個臨時旗標（obj.tested = True）會直接拋 AttributeError: ... no __dict__ for setting new attributes。此外，子類別若沒有自己宣告 __slots__，又會重新長出 __dict__，省記憶體的效果就破功。決定用 slots 前先確認沒有「事後動態加屬性」的需求。

不透過 dataclass 也能直接在手寫類別頂部宣告 __slots__ = ("x", "y")，效果與陷阱完全相同（省記憶體、禁止 __dict__、禁止亂加屬性）；@dataclass(slots=True) 只是幫你把這行根據欄位自動生出來。想精算記憶體又不想用 dataclass 時，手寫 __slots__ 是同一套機制。

⚠️ 陷阱：dataclass 欄位的可變預設值，跟第 3 章的可變預設參數是同一個坑。不能寫 tags: list[str] = []（會被所有實例共享，dataclass 甚至會直接報錯擋你），要用 field(default_factory=list)：

python 
from dataclasses import dataclass, field

@dataclass
class Article:
    title: str
    tags: list[str] = field(default_factory=list)   # ✅ 每個實例各自一個 list

`__post_init__`：衍生欄位與驗證

@dataclass 自動生成的 __init__ 只做一件事——把參數一個個指派到對應欄位。但 Java 的 record 有 compact constructor，能在欄位賦值後插一段「算衍生值」或「驗證」的邏輯。dataclass 的對應就是 __post_init__：__init__ 跑完所有欄位賦值後，會自動呼叫它（如果你有定義）。

python 
from dataclasses import dataclass, field

@dataclass
class Order:
    quantity: int
    unit_price: float
    total: float = field(init=False)        # 衍生欄位：不進 __init__ 參數，由計算得出

    def __post_init__(self) -> None:
        if self.quantity < 0:
            raise ValueError("數量不能為負")
        self.total = self.quantity * self.unit_price
python 
>>> o = Order(3, 100.0)
>>> o
Order(quantity=3, unit_price=100.0, total=300.0)
>>> Order(-1, 100.0)
Traceback (most recent call last):
    ...
ValueError: 數量不能為負

field(init=False) 讓 total 不出現在 __init__ 的參數列（呼叫端不必傳），改由 __post_init__ 算出來填上。這正對應 Java record「不能多宣告非標準欄位、衍生值要靠額外方法」的限制——dataclass 在這點上更靈活。

⚠️ 陷阱：frozen=True 的 dataclass 裡，__post_init__ 不能直接賦值（self.fahrenheit = ... 會觸發 FrozenInstanceError: cannot assign to field 'fahrenheit'，連 mypy 也會擋下來說欄位唯讀）。要繞過唯讀，得退一步直接呼叫 object.__setattr__：

python 
@dataclass(frozen=True)
class Temperature:
    celsius: float
    fahrenheit: float = field(init=False)

    def __post_init__(self) -> None:
        object.__setattr__(self, "fahrenheit", self.celsius * 9 / 5 + 32)

這是 frozen dataclass 設定衍生欄位的標準寫法——object.__setattr__ 繞過了 frozen 攔截賦值的那層 __setattr__。需要更嚴謹的執行期驗證時，pydantic.BaseModel（第 9 章）會是更省力的選擇。

💡 Pythonic 重點：只要類別主要是「裝資料」，預設就用 @dataclass，別手刻 __init__／__repr__／__eq__。需要不可變、可雜湊、可比較的值物件（value object）時加 frozen=True；要算衍生欄位或做輕量驗證就加 __post_init__。其他替代品：typing.NamedTuple（不可變、像 tuple）、enum.Enum（固定的具名常數集合，見下方框）、pydantic.BaseModel（要完整執行期驗證時，詳見第 9 章）。

🔁 Java 對照：Java 的 enum 是核心語言構造，Python 的對應在標準函式庫 enum：Enum（一組具名常數，成員是單例、可用 is 比較）、IntEnum／StrEnum（3.11+，成員同時「是」int／str，能直接拿去算術或字串操作、跟 JSON 互通）。它跟本章主題環環相扣——enum 成員本質是類別屬性、是不可變的值物件、也有漂亮的 __repr__（<Color.RED: 1>）：

python 
from enum import Enum, auto

class Color(Enum):
    RED = auto()
    GREEN = auto()
    BLUE = auto()
python 
>>> Color.RED
<Color.RED: 1>
>>> Color.RED is Color.RED          # 成員是單例，is 比較成立
True
>>> list(Color)                     # 可迭代所有成員
[<Color.RED: 1>, <Color.GREEN: 2>, <Color.BLUE: 3>]

別再用一堆 RED = 1; GREEN = 2 的模組常數（magic number）冒充 enum——enum.Enum 給你型別安全、可迭代、可比較、有名字的真 enum。完整用法可參考官方 enum 文件，這裡先讓你知道 Java enum 在 Python 有一級對應。

`__repr__`、`__eq__`、`__hash__`

這三個 dunder（雙底線方法，dunder = double underscore）直接對應 Java 的 toString／equals／hashCode。用 @dataclass 時它們自動生成；手刻類別時你得自己處理，而且 Python 有一條 Java 沒有的強制規則。

`__repr__`：對照 `toString`（但分兩種）

Python 區分兩個「轉字串」的方法：

  • __repr__：給開發者看的，目標是明確、最好能 eval 還原。repr(obj)、在 REPL 直接打變數、容器內的元素都用它。
  • __str__：給使用者看的，目標是可讀。print(obj)、str(obj) 用它。沒定義 __str__ 時會 fallback 到 __repr__。
python 
class Point:
    def __init__(self, x: int, y: int) -> None:
        self.x = x
        self.y = y

    def __repr__(self) -> str:
        return f"Point(x={self.x}, y={self.y})"   # 慣例：長得像建構呼叫
python 
>>> p = Point(1, 2)
>>> p                  # REPL 用 __repr__
Point(x=1, y=2)
>>> print(p)           # 沒有 __str__，fallback 到 __repr__
Point(x=1, y=2)
>>> [p, p]             # 容器內也用 __repr__
[Point(x=1, y=2), Point(x=1, y=2)]

💡 Pythonic 重點：至少實作 __repr__。一個好的 __repr__ 在 debug、log、測試失敗訊息裡價值連城。預設的 <__main__.Point object at 0x10a...> 對你毫無幫助。__str__ 只在「給終端使用者看的格式」跟 debug 格式不同時才另外寫。

`__eq__` 與 `__hash__`：對照 `equals`／`hashCode`，但 Python 會強制執行契約

跟 Java 一樣，Python 也有「相等的物件雜湊值必須相同」的契約。但 Java 只是寫在文件裡靠你自律；Python 直接用語言機制強迫你遵守：一旦你定義了 __eq__ 而沒有定義 __hash__，Python 會自動把 __hash__ 設成 None，讓這個類別變得不可雜湊。

python 
class Point:
    def __init__(self, x: int, y: int) -> None:
        self.x, self.y = x, y

    def __eq__(self, other: object) -> bool:
        if not isinstance(other, Point):
            return NotImplemented       # 不是同類就交給對方判斷
        return (self.x, self.y) == (other.x, other.y)

🔁 Java 對照：這裡用 isinstance(other, Point) 對應 Java equals 裡的 other instanceof Point——兩者都允許子類別實例通過檢查，於是 Point(1, 2) 可能等於某個 Point3D(1, 2, 0)。Java 社群為此爭論多年：另一派改用 getClass() == other.getClass() 要求型別完全相同，子類別一律不相等（較嚴格、對稱性更乾淨，但破壞了「子型別可替換父型別」的彈性）。Python 慣例是用 isinstance（較寬鬆），跟標準函式庫一致；真要 Java getClass() 那種嚴格相等，條件得寫成 type(self) is type(other)（兩邊類別都要一致才相等）——光寫 type(other) is Point 不夠，因為從子類別那側發起的反向比較會繞過它。return NotImplemented（注意是回傳、不是 raise）則對應「我比不了，交給對方的 __eq__ 試試」，讓 a == b 與 b == a 有機會達成對稱。

python 
>>> Point(1, 2) == Point(1, 2)
True
>>> {Point(1, 2)}                       # 想放進 set？
Traceback (most recent call last):
    ...
TypeError: unhashable type: 'Point'     # Python 擋下來了

（此為 Python 3.12／3.13 的訊息；Python 3.14 起擴充為 cannot use 'Point' as a set element (unhashable type: 'Point')，語意相同。）

為什麼？因為若可雜湊，你會得到「兩個相等卻可能落在不同雜湊桶」的危險物件——這正是 Java 工程師「override equals 忘了 override hashCode」會踩的坑。Java 讓你默默踩，Python 直接讓你用不了 set／dict key，逼你正視。解法是同時定義 __hash__：

python 
class Point:
    def __init__(self, x: int, y: int) -> None:
        self.x, self.y = x, y

    def __eq__(self, other: object) -> bool:
        if not isinstance(other, Point):
            return NotImplemented
        return (self.x, self.y) == (other.x, other.y)

    def __hash__(self) -> int:
        return hash((self.x, self.y))   # 慣例：用同樣參與 __eq__ 的欄位組成 tuple 去 hash
python 
>>> p = Point(1, 2)
>>> hash(p) == hash(Point(1, 2))        # 相等的物件，雜湊值相同 ✓
True
>>> {Point(1, 2), Point(1, 2)}          # 現在可以進 set，且會去重（沿用前面的 __repr__）
{Point(x=1, y=2)}

⚠️ 陷阱：只可雜湊不可變的物件才安全。如果物件放進 set 之後你又改了它參與 hash 的欄位，它就會在錯誤的雜湊桶裡「失蹤」——這跟 Java 把 mutable 物件當 HashMap key 的問題一模一樣。所以「可雜湊」通常跟「不可變」綁在一起。

這也解釋了 @dataclass 的雜湊行為，規則完全一致：

dataclass 設定 __eq__ __hash__ 可放進 set/dict key？
@dataclass（預設） 自動生成 設為 None ❌ 不可雜湊
@dataclass(frozen=True) 自動生成 自動生成 ✅ 可雜湊（且不可變，安全）
@dataclass(eq=False) 不生成（用 object 的身份比較） 繼承自 object ✅ 依身份雜湊

💡 Pythonic 重點：要當 dict key／set 元素的值物件，用 @dataclass(frozen=True) 最省事——它一次給你正確的 __eq__＋__hash__＋不可變，完全符合契約。需要手刻時，記住鐵律：定義了 __eq__ 就要一起定義 __hash__，且兩者用同一組欄位。

本章小結

  1. __init__ 是初始化器不是建構子，self 是明寫的第一參數。obj.m(x) 等價於 Class.m(obj, x)——理解這點，bound method 與 classmethod 都好懂。
  2. Python 沒有 access modifier。封裝靠慣例：_name 是「內部用」的口頭約定，__name 是 name mangling（防繼承撞名，不是 private）。別拿雙底線當存取控制。
  3. 實例屬性讀取會穿透到類別、寫入不會穿透。instance.attr = x 永遠在實例上建屬性，絕不回頭改類別——這跟 Java static 欄位的行為不同。
  4. 可變類別屬性會被所有實例共享，是可變預設參數的近親坑。每個實例該獨立持有的可變狀態，一律在 __init__ 裡建立。
  5. @property 讓你先公開屬性、事後再無痛加上 getter/setter 與驗證，呼叫端語法不變。因此 Python 不需要 Java 那種「預防性 getter/setter」。
  6. @classmethod 收 cls，是替代建構子的首選；cls 參與多型，子類別呼叫工廠會拿到子類別實例。@staticmethod 純粹是命名空間裡的函式。
  7. @dataclass 一行取代 record／Lombok：自動生 __init__／__repr__／__eq__，frozen=True 給你不可變且可雜湊的值物件。可變預設值要用 field(default_factory=...)；衍生欄位／驗證用 __post_init__（frozen 下要 object.__setattr__，對應 record 的 compact constructor）；slots=True 省記憶體但實例沒有 __dict__，無法事後動態加屬性（測試 mock 時要注意）。固定的具名常數集合則用 enum.Enum／IntEnum／StrEnum，是 Java enum 的一級對應。
  8. __repr__／__eq__／__hash__ 對應 toString／equals／hashCode。Python 強制契約：定義 __eq__ 而不定義 __hash__，類別就變不可雜湊。可雜湊通常要搭配不可變。
